Dynamo Under Failure & the Birth of Transactions

An interactive guide to Lecture 14 — how Dynamo survives crashes with hinted handoff and Merkle trees, the gap between theory and practice, and how distributed transactions give us atomic operations across partitions.

Part I

Temporary Failures & Hinted Handoff

Where Dynamo Stands So Far

Before we add anything new, let's catch ourselves up on what Dynamo is and why it exists. Dynamo is Amazon's key-value store — think of it as a giant distributed dictionary that maps a key (like a shopping-cart ID) to a value (the cart's contents). Its entire design philosophy is summed up in one sentence: accept weaker consistency in exchange for better availability and performance. Amazon would rather let you keep adding items to your cart during a partial outage and reconcile any conflicts later than refuse your write because a server is down.

To make this real, Dynamo has assembled a toolkit over the previous lectures. Here is the running scorecard of which problem each tool solves:

ProblemDynamo's Tool
How do we split keys across machines? (partitioning)Consistent hashing
How do we stay available for writes?Vector clocks (let conflicting writes coexist)
How do we tune consistency?Sloppy quorums with tunable R and W
How do we survive crashes? (today)Hinted handoff & Merkle-tree sync
Key Concept — Consistent Hashing & the Coordinator

Consistent hashing places both keys and nodes on a ring by hashing them. A key is stored on the first node clockwise from its position, plus the next N-1 nodes after that (its successors) for replication. The first node responsible for a key is called its coordinator — it drives reads and writes for that key, exactly like the coordinator role we saw in Chord.

Key Concept — Sloppy Quorum (R and W)

Every key is replicated across N nodes. A read returns once R nodes agree; a write succeeds once W nodes acknowledge. If you set R + W > N, every read overlaps every write, giving strong consistency. Applications can lower these values to trade a small risk of inconsistency for higher availability and speed. The "sloppy" part means we don't insist on a strict majority of the designated nodes — a theme that becomes central below.

The Problem: A Node Is Temporarily Down

Why are we learning about failures at all? Because in a data center, machines and networks fail constantly. Dynamo's authors carefully split failures into two kinds, because they demand different responses:

  • Temporary failures — a node is briefly unreachable. The machine is fine; a network switch hiccuped, or there's a transient partition. It will come back.
  • Permanent failures — a disk dies, a rack catches fire. The data on that node is gone for good.

Let's start with temporary failures. Picture the ring with nodes A, B, C, D, E, F, G. Node B is the coordinator for key K, and the replicas live on B, C, D. We want to write with N = 3 and W = 3 — meaning all three replicas must acknowledge before we report success.

Replicas for key K live on B, C, D. With W = 3, all three must ack — but C is temporarily down (red ✕).

Here is the conundrum. If C is down, we will never collect three acknowledgements. We are stuck choosing between two bad options:

The Durability / Availability Bind

Option A: Keep waiting for C to recover so we still have three copies — but then we block the write indefinitely, killing availability. Option B: Give up and report success with only two copies — but then we've lost the durability guarantee the application asked for. The whole point of Dynamo is to escape this false dilemma.

The Fix: Hinted Handoff

The insight is simple: if the intended node is unreachable, borrow a stand-in. Instead of writing to C, the coordinator writes to the next available node past the usual successors — say E. But E records a small note in its metadata: "this key isn't really mine; it belongs to C." That note is the hint.

Key Concept — Hinted Handoff

Hinted handoff lets a write succeed even when a designated replica is down, by temporarily storing the data on a substitute node that holds a hint about the true owner. Periodically, the substitute checks whether the intended recipient is back; once it is, the substitute forwards the data and then deletes its copy, since the key is now properly replicated again.

1

Coordinator B tries to write key K to B, C, D. B and D succeed, but C is unreachable.

2

B writes the third copy to E instead. E stores K plus a hint: "hold this for C."

3

The write returns success — we genuinely have three durable copies (B, D, E), so W = 3 is satisfied.

4

E periodically pings C. As soon as C is back, E forwards the data to C, then deletes its own copy. The system is back to its intended layout (B, C, D).

In-Band vs Out-of-Band Communication

Notice how E's repeated checking of C happens off the critical path. The original write didn't wait on it. Dynamo leans on this pattern constantly: periodic, out-of-band, best-effort communication. Out-of-band means it runs in the background on its own schedule rather than being bolted onto a user request. Because it never blocks a client, it's allowed to be slow — and that freedom is exactly why it scales.

Tuning W: When "Durable" Isn't Durable

Because W is a knob, it's worth asking what extreme settings actually buy you. What does W = 1 mean? It means a write returns as soon as a single node accepts it — the highest possible availability. But there's a catch: if that one node fails before the value spreads, the data is gone. So there is essentially no real durability. In practice almost everyone uses W > 1.

Here's the subtler question the professor pushed on: can you have W > 1 and still have terrible durability? Yes — if all your replicas sit in the same data center and that data center loses power, every copy vanishes at once. This is a correlated failure: the replicas aren't independent, so counting them overstates your safety.

Connection — Echoes of Chord's Successor Lists

Recall how in Chord a successor list had to contain N real nodes, not virtual ones, so a single physical machine couldn't masquerade as several replicas. The same idea returns here at a coarser grain: Dynamo can require the successor list to span at least two data centers, so that a key's replicas are genuinely independent failure-wise. Same principle — "make your replicas truly independent" — applied one level up.

Part II

Permanent Failures & Merkle Trees

Hinted handoff handles temporary blips beautifully — but what if the stand-in node itself dies permanently before it can hand off the data? Say E was holding C's copy and E's disk fails for good. Now that copy is gone, and the system is permanently stuck at two replicas (N = 2) for that key. If this keeps happening across many keys, eventually some key loses all its copies. We need a way to notice and repair shortfalls.

Replica Synchronization (Anti-Entropy)

The fix is replica synchronization: nodes that are neighbors on the ring periodically gossip with each other to compare their data and copy over anything one of them is missing. For key K, the relevant replicas B, C, and D occasionally ask each other, "do we hold the same set of keys and values?" Any key one node is missing gets copied over.

Key Concept — Anti-Entropy via Gossip

Replica synchronization (also called anti-entropy) is a periodic, out-of-band background process where neighboring replicas compare their stored data and reconcile differences. It is the safety net beneath hinted handoff: even when a hinted copy is lost, neighbors will eventually re-replicate the key back up to N copies.

Because this runs in the background (out-of-band again!), it never blocks a client request, so taking a while is fine. But there is a real cost hiding here, and it's worth pinning down precisely.

The Efficiency Problem & Merkle Trees

The naïve way to compare two nodes is to walk through every single key-value pair and check whether the other node has it. If each node stores millions of keys, that's millions of comparisons every sync cycle — and most of the time the two nodes are nearly identical, so almost all that work finds nothing. We want to detect that "these two nodes agree" with one cheap check, and only dig in where they actually differ.

The class brainstormed ideas — tracking changes since the last gossip (amortizes the cost but still scales with the number of writes) — but the winning answer is hashing, organized into a Merkle tree.

Key Concept — Merkle Tree

A Merkle tree hierarchically summarizes a set of key-value pairs using hashes. Each leaf is the hash of one range of keys. Each internal node is the hash of the concatenation of its children's hashes. The root therefore is a single hash that summarizes everything. If two nodes' roots match, their entire datasets match — verified with one comparison.

A Merkle tree over the key range 0–127. Leaves hash sub-ranges; each parent hashes its children. The root H₀ summarizes the whole dataset in one hash.

Walking the Tree to Find a Difference

The payoff is in the search. Suppose node A is missing a "green" key that B has, while B is missing an "orange" key that A has. To find these differences, the two nodes compare hashes top-down, pruning any branch where the hashes already match:

1

Compare the roots. They differ → the datasets differ somewhere. Descend.

2

Compare the two children. Suppose the left child's hash matches on both nodes. Prune it — everything under it is provably identical, so we never look at those keys again.

3

The right child differs, so descend only into it. Repeat: compare, prune matches, follow mismatches.

4

Continue until you reach the specific leaf ranges that disagree. Only those few key ranges get fetched and reconciled — not the whole dataset.

Why This Is a Win

If two nodes are identical, you do exactly one comparison (the roots match, done). If they differ in just a few places, the cost is proportional to the number of differences and the tree's depth — roughly O(d · log n) instead of O(n). You pay for the disagreements, not for the agreement. That's the whole reason Merkle trees show up everywhere from Git to blockchains to Dynamo.

One honest caveat the professor flagged: once you reach a differing leaf, the Merkle tree tells you where data differs, not which version is fresher. To resolve that, Dynamo falls back on the metadata it already keeps — the vector clocks from earlier lectures — which encode causal ordering. At worst the nodes can union the values and let the conflict-resolution machinery sort it out at read time.

Common Confusion — Numbers in the Tree Are Key Ranges

The labels on the tree (0–63, 64–127, …) are ranges of the hash key-space, not individual keys. Consistent hashing already gave us a finite range of key positions (say 0 to 1000) that this group of nodes is responsible for; the Merkle tree just recursively splits that range in half down to the leaves. Don't read them as "key number 63."

Part III

Dynamo in Practice

Now we shift from "how Dynamo achieves correctness" to "how Dynamo actually performs in a real Amazon data center." This part is where the textbook protocols meet the messy realities of December shopping peaks — and where Dynamo repeatedly chooses a simpler protocol with worse worst-case guarantees that nonetheless works well in practice.

Finding Data: Gossip Instead of Finger Tables

When any node receives a GET request for a key, it needs to know which node actually holds that key. In Chord, this was solved elegantly with finger tables — routing tables that let any node reach the target in O(log n) hops. Dynamo looks at finger tables and says: too slow, too complex. Instead it uses gossip.

Key Concept — Gossip-Based Membership

Roughly once per second, each server contacts a randomly chosen peer and exchanges what it knows: which servers exist and which key ranges each one holds. Over time this knowledge spreads to everyone. After gossip converges, a lookup is constant time — a node just knows who has the key. The worst case is worse than finger tables (your info might be stale, forcing extra hops), but in steady state it's a single hop.

Why does Dynamo get away with this when Chord couldn't? It comes down to workload assumptions:

Chord's WorldDynamo's World
Open peer-to-peer network; nodes join and leave constantly.Tightly controlled data center; membership is stable.
High churn demands provable O(log n) routing.Low churn means gossip converges and stays converged.
Could scale to millions of nodes.Hundreds to thousands of nodes in reality.
The Right Tool Depends on the Workload

Chord wasn't wrong — it was designed for the early-2000s dream of massive, churning, decentralized networks. Dynamo runs inside one company's data center where nodes rarely join or leave. Because failures and membership changes are rare, the gossip "best case" is the common case. The lesson: a protocol's quality is inseparable from the deployment it targets. And the lookup table each node keeps (say 1000 entries of range→server) is tiny next to the actual key-value data it stores, so the memory cost is a non-issue.

Reading the Latency Plot

The professor walked through one plot from the Dynamo paper, taken from Amazon's peak season in December 2006. The y-axis is latency on a logarithmic scale (10 ms, 100 ms, 1000 ms = 1 second); the x-axis is the timeline. There are four lines: the average and the 99.9th-percentile tail latency, each for reads and for writes. Three takeaways are worth memorizing:

1

The tail dwarfs the average. The 99.9th percentile sits a full order of magnitude above the average — roughly 100+ ms vs ~10 ms. This is exactly why engineers obsess over tail latency, not just the mean: a small fraction of very slow requests defines the user experience.

2

Real workloads have patterns. There's a clear diurnal (day/night) cycle: more requests during the day overload the system, pushing latency up; fewer requests at night bring it down. Because all four lines track the same load, they rise and fall together.

3

Writes cost more than reads. Writes sit a bit higher because they must actually touch the disk and satisfy durability. But notably, Dynamo keeps the write tail close to the read tail (~2× apart) rather than letting it explode — taming the worst writes is precisely what Dynamo is engineered for.

Connection — The Tail-Latency Theme Returns

Remember the fan-out service from last lecture, where 90th-percentile backend latencies compounded into a much worse top-level percentile? That was a warning about tails. Here we see the tail problem measured in the wild, and Dynamo's whole performance story is about keeping that tail bounded even for the hardest case — writes.

Squeezing More: Write Buffering

Some applications demand even lower 99.9th-percentile write latency. Dynamo offers an opt-in trick: a per-node in-memory buffer. Writes land in RAM immediately and return; periodically a "writer thread" flushes the whole buffer to disk in one batch. Batching amortizes disk cost, and individual writes no longer wait on the disk.

The Cost — Durability Again

If a node crashes while its buffer is still in RAM, those buffered writes are lost — they never hit disk. So write buffering is offered only to applications that explicitly accept this risk. It's the same recurring bargain: push performance harder, give up a sliver of durability, and bet that crashes are rare enough that the data is "almost always" there.

Decoupling Partitioning from Placement

One last practical headache: bootstrapping a new node. When a node joins, it finds its successors and copies the data it should now own. But this scan is so I/O-intensive that it must run at low priority — and during busy seasons, copying a node's data can take up to a full day. Worse, every join/leave also forces neighbors to recompute their Merkle trees and update lookup tables.

The root cause: in plain consistent hashing, a node's position on the ring determines both how data is partitioned and where each partition lives. Change membership and you disturb both at once. Dynamo's fix is to separate these two concerns:

Key Concept — Static Partitioning

Dynamo pre-divides the hash space into a fixed number of equally sized shards, independent of how many nodes exist. Each shard (not each individual key) is placed on the first N virtual nodes after it. When a node joins, you only decide which whole shards to hand off — no linear scan over individual keys. Crucially, Merkle trees are maintained per shard, so they don't need recomputing on every membership change.

The hash space is statically chopped into fixed shards. Nodes (arrows) own whole shards. Adding a node moves entire shards — no per-key scan, no Merkle recomputation.

This doesn't eliminate bootstrapping cost — copying still takes time — but it drops the overhead dramatically by working in shard-sized batches and reusing per-shard Merkle trees.

Theory vs Practice & Dynamo's Impact

Stepping back: after all these modifications — sloppy quorums, hinted handoff, gossip lookup, static partitioning — is this even "consistent hashing" anymore? Do the original guarantees still hold? Honestly, not strictly. And that's the deeper lesson of the lecture.

The Big Idea — Theory Meets Reality

In theory we love protocols we can prove things about. In practice, you take such a protocol, bend it in a dozen pragmatic ways, and end up with something that "works well enough" most of the time. Industry's real decision rule isn't "is this provably correct?" — it's closer to "will this cost us money, clients, or a lawsuit?" Dynamo is the canonical case study of trading provable guarantees for systems that actually ship and scale.

Dynamo's legacy is enormous: it popularized eventually-consistent NoSQL stores, which scale far better than the strongly-consistent SQL systems that came before. But that scalability came with a price — application complexity. Developers now had to reason about stale reads, lost writes, and reconciliation themselves. So the pendulum has begun swinging back toward strong consistency (we'll meet Google Spanner in a few lectures), exploiting new data-center and networking advances. Neither end "wins" permanently; the field oscillates between performance and programmability depending on what applications can tolerate.

Part IV

Transactions & Serializability

We now leave Dynamo and open a new chapter: distributed transactions. The motivation flows directly from sharding, so let's reconnect to that first.

When State Spans Multiple Partitions

Sharding (which we met a few lectures ago) splits state across machines. Its beauty is locality: if a request only touches state on one machine — read X, write X — no other machine needs to be involved. That gives us load balancing and lets us store data too big for any single server.

But what if a single logical operation touches multiple partitions at once? Real examples are everywhere:

  • Transferring money from one account to another — debit one shard, credit another.
  • Adding a meeting to the calendars of multiple participants at once.
  • Aggregating across many rows (summing values that live on different partitions).
  • Training ML on a distributed dataset where you need all the data from a consistent point in time.

For some of these (like money transfer) we need an atomicity guarantee. What does that mean precisely?

Key Concept — Atomicity

Atomicity means a group of operations either all complete or all fail together — never a partial result. A money transfer must not debit your account and then fail to credit the recipient's. Critically, atomicity is about sequences of operations, which makes it different from consistency (which we defined for a single operation at a time).

What Goes Wrong Without It

Consider a tiny client routine that reads a value, adds one, and writes it back — an increment. Now run two clients concurrently against the same key:

// Each client runs increment() on key x
function increment(x) {
  let v = get(x);   // read
  set(x, v + 1); // write
}

If these aren't atomic, the operations can interleave disastrously:

1

Client 1 does get(x) and reads 1.

2

Client 2 does get(x) and also reads 1 (Client 1 hasn't written yet).

3

Client 1 does set(x, 2).

4

Client 2 does set(x, 2) — overwriting with the same value. We expected 3 but got 2. One increment vanished entirely.

Lost Update — an Atomicity Violation

This is the classic lost update. Each individual get and set succeeded, yet the combined effect is wrong because the two-step sequence wasn't treated as one indivisible unit. No single-operation consistency model catches this, because the bug lives in how the sequences interleave.

Key Concept — Transaction

A transaction is a sequence of operations that must execute atomically — as one indivisible unit. A transaction may legitimately succeed or fail: a conditional transfer ("move $10 only if the balance is sufficient") can abort, and that's a valid outcome, not a system crash. The requirement is only that its operations happen all-together or not-at-all.

Defining the Guarantee: Serializability

How do we formally state what atomic execution of transactions means? The professor deliberately mirrored how we defined consistency models like linearizability earlier: take a real, possibly-interleaved execution, and ask whether we can reorder operations across clients to reach an equivalent "clean" execution.

Key Concept — Serializability

An execution is serializable if there exists an equivalent execution (obtained by reordering operations across clients) that is fully serial — meaning all operations of each transaction run consecutively, in some total order, with no other transaction interleaved between them. If every execution of a system is serializable, the system is serializable.

Sequential ConsistencySerializability
About single operations.About transactions (sequences of operations).
Requires a total order of operations.Requires a total order and that each transaction's operations are consecutive within it.
No notion of "operations belonging to a transaction."Adds exactly that extra constraint on top.

Let's make it concrete. Suppose the observed execution is messy — get→1, then someone's set 2, then get→2, then set 3 — interleaved across clients. Is there a serial ordering that explains it? Yes: get 1, set 2, get 2, set 3, where each transaction's two operations sit together. Because we can produce such an equivalent serial order, the execution is serializable.

Why "Permissible Outputs" Are Limited

Take T1 = "transfer $10 from my account to Bob's" and T2 = "read both balances," both starting at $100/$100. Under serializability, T2 may only return (100, 100) — if T2 ran first — or (90, 110) — if T1 ran first. Anything else, like (90, 100) or (100, 110), is forbidden, because it would reflect T2 reading a half-finished transfer. Serializability lets us pick which transaction goes first, but never lets a transaction observe another's partial state.

Part V

Two-Phase Locking & Isolation Levels

Serializability tells us what we want. This part covers how to get it — the classic protocol — and then a weaker, cheaper alternative for when full serializability is overkill.

Two-Phase Locking (2PL)

The most classic way to achieve serializability is two-phase locking. Clients submit transactions to a transaction coordinator. The coordinator acquires locks on every partition the transaction will touch, and only once it holds all of them does it run the transaction, then release the locks.

Key Concept — Two-Phase Locking

2PL has a growing phase (acquire all needed locks) and a shrinking phase (release them after the transaction runs). Because the coordinator holds all locks simultaneously while executing, no other transaction can interleave on those partitions — which is exactly what serializability demands. The two "phases" are acquire-everything-first, then release.

1

The coordinator requests locks on every partition the transaction touches (say partitions P1, P2, P3).

2

Each partition either grants the lock or replies no — perhaps it already lent the lock to someone else, or it can predict the transaction will fail here (e.g. insufficient balance), in which case it aborts immediately without even running it.

3

Once all locks are held, the coordinator executes the transaction and commits.

4

The coordinator releases the locks, freeing those partitions for other transactions.

Disjoint Transactions Run in Parallel

The win: transactions touching different partitions never contend, so they execute concurrently. Only transactions that share a partition must wait on each other. This is the same insight as detecting independent operations and running them in parallel — there's a huge research literature on doing this aggressively, but 2PL is the foundational, "naïve" version.

Two Real Dangers — Deadlock & the Single Point of Failure

Deadlock: if transaction A grabs lock 1 then waits for lock 2, while B grabs lock 2 then waits for lock 1, both hang forever. The standard fix is to impose a global ordering on locks and always acquire them in that order — then a cycle can never form. Single point of failure: the transaction coordinator is now a central bottleneck; if it dies mid-transaction, things stall. (We'll repair this at the end with replication.)

A Weaker Alternative: Snapshot Isolation

Just as consistency had a spectrum (linearizability, sequential, eventual), transactions have a spectrum of isolation levels — different strengths of guarantee. Full serializability is the strongest and most expensive. A very common weaker level is snapshot isolation.

Key Concept — Snapshot Isolation

Under snapshot isolation, each transaction operates on a single consistent snapshot of the database — a real state the database was in at one point in time. It will never read half of one committed transaction's effects mixed with half of another's. It's weaker than serializability because it makes no promise that everything ran atomically; it only promises you read from one coherent point-in-time view.

Why settle for less? Performance. Serializability forces you to actually serialize conflicting transactions, paying that cost on every conflict. Snapshot isolation only needs to find a valid snapshot — it can let more run concurrently. It's supported by systems like Firebase, Oracle, and MySQL (which offers several isolation levels).

The Color-Flip Example — Where SI and Serializability Diverge

Start with a table holding one blue and one green value. Run T1 = "flip all blue → green" and T2 = "flip all green → blue" concurrently. Under serializability, one runs fully before the other, so you end up all-green or all-blue. Under snapshot isolation, both transactions read the same original snapshot, so T1 sees the original blue and turns it green while T2 sees the original green and turns it blue — yielding a swapped result (green, blue) that serializability could never produce. That extra outcome is the price of the weaker guarantee.

SerializabilitySnapshot Isolation
Strongest; transactions appear fully ordered & atomic.Weaker; each transaction reads one consistent snapshot.
Pays the cost of actually serializing conflicts.Cheaper — more concurrency, akin to eventual consistency in spirit.
Color-flip → all-blue or all-green only.Color-flip → also permits the swapped (green, blue) outcome.

Fault Tolerance: Replicate the Partitions

We flagged that both an unavailable partition and a dead coordinator break 2PL. The remedy ties this whole course together: implement every partition as a replicated state machine. Then combine the two big techniques we've studied — sharding to split state across machines, and replication to make each shard fault-tolerant.

Combine both axes: shard horizontally to split state, then replicate each shard vertically (e.g. via Paxos/Zookeeper) so no single machine's failure stalls a transaction.
Connection — Sharding × Replication, Together at Last

Way back in the first sharding lecture, the course planted the idea that you ultimately need both sharding (for scale) and replication (for fault tolerance). Two-phase locking made the need urgent by introducing a single point of failure. Now the two techniques snap together: each shard becomes a replicated state machine, giving us scalable and fault-tolerant transactions.

Putting It All Together

This lecture had two halves, and the thread connecting them is the central tension of the whole course: how much guarantee can we afford, and what do we trade to get it?

The first half finished the Dynamo story. We saw how Dynamo survives temporary failures with hinted handoff (borrow a stand-in node, hand the data back later) and permanent failures with Merkle-tree replica synchronization (compare datasets in O(d·log n), not O(n)). Both rely on Dynamo's favorite trick — periodic, out-of-band, best-effort communication that never blocks a client. Then we watched Dynamo make pragmatic concession after concession: gossip instead of finger tables, write buffering that risks durability, static partitioning to tame bootstrapping. The takeaway crystallized in the theory-vs-practice discussion: a simpler protocol with worse worst-case guarantees can be the right engineering choice when it works well enough and ships.

The second half opened the door to transactions — the tool for operations that span multiple shards, like money transfers. We defined atomicity (all-or-nothing) and saw the lost-update bug that motivates it. We formalized serializability by mirroring linearizability: find an equivalent execution where each transaction's operations stay together. We achieved it with two-phase locking, met its dangers (deadlock, single point of failure), and softened it with snapshot isolation when full serializability is too costly. Finally, replicating each shard as a state machine married sharding and replication — the two pillars the course has been building toward.

Notice how the same trade-off curve reappears in both halves. Dynamo lowers W or buffers writes to gain speed at the cost of durability; transactions drop from serializability to snapshot isolation to gain concurrency at the cost of guarantees. Strong guarantees cost performance; weak guarantees cost programmer sanity. There is no free lunch — only an informed choice of where to sit on the spectrum.

The Big Idea — One Spectrum, Two Stories

Whether you're tuning a key-value store's quorum or picking an isolation level for transactions, you are choosing a point on a single trade-off spectrum between strength of guarantee and performance/availability/simplicity. The art of distributed systems is matching that point to what your application can actually tolerate.

What's Next

Snapshot isolation and serializability set up Google Spanner — the marquee example of the pendulum swinging back toward strong consistency, using clever data-center clocks and networking to make globally-consistent transactions affordable.
