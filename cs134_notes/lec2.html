Part I

Why MapReduce? The Big-Data Problem

Before we learn how a system works, it helps to feel the pain it was built to relieve. So start with a concrete scenario the lecture used: imagine you run a genomics biotech company that produces personalized DNA ancestry reports. Each user uploads megabytes of genome data, and you have millions of users. You want to scan all of that data to find which genes correlate with susceptibility to a particular disease.

This is not a niche problem. The lecture lined up several others that share its shape:

1

Web search ranking. Google's original product ranked web pages by how relevant they'd be to a query. To do that, you have to analyze essentially the entire web and run statistics over it.

2

Ad selection. Choosing which ad to show requires analyzing clicks and actions from billions of users.

3

System diagnostics. Analyzing data-center logs and transactions — data that ranges from terabytes to petabytes in size.

The common pattern: too much data for one machine

What ties these together? Each application has a lot of data — far more than any single computer can hold or process in reasonable time. A single server cannot store the whole web, and even if it could, processing it on one machine would take forever, because one machine simply does not scale to that workload.

So we need two things at once: distributed storage (spread the data across many machines) and distributed processing (compute on it in parallel). This is exactly the gap MapReduce fills.

Key Term — Distributed Processing

Splitting a single computation across many machines so they work on different pieces of the data at the same time, then combining their partial results into one answer. The goal is to finish a job that no single machine could complete on its own.

What do we actually want from such a system?

The professor asked the room to brainstorm the properties a distributed-processing system should have. Their answers, cleaned up and organized, give us the design goals MapReduce was built to hit:

1

Scalability. The more machines we add, the more load (throughput) we can handle. Throwing more hardware at the problem should buy more performance.

2

Fault tolerance. When you run on hundreds or thousands of cheap machines, some will fail mid-computation. The system must make progress and still finish despite failures.

3

Simplicity. An application developer should not have to rewrite all the low-level distribution and fault-handling logic from scratch. The framework hides that.

4

Wide applicability (generality). The framework must serve many kinds of applications — search ranking, ad selection, genomics, log analysis. If it only fit one app, you'd just build that app directly.

The Big Idea — You Can't Have Everything

The single most important theme of this entire course makes its first appearance here: you cannot have high throughput, low latency, and perfect fault tolerance all at once. Every system has to sacrifice something. The art of system design is choosing which property matters least for your use case and giving that one up.

The property MapReduce chooses to sacrifice: latency

If you can't have everything, what does MapReduce give up? Latency. MapReduce is built for batch jobs — large computations that run overnight or weekly. You compute the latest page rankings, the best ads per user, or the latest diagnostics in a big synchronous batch, store the results, and then when a user needs an answer they just look up the precomputed result.

Why Sacrificing Latency Is Safe Here

Because results are precomputed in the background and merely looked up on demand, the user-facing response is still fast — even though the heavy computation that produced it was slow. Insisting on low latency during the batch job would make the system far harder to build, so MapReduce deliberately declares low latency a non-goal.

A strawman solution — and why it fails

Here's the most obvious idea: partition the data across many servers, and have each server process only its own local chunk. Problem solved, right? Let's test it against PageRank.

Why "Just Process Local Data" Breaks

To rank a web page, you can't look at 5 pages in isolation and declare a ranking — a page's importance depends on all the pages that link to it, which live on other machines. There are inter-data dependencies: PageRank needs the whole web, and the genomics analysis needs every user who carries a given gene. A computation that only ever touches local data can't capture these cross-machine relationships.

So the real requirement is a system that does distributed processing but still lets data from different machines come together when the computation demands it. That "bringing together" step is the heart of MapReduce, which we build up next.

Where This Leads

Keep the four goals — scalable, fault-tolerant, simple, general — in your head. In Part III we'll revisit each one and check it off, and the fault-tolerance goal will dominate Parts V and VI. The "you can't have everything" trade-off returns every single time.

Part II

The MapReduce Programming Model

MapReduce, introduced in 2004 and popularized by the open-source Hadoop framework, is really two things bundled together:

Programming interfaceDistributed execution framework
The map and reduce functions you write.The machinery that runs them across a cluster.
Delivers simplicity and generality.Delivers scalability and fault tolerance.

Notice how cleanly this maps onto our four goals: two come from the interface you program against, two come from the framework that executes it. You write two small functions; the framework handles the hard distributed-systems parts.

The two functions: map and reduce

Everything in MapReduce is expressed as keys and values. Each data item is a (key, value) pair. The computation flows through five stages:

1

Partition. The (key, value) data is split across machines. Usually we partition it the way it already sits on disk — if machine 1 already holds a chunk, we leave it there rather than shuffling it around (moving data is expensive).

2

Map. Each machine applies the user's map function to its local partition, producing intermediate (key, value) pairs.

3

Coalesce (the shuffle). All intermediate pairs are regrouped by key: every value tagged with key A is gathered together, every C together, and so on — even though they came from different machines. This is the most expensive and most important step.

4

Reduce. The user's reduce function runs once per key, receiving all the values for that key, and produces the final output.

5

Output. Collect the reducer outputs as the answer.

Data flow: map runs locally per partition, then the orange "coalesce / shuffle" regroups every value by its key so each reducer sees one key's complete list.
A Built-In Assumption: One Key Fits on One Machine

What if a single key has so many values that they don't fit on one reducer? MapReduce simply assumes that doesn't happen. The model relies on there being orders of magnitude more keys than machines, so each key's value-list is small enough to process on a single machine. If that assumption breaks, plain MapReduce can't handle it.

Worked example: word count

The classic MapReduce job is counting how many times each word appears across a set of files. Here's how the two functions are written:

// key = filename, value = the text of that file
function map(key, value):
    for word in split(value):
        emit(word, 1)        // one tuple (word, 1) per occurrence

// key = a word, values = list of 1's, one per occurrence
function reduce(key, values):
    emit(key, sum(values))   // total count for this word

The mapper emits (word, 1) for every word it sees. After the coalesce step, the reducer for a given word receives the list of all the 1's emitted for it, and it sums them. (A student noted you could also just take the length of the list, since every value is 1 — but summing is more general, because it still works if mappers pre-aggregate and emit counts larger than 1.)

Let's trace two tiny files through the whole pipeline:

1

Input. File bar contains "foo bar"; file 2 contains "foo buzz world". They already live on separate machines, so no repartitioning is needed.

2

Map. Machine 1 emits (foo,1) (bar,1). Machine 2 emits (foo,1) (buzz,1) (world,1).

3

Coalesce. Regroup by key: foo → [1, 1], bar → [1], buzz → [1], world → [1]. Note the two foo tuples came from different machines and were brought together.

4

Reduce. Sum each list: foo → 2, bar → 1, buzz → 1, world → 1. Done.

Worked example: PageRank (and chaining stages)

PageRank computes a page's rank as the average rank of the pages that link to it. Assume every page starts with some initial rank from a prior pass.

// key = page W, value = (W's links, W's current rank)
function map(W, contents_and_rank):
    for P in links_of(W):    // every page P that W links to
        emit(P, W.rank)        // "page P, here's the rank of a page pointing to you"

// key = page P, values = ranks of all pages linking to P
function reduce(P, incoming_ranks):
    emit(P, average(incoming_ranks))

The mapper "pushes" each page's rank out along its links. After coalescing by destination page P, the reducer for P has the ranks of every page linking to it, and averages them. That's one round of PageRank.

You Can Chain MapReduce Stages

One pass isn't enough — you repeat the map→reduce cycle, feeding each round's output into the next, until the ranks stop changing (converge). This reveals a general pattern: MapReduce jobs are often pipelines of map and reduce stages, run again and again. A single computation may be many MapReduce rounds glued end to end.

Worked example: joining two tables

Now suppose we have two tables keyed by student ID: one mapping ID → name, another mapping ID → grades on two assignments. We want to find students who improved (grade 2 > grade 1) and produce their names. How do you express a join in MapReduce?

1

Map = identity (almost). The mapper barely transforms anything: it reads a row and emits it keyed by user ID. Row (0001, "Nancy") comes out as (0001, "Nancy"); a grades row comes out keyed by its ID too. The mapper's whole job is to set the key so the shuffle can do the real work.

2

Coalesce does the join. Because both tables are keyed by ID, the coalesce step naturally groups each student's name-row and grades-row together under the same key. The shuffle performs the join for you.

3

Reduce decides. For each ID the reducer now has two values — the name and the grades. It checks whether grade 2 > grade 1, and if so emits the student's name.

A Subtle Optimization: Filter Early

You can do the grade comparison either in the reducer or push some filtering into the mapper over the grades table. If the mapper drops students who didn't improve, you emit fewer intermediate values into the expensive coalesce step. The clever trick: after filtering, a reducer that sees only one tuple for an ID knows that student didn't improve (their grades row was dropped) and stays silent; a reducer that sees two tuples knows they improved and prints the name. Since the coalesce/shuffle is the costliest stage, getting rid of data before it is almost always a win.

Expand: Can I chain extra "map-only" computation between map and reduce?

A student asked whether you can insert extra functional (side-effect-free) computation after the map but before the reduce. The answer: there is no magic "in-between" stage — there's only map and reduce. But you don't need one, because you can chain stages: do map, then another map, then reduce — i.e. map → map → reduce. As long as the data isn't moving between partitions (no shuffle), you can chain as many maps together as you like, either fused into one map function or kept separate. The shuffle is the only thing that forces a boundary.

Part III

Execution & Architecture

We've seen the functions. Now: how does the framework actually run them across a cluster, and what constraints does the model impose on the order of execution?

When can a reduce task start? The synchronization barrier

Tempting answer: as soon as all the A keys are gathered, start reducing A — don't wait for C and W. That's a nice instinct, but there's a catch.

You Can't Reduce Until Every Mapper Is Done

How do you know you've collected all the values for key A? The very last tuple from the slowest, straggling mapper could be an A tuple you still need. So a reducer for A cannot safely begin until every map task has finished — otherwise it might compute on an incomplete list and produce a wrong answer.

Key Term — Synchronization Barrier

A point in a parallel computation that all participants must reach before any may proceed past it. In MapReduce, the barrier sits between the map phase and the reduce phase: no reduce can start until the last mapper completes. It guarantees correctness at the cost of being held hostage by the slowest worker.

Expand: Can we ever break the barrier? (Stream processing & reduction trees)

Yes — but only by changing the programming model. If your reducer must receive a complete list, the barrier is unavoidable. But stream processing systems let data flow through incrementally: you rewrite reduce so it accepts new values one chunk at a time and updates its result as they arrive. Alternatively, you can use reduction trees — combine values pairwise (binary or n-ary) rather than all at once. Both approaches lift the barrier and improve performance, but they make the programming model harder, and not every task (e.g. a global sort) decomposes cleanly into a tree without redundant work. The standard MapReduce assumption in this course is the simple one: the reducer takes the whole list, so the barrier stays.

The master node

Who coordinates all of this? A single master node. (You'll implement one in Assignment 1.) Its responsibilities:

1

Assign tasks. Given the program and knowledge of where the initial data lives, the master hands map and reduce tasks to workers.

2

Track status. It records which tasks are done and which are still running, so it knows when the barrier is satisfied and the computation can advance.

3

Detect & handle failures. If a worker crashes, the master notices and reruns that worker's task elsewhere to recover the missing result.

Checking off our four goals against this design: scalable ✓ (run mappers and reducers in parallel on more machines for more throughput), simple ✓ (you only write map and reduce), widely applicable ✓ (no assumptions beyond expressing work in the stateless map/reduce form). That leaves fault tolerance, which we tackle in Part V — but first we need vocabulary for reasoning about system diagrams.

Two kinds of arrows: the control plane vs. the data plane

Architecture diagrams in this course use two arrow styles — dashed and solid — and they mean genuinely different things. This distinction recurs all term, so it's worth nailing down.

Key Term — Control Plane vs. Data Plane

Control arrows carry commands and coordination: "wake up and do this task," "stop," "I'm done." They must flow between computational components — there has to be a process or thread on each end sending and receiving. Data arrows carry the actual payload data being processed. Crucially, a data arrow need not connect two compute components — it can run between a compute component and storage (you pull data from disk or push results to it).

Control planeData plane
Coordination, commands, statusMovement of the actual data
Always between two compute componentsCan involve storage on one end
"Run task 3", "I'm done", "stop"Reading a partition, writing intermediate output
Why This Vocabulary Matters Later

The next topic — the client–server model — is described as "a key abstraction for the control plane." And in Part V, MapReduce's fault tolerance works by routing intermediate results through the data plane (to storage) rather than sending them directly compute-to-compute. Keeping control and data separate in your head is how you'll reason about every protocol in this course.

Part IV

The Client–Server Model & RPC

We just named the control plane as where components coordinate. The fundamental abstraction for that coordination — used throughout distributed systems and throughout this course's assignments — is the client–server model.

Clients, servers, and requests

Key Term — Client–Server Model

Two roles. A client sends a request asking the server to do something — perform a stateful action, return a result, etc. A server sits in a loop listening for requests; when one arrives it processes it and returns a response. A web backend is the canonical example: the browser (client) says "give me the HTML of this page," and the server replies with it.

This is primarily a control-plane abstraction — it's about asking another component to act. Secondarily it touches the data plane too, since requests and responses can carry data.

Remote Procedure Calls (RPC)

A particularly clean way to use client–server is the Remote Procedure Call. The server publishes an API of methods with specific signatures; the client invokes those methods remotely, passing arguments and waiting for the response.

RPC Makes the Network Look Like a Function Call

From the programmer's side, calling a remote method looks just like calling a local function — you call it with arguments and get a return value. Under the hood, the call travels from client to server, the server runs the code, and the result travels back. RPC hides the network so you can write client code that assumes nothing about the server except its method signatures. Real frameworks like gRPC and Thrift provide this.

Here's a minimal RPC interface — a Greeter service offering one method, SayHello:

service Greeter {
    // takes a HelloRequest, returns a HelloReply
    SayHello(HelloRequest) -> HelloReply
}

message HelloRequest { name: string }
message HelloReply   { greeting: string }

A client just sends its name and gets back "hello, <name>". That signature is the entire contract: the client can be written against it without knowing anything else about the server's internals.

The performance trap: blocking servers

Now the lecture asked a sharp question: would you build Facebook's front end with a server that processes one request fully before touching the next? Let's see what goes wrong.

A Single-Threaded Server Blocks Everyone Behind

If the server handles requests one at a time, every new request must wait in line for all the requests ahead of it to finish. As load grows, this queue grows, and every request suffers a latency hit purely from queuing. The server cannot scale.

The fix is threads. The lecture walked through three designs, each fixing the previous one's flaw:

1

Thread-per-request. A listener thread accepts requests and spawns a fresh thread to handle each one. Now requests run concurrently instead of queuing.

2

The flaw: unbounded threads. Under heavy load you might spawn 100,000 threads for 100,000 requests. They all contend for the same finite CPU and memory, so the machine is overwhelmed and everyone takes the latency hit anyway. (There can also be race conditions if the server holds shared state, though here we assume it's stateless.)

3

The standard fix: a thread pool. A listener thread puts incoming requests into a queue, and a fixed-size pool of worker threads pulls from that queue. The number of threads is bounded, so the machine never gets overwhelmed by thread explosion. (We assume listening is much cheaper than processing, so the listener itself isn't the bottleneck.)

Thread-pool server: a cheap listener queues requests; a bounded set of workers drains the queue, preventing thread explosion.
The Question Left Open

Even a thread pool doesn't answer one fundamental question: what happens when requests arrive faster than the pool can drain them? That deeper problem — backpressure and overload — is left for later in the course. For now, the lesson is just the trade-off space of threading: more concurrency helps, but unbounded concurrency can destroy you.

Part V

Fault Tolerance

We've checked off three of our four goals. The last and hardest is fault tolerance: machines crash, and the computation must survive. This is where the course's signature mindset begins.

Tolerating worker failures: checkpoint to disk

Suppose a worker crashes mid-job. Because every downstream stage depends on its output, the whole pipeline stalls. Hadoop's fix is a "very, very simple trick":

Key Term — Checkpointing

Saving the intermediate output of each stage to durable storage (disk) as the computation proceeds. If a task's worker then crashes, you don't lose progress: a replacement worker reads the saved data back from disk and simply reruns that one task. Because every intermediate result lives somewhere persistent, any single failed task can be recovered and the computation continues.

Notice this is the control/data-plane distinction in action: MapReduce routes intermediate results through the data plane into storage, rather than passing them compute-to-compute, precisely so a crash can't destroy them.

The Cost: Checkpointing Is Slow

There's no free lunch. Writing all your data to disk after every computation step is expensive — disk is orders of magnitude slower than memory. So fault tolerance here is bought with performance. And it interacts badly with the synchronization barrier: if a map task fails, the reduce phase can't proceed until that task is re-run and re-checkpointed, so any single map failure blocks everyone. Later systems like Spark reduce this cost (with a lineage system for recovery instead of constant disk writes), but they still trade away something else — no system escapes the rule.

The bigger threat: what if the master fails?

Checkpointing handles worker failures. But the master is a single point of coordination — if it crashes, can we just start a new master and resume?

Only if the new master can recover the old one's state: which tasks were assigned, which finished. One option is to checkpoint the master's state to disk after every step — but that's a lot of overhead. The better approach, and a centerpiece of this course, is to replicate the master's state across several machines so that if one dies, another already has the state and takes over.

Replicated state machines

Replication sounds easy ("just keep two copies"), but naive replication is subtly broken. The lecture illustrated with a bank account replicated in New York and LA:

Out-of-Order Updates Diverge Forever

A client in Seattle deposits $100. Around the same time, a bank admin adds end-of-year interest. If the two replicas apply these two operations in different orders, they compute different balances — and they will never reconcile. The replicas are permanently inconsistent. (The same bug hides in the two-master scheme: if master 2 learns about task 3 too late, it might re-assign task 3 to another worker — double work.)

The cure is a precise discipline:

The Big Idea — Replicated State Machine (RSM)

Model each replica as a state machine. If every replica (1) starts from the same initial state and (2) applies the same updates in the same order, then every replica is guaranteed to reach the same final state. Deterministic machine + identical ordered input = identical output. This is the foundational technique for fault-tolerant distributed systems, and much of this course is about how to agree on that one shared order efficiently.

Same start state s0 + same update sequence +A, +B, +C in the same order ⇒ both replicas land on the identical final state sF.

State vs. events — a distinction that recurs all term

An RSM replicates a sequence of transitions (events), not just the current state. Why prefer the more expensive thing?

Keeping the stateKeeping the events (transition sequence)
Just the current snapshotThe full ordered history of updates
Cheaper to storeMore expensive — history can grow without bound
Can't tell how you got hereCan replay, roll back, audit; can reconstruct any state exactly
Why Events Are More Expressive

Many different histories can lead to the same final state — so a snapshot loses information about how the system arrived there. The event sequence keeps that information: from the exact sequence of transitions you can always reconstruct the exact state, replay it, or roll back to any earlier point. It costs more (the history can grow large), but it's strictly more expressive — and sometimes that history is exactly what a protocol needs to be correct.

The Mindset This Course Is Teaching You

The professor's stated key outcome: whenever you see code, instinctively ask "what if?" — What if a process crashes? A message is dropped? Delivered twice? Corrupted? Some external service fails? The method for building reliable systems is to enumerate every fault that could occur, reason through what your system does in each case, and confirm you tolerate them all. We'll do this over and over.

Part VI

Clock Synchronization

The RSM left us one urgent problem: how do all replicas agree on the same order of updates? The intuitive answer — use timestamps — is harder than it looks, and chasing it down leads to clock synchronization.

Why ordering needs clocks (and why it's hard)

Let's try the obvious implementation and watch it fail in stages:

1

Attempt: order by time of receipt. Each replica timestamps updates as they arrive and orders by that. Problem: updates can arrive at different replicas in different orders, so receipt-time produces different orderings on different machines — the exact divergence we're trying to avoid.

2

Attempt: order by time of sending. Let the client timestamp each update before sending it, and have all replicas order by that client timestamp. Better — now every replica uses the same number for a given update. Problem: different clients have unsynchronized clocks, so their timestamps aren't comparable.

The Core Challenge — Clocks Aren't In Sync

Clocks on different machines drift apart. If client A's clock reads 10:00:05 while client B's reads 10:00:02 for events that truly happened at the same instant, ordering by their raw timestamps gives a meaningless, machine-dependent result. To order events by physical time, we'd first have to make all those clocks agree — which is itself a hard problem.

Synchronizing clocks: Cristian's algorithm (the heart of NTP)

How do you set one machine's clock from a reference time server? Two naive ideas and their flaws:

  • GPS broadcast. Accurate to the microsecond — but power-hungry and doesn't work indoors.
  • Just ask a server for the time. The server replies with its time — but by the time the reply reaches you, unknown network latency has elapsed, so its timestamp is already stale. You don't know how much time passed in transit.

The fix, which underpins the NTP protocol, is to measure the round trip with four timestamps and estimate the latency. The client records when it sends and receives; the server records when it receives and replies:

1

T1 — client timestamps the moment it sends the request.

2

T2 — server timestamps the moment it receives the request.

3

T3 — server timestamps the moment it sends its response.

4

T4 — client timestamps the moment it receives the response.

Four timestamps. Round-trip network time = (T4 − T1) − (T3 − T2); assume each one-way leg is half of that.

Now reason about what these numbers tell us:

1

Total elapsed at the client: T4 − T1 — the whole round trip as the client experienced it.

2

Time the server spent working: T3 − T2 — subtract this out, since it's not network travel.

3

Total network travel time: δ = (T4 − T1) − (T3 − T2) — both legs combined.

4

Estimate one-way latency. Assuming the network is symmetric (each direction takes the same time), one leg ≈ δ / 2. So when the client receives the response, the true reference time is T3 + δ/2 — the server's send-time plus one more leg of travel.

The Critical Assumption — Symmetric Latency

This whole derivation rests on one assumption: the two network legs take roughly the same time. A student spotted this immediately — "if the latency is different in each direction, this isn't possible." If one direction is, say, 10× slower than the other, the δ/2 estimate is off and your synced clock carries that error. We can measure the total round trip precisely; splitting it in half is the leap of faith.

The deeper limits: drift, precision, and monotonicity

Even with Cristian's algorithm, physical clocks have stubborn problems. A student asked: at nanosecond precision, can you even synchronize at all? Essentially no — below some resolution the latency estimate's error swamps the signal. And there's a nastier issue:

Physical Clocks Can Run Backwards

Because of latency variation and corrections, a synchronized physical clock can sometimes jump backward — it doesn't only move forward. For ordering, that's catastrophic: an event could appear to happen "before" one that truly preceded it. What we really need is a monotonic clock — one guaranteed never to go backward.

Monotonic Is Often Enough — You Don't Need Real Time

Here's the key realization that sets up the next lecture: for ordering, you usually don't need each machine's clock to match real-world time at all. As long as each client's timestamps are monotonic (always increasing), the servers can process that client's requests in a consistent order. A lagging client just makes its requests look uniformly "later," but the relative order stays intact. We don't care about the exact physical instant — only about a stable order.

The Big Idea — Maybe We Don't Need Physical Time at All

Physical clocks are power-hungry, imprecise, asymmetric, and can run backward. So the punchline of this lecture, and the doorway to the next, is: throughout this course we will mostly abandon physical clocks and use logical clocks to order events instead — getting a consistent ordering without ever trusting a wall clock.

Putting It All Together

This lecture looks like four separate topics — MapReduce, RPC, fault tolerance, clocks — but they're a single chain of reasoning, each link forced by the previous one.

We started because one machine can't hold the world's data, which forced distributed storage and processing. The properties we wanted (scalable, fault-tolerant, simple, general) immediately ran into the lecture's recurring law: you can't have everything — MapReduce gave up latency to win the rest. The map/reduce model delivered simplicity and generality; word count, PageRank, and table joins showed the same five-stage skeleton solving wildly different problems, with the expensive coalesce/shuffle doing the heavy lifting (and even performing joins for free).

Running that model surfaced the synchronization barrier (reduce waits for every mapper) and the master node that schedules and monitors — which is exactly the client–server control-plane abstraction, refined into RPC. Asking how a server scales took us through blocking, thread-per-request, and the bounded thread pool — another instance of "more concurrency helps until unbounded concurrency destroys you."

Then came the hard goal, fault tolerance. Worker crashes are handled by checkpointing to disk (buying reliability with slowness — no free lunch again). Master crashes demand replication, and replication done naively diverges forever, which forced the replicated state machine: same start, same ordered updates, same result. That single requirement — same order — is what dragged us into clock synchronization, where we found that physical clocks are imprecise, asymmetric, and can even run backward, so what we truly need is merely a monotonic order, not real time.

The Through-Line

Two ideas thread through everything and will recur all term: (1) every design trades one property for another — name what you're sacrificing, and (2) to build reliable systems, enumerate every fault ("what if it crashes / drops / duplicates / corrupts?") and prove you tolerate each one.

What's Next

We ended on the realization that physical time is too fragile to order distributed events — and that for ordering we don't actually need real time, only a consistent, monotonic order. Next lecture introduces logical clocks: a way to order events across machines without any physical clock at all, sidestepping drift, asymmetry, and backward jumps entirely. That ordering is the missing ingredient that makes replicated state machines — and therefore fault-tolerant masters — actually work.
