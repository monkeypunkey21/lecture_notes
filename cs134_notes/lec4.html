Fault Tolerance & the Chandy–Lamport Snapshot

An interactive guide to Lecture 4 — what can break in a distributed system, how to recover from crashes, and how to photograph a running system without ever stopping it.

Part I

Where We Are — and What Clocks Can't Do

Before this lecture, the course had spent its time on one big question: in a system of separate computers with no shared clock, how do we agree on the order in which things happened? The answer was logical clocks. This lecture is the pivot point where we admit that ordering, by itself, doesn't keep a system alive. To see why, we first need to be crisp about what the earlier machinery actually bought us.

Recap: What Logical Clocks Gave Us

A distributed system is a collection of independent processes (think: separate machines or programs) that coordinate only by sending each other messages over a network. There is no global "now" they can all read. So instead of real time, we use logical time.

Key Concept — Happens-Before

We say event A happened-before event B (written A → B) if at least one of these holds: (1) they are in the same process and A occurs first; (2) A is the sending of a message and B is its receipt; or (3) it follows transitively — there is some C with A → C and C → B. This is the only notion of "order" a distributed system can actually justify.

From that single relation, the course built three tools, each strictly more precise than the last:

  • Lamport clocks — a single integer per event. If A → B then C(A) < C(B). The catch: the reverse is not guaranteed.
  • Vector clocks — one integer per process, so a clock is a tuple like [1, 1, 2]. These capture happens-before exactly: the clocks order iff the events do.
  • Replicated state machines — using logical clocks to keep multiple copies of a service in lock-step agreement about the order of operations.
The One-Way Street That Trips People Up

Lamport clocks only flow one direction. A → B implies C(A) < C(B), but C(A) < C(B) does not imply A → B. Two completely independent events — say, handling two unrelated external requests — can have comparable clock values purely by coincidence. The same one-way logic applies to causation: A caused B implies A → B, but A → B does not mean A caused B (B could just be later and unrelated).

So Are We Done? What's Still Missing

We can now order events, replicate a service, and keep replicas consistent. It is tempting to declare victory. But ask the question the professor asked the room: are we done?

No. Every tool so far quietly assumed that nothing ever breaks. Logical clocks give zero guarantees when a machine crashes, when the network drops a packet, or when a process simply disappears. They order the events that do happen; they say nothing about what to do when the events stop happening.

The Big Pivot of the Course

Ordering is necessary but not sufficient. The hard, multi-lecture problem in distributed systems is fault tolerance: building a system that still meets its specification even when parts of it fail. Everything from this lecture onward is about faults and how to survive them.

A Loose End: Can Two Events Share a Vector Clock?

A student had asked: could two genuinely distinct events ever end up with identical vector clocks? The answer is no, and the informal proof is a nice warm-up in reasoning about these systems. First, recall the two update rules for a vector clock:

// every clock starts at [0, 0, 0]
// Rule 1 — local event (incl. a send):
V[self] = V[self] + 1

// Rule 2 — receiving message with clock M:
for each i: V[i] = max(V[i], M[i])
V[self] = V[self] + 1

Now suppose two events A and B had the same vector clock. The argument rules that out in two moves:

1

They can't be in the same process. Every event a process handles increments that process's own entry in the vector. So no two events in one process share that entry — their clocks must differ. Therefore A and B would have to live in different processes, say P1 and P2.

2

But then a contradiction appears. For P2 to have P1's entry equal to A's value, P2 must have received a message originating from that exact send. Yet by the receive rule, any event in P2 after that receipt has a strictly larger value in its own entry than A does — because that entry only ever grows. So the two clocks can't be equal after all. Contradiction.

Two distinct events always have distinct vector clocks. This isn't busywork: it's exactly the kind of short, informal argument you might be asked to reproduce.

Only vector clocks would let you read ordering off the numbers directly.
Part II

Faults — Naming the Enemy

You cannot tolerate a fault you haven't defined. This part nails down what "fault tolerance" precisely means and then catalogs the kinds of things that actually go wrong, because the right defense depends entirely on the threat.

What "Fault Tolerance" Actually Means

Key Concept — Fault Tolerance

A system S is fault-tolerant if it continues to satisfy its specification (the guarantees it promises to its users) given that some defined class of faults can occur. The phrase "some defined class" is load-bearing — tolerance is always relative to a specific set of failures you've decided to handle.

The First Question You Must Ask

When someone asks you to "make the system fault-tolerant," the correct first response is not to start coding. It is to ask: which faults? A defense against a dropped packet looks nothing like a defense against a malicious adversary. Until the fault model is pinned down, "fault-tolerant" is meaningless.

A Habit Worth Forming

Every protocol in this course will open by stating its fault model and its system model (assumptions about the network, timing, and failures). When you read a new algorithm — including Chandy–Lamport later in this very lecture — find those assumptions first. They tell you exactly what the algorithm is and isn't allowed to promise.

Three Classes of Faults

Distributed-systems failures sort into three buckets, in roughly increasing order of nastiness.

1. Crash Failures

A component crashes: it stops, loses all of its ephemeral (in-memory) state, becomes unreachable, and never comes back. Other processes that try to contact it simply get nothing — their messages go unanswered. Often the system responds by spinning up a fresh process to take over the dead one's role. This is the failure type the course will focus on most.

2. Network Failures

The network misbehaves in two ways: messages get dropped (lost entirely) or reordered (arrive in a different order than sent).

The Cruel Ambiguity

When an expected message doesn't arrive, you cannot tell which of three things happened: the message was dropped, the message is merely very slow, or the sender crashed. From the receiver's point of view these are indistinguishable. This single ambiguity is the source of an enormous fraction of the difficulty in distributed systems.

The common mitigation is retry and deduplicate: resend the message repeatedly, and discard duplicates on arrival. This doesn't truly solve the failure — it relies on the network not being broken forever — but if delivery is at all possible, persistent retrying eventually gets the message through.

And no, TCP is not a free pass: TCP gives reliable delivery over a working wire, but if the physical cable is literally cut, no protocol can paper over that. There is no total delivery guarantee.

3. Byzantine Failures

Byzantine failures are the worst: a component doesn't just stop — it actively lies. Messages or processes get corrupted or maliciously modified. The classic illustration: P1 sends "do you want tomato pizza?" and an adversary rewrites it in flight to "we want terrible pizza." Because the fault can produce arbitrary, adversarial behavior, Byzantine failures are strictly harder to tolerate than crash or network failures, and live in the realm of advanced distributed systems and cryptography.

Keep this scoping in mind — it explains why later algorithms are allowed to assume, for example, that messages aren't maliciously altered.

Part III

Recovering from a Crash — and the Non-Determinism Trap

Now that crash failures are our target, the obvious question is: when a process dies and loses its memory, how do we bring it back? Two strategies dominate — and one of them hides a subtle trap that motivates the rest of the lecture.

Two Strategies: Replicas and Checkpoints

Key Concept — Replication vs. Checkpointing

Replication keeps multiple live copies of a component running, so if one crashes another already holds the state. Checkpointing (used interchangeably here with snapshotting) periodically saves a component's state to persistent storage like a disk; after a crash, the component reloads the latest checkpoint and resumes from there. These are the two big recovery families this course explores.

Checkpointing is the focus of this lecture. The idea is simple: every so often, write your state to disk; after a crash, retrieve the most recent saved state and continue execution from that point. For now we'll assume the disk is a trustworthy, synchronous store (write something, it's safely saved) — a clean abstraction we'll revisit much later when we study a real Azure system that worries about storage failures.

Why Naïve Checkpointing Goes Wrong

Picture a process that takes a checkpoint, then performs some action, then crashes. On recovery it reloads the checkpoint and re-runs that action. Is that safe? Two things can break.

Problem 1 — Actions Aren't Idempotent

Re-running an action repeats its side effects. If the action was "send a text message" or "send an email," recovery sends it twice. An operation is idempotent if doing it multiple times has the same effect as doing it once — and side-effecting actions on the outside world usually are not idempotent.

Problem 2 — The Action Is Non-Deterministic

Even worse, the replayed action might not do the same thing the second time. If "do something" is non-deterministic, recovery might not just resend a text — it might send it to a different person. The system diverges from its original behavior, and now recovery has made things actively wrong rather than merely repeated.

What Is Non-Determinism, and Where Does It Come From?

An operation is non-deterministic when running it again under "the same" conditions can produce a different result. In a distributed system the sources are everywhere:

  • Randomness — random number generators, and things like LLM output.
  • User input — whatever a human types is outside your control.
  • Network delays — messages between machines arrive in timing that varies run to run.
  • Process scheduling — which thread or process the OS runs when.
  • OS interrupts — asynchronous events the OS injects.
Why Distributed Systems Are Especially Hard to Test

Ordinary software already has randomness, user input, and interrupts. Distributed systems pile on network delays and process scheduling across machines. Each new source multiplies the possible executions, so a distributed program can have exponentially many distinct runs from the same starting point. That combinatorial explosion is exactly why testing — and recovery — is so painful here.

The Fix: Log Everything, Then Replay

You might hope to approximate away non-determinism the way you might for testing. But for recovery you can't approximate — if messages arrived in a particular order, you need that exact order back, not a plausible-looking one. The real solution is to remove the non-determinism by recording it:

1

During normal operation, log every non-deterministic input. Every message received, every random number generated, every external input — write it to persistent storage as it happens.

2

During recovery, consult the log instead of the world. When it's time to "receive a message," don't actually receive — look up the logged message. When code wants a random number, pull the logged one instead of rolling a fresh one.

3

Replay deterministically. Because every previously non-deterministic choice now comes from the log, the recovered process re-traces its original execution exactly, side effects and all — turning a non-deterministic program into a reproducible one.

Keep This in Your Back Pocket

We won't implement non-determinism logging for the rest of this lecture or in the assignments — but it is a genuinely important, frequently hand-waved problem. Almost every recovery scheme in distributed systems quietly faces it. Knowing that "log it and replay it" is the standard answer will help you spot what real designs are sweeping under the rug.

Part IV

From One Node to Many — The Snapshot Problem

Checkpointing a single node is easy. But a distributed system is many nodes, and we sometimes need to capture all of them at once — to recover the whole system after a failure, or to relocate every node to a different cluster. The instant we try to photograph many processes together, deep problems appear. This part shows exactly what goes wrong and frames the trade-off that Chandy–Lamport will resolve.

Why Coordinate at All?

The goal is a global snapshot: a single saved state covering every process, such that recovering from it leaves the whole system in a sensible state. The naïve plan is to let each process checkpoint itself whenever it likes and call the collection of latest checkpoints our snapshot. Why isn't that enough?

P1 checkpoints before sending M; P2 checkpoints after receiving it. The recovered state records the receipt but not the send.

In the figure, P1's checkpoint is taken before it sends message M, and P2's checkpoint is taken after it has received M. Recover both from these checkpoints and you get an impossible state: P2 believes it received a message that, according to P1's saved state, was never sent.

The Core Danger of Uncoordinated Checkpoints

The combined global state can be one that the real system could never have reached on its own. After recovery the system finds itself in an illegal state it has no idea how to handle — like a ledger that records a payment received with no matching payment sent. Recovery has manufactured an impossibility.

Uncoordinated vs. Coordinated — and the Cost

So we need coordination. A first attempt: a "stop-the-world" protocol. One process announces "prepare for a snapshot," waits for everyone to confirm, then tells everyone "snapshot now," and all checkpoint together.

This is consistent — but the cost is brutal. Every process must halt its real work while the round of prepare/confirm/snapshot messages flies around. With n processes you exchange on the order of n² coordination messages, and crucially the entire system stalls.

Why "Stop the World" Hurts So Much

For a latency-critical service like a web backend, tail latency (the slowest few percent of requests) is what users feel. If a snapshot freezes everyone, every request unlucky enough to land during the freeze waits far longer — blowing the latency budget, and potentially costing real revenue. The cardinal rule: don't put snapshot work on the critical path.

Do we have to pick? The whole point of this lecture is that the answer is no — Chandy–Lamport gives a consistent, coordinated snapshot with low overhead and without stopping the world. But before we get there, one more crucial idea.

A Distributed System's State Includes the Network

Consider an application where n processes pass a single lock around a ring: P1 → P2 → P3 → P1, and whoever holds the lock may act. Now suppose we draw a snapshot "cut" across all three timelines at carelessly chosen points.

A "lock" message is in flight across the cut — sent before the line, not yet received. If we ignore it, the lock vanishes from the snapshot entirely.

Two opposite disasters are possible. Cut the timelines one way and every process records that it holds the lock — three locks exist, violating mutual exclusion. This is the same happens-before violation as before: a message receipt is captured without its send.

Cut them the other way and no process holds the lock — the lock is lost, and the system can never make progress. Here the problem is different and more subtle: the lock message was in flight — already sent, not yet received — at the moment of the cut. Neither process's state captures it, so it simply disappears.

The Big Idea — Channels Are State

The state of a distributed system is not just the state of its processes. It is the process states plus the state of every communication channel — every message currently in transit. A message that's been sent has already committed to causing an effect, so it is genuinely part of the system. Any correct global snapshot must capture in-flight messages. This single insight is what makes the snapshot problem hard, and it is exactly the thing Chandy–Lamport handles so elegantly.

Key Concept — Two Requirements for a Good Snapshot

(1) Respect happens-before: if event B is in the snapshot and A → B, then A must be in it too. In practice this means a captured message receipt must come with its send. (2) Capture in-flight messages: messages sent before the cut but not yet received are part of the channel state and must be recorded.

In the ring-of-locks example, a snapshot is taken in which no process holds the lock and the lock seems to have disappeared. What went wrong?

Part V

The Chandy–Lamport Algorithm

We want a snapshot that is consistent (respects happens-before, captures in-flight messages) and coordinated, yet cheap and non-blocking. The Chandy–Lamport algorithm delivers exactly this. It was originally designed to evaluate stable properties of a system — but its lasting fame is as the way to snapshot a distributed system for fault tolerance.

Aside: what's a "stable property," and why was the algorithm invented?

A stable property is one that, once true, stays true forever. Two classic examples: deadlock (take a global snapshot, look for a cycle of processes each waiting on the next — if found, the system is deadlocked) and distributed garbage collection (confirm no process anywhere still holds a handle to an object). The original paper aimed at evaluating any such property. In this course, though, the headline use is snapshotting for fault tolerance.

System Model and the Marker Trick

Every protocol states its assumptions; here are Chandy–Lamport's.

Key Concept — System Model

(1) There are n processes, and no process fails while the snapshot is being taken. (2) Any process may initiate a snapshot — there's no special coordinator. (3) Every directed pair of processes has a FIFO unidirectional channel (a separate send-channel and receive-channel each way). (4) Channels never duplicate or lose messages.

The FIFO assumption is the secret weapon. FIFO ("first in, first out") means a channel between two processes delivers messages in the exact order they were sent — no overtaking. If M1 is sent before M2 on a channel, M2 can never arrive before M1. (Note: FIFO only holds per pair of processes, not globally across the whole system.)

The Key Insight

After a process takes its checkpoint, it sends a special marker message down every outgoing channel. Because channels are FIFO, that marker arrives at the other end before any application message the sender produces afterward. The marker therefore acts as a clean barrier on each channel: everything before it belongs in the snapshot, everything after it does not. FIFO turns a single marker into a perfect dividing line.

Markers are distinct from application messages — they're consumed by the snapshot algorithm itself and cause no work for the application. They serve two jobs at once: (1) they tell a process that a snapshot is underway, and (2) they act as the per-channel barrier just described, guaranteeing each captured channel includes exactly the messages received before the marker and none after.

The Algorithm, Step by Step

The initiator

1

Some process spontaneously decides to start. It records its own local state (takes a checkpoint).

2

It sends a marker on every outgoing channel.

3

It starts recording every message that arrives on each of its incoming channels.

When any process Pi receives a marker

Say the marker arrives from Pk on channel C(k,i). (Notation: C(k,i) is the channel where Pk is the sender and Pi the receiver.) There are two cases.

Case A — first marker this process has ever seen

Then Pi (1) records its own state — this is its checkpoint; (2) marks channel C(k,i) as empty; (3) sends a marker out on every outgoing channel C(i,j); and (4) starts recording incoming messages on all other channels (every channel except the one it just marked empty).

Case B — it has already seen a marker before

Then it does not re-checkpoint. Instead it stops recording on C(k,i) and fixes that channel's final state as all the messages it recorded on C(k,i) between when recording started and this marker arrived.

Why empty in Case A?

On the channel the very first marker arrived on, recording would start and immediately end at the same instant — the first marker is also, in effect, the last word that matters on that channel. So there are zero messages to capture: the channel is recorded as empty. On every other channel, the process will eventually receive a second marker, and everything that arrived in between is the channel's contents.

Why does everyone re-broadcast markers?

A natural objection: if the initiator already sent a marker to everyone, why must each receiver also blast markers to everyone? Isn't that duplicate work? No — the point is that every channel must receive a marker to be closed. Each channel needs a marker traveling along it to fix its contents. The initiator's markers only cover the initiator's own outgoing channels; the channels between other processes still need their own markers to be flushed and sealed.

When Is the Snapshot Done?

The global snapshot is complete when every process has received a marker on every one of its incoming channels. The intuition: the marker on a channel acts as a "flush" — once it arrives, all messages that were ever in flight on that channel have been received and recorded, so the channel can be safely closed. We wait until every channel has been flushed and sealed by its marker; only then is the whole snapshot finished.

A Worked Example

Let's trace three processes. P3 initiates. Watch how each process's "first marker" determines its checkpoint, and which application messages end up recorded as channel state.

4

Because M1 and M2 were sent before P1's checkpoint but their channels get closed by later markers, they are exactly the in-flight messages recorded as channel state. Every other application message in this run was exchanged after the relevant checkpoints, so none of those are recorded.

Two Pieces of Saved Information — Don't Conflate Them

Each process saves two distinct things. (1) Process state — captured instantaneously when it sees its first marker. (2) Channel recordings — captured incrementally as messages arrive, and finalized when the closing marker for that channel arrives. Both together form the global snapshot. Whether you store them in the same place doesn't matter; what matters is that you keep both.

Crucially — Normal Work Never Stops

Throughout all of this, the system keeps running exactly as before. A process applies incoming messages to its real state as usual; the only additions are taking a checkpoint, emitting markers, and logging the relevant messages to the side. Nothing blocks, nothing stalls. That's the entire payoff: a consistent, coordinated snapshot with no work on the critical path.

Subtlety: recorded messages vs. checkpoint state — and why we keep them separate

A message that had already been received before a process's checkpoint is baked into that process's saved state — the process already acted on it, so its effect is captured. A message still in flight at checkpoint time isn't in any process's state, so it must be recorded separately as channel state and replayed on recovery.

Why keep "state" and "recorded messages" as two separate things instead of applying the messages to produce one final state? They are equivalent in correctness — you could replay the recorded messages into the state and resave a single combined state. The reason the algorithm records them separately is that channels are closed incrementally (each channel finalizes when its own marker arrives, not all at once), so it's cleaner to accumulate messages as they come and finalize per channel. Storing state plus messages, versus storing one merged final state, recovers to the same place.

How Many Markers? And a Special-Case Escape Hatch

With n processes, every process sends a marker to every other process — to everyone except itself. That's n × (n − 1) = n² − n markers total.

Connection — n² Messages, but Off the Critical Path

The stop-the-world protocol from Part IV also cost on the order of n² messages — but it blocked the whole system while sending them. Chandy–Lamport sends a comparable number of marker messages, yet no one ever stops working. The markers are just extra traffic riding alongside normal messages. Same message count, completely different cost, because the work isn't on the critical path.

The Frontier

The collection of all the checkpoint points across processes is often called the frontier of the snapshot. Every message sent from inside the frontier must end up captured in the snapshot — either already absorbed into some process's state, or recorded as channel state. The frontier is the visual boundary marking where the global snapshot "ends."

Finally, a lesson that recurs throughout the course. Return to the ring-of-locks application. There exist snapshots that technically violate happens-before — recording a lock received without its send — yet are still perfectly correct for that specific application, because all that matters is that exactly one process ends up holding the lock and the system can progress. Chandy–Lamport is a way to get a correct snapshot, not the only way.

The Big Lesson — General Is Hard, Specific Is Easier

A protocol that must work for every application has to be very careful and very general. But if you can lean on assumptions about your specific application — like "the lock just needs to live somewhere in the ring" — you can often get away with something simpler. This is how much of distributed-systems research and design actually works: find the special structure in your problem, then exploit it.

Putting It All Together

This lecture was the course's turn from ordering to survival. Here's the spine that connects every part.

We opened by acknowledging that logical clocks (Part I) — Lamport clocks, vector clocks, replicated state machines — solve ordering but say nothing about failure. That gap defined the new mission: fault tolerance (Part II), which is always relative to a declared fault model. Among crash, network, and Byzantine faults, the course commits to crash failures as its main quarry.

To survive a crash you recover, and the cheapest recovery tool is checkpointing (Part III). But naïve replay collides with non-determinism — re-running can repeat side effects and diverge — so we log non-deterministic inputs and replay them deterministically. The moment we scaled checkpointing from one node to many (Part IV), two new failure modes appeared: inconsistent global states that violate happens-before, and lost in-flight messages, because a distributed system's true state includes its channels. That left us trapped between cheap-but-inconsistent uncoordinated snapshots and consistent-but-world-stopping coordinated ones.

Chandy–Lamport (Part V) escaped the trap. By riding on FIFO channels and a single clever idea — a marker that acts as a per-channel barrier — it captures both process state and in-flight channel state, producing a consistent, coordinated snapshot for about n² − n marker messages, all without ever stopping the system's real work. Watch how vector-clock intuition (preserving causal order) quietly underwrites the marker barrier, and how the course's recurring lesson — exploit your application's specific structure — let even the ring-of-locks tolerate a "technically wrong" snapshot.

The Through-Line

The hardest thing to save in a distributed snapshot is the in-flight messages — the system state living inside the network, not inside any process. Chandy–Lamport's entire cleverness is a mechanism to capture exactly those, cheaply and without pausing the world.

What's Next

Chandy–Lamport assumes no process fails during the snapshot — a strong assumption (if a node dies before closing its channels, the snapshot is invalid). The next lectures relax exactly these assumptions, asking: how do we tolerate crash failures without relying on checkpoints? That's where replication returns in force and the course builds toward consensus. Keep the marker/barrier idea handy — the discipline of "flush the channel, then seal it" reappears whenever distributed processes must agree on a clean cut through time.
