Consistency Models & Linearizability

An interactive guide to Lecture 6 — what it means for a replicated service to behave "correctly," and the ladder of guarantees from linearizable down to causal.

Part I

Where We Are & Why We Care About Ordering

Up to now this course has been building two parallel tracks. There is the systems track — messy, real-world machinery like MapReduce — and the foundations track — the clean theory underneath, like logical time and snapshots. The last lecture introduced primary-backup replication, a practical way to keep a service running even when a machine dies. Today we cross fully into foundations and ask a deceptively hard question: when we run many copies of a service, what does it even mean for them to behave correctly?

Where This Sits In The Course

This lecture builds directly on primary-backup (last time) and on the happens-before / logical-clock ideas from the time-and-snapshots lectures. It sets up the next two lectures on consensus, and later ones on weak consistency and distributed transactions. Consistency models are the vocabulary the rest of the foundations half will speak in.

A Quick Recap: Primary-Backup

In primary-backup replication, one replica is the primary and the others are backups. The backups stay in sync with the primary and are ready to take over the instant it crashes. A separate view service tracks the current configuration — who is primary, who is backup — so clients know where to send requests.

The view service only advances to a new "view" after the primary has acknowledged the current view, and the primary only acknowledges after confirming it can sync with all its backups. Two details from last time matter today: clients cache the view (so they don't ask the view service on every request), and crucially, to avoid split brain the primary must check with its backups before serving any client request — including reads.

Correction From Last Lecture

A clarification the professor opened with: in this service's architecture, the primary must always contact the backup before responding — even for reads. An earlier slide suggested reads could skip the backup as an optimization; in our setting that optimization causes split brain and is wrong. When in doubt: contact the backup.

Why We Don't Actually Need Input Queues

Last lecture, while replicating a MapReduce master, the professor claimed we'd have to replicate the master's input queues alongside its state. He corrected that here, and the reasoning is worth internalizing because it reveals how this whole request model works.

A queue is appropriate when a component fires a message into a one-way channel and then forgets about it — fire-and-forget. But our clients don't do that. A client sends an RPC to the primary and then blocks, waiting for a response. The request lives in the client's hands until it completes.

1

A client sends a request to the primary and waits — it does not move on.

2

The primary picks up one request, parses it, does the work.

3

If the primary crashes mid-request (or the network drops it), the client simply sees an error.

4

The client is responsible for retrying. Nothing was ever "parked" in a queue, so there is no queue to replicate.

The Mental Model Shift

There is no expectation that "because I made the request, the primary now owns it and will eventually tell me." The client stays on the hook until it sees a completion. This synchronous, blocking, at-the-client model is exactly why we can reason about correctness in terms of when a request started and when its response arrived — the foundation of everything in this lecture.

What Ordering Do We Actually Want?

Picture a classic example from earlier in the course: a bank account replicated across two machines, with clients in different cities issuing deposit, pay_interest, and get_balance operations. With multiple replicas processing concurrent requests, which orderings of operations should the system be allowed to produce? The class brainstormed, and three themes emerged.

1

Some relationship to real-world time. If I deposit money and that finishes, a later request should reflect it — the system shouldn't pretend my completed deposit never happened.

2

A "happens-before" between my own operations. If I deposit $500 and then try to withdraw $1,200, the withdrawal should see the deposit. Operations I issue in sequence should be seen in that sequence — but, a student noted, only per account. Two users touching different accounts never interact, so their operations needn't be ordered relative to each other.

3

Atomicity. A deposit that does read-modify-write (read account → add amount → write account) must not interleave with another deposit, or money gets lost.

Scope For Today

Today's lecture touches all three, but we assume atomicity is handled for us by the system. Our focus is purely on ordering: which sequences of completed operations a replicated service is allowed to expose to clients.

Three Properties Worth Wanting

Generalizing the brainstorm, here are three properties that, together, describe an ideal-feeling service. Each one exists to make life easier for the application programmer sitting on top.

1

Total order of writes. All writes are placed in one global order — there's a fact of the matter about which write was first, second, third. From the client's view, it's as if a single thread processed every request in order. That's the easiest possible thing to program against: deposit money, then buy coffee, and the coffee purchase will see the money.

2

Reads never go backward. Once a read returns a value, every later read returns that value or a newer one. The service can never take back something it has already shown you.

3

Reads see the freshest completed write. Once a write completes, every later read returns that write's value (or a newer one). If your deposit completed, you can immediately spend the money.

The Big Idea Behind All Of This

Stronger consistency exists to make applications easier to write. If your service promises these properties, you never have to write defensive code like "maybe my write hasn't landed yet, so let me re-check." You get to pretend there's a single, in-order machine behind the curtain. The entire rest of the lecture is about how much of this guarantee you can keep — and what it costs.

Part II

Linearizability — The Strongest Guarantee

We listed three desirable properties. Now we name the consistency model that delivers all of them at full strength. That model is linearizability, introduced by Herlihy and Wing. Before the formal machinery (Part III), let's nail the intuition.

What Linearizability Promises

Key Concept — Linearizability

Linearizability is the strongest consistency condition. It guarantees a total order of writes, and it guarantees that every operation observes the latest completed write. Equivalently, it gives single-copy semantics: all the visible effects look exactly as if there were a single, non-replicated copy of the data processing operations one at a time, in an order consistent with real time.

Let's ground this with the bank example the professor walked through. Start with a balance of $1,000. Client 1 issues a deposit of $100, and before it finishes also issues pay_interest (which means "add 1% interest"). Meanwhile Client 2 issues a get_balance. What value may that get_balance return?

1

The deposit's response has already been sent to Client 1 — the primary told the client the deposit is done. So the result must include the $100. Balance is at least $1,100.

2

The interest request has arrived, but no response has been sent. Has it "happened"? Either answer is acceptable. Until the system confirms completion, it's free to have done it or not.

3

So get_balance may return $1,100 (deposit only) or $1,100 + interest. Both are linearizable. What it may never return is anything below $1,100 — that would erase a completed write.

"Done" Means Responded, Not Sent

The moment that matters is not when a request is sent into the system — it's when the system responds that it's done. Think of tapping Apple Pay: until the screen says "Approved," the payment hasn't truly gone through, even though you've already tapped. If two people race to overdraft the same account, exactly one response comes back first, and that's the one that "happened first."

One more constraint from the example: if a second get_balance comes after the first one returned, it must see everything the first one saw — never less. Reads never travel backward in time.

Single-Copy Semantics: Why It's Worth Paying For

The payoff of linearizability is that users are completely oblivious to replication. From the outside, a linearizable system and a real single-machine system are indistinguishable — and this is provable, not just intuitive (the original paper proves the equivalence to a sequential single copy).

Why This Matters For The Application Developer

If your service is linearizable, you never write code like "I just did a write, but maybe my read won't reflect it yet, so let me add a retry loop." That defensive mental gymnastics disappears. You program as if there's one machine doing one thing at a time. This is exactly the "easier to program" payoff from Part I's big idea — made concrete.

A Tempting But Wrong Fix

A student asked: could we just delay the acknowledgement so the client never notices a stale-looking order? You could — but only by increasing latency for no benefit. The client already can't tell which of two overlapping operations finished first, so there is nothing to hide. Adding delay just makes the system slower while fixing a "problem" that was never observable. (More on why overlap gives this freedom in Part III.)

Part III

Traces, Specifications & the Formal Definition

The intuition in Part II — "looks like a single copy" — is exactly right, but to check whether a real, messy, concurrent execution is linearizable we need a precise vocabulary. This part builds that vocabulary one term at a time. It's a little pedantic, which the professor admitted, so we'll go slowly and define every word.

Executions and Traces

We model a system as multiple clients sending requests. An execution (or trace) is a sequence of two kinds of events: request events (with arguments) and response events (with return values).

// a trace as a raw event sequence
request  deposit(100)
response ok
request  balance()
response 1100

That linear text is hard to reason about, so we draw traces. Each client gets its own horizontal line, and each operation is a box: the left edge is the request, the right edge is the response. The box's width is real time, so the picture shows directly which operations overlap and which are cleanly separated.

A trace drawn as boxes. The left edge of each box is the request; the right edge is the response. Here C1's deposit finishes before C2's balance even starts.

Sequential Traces

Key Concept — Sequential Trace

A trace is sequential if no two operation boxes overlap — every request finishes before the next one starts. We also assume each individual client is sequential: it has at most one pending request at a time.

The professor stressed how pedantic this definition is, with a sharp example. Consider one client doing enqueue(x) and another doing dequeue() → x on a FIFO queue, where the two boxes overlap. Is this a valid FIFO behavior? Yes — x went in and x came out. But is it sequential? No — the boxes overlap. "Valid for the data structure" and "sequential" are two completely separate questions. Slide the dequeue box to the right so they no longer overlap, and only then is it sequential.

Linearizations

Key Concept — Linearization

A trace T1 is a linearization of a trace T2 if it is obtained by making the overlapping operations non-overlapping in some chosen order, while preserving the order of any operations that were already non-overlapping.

The intuition: each operation really "takes effect" at some single instant inside its box. If two boxes overlap, we genuinely cannot know which instant came first, so we're free to pick either order. If two boxes don't overlap, the earlier one definitely took effect first, and that order is locked.

Two overlapping operations can be linearized two ways. Because they overlap, either order is a legitimate reading of "what really happened."
Linearization Adds No New Behavior

A student worried: doesn't picking an order "invent" something? No. At the end of the day, A and B each really took effect in some order — we just couldn't observe which. Choosing one is allowed precisely because the overlap made both orders externally possible. Linearization never introduces behavior the system couldn't already have exhibited.

Specifications: What "Correct" Even Means (FIFO Queue)

We can now describe a system's whole observable behavior as a set of traces. The specification of an object is the set of sequential traces that count as correct for it. A specification is sequential when all its traces are sequential — important because we reason about correctness "one request at a time."

Take a real FIFO queue with enqueue(x) and dequeue() → x (dequeue removes and returns the oldest item). Which sequential traces belong to its specification?

Sequential traceIn the FIFO spec?
enq(x); deq()→xYes — oldest item out is x. Valid.
enq(x); enq(y); deq()→zNo — z was never enqueued. Not a FIFO trace.
enq(x); enq(y); deq()→yNo — oldest is x, not y. Not FIFO.

The specification is simply the collection of all the sequential traces that respect FIFO order. To ask "is this object a FIFO queue?" we ask: is this execution one of the traces in the set? Note both bad rows above are perfectly sequential (nothing overlaps) — they just aren't FIFO. Sequential is about timing; the spec is about meaning.

The Formal Definition

Everything assembles into one clean statement.

Key Concept — Linearizable Implementation

Given a sequential specification spec (a set of sequential traces), an implementation (also a set of traces, mostly non-sequential) is linearizable if: for every trace T the implementation can produce, there exists a linearization T' of T such that T' ∈ spec.

In words: our real distributed replicas emit all sorts of messy, overlapping executions. The implementation is linearizable if every one of them can be untangled (by reordering only overlapping operations) into a clean sequential trace that the specification already approves. We don't care how the system works internally — only whether its observable traces can always be explained this way.

Linearizability Is Always Relative To A Spec

Online discussions often say a system "is linearizable" with no qualifier — but there is always an implicit specification underneath. You can only define linearizability against some notion of how the object is supposed to behave (a register, a queue, a counter). No spec, no meaning.

How To Check Linearizability

Given a single observed execution, here's the recipe — it's essentially a search problem.

1

Fix the specification you're checking against (e.g. "register" or "FIFO queue").

2

Keep the order of all non-overlapping operations fixed — you're not allowed to touch those.

3

Try every possible ordering of the overlapping operations.

4

If any such ordering yields a sequential trace in the spec, the execution is linearizable. To prove it's not linearizable, you must exhaustively show that none of the orderings work.

To show an entire distributed system is linearizable, you check that all its possible executions are. In practice that's done two ways: by proof (mathematically characterize every execution and show each is linearizable), or by a dynamic monitor that watches runtime executions and raises an alarm the moment a non-linearizable one appears.

Worked Examples: A Register

Let's apply the recipe to a sequential register (one cell, operations set and get, initial value 0). For each trace below, ask: is it linearizable?

Three register executions (initial value 0). ① and ③ fail; ② passes thanks to overlap.
1

set(1) then get → 0, non-overlapping. Not linearizable. The set finishes before the get starts, so their order is locked: the get must see 1, not 0. No reordering is even allowed.

2

set(1) overlapping get → 0. Linearizable. Because they overlap, we may place the get first: it reads the initial 0, then the set lands. Valid register behavior.

3

set(1) and set(2) overlapping, then get → 1 and get → 2. Not linearizable. A single register can settle on either 1 or 2, but not both. We can reorder the two sets to explain one get, but never both at once. One client seeing 1 and another seeing 2 cannot be a single register.

A Useful Contrast

A student asked: what if example ③ were set(x,1) and set(y,2) — two different variables? Then it would be linearizable, because the gets touch independent cells and can each be explained on its own. The conflict in ③ exists only because both sets and both gets are fighting over the same register.

Why You Can't Just Delay To Fake It (Revisited)

Recall the Part II "tempting fix." This is the deeper reason: an operation's true effect is a single point inside its box. With non-overlapping boxes that point order is fixed by real time, so no delay can change which one was first. You can only exploit reordering freedom where boxes overlap. If your friend's request started after yours finished and they see a state older than yours, that's a genuine violation. If their request merely overlapped yours, network lag explains everything and nothing is broken.

Linearizability Is Compositional

A standout property, proven in the paper: if every object in your system is individually linearizable, then the whole system is linearizable. You can build linearizable components and freely compose them — the guarantee survives. Keep this in mind; the very next model we meet does not have this property, and that contrast is one of the most important takeaways of the lecture.

Part IV

Sequential Consistency — Dropping Real Time

Linearizability is wonderful to program against, but it has a price we'll quantify here. So why would anyone want anything weaker? Because enforcing linearizability is expensive, and weaker models can buy serious performance with acceptable trade-offs. The first step down the ladder is sequential consistency, introduced by Leslie Lamport in his paper "How to Make a Multiprocessor Computer That Correctly Executes Multiprocess Programs."

The Idea: A Global Order, But Ignore Real Time

Key Concept — Sequential Consistency

An execution is sequentially consistent if there exists some single sequential order of all events that explains it — and each client's own operations appear in that order in the sequence they issued them. The crucial relaxation from linearizability: we may reorder operations across different clients arbitrarily, ignoring real-world time.

Linearizability says "the global order must respect real time." Sequential consistency says "there must be a global order, but it need not match real time across clients." Why is dropping real time defensible? Because if clients only ever talk to the system through requests, how could one client even know that another client's operation finished before its own began? Without an outside channel, they can't — so from their perspective any consistent global order is fine.

The Catch — A Hidden Side Channel

That argument breaks if two clients have an external communication channel (a phone call, a shared whiteboard). Then real-time ordering does become observable, and sequential consistency can produce surprising results. Hold this thought — it returns when we discover sequential consistency is not compositional.

Here's the canonical contrast. Under linearizability, if set(x,1) finishes and then get(x) starts (non-overlapping), the get must return 1. If the writer is in New York and the reader in Los Angeles, the replicas must coordinate within microseconds — wildly expensive. Under sequential consistency, an execution where the LA read returns the old value 0 (even though the NY write finished first in real time) is perfectly legal, as long as a single global order exists.

Worked Examples

For each register execution (initial 0), classify it: linearizable, only sequentially consistent, or neither?

C3: set(x,1) then sees order 1,2; C4: set(x,2) then sees order 2,1 Neither. Two clients disagree on the order of the same two sets. No single global order can place set(1) before set(2) and after it. Same-client order is locked, so no reordering rescues it.
C1: set(x,1); C2: get→2 then get→1 (with a set(2) somewhere) Neither. Forcing the first get to see 2 pushes set(2) late, but then the later get→1 can't see set(1). The constraints form an impossible cycle.
A trace already in program order, no overlaps, but real time violated only across clients Only sequentially consistent. No overlap means it's not a valid linearization (can't reorder anything), but we can still build a legal global order by interleaving across clients — e.g. slot a get between two sets.
The Two-Part Argument You Must Give

To show an execution is "only sequentially consistent," you owe two claims: (1) it is sequentially consistent — exhibit a legal global order, e.g. set1, get1, set2, get2; and (2) it is not linearizable — show no valid real-time-respecting reordering exists (often because nothing overlaps, so it isn't even a valid linearization to begin with).

The Surprise: Sequential Consistency Is Not Compositional

Now the deep result. Consider two registers, X and Y, each individually sequentially consistent. Combine them in one system. Is the combination sequentially consistent? Not necessarily.

The professor sketched the construction (and noted in class that an earlier version of the slide had a bug he later fixed): take an execution where, to make X's part work, you must move a get(x) before its set(x); and simultaneously, to make Y's part work, you must move a get(y) before its set(y). Each fix is fine alone. But the two required reorderings conflict — satisfying one forces the other to fail. There is no single global order that satisfies both, so the combined system is not sequentially consistent even though each object is.

Compositionality: The Sharpest Distinction

This is the lecture's most important structural insight. Linearizability composes — linearizable parts make a linearizable whole. Sequential consistency does not — sequentially-consistent parts can combine into something that is not sequentially consistent. That single difference is a major reason linearizability, despite its cost, is so valued: you can reason about components independently.

Implementing Sequential Consistency

We want one global order of all events, but we're freed from matching real time. How do we get a total order? With tools already in this course.

1

Use Lamport clocks to impose a total order on events — exactly the logical-time machinery from earlier lectures.

2

Or force a single primary to perform all writes, so the write order is unambiguous.

3

Because real time across clients no longer matters, you can serve reads from stale read-replicas or caches. A read might see an old value — fine, as long as a single global order over all events still exists.

Still Not Cheap

Read-replicas and caches help, but sequential consistency still blocks all writes to maintain the global order. Writes must serialize and synchronize. You save on reads, not on writes. That residual cost is the motivation for going even weaker — which is where causal consistency comes in.

Part V

Causal Consistency & the Whole Spectrum

Sequential consistency still forces a total order on all writes — and that ordering blocks writes globally. The next step down keeps only the orderings we genuinely need and drops the rest, buying much cheaper synchronization. This is causal consistency, introduced in 1995.

The Idea: Preserve Only Happens-Before

Key Concept — Causal Consistency

Causal consistency requires that the order of causally related writes be preserved in what reads observe. Formally, if W1 happened-before W2 (using the happens-before relation from the Lamport-clock / vector-clock lectures), then any read that sees the effect of W2 must also see the effect of W1. There is no requirement on a total order for unrelated writes.

Recall happens-before has exactly two sources: (1) two events on the same client are ordered by program order, and (2) a message from one process to another orders the send before the receive. If two writes share neither relationship, causal consistency lets reads see them in any order — even different orders for different clients.

The Intuition: A Social Media Feed

It's fine if you don't see every post in the world. But if you see a comment, you must also see the post it replies to; if you see a retweet, you must see the original tweet. The comment causally depends on the post, so that one ordering is sacred. Everything causally unrelated can arrive in any order. The payoff: replicas can synchronize far more lazily, because they only need to respect dependencies, not a global clock.

Worked Examples

Reusing the register executions, now ask: are they causal? (We already know whether each is linearizable or sequentially consistent.)

1

Two clients each write, a third reads them in one order, a fourth reads them in the opposite order. Causal. The two writes are on different clients with no message between them — so neither happens-before the other. With no causal link to preserve, readers may disagree on their order freely. (This same execution was neither linearizable nor sequentially consistent — causal is strictly weaker.)

2

A client reads set(x,1), and based on that issues set(x,2); later a read sees 2 but not 1. Not causal. Here set(1) happens-before set(2) (same client, program order, the second even depends on having read the first). A read that sees 2 is obligated to also see 1. Seeing 2 but not 1 is a causality violation — it's "turning back in time" to a state before the one you already knew about.

3

The previously "only sequentially consistent" execution. Causal. Sequential consistency demands a total order, which is strictly stronger than respecting only causal order. So anything sequentially consistent is automatically causal.

Why "Turning Back Time" Is The Real Violation

In example 2, one client saw 1, then caused 2, then saw 2 — from its viewpoint the system has reached state "2," which is built on top of "1." If a read then surfaces 2 without 1, that client is forced to witness a past state of a system it already saw move forward. That's the precise harm causal consistency forbids. Crucially, if those two reads are on different clients with no shared history, each can legitimately have its own "final state," and there's no violation.

The Whole Spectrum at a Glance

We now have a ladder. Every step down is weaker (allows more executions, harder to reason about) but cheaper to implement.

ModelOrder guaranteedCost / notes
LinearizableTotal order of all ops, respecting real timeMost expensive; single-copy semantics; compositional
SequentialTotal order of all writes; same-client order kept; ignores cross-client real timeCheaper reads (caches/replicas), but writes still serialize; not compositional

The consistency ladder. Left = stronger guarantees, easier to program, more expensive. Right = weaker, cheaper. Many more conditions live between these — a 2016 survey maps dozens, some not even comparable to each other.
Consistency Models Don't Fully Describe Your System

A subtle point the professor stressed: a consistency condition is a property, not a complete description of correctness. The set of executions allowed by "causal" includes many your real system would never produce — even nonsensical ones. A condition is a lens: it rules out some bad behaviors (like turning back time) while staying agnostic about everything else. To pin down a real system you intersect several such properties, gradually carving out something that resembles — but never exactly equals — your actual system.

Putting It All Together

This lecture answered the question we opened with: what does it mean for a replicated service to be "correct"? The answer isn't a single yes/no — it's a ladder of consistency models, each a different bargain between how easy a service is to program against and how expensive it is to build.

We started from the synchronous, at-the-client request model (the reason there are no input queues to replicate) and from the three properties an ideal service would have: a total order of writes, reads that never go backward, and reads that see the freshest completed write. Linearizability delivers all three by promising single-copy semantics that respect real time — and, uniquely, it composes. To check it precisely, we built the formal machinery of traces, sequential traces, linearizations, and specifications: an execution is linearizable if every messy concurrent trace can be untangled — reordering only overlapping operations — into a sequential trace the spec already blesses.

Then we walked down the ladder. Sequential consistency keeps a total order but abandons real-time ordering across clients — cheaper (caches and read-replicas become legal) but it loses compositionality, the single sharpest contrast with linearizability. Causal consistency goes further, preserving only happens-before relationships, which lets replicas synchronize lazily; the social-media feed is the mental model — see the comment, you must see the post. Each step trades a guarantee for performance, and each guarantee maps directly onto how much defensive logic the application programmer is spared.

The One Sentence To Remember

Stronger consistency makes applications easier to write but costs more to implement — and the art of distributed systems is choosing the weakest model your application can tolerate, so you pay only for the guarantees you actually need.

What's Next

We now know what we want (a total order of writes for the stronger models) but not yet how a set of replicas can agree on one despite crashes and network failures. That's consensus — the next lecture, and the engine that makes linearizable and sequentially-consistent systems actually buildable.
