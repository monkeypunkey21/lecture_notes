Consensus & the Paxos Protocol

How a group of unreliable machines — some crashed, some partitioned, some asleep — can still agree on a single value that can never be taken back.

Part I

Causal Consistency, Made Precise

Before we reach the heart of today's lecture — consensus — we owe a debt from last time. The professor admitted the definition of causal consistency was "a little bit hand-waved." Distributed-systems definitions are notoriously unintuitive, so the right way to learn them is not to expect them to click immediately, but to nail down the precise definition and then test it against concrete executions. Let's do exactly that.

First, some grounding vocabulary, since we'll lean on it all lecture. A client is an actor (a process, a node, a person) issuing operations. A set writes a value into the system; a get reads one back. An execution (or trace) is the recorded sequence of all these operations as they actually happened across all clients. A consistency model is a rule that says which executions are "allowed" — which ones a correct system is permitted to produce.

Two Kinds of "Happens-Before"

Causal consistency, defined in a 1995 paper, is built on a single relation called happens-before. The subtlety is that this relation is generated by two distinct rules, and you have to track both:

Key Concept — The Happens-Before Relation

Operation A happens-before operation B if either of these holds:

(1) Program order. A and B are issued by the same client and A comes first. The client did A, then B.

(2) Reads-from. B is a get that returns the value written by A (a set). Reading a value means you must have come "after" the write that produced it.

Happens-before is the transitive closure of these two: chain them together and the relation flows through.

A crucial fact about the reads-from edge: a get reads from exactly one set (or from the system's initial value). There's no ambiguity about which write a read observed — it saw precisely one write's value.

Two happens-before edges. The red program-order edge links one client's own operations; the green reads-from edge crosses clients, linking a write to the read that observed it.

The Definition Itself

Key Concept — Causal Consistency

An execution is causally consistent if, for every client, there exists a linearization (a single total order) of that client's own operations plus all writes in the trace, such that the order preserves the happens-before relation.

Read that twice, because the placement of "for every client" is the whole game. There is no single global order that all clients agree on. Instead, each client independently gets to build its own consistent timeline. Client 1 may order events one way; Client 2 may order them another way. Both can be valid. The only thing each client's private order must respect is happens-before.

Why "Plus All Writes"?

When we build Client 1's private timeline, we don't only include Client 1's operations — we fold in every write from every client. Reads stay local to the client, but writes are shared facts the whole system must account for. This is what lets one client "see" another client's writes while still ignoring the other client's read ordering.

Why Sequential Consistency Implies Causal

The lecture spent real time on a claim worth understanding deeply: causal consistency is weaker than sequential consistency. "Weaker" means more permissive — it allows strictly more executions. Equivalently, the implication runs the other way: if an execution is sequentially consistent, it is automatically causally consistent. Let's first define the stronger model.

Key Concept — Sequential Consistency (SC)

An execution is sequentially consistent if there exists a single global total order of all operations — one timeline everyone shares — that (a) respects each client's program order, and (b) is explainable under the object's sequential specification (a read returns the value of the most recent write in that order).

There are two reasons SC is stronger. The obvious one: SC demands one global order, whereas causal lets each client have its own. A single global order can always be "projected down" to give each client an order, but per-client orders can't always be stitched into one global order. So global ⇒ per-client, but not the reverse.

The subtle reason — the one the class wrestled with — is this: does SC's global order automatically preserve the reads-from edge? If it does, then everything causal consistency requires is already guaranteed by SC, which proves the implication. The answer is yes, and here is exactly why:

1

Suppose Client 2's get x returns 2, the value written by Client 1's set x=2. That's a reads-from edge.

2

SC's global order isn't any order — it must satisfy the sequential specification. A read must return the value of the latest preceding write in that order.

3

Now try to place get x before set x=2 in the global order. Then the latest write before the read would be some earlier value (say the initial 0), so the read would have to return 0 — not 2.

4

That contradicts what actually happened: the read returned 2. So any valid SC order is forced to put the write before the read. The reads-from edge is preserved automatically — it's not an extra rule, it falls out of the sequential specification.

The Big Idea

You never preserve reads-from "by hand" in sequential consistency. It comes for free, because a global order that explains the values actually read cannot possibly put a read ahead of the write it read from. This is why every SC execution is also causal: causal asks for exactly the edge that SC already cannot violate.

The Consistency Hierarchy

Putting last lecture's models in a line, from strongest (most restrictive) to weakest (most permissive):

ModelGuaranteesReal-time order?
LinearizabilityGlobal total order + real-time orderYes — if A finishes before B starts in wall-clock time, B sees A
Sequential consistencyGlobal total order, program order respectedNo
Causal consistencyPer-client orders, happens-before preservedNo

The arrows of implication point downward: linearizable ⇒ sequentially consistent ⇒ causally consistent. If a system satisfies a stronger model, it automatically satisfies every weaker one. A linearizable system is also SC and also causal; an SC system is also causal.

Only Linearizability Respects Real Time

Consider a set that completes, and then a get that starts after it finished in wall-clock time. Only linearizability forces that later get to see the set's value. Under SC and causal consistency you're allowed to reorder them — that "later" get could legally return the old value (e.g. 0), because those models have no notion of real-time ordering at all.

How to Actually Answer These Questions

The professor's repeated advice: don't trust intuition, follow the definition mechanically. To check causal consistency, gather each client's operations plus all writes, draw the happens-before edges, and ask "can I find an order with no cycle?" If the required edges form a cycle, no valid order exists and the property fails. We'll use exactly this cycle-hunting technique on consensus too.

Part II

The Consensus Problem

Consistency models told us what guarantees a replicated system offers readers and writers. Consensus is the deeper machinery that lets a set of unreliable machines agree on a value in the first place. The professor flagged this upfront: it's harder than consistency, and the protocol we'll meet is "kind of hard to understand, and even harder to understand why it's correct." So let's build the motivation carefully before touching any mechanism.

Where Primary-Backup Replication Breaks

Recall the story so far in the course. We learned to use logical clocks to order events, then primary-backup replication to tolerate failures: one node is the primary handling requests, others are backups, and a view service decides who's primary. When the primary dies, the view service installs a new view promoting a backup, and the system limps onward.

But ask the sharp question: when can a primary-backup system not serve requests? The class enumerated the failure modes, and they're all instances of one disease:

1

A new backup hasn't finished syncing — it switched into the view but hasn't caught up on state, so it can't safely serve.

2

Both primary and backup fail at the same time. Nothing left to serve.

3

A network partition splits the nodes: the backup can reach the view service but the primary can't, or vice versa.

4

The view service itself goes down — and it's a single point of authority, so no one can change views.

The Root Cause: Hierarchy

Every one of those failures traces to the same flaw: there is a hierarchy among the nodes. One node (the view service) is the boss; one node (the primary) is privileged. Lose the boss, and the whole system stalls — even when a perfectly healthy majority of machines is still alive and connected. We want to dissolve that hierarchy.

The goal, stated precisely: make replication tolerant to network partitions so that operations don't block even when some machines are unavailable. Concretely, with 5 nodes, if 2 fail but a majority (3) stays alive, the system should keep responding — and keep being correct. No single privileged node whose death freezes everything.

The Group-Trip Analogy

Here's the mental picture the professor used. You and a group of friends need to decide where to travel. You coordinate over a group chat — but, realistically, not everyone is looking at the chat at any given moment. Some have it muted, some are offline, some are asleep. This is exactly a distributed system: at any instant, some nodes are crashed, partitioned, or simply not paying attention.

The thing you want to guarantee: once a destination is chosen, it cannot be overwritten. The friends who muted the chat can't show up one day before the trip and re-litigate the decision. If the protocol completes and picks Paris, Paris is final — even for the people who weren't watching.

Defining Consensus

Key Concept — Consensus

Consensus is a protocol run among a set of nodes that, despite failures, lets them all agree on a single value — and once that value is chosen, it cannot be changed. Each node starts with a proposal; the protocol drives them to a single decided value that every node returns.

To keep the mechanics clean, this lecture restricts attention to binary consensus: every node proposes either 0 or 1, and we want them all to end up returning the same one — all 0, or all 1 — despite failures along the way. Each circle in our diagrams is a node: a machine, an actor, a person.

Nodes enter with conflicting proposals (1, 0, 1). The consensus protocol funnels them to a single agreed value (here, all 0) — despite failures in the middle.

What Consensus Buys Us

Why care so much about this one primitive? Because consensus is a foundational building block — once you can reliably agree on a single value despite failures, an enormous amount of distributed infrastructure becomes possible:

You can agree on......which gives you
Who is the next primaryLeader election without a fragile single view-service
The next request to executeA replicated state machine — every replica runs the same operations in the same order
Who holds a lock right nowDistributed locking / mutual exclusion
Which nodes are alive and reachableReliable membership / failure detection
What financial transactions occurredBlockchain and cryptocurrencies (built directly on consensus)
Where This Lecture Is Heading

Our target use is the replicated state machine: a set of replicas continuously receiving client requests, using consensus to agree on the next operation to execute. Today we build the agreement block in isolation. Next lecture we plug it into a full request-processing system — and you'll build exactly this in Assignment 3.

Part III

A Side Quest: Safety & Liveness

Before we can say what we want from a consensus algorithm, we need vocabulary for talking about correctness properties in general. This short detour pays off immediately: the requirements of consensus turn out to be one of each kind.

Systems as Sets of Traces

Recall how we model a system formally. You sit at the boundary, issue operations, and observe what comes back. Each complete observation is an execution trace. The specification of a system is simply the set of all traces it can possibly produce.

A property P is also just a set of traces — the "good" ones. We say the system satisfies P when every trace the system can produce lies inside P's set. In set language: the system's traces are a subset of the property's traces.

Key Concept — Proving vs. Disproving a Property

To disprove that a system satisfies P, you only need one counterexample: a single trace the system can produce that's outside P. To prove it, you must show all traces are inside P — a far heavier burden.

Safety Properties — "Nothing Bad Happens"

Key Concept — Safety

A safety property says the system never reaches an undesirable state. Nothing bad ever happens.

How do you disprove a safety property? You exhibit a finite trace in which the bad thing has already occurred. The instant the system enters the forbidden state, you've got your counterexample — you don't need to watch any longer. Examples of safety in consensus: "two nodes never decide different values," "a decided value is never un-decided."

Liveness Properties — "Something Good Eventually Happens"

Key Concept — Liveness

A liveness property says something good eventually happens — the system makes progress, a request eventually gets answered, the protocol eventually terminates.

Disproving liveness is fundamentally harder, and the reason is beautiful. A finite trace can never disprove liveness — because someone can always object, "but what if you ran it ten steps longer? Maybe the good thing happens then." To genuinely violate liveness you need an infinite trace in which the good thing never happens. Typically you find this as a loop: the system enters a cycle of states and never escapes.

Deadlock Is Only a Special Case

A deadlock (the system frozen, unable to move) is one way to violate liveness — but it's not the only way. A livelock is worse and sneakier: the system keeps taking steps, keeps looking busy, but loops forever without ever making real progress. That's why "find an infinite trace" — not just "find a stuck state" — is the correct counterexample for liveness.

SafetyLiveness
Slogan"Nothing bad happens""Something good eventually happens"
CounterexampleA finite trace reaching a bad stateAn infinite trace where the good thing never occurs
DifficultyEasier to prove & checkVery hard — active research frontier
Why This Asymmetry Matters

Because safety is checkable with finite reasoning, most verification and formal-methods work on distributed systems targets safety. Liveness needs reasoning about infinite executions, fairness, and progress — it's genuinely state-of-the-art to prove rigorously. Keep this asymmetry in mind: throughout distributed systems, "is it safe?" is the tractable question and "is it live?" is the hard one.

Option C is too narrow: a deadlock is one liveness violation, but a livelock — looping forever while still taking steps — also violates liveness without any deadlocked state.
Part IV

What We Actually Want From Consensus

Now we can state the requirements precisely — and, instructively, by trying naive answers and watching them fail. Three properties together pin down "correct consensus."

Agreement, Termination, Validity

Key Concept — The Three Consensus Properties

Agreement (safety): All nodes that decide, decide the same value. No two nodes pick differently.

Termination (liveness): The protocol eventually finishes — every (correct) node eventually decides.

Validity (safety): The decided value must be one that was actually proposed by some node. You can't agree on a value nobody suggested.

Notice the side-quest paying off: termination is a liveness property ("good thing eventually happens — everyone decides"), while agreement and validity are safety properties ("bad thing never happens — disagreement, or a fabricated value"). And recall the lesson: the liveness piece (termination) will be the hard part to guarantee.

Why the Trivial Answers Fail

The professor invited the class to find the simplest protocol satisfying agreement and termination — simpler even than majority voting. The winning trick is almost a cheat:

The "Always Output 0" Cheat

Protocol: every node ignores all proposals and always decides 0. This trivially satisfies agreement (everyone picks 0, so they all match) and termination (no messages needed; everyone decides instantly). But it's useless — it decides 0 even when every single node proposed 1. The "decision" reflects nothing about what anyone wanted.

This is precisely why validity exists. Validity outlaws the cheat: the agreed value must be one of the proposed values. You cannot pick an arbitrary constant and call it a day. With all three properties — agreement, termination, validity — the trivial protocols are dead, and we genuinely have to do work to reach agreement.

The Big Idea

Consensus = Agreement + Termination + Validity. Drop any one and the problem becomes trivial or meaningless: without validity you cheat with a constant; without agreement you don't have consensus at all; without termination you can stall forever. All three together are what make consensus genuinely hard — and what Paxos is built to satisfy.

What We're Deliberately Ignoring

One might also want fairness (a principled, non-arbitrary reason for which value wins) or fast real-time performance. Classic consensus protocols like Paxos don't promise fairness — proposals are treated as arbitrary, like picking a paint color. For this lecture we set fairness and performance aside and focus purely on the three core properties.

Correct — C. Always-decide-0 actually satisfies agreement (all nodes match) and termination (everyone decides instantly), which is exactly why it's a tempting cheat. What it violates is validity: if every node proposed 1, the decided value 0 was never proposed by anyone. Validity is the property that forces the decision to mean something — it must be one of the actually-proposed values.
Part V

Paxos: The Protocol

Now the main event. Paxos, introduced by Leslie Lamport, is the foundational consensus protocol. Lamport framed it as a myth about legislators on the Greek island of Paxos passing laws while constantly travelling — a parable for nodes that keep going offline. Despite the whimsy, Paxos (and its descendants Raft, Zookeeper's Zab, and others) underpins essentially all consensus deployed in industry today. It's hard, but, as the professor put it, "actually pretty fun of an algorithm."

Three Roles

Paxos splits responsibility across three role types. In a real deployment every node plays all three roles at once, but to learn the protocol we pretend they're separate nodes:

RoleJob
ProposerProposes values and drives rounds of the protocol forward.
AcceptorThe voters. They promise and accept; a majority of them choosing a value is the decision.
LearnerPure observers. They watch for a majority acceptance and announce "consensus reached."
Two Foundational Assumptions

(1) Every process knows the number of acceptors in advance. This is essential — it's how anyone can compute what counts as a majority. (2) Acceptors can persist a little state to disk: the proposal number they've promised, and the value they've accepted. Paxos is, at heart, a careful majority vote.

The Big Idea

The single most important guarantee: once a majority of acceptors has accepted a value, no different value can ever be accepted in any later round. The decision is locked. No matter what messages fly afterward, no matter who crashes, the system cannot change its mind. Everything subtle in Paxos exists to preserve this one property.

One more crucial distinction before we trace the protocol — the difference between a number and a value:

Proposal number (N)Value (val)
A unique, ever-increasing tag used to decide who wins a round.The actual thing being agreed on (a request, a transaction, 0/1).
Internal bookkeeping; clients don't care about it.The only thing clients ultimately care about.

Phase 1 — Prepare

A proposer kicks off a round by choosing a proposal number N and broadcasting prepare(N) to all acceptors. Note: it sends the number only — not the value yet. Two requirements on N:

1

Higher than any N this proposer has used before. Each new attempt uses a strictly larger number.

2

Globally unique across proposers. Achieved by partitioning the number space: proposer 1 uses 1, 3, 5, 7, …; proposer 2 uses 2, 4, 6, …. No two proposers can ever pick the same N.

Why "Higher Than This Proposer's Previous N" — Not Globally Highest?

It's tempting to demand "higher than any N ever used by anyone." But computing the global maximum would require the proposers to agree on a number first — and agreeing is consensus itself. You'd need to run consensus to run consensus: hopeless circularity. So each proposer only tracks its own last number, and uniqueness is handled cheaply by the disjoint number pools. No global coordination required.

What an acceptor does on receiving prepare(N)

Each acceptor keeps a local variable NP = the highest proposal number it has promised so far. On receiving prepare(N):

if N <= NP:           # already promised something at least this high
    reply "prepare failed"
else:               # N is the highest we've seen
    NP = N            # remember this promise (persist it)
    reply "promise(N)"  # * there's an asterisk here — Part VI

A promise is a commitment: "I will not accept any proposal numbered lower than N from now on." Let's trace it. Proposer 1 sends prepare(5) to acceptors A1, A2, A3. Suppose A1 and A2 had previously promised 4:

1

A1 asks: "have I promised something >= 5?" Its NP = 4 < 5, so no. It sets NP = 5 and replies promise(5).

2

A2 does the same: 4 < 5, so it promises 5 too.

3

Two of three acceptors have promised — that's a majority. Milestone 1 reached.

Milestone 1 — Majority Promised

A majority of acceptors have promised on the proposal number (not a value yet!). They will now refuse any proposal with a smaller number. The proposer can safely move to Phase 2.

Phase 2 — Accept

Having collected promises from a majority, the proposer now reveals its value. It broadcasts accept(N, val) to the acceptors — same N as before, plus its proposed value.

What an acceptor does on receiving accept(N, val)

if N < NP:            # someone promised a higher number meanwhile
    reply "accept failed"
else:               # N >= NP: still the leader of this round
    NA = N            # remember accepted number
    VA = val          # remember accepted value (persist both)
    reply "accepted"
    broadcast "accepted(N, val)" to learners
A Subtle < vs <=

In prepare, the failure test is N <= NP. In accept, it's N < NP — strict. Why the difference? If N == NP at accept time, this is precisely the proposal we already promised, so we should accept it. Equivalently: accept whenever N >= NP. Getting these comparisons exactly right is one of the things you have to sit down and convince yourself of.

Continuing the trace: P1 sends accept(5, X) to A1 and A2. Neither has promised anything higher than 5, so:

1

A1 checks: promised higher than 5? No. It sets NA=5, VA=X and replies "accepted X."

2

A2 does the same: accepts X with number 5.

3

A majority of acceptors have now accepted the same value X. Milestone 2 — consensus is reached.

Milestone 2 — Consensus Reached

The moment a majority of acceptors accept the same value, the decision is final. There is no going back. Even if every subsequent message is dropped and every node crashes, the system can never decide a different value. This is the heart of Paxos. (Note: a mere promise from a majority is not consensus — only a majority acceptance is.)

A Majority Is Enough — You Don't Need Everyone

In the trace, A3 never received any message — maybe it was partitioned, maybe every message to it was dropped. It doesn't matter. A1 and A2 alone form a majority of three, so consensus still holds. In practice you'd broadcast to everyone, but the protocol's correctness only requires a majority to participate. This is exactly the partition-tolerance we set out to achieve in Part II.

Phase 3 — Learn

When an acceptor accepts a value, it also broadcasts accepted(N, val) to all the learners. A learner waits until it hears "accepted" for the same value from a majority of acceptors. At that point it knows consensus has been reached.

Milestone 3 — A Learner Knows

This is a local, per-learner milestone: each learner independently reaches it once it has heard a majority of matching acceptances. The learner is effectively the client of the consensus block — the entity that wanted the decision in the first place. The instant a learner hits Milestone 3, the system can safely act on the value: run the request, commit the transaction, install the new primary. In Assignment 3, this is exactly when you execute the agreed-upon operation.

One Acceptor Accepting ≠ Consensus

What if only A1 accepts and its message to A2 is lost? A1 still broadcasts "accepted" to the learners. Does a learner reach Milestone 3? No. A single acceptance proves nothing — that acceptor might be in the minority while the majority accepts a different value. A learner reaches Milestone 3 only on hearing from a majority. "I accepted X" is not "the system chose X."

Message flow. P1 sends prepare/accept to acceptors (solid orange); A1 and A2 broadcast "accepted" to all learners (dashed green). A3 is unreachable but a majority still decides. Each learner reaches Milestone 3 on hearing a majority.

Part VI

Competing Proposers — Filling In the Asterisk

We've been carrying two asterisks — one in the prepare logic, one in the accept logic — and now we discharge them. Everything so far assumed a single, well-behaved proposer. Real systems have multiple proposers racing, and incomplete rounds left behind by crashes. This part is what makes Paxos correct, and it's the part that makes Paxos famously hard.

The Scenario

Suppose Proposer 1 is mid-round: it sent prepare(5), got promises, and even got A1 and A2 to accept value X with number 5. Now Proposer 2 enters and sends prepare(6) to A2 and A3. What should A2 — which has already accepted 5, X — reply?

Our original prepare logic only checked "have I promised higher?" It said nothing about the case where the acceptor has already accepted a value. That's the gap the asterisk marks. Here is the full prepare logic:

# Acceptor handling prepare(N) — the complete version
if N <= NP:
    reply "prepare failed"
elif already_accepted:           # I have some NA, VA on record
    NP = N
    reply "promise(N, NA, VA)"      # promise — AND report what I accepted
else:
    NP = N
    reply "promise(N)"              # plain promise, no prior value
The Key Move — Promises Carry Accepted Values

If an acceptor has already accepted some (NA, VA), its promise to a higher number isn't empty. It says: "Yes, I promise you with your bigger number 6 — but be aware I already accepted value X at number 5." The promise piggybacks the previously-accepted value so the new proposer learns about it.

So A2 replies promise(6, 5, X): it promises P2's number 6 (legal, since 6 > 5), but discloses that it already accepted X at 5. A3, which accepted nothing, replies a plain promise(6).

The Proposer Must Adopt the Accepted Value

Now the second asterisk, in Phase 2. When a proposer collects its majority of promises, it checks: did any promise carry a previously-accepted value?

# Proposer, after receiving promises from a majority
if any promise contained a (NA, VA):
    val = VA of the promise with the highest NA   # adopt it!
else:
    val = my_own_proposed_value                  # free to use my value
send "accept(my_N, val)" to a majority
It Uses Its Own Number, But Someone Else's Value

This is the trickiest line in Paxos. P2 wanted to propose Y. But because A2's promise revealed that X was already accepted, P2 must abandon Y and propose X instead. So P2 sends accept(6, X) — its own number 6, but the previously-accepted value X. If multiple accepted values came back, it picks the one with the highest NA. Number = mine; value = the existing one.

The Big Idea — Why Adoption Preserves Safety

This adoption rule is exactly why "once a majority accepts X, no other value can ever be accepted." Any future proposer, to assemble a majority of promises, must overlap with at least one acceptor from the majority that accepted X (two majorities always intersect). That acceptor reports X, forcing the new proposer to re-propose X. The decided value propagates forward unchanged, round after round.

Why Continue At All? Why Not Just Stop?

A natural objection: if P2 discovers X was already accepted, why bother sending accept(6, X)? Why not give up — "someone already chose, forget it"? The answer reveals the deepest purpose of Paxos:

Completing an Abandoned Decision

Imagine the original round was incomplete: P1 got A1 to accept X, but the message to A2 was dropped, and then P1 crashed. Now exactly one acceptor holds X — that's not a majority, so there's no consensus, and the system is stuck. We need another proposer to come along, send prepare(6), learn that X was accepted, and drive X to a full majority. By re-proposing X, P2 forces A3 to also accept X — completing the half-finished decision rather than abandoning it.

So continuing isn't optional politeness — it's the mechanism that rescues the system from incomplete states. Without it, a single dropped message plus a crashed proposer would freeze consensus forever. Proposers therefore keep retrying with ever-higher numbers (P1 escalates to 7, P2 to 8, …) until the protocol completes. That relentless retrying is the liveness engine — and recall, liveness is the hard part.

Clients Only Care About the Value, Not the Number

A learner might see X accepted at number 5 by some acceptors and at number 6 by others. Does that conflict? No — the number is internal bookkeeping. The client only cares which value won: run request X or request Y? As long as a majority accepted X (whatever the numbers), consensus is on X. Mixed numbers, same value — totally fine.

Acceptors Never "Finish"

An acceptor never knows the global protocol is done. It only knows its own state ("I accepted X at 5"). It stays forever ready to answer new prepares and accepts — replying to prepares with its accepted value if it has one, or a plain promise if not. The protocol simply quiesces: once everyone knows the decision, no one bothers sending more messages, but nodes never formally stop running.

Deep-dive: walking through the full incomplete-round recovery

Start: P1 sends prepare(5), gets promises, sends accept(5, X). The message to A1 lands — A1 accepts X — but the message to A2 is dropped, and P1 crashes immediately after. State of the world: A1 has (5, X); A2 and A3 have nothing accepted. No majority ⇒ no consensus ⇒ system stuck.

# Recovery by Proposer 2
1. P2 sends prepare(6) to all acceptors
2. A1 replies promise(6, 5, X)   # discloses its accepted value
   A3 replies promise(6)         # nothing accepted
3. P2 has a majority of promises; one carried (5, X)
   => P2 adopts val = X (highest NA among reported)
4. P2 sends accept(6, X) to A1, A3
5. A1, A3 accept (6, X)  => majority accepts X
   => Milestone 2: consensus on X, finally complete

The half-finished decision X was driven to a full majority by a brand-new proposer that never wanted X in the first place. That is Paxos completing an incomplete consensus — exactly what "why continue?" was about.

Check Your Understanding
Proposer 2 wants to propose Y. During its prepare phase, an acceptor's promise reports that value X was previously accepted (at number 5). What value does P2 send in its accept message, and with what number?
A
accept(5, X) — it reuses both the old number and the old value.
B
accept(6, Y) — it ignores the old value and pushes its own.
C
accept(6, X) — its own higher number, but the previously-accepted value.
D
It stops entirely, since a value was already accepted.
Correct — C. P2 must adopt the previously-accepted value X (the one with the highest reported NA) but uses its own proposal number 6 — so it sends accept(6, X). This is the rule that guarantees an already-accepted value can never be displaced. Option B (pushing Y) would violate safety. Option D is wrong because not continuing is exactly the bug: a dropped message plus a crash could leave the decision stuck at one acceptor, and only by continuing does P2 drive X to a full majority and complete consensus.

Putting It All Together

This lecture moved from describing agreement to engineering it. We opened by sharpening causal consistency — building it from the two happens-before edges (program order and reads-from), proving that sequential consistency automatically implies it, and slotting it into the hierarchy linearizable ⇒ sequential ⇒ causal. That hierarchy is about what readers observe; the rest of the lecture asked the harder question underneath it: how do unreliable nodes decide anything at all?

The answer began with a problem. Primary-backup replication stalls whenever a privileged node — the primary or the view service — fails or is partitioned, because it has a hierarchy. Consensus dissolves that hierarchy: agree on one value, despite failures, as long as a majority survives, and never take it back. We saw it's the building block under leader election, distributed locks, membership, blockchains, and — our real target — the replicated state machine.

To say what we wanted, we took the safety/liveness detour, and it organized everything: consensus needs agreement (safety), validity (safety), and termination (liveness). Validity kills the "always output 0" cheat; termination is the liveness obligation that, true to form, turns out to be the hard one to guarantee.

Finally, Paxos delivered all three. Its three phases — prepare (lock in a number via a majority of promises), accept (drive a value to a majority), learn (observe the majority acceptance) — hit three milestones, of which only Milestone 2, majority acceptance, is irrevocable consensus. And the two asterisks we deferred carry the whole correctness argument: because promises report previously-accepted values, and because new proposers must adopt those values, an accepted value propagates forward unchanged across every future round. That is why a majority acceptance can never be undone — and why a stuck, half-finished decision can always be completed by a fresh proposer rather than abandoned.

The One Sentence to Remember

Paxos lets any majority of nodes agree on a single value that, once a majority has accepted it, can never be changed — achieved with no privileged node, by having every new proposer first learn and then re-propose whatever value might already have been accepted.
