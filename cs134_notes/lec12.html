Sharding, Consistent Hashing & the Chord Protocol

An interactive guide to Lecture 12 — how to split a system's data across many machines, find any item quickly, and survive nodes joining and leaving without moving everything.

Part I

Why We Need Sharding

Everything in this course up to now has been about replication: making several copies of a system so that if one machine fails, the others keep going. We built ordering (Lamport and vector clocks), fault tolerance (primary–backup, Paxos), and a whole menu of consistency models (linearizability, sequential, causal, eventual). All of that assumed one thing without questioning it: every replica holds the entire system and handles every request. This lecture is where we finally challenge that assumption.

Two Problems with Full Replication

If every replica is a complete copy of the system, two problems appear the moment the system gets big. Both come down to the same uncomfortable fact: your system can only be as capable as your weakest machine.

Problem 1 — It Doesn't Scale

If every replica must handle every operation, then adding replicas does nothing to share the workload — each one still does the full amount of work. Worse, the weakest replica is the bottleneck: the slowest machine in the group caps how fast the whole system can process requests. You can't grow throughput by adding machines.

Problem 2 — It Doesn't Fit

If every replica must store the complete state, then your smallest machine has to be big enough to hold all of it. A small, cheap machine literally cannot be used as a replica because the data won't fit. Your storage capacity is capped by your smallest box.

Notice these two problems are about load and capacity, not about crashes. Replication already solved crashes. These are new problems that replication, by its very design, makes worse.

Vertical vs. Horizontal Scaling

Suppose you run a service and you hit these limits. What are your options? There are two, and the lecture wants you to feel why one of them won.

Vertical ScalingHorizontal Scaling
Buy one bigger, more powerful machine.Buy more small, cheap machines.
Simple: no new protocol needed.Needs protocols to coordinate the machines.
Very expensive — a giant machine costs far more than the equivalent aggregate power in small boxes.Much cheaper per unit of capacity, and you keep adding cheap boxes as load grows.
Hits a ceiling: there's only so big a single machine can get.Scales as far as your protocols allow.

For a while, "just buy a bigger machine" was genuinely how companies coped. But a big machine is dramatically more expensive than a collection of small machines with the same total processing power and storage. So the industry moved to horizontal scaling — and that move is exactly what creates the need for the rest of this lecture. Small machines can't each hold everything, so we need a principled way to spread the data out. That technique is sharding.

What Sharding Is

Key Concept — Sharding

Sharding is a technique that partitions a system's state across multiple machines. Instead of every machine holding the whole dataset, each machine (a shard) holds only a slice of it. The goal: take a set of data items and split them across a set of machines.

Picture a single machine holding three keys, X, Y, and Z. Sharding might put X on machine 1, Y on machine 2, and Z on machine 3. Now ask: which of our two problems does this fix?

Sharding splits the keyspace. Each shard stores only its slice — so no single machine must hold the whole dataset.
Sharding Reliably Solves Storage, Only Sometimes Solves Load

The storage problem is solved: state is genuinely spread out, so no machine needs to hold everything. The load problem is only partly solved. If a request needs just X, only machine 1 is involved — great. But a request that asks for X, Y, and Z must touch all three shards. So we can't always guarantee that each machine processes fewer requests. Splitting the data does not automatically split the work.

Part II

Sharding Is Not Fault Tolerance

This is the single idea the professor said to take away above all others: if you remember only one thing from this lecture, remember that sharding and replication are different things, even though both add machines to your system.

Two Goals That Happen to Look Alike

Sharding adds machines. Replication adds machines. It's tempting to lump them together. But they serve opposite purposes, and confusing them leads to broken designs.

The Big Idea — Orthogonal Techniques

Replication exists for fault tolerance: keep extra copies so a crash doesn't lose data. Sharding exists for performance: split the data so the system can scale. They are orthogonal — you can do either, both, or neither. Sharding by itself gives you zero fault tolerance: if a shard fails, the state on it is lost forever, because there is no copy anywhere else.

Footgun — "We Sharded, So We're Safe"

A naive engineer reasons "we have lots of machines now, so we're protected against failure." Wrong. Sharding spreads data out but never duplicates it. A single shard failing destroys its slice of the data permanently. To get fault tolerance, you must also replicate — for example, keep replicas of each shard.

Hybrids: HDFS in Practice

Real systems usually want both performance and reliability, so they combine the techniques. The canonical example is the Hadoop Distributed File System (HDFS).

HDFS shards everything — it splits files into chunks spread across many machines — and it replicates each chunk, by default storing it on (say) two, three, or four machines. So a chunk X lives on two machines, Y on two machines, and so on. Sharding gives the scalability; the extra copies give the fault tolerance. When you design a storage system, you genuinely have to think about both questions separately: "How do I store data reliably?" and "How do I scale to handle more load?"

Sharding and replication composed: every key is on two distinct machines, so the data is both spread out (scalable) and duplicated (survives one failure).

The Design Problems Sharding Forces Us to Solve

The goal "partition data across machines" sounds simple, but the moment you try it, a swarm of hard questions appears. The class brainstormed these; here are the ones that matter, because the rest of the lecture is dedicated to solving them.

1

Finding items efficiently. Given a key, which machine has it? A client needs to know which shard to talk to, ideally without an expensive search.

2

Uniform partitions. We want keys spread evenly across machines — even with respect to storage (how many items each holds) and with respect to usage (how many requests each gets). These are different! An even split of items can still be a terrible split of load.

3

Communication. Do machines need to talk to each other to locate an item? Tightly-coupled data (asking for A almost always means also asking for B) may end up on different shards, forcing cross-shard communication.

4

Churn — a changing set of machines. This is genuinely new. With replication you fix the count ("3 replicas, replace failures"). With sharding for scale, you might add or remove many machines throughout the day as load fluctuates. How do we cope when the membership keeps shifting?

5

Load skew. Even a perfectly even split of items can be lopsided if some items are hammered far more than others — small numbers used more than big ones, words starting with "A" more than "Z".

What We'll Tackle Today vs. Later

This lecture solves item lookup, uniform partitioning, and churn (adding/removing machines). The cross-shard problems — what happens when one operation needs data from several shards at once, and keeping that data consistent — are deferred to a future lecture on distributed transactions. Keep that thread in mind: it's the natural sequel to today.

Part III

Data Lookup — Finding the Right Shard

Of all the design problems, the most fundamental is the first: given a key, which machine holds it? If you can't answer that cheaply, nothing else works. There are two broad families of answers — centralized and decentralized — and the trade-off between them drives the rest of the lecture.

Centralized Lookup — The HDFS Name Server

The most obvious approach: keep one special server, a lookup service, that knows the location of every item. When a new data server joins, it contacts the lookup service to learn what to fetch. When you create a new item, you ask the lookup service to assign it to some machine. To find anything, you ask the lookup service first.

Key Concept — The HDFS Name Server

HDFS uses exactly this design. A single name server holds the map of where each piece of data lives. Separate data servers hold the actual data and join or leave the system as needed. Every lookup goes through the name server.

Pro of Centralized LookupCon of Centralized Lookup
Simple. One straightforward protocol — ask the name server. This simplicity is exactly why Hadoop chose it.The name server is a bottleneck. Every lookup, every new item, every joining server must go through it. Nothing in the system works without it.
Easy to reason about and operate.Limited scalability. A single server must know about everything in the system, so it caps how large the system can grow.

Why Does a Bottleneck Design Work for HDFS Anyway?

This is the subtle, exam-worthy point. A central name server should be a scalability nightmare — yet HDFS ships it and it works fine. Why? The answer is about the ratio of lookup work to data work, and it comes from HDFS's coarse granularity.

It's All About Granularity

HDFS does not shard at the level of tiny items like X, Y, Z. It shards in huge chunks — think 1 GB at a time. You never ask "where is this little record?"; you ask "where is this 1 GB chunk?", then go read the whole chunk and do your work. Lookups are infrequent (one per giant chunk) and cheap (they only move location metadata — keys, not whole rows). So the name server handles very few, very light requests relative to the enormous data work the data servers do. The ratio makes the bottleneck negligible.

This works because of what HDFS is for: massive batch processing. Recall the genomics / log-processing applications from early in the course — you grok through an entire dataset, reading every item, rather than randomly fetching one key. In that workload, random access to individual items is unnecessary, so fetching huge chunks via infrequent lookups is exactly right. The centralized design and the batch-processing model fit each other perfectly.

Going Decentralized

But this course cares about systems with many small items and random access — like the key-value store in the assignment. There, a single name server fielding a lookup for every tiny key really would melt. So we need a decentralized approach.

Key Concept — Decentralized Lookup

In a decentralized scheme there is no single node that knows everything. Instead, each node can determine where to find an item — possibly by communicating with a few other nodes, but never by consulting one all-knowing directory. The rest of the lecture builds protocols of this kind.

Two Strawmen (So We Know What to Avoid)

Before the real solution, two deliberately bad ideas sharpen our intuition for what "good" must mean.

1

Randomly assign each item to a node. The distribution is even — but in a possibly-bad way (it ignores load and coupling). Worse, lookup is impossible: since you placed the item randomly, finding it later means searching every node. No structure means no efficient lookup.

2

Put everything on one machine. The opposite failure: lookup is trivially easy (it's all in one place), but there's no spread at all — we've thrown away the entire point of sharding.

The lesson: a good scheme needs both an even spread and an efficient, structured way to locate any item. That tension is what consistent hashing resolves.

Part IV

Hashing and Its Hidden Cost

We need a decentralized way to map keys to nodes that gives an even spread and a cheap lookup. The first honest attempt is range partitioning, which fails for a subtle reason — and fixing it leads us straight to hashing.

First Try: Range Partitioning

Assign keys to nodes by numeric range: keys 0–100 go to node 1, 100–200 to node 2, and so on. Lookup is easy (compute which range a key falls in — a "delegator function"). The item distribution can even be made even. So what's wrong?

The Problem — Skewed Request Distribution

Items may be evenly spread, but requests are not. Real datasets are skewed: small numbers get used more than big ones; words starting with "A" more than "Z". A linear split puts all the popular low keys on one node, hammering it while others idle. Range partitioning gives even storage but lopsided load — exactly the storage-vs-usage distinction from Part II.

What a Good Hash Function Does

The fix is to destroy the natural ordering of keys so that adjacent, similar keys land far apart. That's what a hash function is for.

Key Concept — Hash Function

A hash function takes values from an input space (keys) and maps them into an output space (think: a range of integers). A good one is:

• Deterministic — the same key always hashes to the same value (so we can find it again later).
• Apparently random / uniform — it spreads keys across the output space evenly, scrambling any structure in the input so popular, clustered keys are scattered.

Hashing won't perfectly fix load skew — some individual keys are still hotter than others — but by scattering keys uniformly it breaks up the systematic skew of range partitioning. Hash functions are the key building block of essentially every sharding scheme in the distributed-systems literature.

Modulo Hashing

The simplest way to turn a hash into a server assignment: take the hash of the key, then reduce it modulo the number of servers.

# with 4 servers, place key k on:
server = hash(k) % 4   # → 0, 1, 2, or 3

For example, to store student transcripts across four servers, you might hash on date of birth and take the result mod 4. The benefits are real: no communication is needed (any client computes the location itself) and it is extremely fast — you just run the hash and you're done.

The Hidden Cost: Rehashing on Membership Change

Modulo hashing has a fatal flaw that surfaces the instant the set of servers changes — and remember, with sharding-for-scale the membership changes constantly. The number of servers is baked into the formula. Change it, and almost every key's assignment changes.

Modulo hashing's disaster: going from 4 servers to 5 changes the divisor, so nearly every key maps to a different server and must be physically moved.

How Much Must Move? The Theoretical Floor

Modulo hashing moves everything — but that's a flaw of the scheme, not a law of nature. What's the minimum data movement physically required when you add a node and want to return to an even split?

The Lower Bound — keys ÷ nodes

If you have an even split and add one node, that new node must end up holding its fair share. The minimum number of keys you must move to it is (total keys) ÷ (number of nodes). You can't do better than this — the new node has to get some data to pull its weight. Modulo hashing moves vastly more than this floor; a good scheme should move only about this much.

So we have our target. We want a decentralized hashing scheme where adding or removing a node moves only ≈ keys/nodes of the data, not all of it. That scheme is consistent hashing.

Part V

Consistent Hashing — The Ring

A Note on the Name

"Consistent hashing" has nothing to do with the consistency models (linearizability, causal, etc.) from earlier lectures. It's an unfortunate name collision. Here "consistent" just means the mapping stays mostly stable when nodes come and go.

The Core Trick: Hash Onto a Circle

Modulo hashing failed because the assignment depended on the count of servers. Consistent hashing removes that dependency with one elegant move: present the hash output as a circle (a ring) rather than a flat number line, and place both keys and servers on it.

Key Concept — Consistent Hashing

Imagine the hash output space wrapped into a ring (say positions 0 through 16, then back to 0). Hash each data item to a point on the ring. Also hash each node's ID (e.g. its IP address) to a point on the same ring. Then: each node is responsible for the keys lying between its predecessor and itself — the arc of the ring leading up to that node.

The ring. Each key (blue) is owned by the first node (orange) clockwise from it — i.e. each node owns the arc from its predecessor up to itself.

Locating a Key

To find a key's home: hash the key to its point on the ring, then walk clockwise to the next node — that node owns it. If a node keeps a sorted list of the server positions, it can do this with a binary search instead of a linear scan. Simple and fully decentralized.

This Is Used Everywhere

Consistent hashing is not a toy. It powers production databases including MongoDB, Cassandra, and Memcached. Today we study it through Chord, a foundational 2001 paper that worked out the details and won a "Test of Time" award at SIGCOMM in 2011.

Why the Ring Wins: Minimal Movement on Join & Removal

Here's the payoff. Recall the floor from Part IV: adding a node should move only ≈ keys/nodes. The ring achieves it — and does something even nicer.

Adding a Server

Drop a new node S5 onto the ring between S1 and S2. S5 now owns the arc that previously belonged to S2 (the part now falling before S5). So S2 hands those keys — and only those — to S5. Every other server is completely oblivious; they keep serving requests as if nothing happened.

1

New node S5 lands on the ring between S1 and S2.

2

The keys in S5's new arc were previously owned by S2 (its successor). Only S2 needs to send keys.

3

On average this is the minimum number of keys (≈ keys/nodes), and only one server does any work. All other servers are untouched.

Removing a Server

Symmetric. Remove a node and its keys flow to its successor (the next node clockwise), which now owns the merged arc. Again only one server is involved, moving the minimum amount.

The Big Idea — Chord Minimizes Movement and Disruption

Consistent hashing's headline property: when nodes join or leave, on average it moves only the minimum number of keys (≈ keys/nodes), and only one server sends data while everyone else stays oblivious. Both join and removal have these same characteristics. This is precisely the churn problem from Part II, solved.

The Catch and the Fix: Virtual Nodes

The keyspace is huge and the number of servers N is small. When you hash only N server IDs onto a giant ring, you can get unlucky: by chance, two servers land close together (tiny arc, little data) while another spans a huge arc (overloaded). On average the split is even, but any particular random placement can be badly lopsided.

Why Not Just Space Servers Evenly?

Tempting, but it defeats the purpose. The whole benefit of hashing is that no coordination is needed — each server's position is computed independently from its ID. To place servers at hand-picked even positions, you'd need a decision process and broadcasts to agree on who sits where, reintroducing exactly the communication hashing let us avoid.

Key Concept — Virtual Nodes

Instead of one ring position per server, give each server multiple IDs — combine its IP with several random identifiers, hash each, and scatter them around the ring. These are virtual nodes. With N servers and V virtual nodes each, every virtual node owns on average 1/(V·N) of the space. The larger V, the more you'd have to get unlucky for the load to be uneven — so larger V means better load balancing.

A Free Bonus: Heterogeneity

V is a knob we can vary per server. Give a powerful machine a large V and a weak machine a small V, and the powerful one naturally claims more of the ring. So virtual nodes also let us account for heterogeneous hardware — different machines with different specs — for free, with no special mechanism.

One small cost: with virtual nodes, a joining server may need to pull keys from up to V different successors instead of just one. That sounds worse, but it's also a benefit — the data movement is now spread across many senders, so we can load-balance who sends what and avoid a single network bottleneck.

Part VI

Chord Lookup — Finger Tables in O(log n)

We can place keys and minimize movement. The last question is routing: a node receives a request for some key it doesn't own. How does it find the node that does? The answer evolves from a slow baseline to Chord's clever O(log n) structure.

The Baseline: Know Only Your Successor

The minimal routing state: each node knows only its successor (the next node clockwise) and nothing else. To look up a key: if you don't own it, forward the request to your successor; repeat. Eventually you reach the node whose arc contains the key.

Successor-only routing: correct but slow. Finding a key can take a linear number of hops in the number of nodes.
The Downside — Linear Hops

This works and needs almost no state, but it's slow: a lookup can take O(n) network messages — one per node — in the worst case. With many (virtual) nodes, that's far too many round trips. We care about message count, not local computation, so this is the metric that matters.

The opposite extreme — every node knows every other node — gives O(1) lookups but is infeasible: Chord targets systems with a huge number of nodes, so storing all of them everywhere is prohibitive. We want something in between.

The Fix: Finger Tables (Binary Search on the Ring)

The "in between" is to keep just log n pointers per node, chosen so that lookups become a binary search. The insight, which the class derived: from each node, keep a pointer to the node about half the ring away, another a quarter away, an eighth, a sixteenth, and so on — each entry covering half the distance of the previous.

Key Concept — Finger Table

Each node maintains a finger table of log n entries. Entry i points to the node responsible for the position 2i away around the ring (the successor of that position). By threading lookups through successive finger tables, every node sits at the root of a binary lookup tree over the whole ring — so a lookup halves the remaining distance at each step.

The Lookup Rule

At the current node, to find key K: scan your finger table for the highest entry F whose position is still less than K (the farthest finger that doesn't overshoot the target). Forward the request to F and let it continue. If no such finger exists — i.e. all your fingers are past K — then your own successor owns the key and you're done.

Why Scan the Whole Table?

We deliberately don't care about local computation cost — checking every finger-table entry is cheap. What we minimize is network messages. So we always take the biggest legal jump to cover the most ring per hop, even though finding it means scanning the whole table.

A Lookup, Step by Step

Here is the lecture's worked example: resolve key 26 starting from node 1, on a ring whose nodes include 1, 4, 9, 11, 14, 18, 20, 21, 28. Node 1's finger entries point to (among others) nodes 4, 9, 11, 14, and 18.

1

At node 1, target 26. Scan node 1's fingers for the largest that is still below 26. That's 18. Jump to node 18.

2

At node 18, target 26. Its largest finger below 26 is 20. Jump to node 20.

3

At node 20, target 26. Its largest finger below 26 is 21. Jump to node 21.

4

At node 21, target 26. No finger is still below 26, so the rule says: my successor owns it. Node 21's successor is 28, and 28 owns the arc containing 26. Done — found in a handful of hops, not a linear walk.

Expand: a second trace — resolve key 12 from node 28

Starting at node 28, target 12 (remember the ring wraps around past 0): take the biggest legal finger each step — 28 → 4 → 9 → 11, then the next step reaches 18, the successor that owns 12's arc. Notice that a node always knows the keys it owns itself, so if the lookup ever lands on the owning node it stops immediately rather than hopping further.

Why O(log n)?

Each finger entry covers a distance double the previous, so each hop at least halves the remaining gap to the target — exactly like binary search. Halving a space of size n takes log n steps, so lookups complete in O(log n) hops in the worst case (with some constant factors). We went from O(n) to O(log n) by storing just log n pointers.

The Hard Part: Joining the Ring

Lookups are settled. The genuinely tricky part is what happens when a node joins: every affected finger table must be fixed and data must move. A new node N joins knowing one arbitrary existing node N0.

1

Find your successor. Ask N0 to look up N as if it were a key — the ordinary finger-table lookup — to find N's successor on the ring.

2

Find your predecessor. Ask that successor "who is your predecessor?" — that's now N's predecessor. Update the successor's predecessor pointer to point at N.

3

Fill your own finger table. For each entry, compute its target position and ask the system to find that position's successor (again, just like a key lookup). Easy.

4

Fix everyone else's fingers. The hard part: which other nodes have a finger that should now point at N? We must not scan all of them.

The Key Optimization for Step 4

A node P gets N as its i-th finger entry only if P precedes N by at least 2i−1. In other words, a node too close to N could never have N as a far finger entry. So for finger entry i, we jump straight to the relevant region: find the predecessor of the position N − 2i−1, check whether that node's i-th entry should now be N (i.e. whether N beats its current entry), update if so, then walk to its predecessor and repeat for the same entry until we pass out of range. This examines only the few nodes that could possibly be affected — as few as i in the best case — instead of the whole ring.

Expand: worked join — node 6 joins a ring of {0, 1, 3}

Chord finger tables are written with a start column: for entry i at node n, start = n + 2i, and the entry's value is the successor of that start. (The "interval" column is just the range between consecutive starts — purely a visual aid, derivable from the starts, so it carries no real information.)

Node 6 joins. Its starts are 6+1=7, 6+2=0 (wrapping), 6+4=2. The successors are: succ(7)=0, succ(0)=0, succ(2)=3 — so node 6's finger table is [0, 0, 3].

Now existing nodes update. Node 0's last entry, whose start is 0+4=4, previously pointed past 6; its successor is now 6. Likewise an entry with start 5 now has successor 6, and start 4 → successor 6. The entry from 7 still points to 0 (unchanged). The protocol finds exactly these by jumping to the predecessor of 6 − 2i−1 for each entry and walking backward, rather than touching every node.

Separating Correctness from Performance

Fixing every finger table on every join means lots of communication — each join touches many nodes. That's expensive under heavy churn. Chord's escape is one of its most important and reusable ideas, and it returns next lecture in Dynamo.

The Big Idea — Correctness vs. Performance

Chord separates correctness from performance. For correctness, a joining node needs only one thing immediately: its successor. With the right successor, requests can always be forwarded to where they belong and the node knows which keys it owns. The finger tables and predecessor pointers are not needed for correctness — they're a performance optimization, repaired lazily by a periodic stabilization protocol.

Successor pointersFinger tables & predecessors
Required for correctness — the system can't run without them.Only for performance — speed up lookups from O(n) to O(log n).
Set up immediately when a node joins.Updated lazily/periodically via stabilization.
Tell you where to forward and which keys you own.Let you take big binary-search jumps.

So performance degrades gracefully: with intact fingers, lookups are O(log n); with stale fingers but correct successors, the worst case falls back to following successor pointers — "not fantastic, not terrible either." And there's a reassuring theorem: take a stable ring of n nodes (good fingers), then add n more nodes with correct successors but no fingers at all — lookups still take O(log n) with high probability. The intuition: you only follow slow successor hops until you reach any node that does have good fingers, and with half the nodes fingered, you reach one quickly and jump from there.

As long as every node knows its correct successor, requests can always be forwarded to the right owner; fingers only make those lookups faster (O(log n) instead of O(n)).
C
Finger tables are required for correctness, but successors are optional.
D
Without finger tables, lookups return wrong answers, just more slowly.
Correct — B. Correctness only needs the successor chain: with correct successors, a request can always be passed clockwise until it reaches the owning node, and each node knows which keys it owns. Finger tables merely let you skip ahead in binary-search fashion, turning O(n) hops into O(log n). So fingers can be stale or missing and lookups are still correct (just slower) — which is exactly why Chord repairs them lazily via stabilization. D is wrong: missing fingers slow lookups but never make them incorrect.

Putting It All Together

This lecture closed out the "foundations" arc of the course by attacking a problem replication never could. Every earlier technique — clocks, Paxos, the consistency-model menu — assumed every replica holds everything and handles everything. That assumption caps the system at the capability of its weakest machine. Sharding breaks it by partitioning state across machines, enabling cheap horizontal scaling.

The whole lecture is one chain of "this fails, so we fix it":

1

Sharding solves storage and (sometimes) load — but provides zero fault tolerance. Sharding and replication are orthogonal: performance vs. reliability. Real systems like HDFS compose both.

2

We need to find items. Centralized lookup (HDFS name server) is simple but a bottleneck — tolerable only because HDFS's coarse 1 GB granularity makes lookups rare. For small-item workloads we go decentralized.

3

Range partitioning skews load; modulo hashing spreads load but moves nearly all data whenever membership changes — fatal under churn.

4

Consistent hashing (the ring) makes assignment independent of node count: join/remove moves only ≈ keys/nodes, with just one server involved. Virtual nodes fix unlucky splits and handle heterogeneous hardware.

5

Chord's finger tables route any lookup in O(log n) hops via binary search on the ring, while separating correctness (successors) from performance (fingers) so the system stays correct and fast even under heavy churn.

If you carry one sentence out of today, make it the orthogonality principle: sharding improves performance, replication provides fault tolerance, and although both add machines, they are fundamentally different goals. And keep the correctness-vs-performance separation in your toolkit — it's a design pattern, not just a Chord detail.
