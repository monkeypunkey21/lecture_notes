Putting It All Together
CS134 · Distributed Systems

Logical Time: Clocks, Causality & Vector Clocks

How replicas in a distributed system agree on the order of events when no two computers can ever agree on what time it is.

Part I

Why Physical Clocks Fail Us

Last lecture left us with a hard problem. We were building a replicated state machine: several copies (replicas) of the same service, each running the same code, so that if one machine dies the others keep going. The catch is that a replicated state machine only stays consistent if every replica applies the same updates in the same order. If a deposit and an interest payment hit two copies of a bank account in different orders, the two copies end up holding different balances.

So the whole game reduces to one question: how do replicas agree on an order of events? The most obvious idea is to stamp every event with the time it happened and then sort by timestamp. This part is about why that obvious idea quietly breaks — which is exactly the motivation for everything that follows.

First: What Do We Even Use Timestamps For?

Before attacking clocks, it helps to be precise about what we want from them. In code, timestamps serve two different jobs:

Two Uses of Time

Points in time — "this event happened at a specific moment." Examples: the class starts at 4:00, the browser cache entry expires at 5:30, this log line was printed at 12:01.

Durations / intervals — "the gap between two events." Examples: the class lasts 50 minutes, the program ran for 3.2 seconds, retry after 200 ms.

Keep these two jobs in mind — the punchline of this part is that the clocks available to a real computer are each good at one job and bad at the other.

Time-of-Day Clocks

A time-of-day clock reports the real wall-clock time — "it is 3:00:00 PM." Because every machine drifts (a cheap crystal oscillator runs slightly fast or slow), these clocks are periodically corrected by NTP, the Network Time Protocol, which under the hood uses an algorithm like Cristian's algorithm from last lecture to ask a time server "what time is it?" and adjust.

Here is the footgun. Suppose a process prints the time, NTP quietly corrects the clock in the background, and the process prints the time again:

print(now())     # 3:00:00 PM
# ... NTP runs behind the scenes, decides the clock was 1s fast ...
print(now())     # 2:59:59 PM   ← time went BACKWARDS
Time Can Jump Backwards

Because NTP corrects the clock, a time-of-day clock can jump forward or backward. If you measured a duration by subtracting two readings, you could get a negative number. Cloudflare once suffered a serious outage from exactly this: a backwards leap-second adjustment made a duration go negative and crashed a core service. Time-of-day clocks are fine for "roughly what time is it," but unsafe for measuring durations or comparing fine-grained order.

Monotonic Clocks

The fix for measuring durations is a monotonic clock — a counter that only ever moves forward, never jumps back, and is never corrected by NTP. In Python this is time.monotonic(); nearly every language has an equivalent. Subtract two monotonic readings and you get a trustworthy elapsed time.

But a monotonic clock buys safety by giving up meaning. Each machine's monotonic counter started at some arbitrary moment (often boot) and ticks at its own rate. There is no shared zero, so a monotonic reading on machine A is simply not comparable to one on machine B. It answers "how long?" beautifully and "when, globally?" not at all.

ClockPoint in time?Duration?
Time-of-day (NTP)Yes — gives real wall timeNo — can jump backward
MonotonicNo — no shared zero across machinesYes — only moves forward

The Two-Process Problem

You might hope time-of-day clocks at least let two machines agree on the global state at a given instant. They don't. Consider processes P1 and P2, both starting with x = 0, whose clocks are slightly out of sync. P1 sets x = 1 and tells P2.

At the red cut (3:00 PM), P1 already has x=1 but P2 still reads x=0 — a globally inconsistent state even though both clocks "agree" it is 3 PM.

At the instant both clocks call "3:00 PM," P1 has x = 1 but P2 still has x = 0. A snapshot taken "at 3 PM" across the system is internally inconsistent. The wall clock simply does not give us a faithful global view of which event came first.

The Realization

We have been chasing the wrong target. We don't actually need to know what real time each event happened at. For a replicated state machine, the only thing we care about is that all replicas agree on the relative order of events. Real time is a distraction; what matters is order. That shift in goal is the seed of the whole lecture.

Part II

What We Actually Want From an "Order"

We just decided that order, not real time, is the goal. But "order" is vague. Before designing any mechanism, good engineering practice is to write down the requirements precisely. This part nails down exactly what a good ordering scheme must guarantee — the checklist we will hold every later design up against.

Formalizing "Order" as a Function

Let's give each replica a function that assigns a number to every event. Write Cᵢ(A) for the timestamp that replica i gives to event A. The whole point of these numbers is comparison: we want

Cᵢ(A) < Cᵢ(B)   means "A is ordered before B"

So the design question becomes concrete: what properties should this function Cᵢ have so that replicas stay consistent? In lecture the class brainstormed these, and three rose to the top.

The Three Properties

Property 1 — Agreement

All replicas must agree on the ordering of every pair of events. Formally, for any two replicas i and j, Cᵢ and Cⱼ rank every pair of events the same way. If replica 1 thinks A came before B, replica 2 must think so too.

Property 2 — Total Order

Every pair of events must be ordered — there are no ties. For any two distinct events A and B, either C(A) < C(B) or C(B) < C(A). A tie would leave a replica with no rule for which update to apply first, reopening the door to inconsistency.

Property 3 — Accuracy (the "happens-before" property)

If event A really did happen before event B, the clocks should reflect it: A happening before B should imply C(A) < C(B). We don't want the mechanism to invent an order that contradicts a real cause-and-effect that already occurred.

Why Agreement Is the Liberating One

Notice what Agreement quietly buys us. If every replica is guaranteed to agree, then it doesn't matter what real-world time an event "really" happened at, or whether replica j's internal numbering looks weird. As long as each replica's ordering is self-consistent and they all match, the system is correct. This is precisely why we are free to abandon real time in favor of invented "logical" numbers in the next part.

One question kept coming up in lecture: what if two events happen at exactly the same time on different machines? The professor's answer is worth internalizing: in a distributed system there is no such thing as "exactly the same time." You can always say one happened a hair before the other — and if you genuinely can't relate them, that's a real category we'll name later (concurrent). Hold that thought; it becomes the punchline of Part VI.

Part III

Happens-Before & Causality

Where We Are

Part II's Accuracy property said the clocks must respect when "A happened before B." But we never defined what "happened before" means when there is no global clock to consult. This part pins that down — it is the single most important definition in the lecture, introduced by Leslie Lamport.

Defining "Happens-Before" Without a Clock

Picture three processes drawn as horizontal timelines, with dots for events and arrows for messages between them. The deep idea is that we can decide some events happened before others using only the structure of the program, never consulting a clock. There are exactly three rules.

1

Same process, in order. If A and B happen on the same process and A comes first in the program, then A happened before B. (A single process executes its own events in a definite sequence.)

2

Send before receive. If event B is the sending of a message and event C is the receiving of that same message, then B happened before C. You cannot receive a message before it was sent.

3

Transitivity. If A happened before B and B happened before C, then A happened before C. This is the "pedantic" rule the class kept forgetting — but it is what lets us chain the first two rules across many processes.

Key Concept — The Happens-Before Relation (→)

We write A → B ("A happened before B") when it can be derived from the three rules above: same-process ordering, send-before-receive, or transitivity through other events. Crucially, → is defined entirely by program structure and message flow — no physical clock is ever consulted.

Spacetime Diagrams

These three-timeline pictures are called spacetime diagrams (or Lamport diagrams). They are the standard way to visualize happens-before in the literature, so you will see them constantly. Let's trace one.

Dots are events; arrows are messages. B sends to C; F sends to C. From the rules: A→B, C→D, F→G, B→C, F→C, and transitively A→C, A→D, B→D, F→D, and so on.

Now the subtle case. Does F happen before E? (Imagine E is an event on P2.) Thumbs down: there is no chain of same-process steps and message arrows linking F to E. F→G, sure — but nothing connects G or F to E. When no chain exists, the relation simply does not hold.

"Not Ordered" Is a Feature, Not a Gap

A natural worry: if some event on P3 has no message connecting it to the others, can we order it at all? The answer is we don't need to. If an event happened completely independently — touching none of the others, influenced by none of them — there is no reason to force an order on it. Happens-before deliberately only relates events that could have influenced each other. Independence is information, not a missing answer.

Why It's Really About Causality

Step back and ask what happens-before is secretly capturing. If A → B, then A could have caused B — but B could not have caused A. Contrapositively: if A genuinely caused B, then certainly A → B. The relation is a conservative over-approximation of "could be a cause."

Why care? Because violating causality produces baffling user-facing bugs. The lecture's example: Alice broadcasts "I lost my dog!" Bob, having seen it, broadcasts "I'm so sorry to hear that." A third user, Carol, due to a slow network, receives Bob's reply before Alice's original message — so Carol sees Bob apologizing for nothing.

The Big Idea — Preserve Causality

The fundamental job of logical clocks is to preserve causal order: if one event causally led to another, every observer should see them in that order. Get this wrong and users see effects before causes — replies before messages, comments before posts. Almost everything in this lecture exists to make causal anomalies impossible.

A Quick Detour: Network Models

The Alice/Bob/Carol delay raises a question — how slow can the network be? We model this with two assumptions about message delivery:

Synchronous modelAsynchronous model
There exists a bound n such that every message is delivered in less than n (say, 1 second).A message can take an arbitrary, unbounded amount of time to arrive.
Easier to program: set a timeout of n and you know a message is lost.Harder: a timeout never tells you whether a message is lost or just slow.
Convenient but unrealistic for the real internet.Pessimistic but realistic across regions and data centers.

Which should we build on? The synchronous model is easier for the developer, but it is a fantasy on a wide-area network. The synchronous model permits strictly fewer behaviors (messages always arrive within n); the asynchronous model permits all those behaviors plus arbitrary delays.

The Engineering Move

Always design for the harder (asynchronous) model. If your system is correct under arbitrary delays, then it is automatically correct on any synchronous network too — you get those guarantees "for free." Build robust to the worst case, and the easy case takes care of itself. (Inside a tightly controlled data center the synchronous assumption can be reasonable, but geo-distributed, we assume asynchronous. The rest of the course does too.)

Part IV

Lamport Clocks — Numbering Events

We have a relation, →, defined by program structure. But a relation isn't a number, and to sort events into a queue we need actual numbers. Lamport's logical clock is the algorithm that assigns each event an integer C(event) such that A → B implies C(A) < C(B) — turning the abstract relation into concrete, comparable timestamps. And it does so with a startlingly simple rule.

The Algorithm

Every process keeps one integer counter, all starting at 0. Three rules update it:

1

Local event: before recording any local event, increment your own counter by 1, and stamp the event with the new value.

2

Sending a message: attach your current counter value to the outgoing message.

3

Receiving a message: set your counter to max(local, message) + 1, then stamp the receive event with that value.

Rule 3 is the clever bit. Just incrementing on receive isn't enough — if the local clock is 2 and a message arrives carrying clock 3, simply doing 2+1 gives 3, which doesn't beat the sender. Taking the max first guarantees the receive event's timestamp exceeds both the local history and the sender's history, preserving send-before-receive. Let's trace it on a small diagram (P1: A, B; P3: F; with B sending to an event C on P1, and a message reaching C):

A=1, B=2 on P1; F=3 on P3. F sends a message carrying clock 3 to event C on P1. C = max(local 2, msg 3) + 1 = 4.
These Are Not Synchronized Clocks

It is tempting to read these integers as a synchronized global time. They are not. The numbers on different processes are not on a shared scale and say nothing about real time — F=3 and B=2 doesn't mean F happened "after" B in any wall-clock sense. They only encode the happens-before relation. This is the entire reason we abandoned physical clocks: these logical numbers cannot jump backward, because the protocol forbids it.

The Catch: Only a Partial Order

Lamport clocks faithfully deliver Accuracy (A → B ⟹ C(A) < C(B)) and Agreement. But check the three properties again — we are missing Total Order. Two unrelated events on different processes can receive the same integer.

In the running example, A and D might both be 1, B and E both 2 — concurrent events sharing a timestamp. Worse, the converse of accuracy fails:

The One-Way Street

A → B implies C(A) < C(B). But C(A) < C(B) does not imply A → B! A smaller Lamport timestamp could mean A happened before B — or it could mean they are concurrent and the numbers just landed that way. Lamport clocks keep one direction of the implication and lose the other. Remember this gap: it is exactly what vector clocks (Part VI) will repair.

Forcing a Total Order: Break Ties with Process ID

We need some rule to order two events that share a Lamport number, and every replica must break the tie the same way (or Agreement breaks). The trick relies on one fact:

Key Insight — Same Process, Never Equal

Two events on the same process can never have the same Lamport timestamp, because each local event increments the counter. Therefore the (timestamp, process-id) pair is globally unique, and we can break ties by process id.

So every event is timestamped with the pair (clock, pid), and we compare pairs lexicographically:

# (clock, pid) of event A is "less than" that of B when:
clockA < clockB
  or (clockA == clockB and pidA < pidB)

The process ids here are not the operating-system PIDs — they are agreed-upon unique indices the system assigns to each participant up front (every node knows everyone's index). With this lexicographic rule, any two distinct events compare strictly, all replicas compare them identically, and we finally have all three properties: Agreement, Total Order, and Accuracy.

Part V

Applying Updates Safely

Closing the Loop

Parts I–IV built a total order on events. But ordering alone isn't the goal — the goal (from the very start) was replicas applying updates in that order without diverging. The trap: a replica can't apply the lowest-timestamped update it has seen so far, because a lower-numbered update might still be in flight. This part is how we know it is finally safe to act.

The Replicated Bank, Concretely

Two replicas of a bank account. A client in Los Angeles deposits \$100; an admin in New York adds 1% interest. Each request goes first to its local replica, gets a Lamport timestamp, and must eventually be applied in the same order at both replicas. Let's trace the timestamps. R1 starts at clock 2 (id 1), R2 starts at clock 7 (id 2):

1

LA deposit arrives at R1 → timestamped 3.1 (clock 3, pid 1). NYC interest arrives at R2 → timestamped 8.2. Nothing is applied yet — we have only timestamped the requests.

2

Each replica forwards its event to the other. R2 receives 3.1 and stamps the receive 9.2 = max(8,3)+1. R1 receives 8.2 and stamps it 9.1 = max(3,8)+1.

3

Now R1 has heard from R2 an event timestamped 8.2. So R1 knows R2 has no un-sent update earlier than 8.2 — and therefore none earlier than 3.1. R1 can safely apply its own 3.1 deposit: nothing smaller can ever arrive.

4

By the same logic R1 can then apply 8.2 (nothing lies between 3.1 and 8.2). And R2, having heard 3.1 from R1, knows R1 will never send anything below 3.1, so R2 may apply 3.1 too. Both replicas apply 3.1 then 8.2 — same order. Consistent.

The Hold-Back Queue

The mechanism behind that trace is a per-replica hold-back queue. Every arriving update is placed in the queue and sorted by its (clock, pid) timestamp. You may not apply an update the instant it arrives — you must wait.

The Delivery Rule

You may apply (deliver) the update at the head of the queue only once you have received a message with a higher timestamp from every other replica. At that point you are certain no replica can still send anything smaller than the head — so the head is final, and applying it can never be contradicted later.

The Assumption That Makes It Work: FIFO Channels

Why is "I heard 8.2 from R2" enough to rule out a sneaky earlier 5.2 still in flight? Because we assume each pair of replicas communicates over a FIFO (first-in-first-out) channel.

Key Concept — FIFO Channels

On a FIFO channel between two participants, if you receive a message from a sender, you have already received everything that sender sent before it. Messages from one source arrive in send order. This is guaranteed in practice by protocols like TCP. So receiving 8.2 from R2 proves R2 has nothing earlier left to deliver to you.

FIFO Is Per-Channel, Not Global

FIFO only orders messages along a single sender→receiver channel. It does not order messages across different senders — which is exactly the Alice/Bob/Carol anomaly: Alice→Carol and Bob→Carol are separate channels, so Carol can see Bob's reply before Alice's message. FIFO per channel is what we assume; cross-channel causal ordering is a harder, separate problem.

Pulling it together, this scheme rests on three standing assumptions: (1) every replica knows all the other replicas, (2) channels between each pair are FIFO, and (3) every replica runs the Lamport-clock protocol. Because everyone runs the protocol, R1 knows R2's clock can only increase past 8 — there is no skew to fear, since these are logical, not physical, clocks.

Heartbeats: Don't Wait Forever

There is a cost. A replica can't apply the head of its queue until it hears a higher timestamp from every other replica — so a single quiet replica stalls everyone. The whole system moves at the speed of its slowest, most silent node.

The Fix — Periodic Heartbeats

Have each replica periodically send a tiny "tick" message carrying just its current clock, even when it has no real update. In the bank example, R1 and R2 later send heartbeats bumping their clocks to 10.1 and 10.2; now each has heard a higher number from the other and can apply any held-back update. The cost is modest network overhead — even with 20 nodes, occasional empty clock messages are cheap — and the course will return to ways of reducing this coordination later.

The Honest Caveat: We Reorder Reality

The LA client may have clicked "deposit" before the NYC admin clicked "interest" in real wall-clock time — yet Lamport order might apply them the other way. That's allowed: the two clients never communicate, so neither happens-before the other, and users only learn the result when their transaction is reported done (like a credit-card "transaction succeeded"), not when they click. The one thing that would break this is a bypass channel — if the admin phoned the client to coordinate, real causality would escape the system's view. We assume no such side channel exists.

Part VI

Vector Clocks — Recovering Concurrency

Lamport clocks got us a working replicated state machine, but they have a real blind spot we flagged in Part IV: C(A) < C(B) does not tell you whether A actually happened before B or whether they were merely concurrent. The two cases look identical. Often we genuinely need to tell them apart — and that need motivates vector clocks.

Why We'd Want Concurrency Back

Total order is sometimes overkill. Some operations commute — applying them in either order gives the same result — so forcing an order on them only creates needless waiting:

  • Updates to two different bank accounts never need ordering.
  • Two deposits to the same account commute (a+b = b+a), so their order is irrelevant.
  • Two Facebook posts from different users needn't be ordered — but a comment that replies to a post must come after it.

What we really want to order is only the causally related events. That is causal ordering, and it usually suffices. To get it we must be able to detect concurrency precisely — and detect it we cannot, with a single Lamport integer. Beyond replication, this precision is invaluable for debugging, testing, and monitoring distributed executions, where you need to know the exact order — or genuine lack of order — between events.

The Vector Idea & Algorithm

Key Concept — Vector Clocks

Instead of one integer, label each event with a vector of N integers, one per process in the system. Entry V[i] is the count of events on process i that causally precede this event. The vector is a compact record of "how much of every process's history this event is aware of."

All vectors start at [0, 0, 0] (for three processes). Two update rules:

1

Local event on process j: increment only your own entry, V[j] += 1. (This is why the first event on P1 is [1,0,0], not [0,0,0] — increment first, then stamp, so no two events on a process share a vector.)

2

Receiving a vector D on process j: take the element-wise max of your vector and D (a merge, absorbing everything the sender knew), then increment your own entry V[j] += 1.

Let's trace. P1 does two local events; then sends to P3; P3 also has its own local event:

P1: 100, 200. P3's own event: 001 (it has heard from nobody). On receiving 200, P3 merges: max(001, 200)=201, then +1 on its own entry → 202.

Notice P3's independent event is 001, not 100 — P3 has received no message, so it knows nothing about P1's or P2's progress. The merge step is the whole point: a vector ends up holding the union of the freshest information from every process the event causally depends on. No single process knows who has the freshest data overall; the element-wise max ensures the receiver ends up with the best of both.

Comparing Vectors & Detecting Concurrency

Now the payoff. We compare two vectors element by element:

RelationDefinitionMeaning
VA = VBevery entry equalsame event (uninteresting)
VA < VBevery entry ≤, and at least one strictly <A → B (A happened before B)
VA ∥ VBneither ≤ the other (one bigger somewhere, smaller elsewhere)concurrent

That third row is what Lamport clocks could never express. Take VA = [1,1,0,0] and VB = [0,0,1,1]: A is bigger in the first two entries, B is bigger in the last two. Neither is ≤ the other, so there is no message chain linking them — they are genuinely concurrent, A ∥ B.

The Big Idea — Vector Clocks Capture the Exact Partial Order

Lamport clocks kept only one direction: A → B ⟹ C(A) < C(B), losing the converse. Vector clocks restore it: VA < VB if and only if A → B, and incomparable vectors mean exactly "concurrent." The vector encodes the true happens-before partial order with no loss of precision — recovering the very information Lamport clocks threw away.

What "Concurrent" Really Means Here

Concurrent does not mean "happened at the same wall-clock instant." It means the two events are incomparable under happens-before — no chain of messages connects them, so from the system's standpoint neither caused the other. They could have occurred seconds apart in real time; what matters is that no causal relationship exists to order them.

Why a vector entry might already exceed the incoming one

When merging, can your local vector already hold a bigger value than the message for some entry? Yes — for entries other than the sender's own. Suppose process 3 is at [0,0,5] (lots of local events) and process 2 sends you [0,1,0]. After merge you'd take max per entry. The guarantee is narrow: when you hear directly from process i, you can't have a higher value for i's own entry than i itself reports. But for everyone else, you may already know more — perhaps you've heard from them a thousand times while the sender hasn't heard from them at all. That's exactly why we take the element-wise max: nobody knows who holds the freshest information for which process, so we union the best of both.

Putting It All Together

This lecture is one long argument, and every part is a move in it. We started (Part I) with a need from last lecture — replicas must apply updates in the same order — and discovered that physical clocks can't deliver it: time-of-day clocks leap backward, monotonic clocks can't be compared across machines, and a wall-clock "snapshot" of the system is internally inconsistent.

That failure forced a reframing (Part II): we don't need real time, we need an order with three properties — Agreement, Total Order, and Accuracy. Accuracy referenced "happens-before," so we had to define it (Part III) from program structure alone — same-process order, send-before-receive, transitivity — and saw it is really a conservative model of causality, the thing whose violation makes users see replies before messages.

Part IV turned that relation into numbers with Lamport clocks (max+1 on receive), then patched in a total order by breaking ties with process ids. Part V cashed all of this in: a hold-back queue plus FIFO channels lets a replica know precisely when an update is final and safe to apply — at the cost of waiting on the slowest node, softened by heartbeats. Finally Part VI repaired Lamport's one blind spot — the inability to tell "before" from "concurrent" — with vector clocks, which encode the exact causal partial order.

The One Sentence to Remember

In a distributed system you cannot agree on what time it is, but you can agree on what happened before what — and logical clocks (Lamport for a total order, vectors for the exact causal order) turn that agreement into concrete, comparable timestamps that keep replicas consistent.
