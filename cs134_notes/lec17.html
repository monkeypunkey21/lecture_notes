CS134 · Distributed Systems

Delos & the Virtual Log

How Facebook built a production database that can swap out its consensus protocol on the fly — by virtualizing the log the way an OS virtualizes memory.

Part I

The Migration Problem

Before we meet Delos, we need to understand the pain it was invented to cure. The whole system is an answer to one stubborn engineering question, so let's start there.

Why Would Anyone Swap a Consensus Protocol?

A consensus protocol is the machinery a distributed system uses to get a set of machines to agree on a single ordered sequence of decisions — for example, "the order in which writes happened" — even when some machines crash or messages get delayed. You have already met several in this course: Paxos, ZooKeeper's ZAB, and Raft. There are many more (Corfu, and others), and the field is still very active research.

Why does anyone care about improving a protocol that already works? Because of scale. At a hyperscaler — Google, Meta, Amazon — a one-to-three-percent reduction in the CPU time or network traffic a protocol consumes translates directly into buying, powering, and maintaining measurably fewer machines. At billions of operations per day, single-digit percentage wins are worth real money. So companies genuinely want to experiment with new consensus protocols.

Key Concept — Consensus Protocol

A protocol that lets a group of machines agree on a totally-ordered log of decisions despite crashes and message delays. Paxos tolerates a minority of failures; primary-backup tolerates one. The cost of a protocol — its messages and CPU per decision — is exactly what hyperscalers want to shave down.

Why Is Migrating a Protocol So Painful?

Designing a better protocol on paper is the easy part. The hard part is deploying it into a system that is already running and serving live traffic. Walk through what it actually takes:

1

Implement it. Writing a correct consensus implementation is a heavy, error-prone effort — far harder than the clean pseudocode in the paper.

2

Test it — including in production. A proof of correctness is not enough. As Dijkstra warned, you can prove a program correct and still have an incorrect implementation. You eventually have to run the new protocol against real traffic.

3

Stop or restart the system to cut over. Most cutovers require a manual pause — but you cannot just turn off Instagram for a few hours while you migrate consensus.

4

Consolidate state across the boundary. The data written under the old protocol and the data written under the new one must remain one coherent history, and you must keep serving reads of both.

5

Be ready to roll back. If the new protocol reveals a bug in production, you need a clean way to retreat to the old one.

The Core Obstacle

Every one of these steps is hard, and together they make experimenting with a new protocol so risky that real systems rarely do it. A proof of correctness does not save you: "do not trust software I have proven correct but not tested." The result is that good protocol ideas never make it into production.

The Goals of Delos

Delos (named, like Spanner before it, after a Greek island) is the system in this lecture. It comes from a 2020 OSDI paper by engineers who were then at Facebook. Its stated goals were strikingly pragmatic:

  • Build a new fault-tolerant database that can reach production in 6–9 months.
  • Then allow incremental performance improvements over time, without disruptive rewrites.
  • Fill roughly the same role as Chubby or ZooKeeper — a highly reliable store for coordination and metadata.

To get there, Delos needed one capability above all: easy, seamless migration across consensus protocols. That single capability is what unlocks experimentation, optimization, and the ability to ship fast first and improve later. The rest of this lecture is the story of the one abstraction that makes it possible.

The Big Idea

Delos's bet: if you can make swapping the consensus protocol as routine and invisible as any other operation, then all the risk drains out of experimentation. Ship a simple version now, improve the engine underneath later — without ever stopping the system.

Part II

The Virtual Log Abstraction

If swapping the protocol is the goal, the question becomes: how do you build a database that doesn't even notice when the protocol underneath it changes? Delos's answer is its foundational abstraction — the virtual log — and it is best understood by analogy to something you already know from operating systems.

Builds On — Logs as the Spine of Replication

Throughout this course, replicated systems have been built on a shared, ordered log: every node applies the same operations in the same order and ends up in the same state. Delos keeps that idea intact — applications still see one ordered log — but virtualizes what sits underneath it.

Loglets and the Virtual Log

Recall virtual memory: a program sees one flat, continuous address space, while underneath, the OS maps different ranges to different physical pages — some in RAM, some swapped to disk — and the program neither knows nor cares. Delos does the same trick for the log.

The application sees a single, continuous, ever-growing log. Underneath, that log is stitched together from a sequence of loglets. A loglet is a chunk of the log with its own concrete implementation. One stretch might be backed by ZooKeeper, the next by Delos's own native protocol, an older stretch by cheap cold storage. The virtual log layer can transparently switch from one loglet implementation to the next, as many times as it likes.

Key Concept — Virtual Log & Loglet

A loglet is a contiguous segment of the log with a single concrete implementation (ZooKeeper, native, backup-to-cold-storage, …). The virtual log stitches loglets into one seamless log. The application just appends and reads; it never sees the seams or the implementation changes underneath.

The application sees one log from 0 to ∞; underneath, ranges map to different loglet implementations. The newest range is the active loglet that takes appends.

The Log-Space API

Think of the virtual log as a position space from 0 to ∞, where each range of positions is mapped to a loglet. On top of that mapping sits a small, ordinary-looking log API:

append(record)        # add a record at the end of the log
checkTail()           # return (final position, sealed?) — where does the log end?
readNext(from, to)     # read a range of positions

The routing is mechanical: append and checkTail always go to the latest (active) loglet, since that is where new records land and the tail lives. readNext dispatches to whichever loglets cover the requested range — a read of positions 40–160 in the figure above touches all three loglets.

So far this looks exactly like a plain log. A student in lecture asked precisely this: how is it any different from just splitting a normal log into pieces? The answer is the two extra operations that come next — and the fact that each loglet can run a cheaper protocol than the whole log appears to offer.

The Two Special Operations: Seal & Reconfigure

On top of the ordinary log API, the virtual log adds the two operations that make migration possible:

Key Concept — Seal & Reconfigure

seal finalizes a loglet: a sealed loglet rejects all future appends and becomes read-only. Before you can add a new loglet, every earlier (non-last) loglet must be sealed. reconfigure installs a new loglet — it proposes a new mapping from log ranges to loglets. Reconfigure is a strongly consistent operation that can succeed at most once for a given configuration.

Builds On — Sealing in Azure Storage

You saw seal before in the Azure Storage lecture: an extent is sealed before a new one is started, so its length is frozen and unambiguous. Delos reuses exactly that idea. Sealing a loglet is what lets the system pin down "this segment is finished and immutable" before moving on.

Switching Loglets Safely

Now we can answer the central question: what does a client do to switch loglets — say, from the ZooKeeper loglet to the native one? Naively it is just "seal, then reconfigure," but two subtle gaps make it a four-step dance.

1

Seal the current loglet to terminate it. It now rejects new appends and is read-only.

2

CheckTail to find where the loglet actually stopped. This is not redundant: a loglet can hold more than you last thought it did, so you must ask where it truly ended. CheckTail returns both the final position and the sealed bit.

3

Reconfigure, proposing the new loglet and the new range-to-loglet mapping. But if two clients reconfigure concurrently, only one wins — and you don't yet know which.

4

CheckTail again to learn which new loglet actually took effect, before you resume appending. Both seal and reconfigure are operations where "I attempted it" does not tell you "it finished, and here is the result" — so you must re-check.

Why "Attempt" ≠ "Knowledge"

The recurring theme in both gaps is the same: seal and reconfigure can be racing against other clients, so issuing the operation does not tell you its outcome or final position. The cure each time is a checkTail — go find out the ground truth before you act on it. This "do, then re-read the truth" pattern returns again and again at the loglet level.

In the common case, none of this happens: clients just append, checkTail, and readNext like any log. Only when a failure is detected or a reconfiguration is desired do they pay for the checkTail → reconfigure → checkTail detour and then carry on. The result is a clean log abstraction with strong guarantees that can still be re-engineered underneath at runtime.

Part III

Building a Database on the Virtual Log

We now have a log we can append to, read, and re-engineer underneath. But a log is not a database. This part answers the first of Delos's two big construction questions: how do you build a real, strictly-serializable database on top of the virtual log? (Part IV and V answer the second: how do you build the virtual log itself.)

Why This Matters — An Interview-Shaped Question

"You're given a log and an in-memory database library — design a strictly-serializable database" is exactly the kind of architecture question asked in distributed-systems interviews. In practice you rarely build from scratch; you compose known components to hit your requirements. Treat this part as a worked example of that skill.

LogDB and State-Machine Replication

The components are: a set of replica nodes, an in-memory database library called LogDB running on each replica, and the fault-tolerant virtual log underneath. Each replica keeps the entire database state in memory.

The replicas stay consistent through state-machine replication: every replica applies the same operations, taken from the shared log, in the same order, so they all reach the same state. The virtual log is the single source of truth for that order. The goal is to support two kinds of transactions: strictly-serializable read-write transactions, and strictly-serializable read-only transactions — and, crucially, they get separate flows, because read-only is simpler and shouldn't pay the read-write cost.

Key Concept — State-Machine Replication

Treat the database as a deterministic state machine. If every replica starts from the same state and applies the same ordered log of commands, every replica ends in the same state. Consistency reduces to agreeing on the log order — which the virtual log already provides.

Why a Log AND an In-Memory Database?

A natural question: if the log already records every operation, why keep a separate live database? Because answering "what are the values of these 100 keys?" by replaying millions of log operations would be hopeless. The live in-memory view exists so reads are fast; the log exists to record full history and to bring a fresh replica up to date (replay the log, or copy a peer's state then tail the log). Old log entries can even be pushed to a cheap backup loglet in cold storage while the live view stays hot.

Read-Write Transactions

Here is the flow for a read-write transaction. Notice that the replica writes the whole transaction into the log first, and only then executes anything.

1

Append the entire transaction to the virtual log — without running it yet. You are simply recording your intent at some position in the global order.

2

CheckTail to synchronize: find out where your transaction actually landed and everything that precedes it.

3

Catch up by replaying. Execute, in order, every transaction from where this replica left off up to your transaction. You must do this — otherwise your in-memory state doesn't reflect the real system state, and you can't correctly run your own transaction.

4

Execute your transaction and return. Once you reach your transaction in the log and run it against now-current state, you return the result to the user.

Why You Can't Just "Append and Done"

Suppose the last time you checked, the log ended at position P. You append your transaction — but you don't know whether three other transactions slipped in between P and yours. So you cannot execute against your stale snapshot. You must append, then re-read everything from P up to your own position, and apply it all in order. This is the "attempt, then re-read the truth" pattern again, now at the database layer.

Why not just take a lock and run transactions one at a time?

A student proposed: why not grab a lock, run the transaction, then release? The problem is what a lock costs in a distributed setting. A lock requires extra communication among replicas to agree on who holds it, and while one replica holds it, everyone else is blocked — their requests stall.

The log-based design avoids this entirely. Every replica can append its request to the log at full speed, concurrently, with nothing blocking it. The only "extra work" a replica does is replay the entries before its transaction — but that work isn't waste: it is exactly how the replica catches its in-memory state up to the rest of the system. So the apparent cost of replaying is doing double duty as necessary synchronization.

Read-Only Transactions

Read-only transactions get a lighter, separate flow. The trick is to pick a linearization point — a single position in the log that the read is logically "as of."

1

CheckTail to pick a linearization point. The instant the read arrives, check the tail. This guarantees the read will observe everything that committed before it.

2

Synchronize the in-memory state up to that exact point — the same catch-up as in the read-write flow.

3

Run the read-only transaction against that synchronized snapshot and return.

Why CheckTail Prevents Stale Reads

Imagine someone writes a value and their write completes. If a later reader skipped checkTail and answered from a stale snapshot, it could return the old value — violating strict serializability. By checking the tail first, the reader refuses to answer until it has seen everything that already committed. CheckTail is the mechanism that turns "fast local read" into "linearizable read."

Optimization — Batch Reads Behind One Linearization Point

If several read-only transactions pile up at one replica, you don't need a separate checkTail for each. Pick a single linearization point, synchronize once, and answer them all together. It's still correct — they all observe the same committed prefix — and it saves a round-trip per read.

Part IV

The VirtualLog Layer: MetaStore & Reconfigure

We've built a database on top of the virtual log. Now we descend below it. Building the virtual log breaks into two sub-questions: how is the virtual log layer itself implemented, and how is an individual loglet implemented? This part answers the first; Parts V–VI answer the second.

Splitting the Control Plane from the Data Plane

The key architectural move is a clean split:

Control Plane — the VirtualLogData Plane — the Loglet
Owns reconfigure — the fundamental control operation.Owns append, checkTail, readNext — the data operations.
Decides the loglet mapping (which range → which loglet).Also owns seal (a control op, but it lives in the loglet).
Rare, must be bulletproof.Frequent, must be fast.
The Big Idea

Push all the expensive coordination into a tiny control plane that runs rarely, and make the high-traffic data plane as light as possible. This is the move that lets each loglet skip full consensus — the hard agreement work happens once, at reconfiguration time, not on every append.

The MetaStore

The VirtualLog stores exactly one thing: the current configuration — the mapping from log ranges to loglets — and it must implement reconfigure on it. So let's reason about the requirements before naming the mechanism.

Does it need to be fast?

No — not very. Reconfiguration is rare; it is not on the critical path of appends. (In practice, over months, Delos saw maybe hundreds to a few thousand reconfigurations, against billions of transactions per day.) It should be reasonably quick, but raw speed is not the priority.

Does it need to be fault-tolerant and highly available?

Yes — and this is the crucial requirement. Reconfiguration is exactly what you do when things go wrong: a loglet has failed and you need to switch away from it. If reconfiguration depended on a single server being up (like primary-backup), a catastrophe could leave you unable to reconfigure at all. So it must be highly available — able to make progress even when many replicas are down.

Key Concept — The MetaStore

The MetaStore is a highly-available store holding a single pair: (currentChain, version). currentChain is the range-to-loglet mapping; version is just a counter (1, 2, 3, …). It supports two operations: get() returns the chain and version, and conditionalWrite(newChain, V) writes the new chain only if the current version is still V.

That conditionalWrite is the whole trick for handling concurrent reconfigurations. Two clients both read version V and both try to install a new chain. The first to succeed bumps the version to V+1. The second's write — still conditioned on V — now fails, because the version no longer matches. So exactly one writer wins; the loser must re-get and try again.

The conditional write makes reconfiguration a single-winner race: whoever bumps the version first invalidates every other write conditioned on the old version.
How is the MetaStore actually implemented?

The very first implementation was a single file in ZooKeeper — quick to stand up and good enough to get Delos running. Later they replaced it with an internal, "very naive" Paxos implementation. It is deliberately slow, and they genuinely do not care that it's slow: the only requirements are that it is correct and that it stores this one (chain, version) pair reliably and highly-availably. This is the control plane embodying its own principle — correctness and availability over speed.

Part V

The Native Loglet

The control plane is settled. Now for the data plane: an actual loglet. Delos's first loglet just wrapped ZooKeeper (easy, got them to production in 8 months). Later they built the native loglet and got a 10× latency improvement. There is also a backup loglet on HDFS for pushing old data to cold storage. The rest of the lecture is about how the native loglet works — and why it can be so much faster.

Why a Loglet Doesn't Need Consensus

Here is the payoff of the control/data split. A normal log implementation runs full consensus (Paxos: leader election, the works) on every append. The native loglet does not need consensus or leader election at all. Why is that okay?

Because all the heavy coordination has been offloaded to the MetaStore and VirtualLog. The one thing the loglet truly must guarantee is: after a seal succeeds, new appends are rejected. That's enough — because the moment anything goes wrong with a loglet, we don't repair it in place; we seal it and reconfigure to a fresh loglet (possibly on entirely new servers). The expensive "what's the new configuration?" decision is made highly-available by the MetaStore, not by the loglet.

Builds On — Azure Storage's Layering

This mirrors Azure Storage exactly: there, the stream layer used Paxos to map extents to servers (the rare, must-be-reliable control decision), while each extent ran cheap primary-backup replication for its actual data. Delos does the same: heavyweight agreement only at reconfiguration; lightweight replication inside each loglet.

The Big Idea

By making "switch to a new loglet" the universal recovery action, each loglet is freed from running consensus. It only has to honor one promise — sealing stops appends — and can otherwise use a faster, lighter protocol. That is the source of the native loglet's 10× win.

Log Servers and the Sequencer

The native loglet is a set of nodes acting as log servers, plus one designated node that also runs a sequencer component. The sequencer plays the role of a primary: all appends funnel through it.

Each log server keeps:

  • a local on-disk log (and the native loglet has no gaps within a server's accepted prefix conceptually — every slot of the loglet ultimately holds an entry);
  • a single sealed bit — once set, the server rejects all appends to its local log;
  • a local tail — the first unwritten position in this server's local log.
Key Concept — The Sequencer

One log server hosts the sequencer. Every append goes to it; it assigns the append a position in the loglet and forwards it to all log servers. Because a single component assigns positions, no two entries ever disagree at the same position — there is never a conflict to resolve. The cost: the sequencer is a single point through which appends must pass.

Local Commit vs. Global Commit

To reason about correctness we need two precise notions of "committed," plus a "best-effort" notion of how much each server knows.

Key Concept — Three Definitions

Locally committed: a command is locally committed on a log server once it has been written and fsync'd to that server's local log.
Globally committed: a command is globally committed when (a) it is locally committed on a majority of log servers, and (b) all previous commands are also globally committed.
Known tail: each server's best guess of the global tail. It always trails the true global tail (keeping it exact would need constant communication). It plays the "learner" role — what this server currently knows about how far the system has progressed.

Local tails differ per server. The global tail is bounded by the majority. Entries a server holds beyond the global tail (here on S3) are exactly the ambiguous "zombie" writes we'll worry about in Part VI.
Why the Two-Part Global-Commit Rule Lets You Skip Gaps

A log server may commit position n if either (a) it already holds n−1 locally, or (b) n−1 is globally committed (its known tail is past n−1). That second clause is powerful: a slightly-behind server can accept new entries even with a gap in its own local log, because it knows the whole system already committed everything up to that point. There are no gaps at the loglet (global) level — only, possibly, in a slow server's local copy.

Why Allow Local Gaps at All?

To avoid forcing slow servers to catch up. Filling every local gap would mean waiting on potentially many operations. By letting a lagging server keep gaps — as long as a majority holds each entry — the protocol never stalls on the slowest node and uses resources far more efficiently. No data is lost: every entry lives on a majority, so it survives any minority failure.

Why is it safe for a slow log server to have gaps in its local log (i.e., be missing some earlier entries) while still accepting newer ones?

Part VI

Native Loglet Operations & Fault Tolerance

With the setup in hand, we can implement the loglet's three data operations — and confront the one genuinely tricky corner case that the design creates. Then we'll tally exactly how many failures each operation can survive.

Seal and Append

Seal — the easy one

To seal, contact all log servers and ask each to set its sealed bit. If a majority respond success, the loglet is sealed, and future appends return an error. Simple. One subtlety to file away: the servers that respond will not all have the same local tail — a majority may have sealed, but a minority might hold extra entries, and even within the sealing majority the local tails can differ. We'll have to deal with that.

Append

1

The client sends the append to the sequencer, which assigns it a position and forwards it to all log servers.

2

The sequencer counts successful local commits. When a majority respond success, the append is complete and it returns success to the client.

3

If a majority of servers respond that they are sealed, the append fails. In all other cases (timeouts, etc.), retry after a timeout until either sealed or successful.

Zombie Writes and Copying Suffixes

Now the hard part — and the most important idea in the lecture. Recall that when we seal, the responding servers can have different local tails. Some servers may hold an entry that others don't. What do we do with such a dangling suffix during checkTail after a seal?

Here is the danger. The sequencer assigned that entry a position and forwarded it; a few servers stored it. Maybe it reached a majority and then some of those servers crashed — in which case it was committed. Or maybe it only ever reached a minority — in which case it was not. From the survivors alone, you cannot tell these two cases apart.

Key Concept — Zombie Write

A zombie write is an append that was given a sequence number and stored on a few nodes, but was not confirmed accepted by a majority. After a seal, a zombie write looks identical to a genuinely-committed entry that simply lost some replicas to crashes. You must decide whether the loglet ends before it or includes it.

So at seal time you face a fork, and you must commit to one rule:

Under-approximate (drop the suffix)Over-approximate (keep / copy the suffix)
Discard any entry not provably on a majority.Copy the dangling suffix across servers so it is on a majority.
Violates strong consistency: an append that really did commit (and was reported as committed to a client) could be lost forever.Safe: no committed entry is ever lost. At worst, a never-acknowledged entry gets promoted to committed.
Rejected.This is what the native loglet does.
Why Dropping the Suffix Is Unacceptable

Suppose the entry really was committed and the client was told "your payment went through," but then enough replicas crashed that it now sits on only a minority. If we dropped it during seal, we'd move to a new loglet with no record of it. Later, when someone reads, the payment is simply gone — "you never paid." Losing an acknowledged write is a fatal consistency violation, so Delos always copies the suffix.

Why is copying always safe? Because the sequencer guarantees there is never a conflict: at any given position, every server that holds anything holds the same thing. So "repairing" by copying a suffix from the servers that have it to those that don't can never overwrite a different value — there are no conflicts to resolve. Copying can only ever promote a minority entry to a majority, never corrupt one.

The Contract This Forces on Clients

Over-approximating means a never-acknowledged write can become committed — so a client cannot trust that a failed append really failed. The contract: if an append fails, that does not guarantee it didn't go through. Only a subsequent checkTail gives a definitive answer. This is exactly like an online card payment that reports failure but actually succeeded — you must re-check. The reassuring half: if an append succeeds, it definitely succeeded. And appends rarely fail anyway — a failed append means the sequencer failed, which is uncommon.

CheckTail — The Most Complex Operation

CheckTail returns the global tail and the seal status. You send it to all nodes, wait for a majority of responses, and then run a small state machine. (We assume exactly a majority responds; extra responses only help performance.) The cases:

1

None sealed, tails agree: just return that common tail.

2

None sealed, tails differ: wait until the known tail exceeds the max tail seen across responses, then return. (More on why below.)

3

All sealed, tails agree: the clean case — just return the agreed tail.

4

All sealed, tails differ: repair by copying suffixes across servers (safe, per the sequencer guarantee), then return the repaired tail.

5

Some (not all) sealed: send seal to all servers, then retry checkTail — driving the loglet toward the fully-sealed cases above.

Why Wait for Known Tail > Max Tail?

The max tail is the largest local tail any responding server reported. But a max tail does not mean that position is committed — it could be a minority (zombie) entry. The known tail, by contrast, only advances once a position is globally committed. By waiting until the known tail passes the max tail, checkTail guarantees that everything up to what it returns is genuinely globally committed — which is exactly what strict serializability requires.

SealFNeeds a majority to set the sealed bit, which survives F failures.
CheckTailFNeeds a majority to respond, then runs the state machine. As long as a majority is up, it returns.
Append0Funnels through the single sequencer. If the sequencer is down, no appends — a single fault stops appends.
The Big Idea

The native loglet makes a deliberate trade. It can survive many failures and still close the log (seal + checkTail), so it can always reconfigure to a new loglet and keep the system moving. The price: while the sequencer is down it cannot append. That's acceptable precisely because the recovery action — seal and reconfigure to a fresh loglet, on fresh servers if needed — remains highly available.

Putting It All Together

Every part of this lecture is one move in a single argument. Part I posed the problem: experimenting with consensus protocols in a live system is so risky that good ideas never ship. Part II gave the cure — the virtual log, which hides a chain of swappable loglets behind one continuous log, with seal and reconfigure as the levers for switching them, the way virtual memory hides physical pages behind one address space.

From there the system splits cleanly in two. Part III built upward: a strictly-serializable database via state-machine replication, where read-write transactions append-then-replay and read-only transactions pick a linearization point with checkTail. Parts IV–VI built downward: the control plane (the highly-available MetaStore with its single-winner conditional write) absorbs all the expensive coordination, which is exactly what frees the data plane (the native loglet) from running consensus at all. The loglet only promises one thing — sealing stops appends — and so it can use a fast sequencer-plus-majority protocol, paying for that speed with the "zombie write" corner case, which it resolves by always over-approximating and copying suffixes.

Notice the one idea that threads through every layer: "attempt" is not "knowledge." A reconfigure might lose a race; an appended transaction might have others slipped in front of it; a sealed loglet might hold a zombie suffix; a failed append might actually have committed. Every time, the cure is the same — go re-read the ground truth with a checkTail before you act. That single discipline is what lets Delos be both fast and correct.

The results vindicate the design: Delos went from scratch to production in 8 months on a ZooKeeper loglet, then earned a 10× latency win by swapping in the native loglet underneath a running system that was already serving 1.8 billion transactions a day — managing the metadata for Facebook's entire server fleet, exactly the Chubby/ZooKeeper-class role it set out to fill.
