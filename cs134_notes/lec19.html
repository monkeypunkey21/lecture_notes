Lessons Learned & the Road Ahead

The final lecture — distilling the whole course into three durable lessons for building systems, then touring the open frontier: serverless, data processing, reliability, verification, and beyond.

Part I

The Map of the Course

Why End With a Recap?

This is the last lecture of the quarter, and the last one for a reason. Across the term we built a large toolbox — and a toolbox is only useful if you can see how the tools relate. Before looking forward to research frontiers, the professor steps back to draw the map: what did we actually cover, and what are the few high-level ideas that survive after the details fade?

If you are coming to this page cold, here is the one-paragraph version of the whole course. A distributed system is a set of separate computers (nodes), connected by a network, that cooperate to look like a single reliable service. The hard part is that nodes can crash, messages can be lost or delayed, and there is no shared clock — yet clients still want correct answers and good performance. Everything in the course is a technique for coping with that reality.

The Topics We Covered

In rough order: clocks (ordering events without a shared time); MapReduce (a first distributed application); snapshots (capturing a consistent global state); replication & fault tolerance via primary-backup and Paxos; consistency models from strong to weak; consensus (and a practical system, ZooKeeper); sharding (scaling out by splitting data); transactions (guarantees across multiple shards); and real systems — Dynamo, Azure Storage, Spanner, NCas, and more.

Notice the shape: foundations first (clocks, snapshots), then the two great problems — fault tolerance (staying correct when nodes die) and scalability (staying fast as load grows) — and finally the systems that combine these into something deployable. The whole course is a layered tower; each layer assumes the one below.

NCas, Caching & the Real-Time Ordering Trap

The lecture opened by revisiting three quiz questions, because they expose subtle ideas worth nailing down. The most important concerned NCas — the caching layer studied last time (modeled on Facebook's Memcache-style infrastructure). Let's reconstruct the scenario carefully, because it is a classic correctness pitfall.

The frontend sends read requests. Several of them hit the cache and return immediately with the cached value. But here is the catch: before those reads were even issued, a write had already happened that eventually invalidates the very value those reads are returning. The writer finished its work first in real time — yet the readers never see it. They serve stale data.

The write completes (in real time) before the reads even begin — yet the reads return the old value. Real-time order is violated.
Key Concept — Real-Time Ordering Guarantee

A system respects real-time ordering if an operation that finishes before another operation starts is seen as happening first by everyone. Concretely: if a write completes at wall-clock time t, any read that starts after t must reflect that write (or a later one). This is the property at the heart of linearizability, the strongest single-object consistency model.

NCas Trades Real-Time Order for Speed — On Purpose

NCas does not provide real-time ordering guarantees, and that is a deliberate design choice, not a bug. Invalidating the cache on the critical path of every write would make writes slow. Instead NCas invalidates lazily, off the critical path. The price is that reads can briefly see stale data. This is the course's central theme in miniature: a real-time guarantee was sacrificed to win a performance benefit.

The Diamond Pattern

The third review question was about the diamond pattern, a microservice topology where a request fans out and reconverges: one service calls a downstream service that is also reached by another path, so two routes meet again at the same point — forming a diamond shape in the call graph.

The diamond: A reaches D through both B and C. Service D (or its cached data) is touched along two paths.

The subtle point the professor wanted to correct: the diamond pattern is about caching correctness, not about programmability or performance. If you naively add caches to a diamond, you can get wrong results — because the same underlying data is reached through two different routes, and one route's cache may go stale relative to the other. The fix seen last time was to track every service visited along a request's path, and track which services are listed by the cached items, so invalidations propagate correctly across both arms of the diamond.

Connecting Back

Both review points are the same lesson wearing different clothes: caching is a performance optimization that quietly threatens correctness. NCas threatens real-time order; the diamond threatens cross-path consistency. Hold onto this — it is the bridge into Part II, where we name the principles that let us reason about exactly these trade-offs.

Part II

Three Distilled Lessons

If you remember nothing else from this course in five years, the professor wants it to be these three lessons. They are not specific to any one system — they are how to think when you sit down to build any system, in academia or industry. Every system in the course was presented through this exact lens.

Lesson 1 — Agree on the System Goals First

Why this matters: the single most common way to build a bad system is to start writing code before deciding what the system is for. Without explicit goals you cannot tell whether a design is good — "good" is only meaningful relative to a target.

So before any design or implementation, agree on the requirements, and treat them as goals. This is precisely how every system in the course was presented: first, what is this system trying to achieve? Once the goals are pinned down, the design "falls out for free" — the right structure becomes almost obvious given the constraints.

Requirements come in several flavors. They can be about correctness (e.g. linearizability), about performance (latency, throughput), or about other properties entirely — for example, NCas required that it could be incrementally deployed and that it needed no modification to the user application. Those are real, design-shaping requirements that have nothing to do with correctness or speed.

The Big Idea — Goals Imply Non-Goals, and Non-Goals Are a Gift

Naming your goals automatically names your non-goals — the things you are happy to ignore. NCas's non-goal: it does not care about write-heavy workloads. If an application writes far more than it reads, caching is the wrong tool — you pay constant invalidation cost for little benefit. By declaring write-heavy workloads out of scope, NCas is free to optimize aggressively for the read-heavy case it actually targets.

Why goals exist at all: there are always trade-offs

The deep reason goals are unavoidable: no system works perfectly in every case. Performance and correctness, latency and consistency, simplicity and generality — these pull against each other. A system that is great in some cases must, somewhere, be worse in others. If you do not choose your goals deliberately, you will still make trade-offs — you will just make them by accident, and you may sacrifice exactly the cases you cared about most.

Goals Are How You Steer the Trade-offs

Identifying goals and non-goals is the mechanism by which you navigate trade-offs correctly. It tells you which property you are allowed to sacrifice in order to gain another that matters more. CAP, linearizability-vs-availability, the latency cost of strong consistency — all of these are choices, and your goals are the rulebook for making them on purpose instead of by chance.

Lesson 2 — Think Defensively

Why this matters: a system that only works in a clean lab is not a real system. Production environments are hostile. To build something robust — able to tolerate faults and adversarial conditions — you must enumerate the corner cases before they happen to you.

The defensive questions to ask about any system: Where will it run? Who can interact with it? What is the worst thing that can happen in this environment? Then get specific:

1

Can messages be lost? Networks drop and delay packets. A protocol that assumes reliable delivery is already broken.

2

Can components crash? Nodes die mid-operation, mid-write, mid-handshake. What state are you left in?

3

Can the system get overloaded? Demand can exceed capacity. Does the system degrade gracefully or collapse?

This is exactly the methodology the course used over and over: first catalog the kinds of faults that can occur, then ask, at each step, "what could go wrong here?" Defensive design is not pessimism for its own sake — it is the discipline of preparing for bad things by first naming them. You cannot defend against a failure mode you have never articulated.

The Trap of the Sandbox

The most dangerous assumption is that your system runs in a friendly "sandbox bubble." Anything destined for production — anything real users or adversaries touch — must be designed against the full menagerie of failures. A system that handles only the happy path is not robust; it is a demo.

Lesson 3 — Specifications Are Very, Very Important

Why this matters: the most powerful idea in the whole course is that you can describe what a system promises without describing how it delivers. That separation is what lets two different people — the client and the system builder — reason independently and still agree.

Key Concept — A Specification Decouples "What" From "How"

A specification is a contract that states what a system guarantees, while saying nothing about its implementation. Linearizability is the textbook example: it tells a client exactly what behavior to expect (operations appear instantaneous and in real-time order) without prescribing whether you use Paxos, primary-backup, a single server, or anything else. The spec is the interface; the implementation is hidden behind it.

This decoupling pays off on two independent levels. As a client, you can reason about what you are allowed to build on top, using only the contract. As a system builder, you can reason about how to satisfy the contract, freely changing the implementation. As long as the contract holds, neither side needs to understand the other's internals. We saw this with consistency conditions like linearizability and sequential consistency, but also with isolation levels for transactions and with the properties of consensus.

Specs make systems comparable

There is a second, subtler payoff. Specifications give us a common language to compare systems. If system X "offers linearizability" and system Y "offers sequential consistency," we can immediately rank their guarantees, because we can compare the specs. Without specifications, comparing two systems would mean comparing two messy implementations directly — with no shared abstract model, that is nearly impossible.

The Three Lessons Reinforce Each Other

These are not independent. Goals tell you which guarantee to aim for; specifications are how you write that guarantee down precisely; defensive thinking is how you make sure your implementation actually upholds the spec under failure. Goals → spec → robust implementation is the whole arc of system design in three words.

Part III

Serverless & Data Processing

The rest of the lecture is a guided tour of open problems — places where the field is still moving, and where the course's fundamentals get applied to new questions. Think of it as a map of where to go next. We begin with two of the hottest practical areas: the serverless cloud, and large-scale data processing.

Serverless Functions

Why this matters: the dominant trend in cloud computing is offloading more and more responsibility to the provider. Understanding serverless is understanding where the industry is heading.

The cloud evolved through stages. First it was simply renting physical machines for a while. Then virtual machines (VMs). Then lightweight containers. Now the frontier is functions as a service — serverless.

Key Concept — Serverless / Function-as-a-Service

In a serverless model, you write a function that handles an incoming event — say, an HTTP request — and hand it to the provider (AWS Lambda, Azure Functions, Google Cloud Functions). The provider does everything else: allocates VMs to run it, load-balances across many VMs under heavy load, and scales VMs up and down with demand. "Serverless" does not mean there are no servers — it means someone else manages the servers.

# You write only this. The provider runs it for you.
def handler(event):
    # 'event' is an incoming HTTP request, queue message, etc.
    result = process_request(event)
    return result

The economic model is pay-per-execution. When a load spike arrives, the provider allocates VMs for you; when it subsides, the provider deallocates them. You are not billed for VM allocation — you are billed per execution. The provider of course marks this up, so a unit of compute costs more than renting raw VMs yourself. What you buy with that premium is freedom from VM management, scaling, and upgrades.

The Stateless Problem

Here is the catch that makes serverless interesting from a distributed-systems view: serverless functions are short-lived and have no permanent state. Why? Because if a function held state, the provider could not freely kill and migrate VMs to balance load — it would have to track, replicate, and manage that state. So providers impose a rule: keep any state in an external service (S3, DynamoDB, etc.); the function itself stays stateless.

Statelessness Creates Hard New Problems

Pushing all state outside the function does not make state problems vanish — it relocates them. Two big questions follow: (1) How do you give serverless a correct, efficient, fault-tolerant abstraction for state? (2) How do you build rich applications on top of something with no internal state? These are active research areas, not solved problems.

The lecture pointed at concrete work. One paper provides fault-tolerant, transactional state for serverless workflows — assuming an external store, it tackles idempotence: if a function crashes mid-execution and restarts, how do you avoid corrupting state by applying an effect twice? Another, "From Laptops to Lambdas," shows you can build surprisingly powerful things on top of stateless functions — for instance, parallelizing a Chrome compilation across thousands of serverless functions so it finishes in seconds instead of hours.

Idempotence Is the Serverless Fault-Tolerance Workhorse

Because a serverless function can crash and be retried at any moment, the safest design makes each operation idempotent — applying it twice has the same effect as applying it once. This is the same defensive instinct from Lesson 2, applied to a new environment: assume the crash will happen, and make the retry safe.

MapReduce → Spark

Why this matters: the course saw MapReduce only briefly. Spark is the natural next step, and its core idea is a clean, memorable systems insight.

First, the problem with MapReduce. MapReduce achieves fault tolerance by writing every stage's output to disk. One MapReduce stage runs, dumps its result to disk; if you chain many stages into a pipeline, you pay a disk write after every stage. For multi-stage jobs that is enormous, wasted I/O — you only actually care about the final output.

Key Concept — Spark's Lineage

Spark keeps intermediate data in memory and records lineage: metadata describing how each piece of data was derived from the inputs. It persists only the original inputs, not the intermediate stages. If a worker fails, Spark consults the lineage to find exactly which piece of data was lost and which input it came from, then re-runs only that slice of the pipeline — instead of re-storing and re-running everything.

MapReduce checkpoints to disk between every stage; Spark keeps data in memory and recovers lost pieces via lineage.
Recompute vs. Re-store — A Recurring Trade-off

Spark trades storage for recomputation. Rather than persist everything so it can be reloaded after a crash, it persists almost nothing and is willing to recompute the lost slice from lineage. This is a deep, reusable pattern: when computation is cheap relative to I/O, remembering how to rebuild data can beat storing the data itself.

Stream & Actor Systems

Data processing has a different "flavor" from the fault-tolerance-centric core of the course. It cares overwhelmingly about throughput (data crunched per second) and about efficient communication, and relatively little about latency or fine-grained fault tolerance. The course leaned the other way — toward faults and consistency — so these are complementary worlds.

Two directions worth knowing:

  • Stream processing — instead of batch jobs over a fixed dataset, you process a continuous flow of incoming events as data-flow pipelines, aiming for high throughput and fault tolerance on a never-ending stream. (The Naiad / Timely Dataflow line of work, now commercialized as Materialize, pushes this view.)
  • Actor-based systems like Ray — an alternative model where computation is organized around lightweight actors; heavily used for distributed reinforcement learning and ML workloads.

Part IV

Reliability & Correctness

The course mostly asked "how do we build a correct system?" But in the real world there is a second, equally hard question: "how do we know our deployed system is behaving as expected — and how do we keep it alive under stress?" This part surveys the research that lives there.

Overload Control

Why this matters: real systems regularly face load beyond their capacity. The naive failure mode is catastrophic — the system simply crashes. The goal of overload control is for a system to degrade predictably under heavy load: stay up, keep its SLOs (Service Level Objectives — promises like "99% of requests answer within 100 ms") for at least some requests, rather than collapsing for all of them. There is active research on this for microservice applications specifically.

Metastable Failures

Why this matters: this is a failure mode so subtle that almost no one thought about it until roughly a decade ago, and it is now a serious concern at companies like Amazon.

Key Concept — Metastable Failure

A system is vulnerable to a metastable failure if, under high load, a triggering event can push it into a bad stable state that it cannot escape on its own — even after both the original trigger and the high load are removed. The system stays stuck and requires manual recovery (an operator restarting it, draining queues, etc.). The bad state is "metastable": self-sustaining once entered.

The classic example is retries amplifying load. Many systems retry on timeout: a client that hears nothing for 10 seconds re-sends its request. Now walk through the death spiral:

1

The system is running near capacity. Load is high but it is coping.

2

Something bad happens — a single node crashes, or one machine gets briefly bottlenecked. The system can no longer keep up with incoming requests.

3

Requests start timing out. Each timed-out client retries, adding a duplicate request on top of the original.

4

Now the system faces the original load plus a flood of retries. It falls further behind, causing more timeouts, causing more retries — a self-reinforcing loop.

5

The system is fully choked. Even if you fix the crashed node and the load returns to normal, the backlog of retrying requests keeps it jammed. It will not recover until an operator manually drains the queue.

A "Fix" Can Be the Trap

Retries are a sensible, common reliability tactic — yet here they are the very thing that prevents recovery. This is the unsettling lesson of metastable failures: a mechanism that helps in the normal case can become the engine of an unrecoverable state under load. Defensive design (Lesson 2) means asking not just "does my retry help?" but "what happens when everyone retries at once?"

Verification, Testing & Model Checking

Why this matters: "correctness" means does the system satisfy its specification? — and there is a whole research ecosystem devoted to answering that with more than hope.

The techniques form a spectrum of rigor:

  • Testing — run the system on many inputs and check outcomes.
  • Model checking — systematically explore reachable states. The P language lets you describe a distributed protocol and then test or model-check it systematically.
  • Full verification — mathematical proof that an implementation meets its spec. IronFleet was a landmark: the first full-fledged proof of a real-world distributed system implementation satisfying both safety and liveness properties.
Even Verified Systems Can Have Bugs

A humbling result: researchers took a verified system and still found bugs in it — because the assumptions behind the proof were imperfect. A proof only guarantees the system is correct relative to its assumptions; if those assumptions don't match reality, bugs slip through. Verification is not a panacea — but for truly critical systems, it is still worth doing.

Monitoring & Tracing

Building a correct system is only half the battle; once it is running, you need to see inside it. When a production system has a problem, can you even tell? And if so, can you locate and fix it? Distributed tracing tackles this by reconstructing causal relationships between events across many machines — so you can follow a single request's journey through dozens of services and find where it went wrong. Closely related is profiling: measuring where time actually goes, so you know what is fast and what is slow.

Part V

Frontiers — Where the Field Is Headed

The final batch of open problems pushes on the course's deepest assumptions: what consistency we settle for, how hardware is organized, what we assume about node behavior, and even how we frame classic problems like consensus.

Weaker, Smarter Consistency

The course studied consistency from strong (linearizability) to weak (eventual). Research is filling in the rich middle ground, because strong consistency is expensive and weak consistency is sometimes too weak:

  • Monotonicity — generalizing eventual consistency to identify systems that work correctly without ever needing stronger guarantees. If your computation is monotonic, eventual consistency is enough.
  • Strong causal consistency — strong enough to guarantee nearly everything you'd want, yet implementable far faster than sequential consistency.
  • Impossibility results — continued study of what you provably cannot achieve, mapping out the fundamental trade-off boundaries of the design space.
This Is Lesson 1, Generalized

Every item here is the goals-and-trade-offs lesson at the frontier. The research question is literally: what is the weakest (cheapest) consistency that still meets our goals? Picking the right point on the strong-to-weak spectrum is exactly the trade-off navigation the course kept emphasizing.

Disaggregation

Why this matters: data-center networks are getting dramatically faster, and that changes what hardware layouts are possible.

Key Concept — Resource Disaggregation

Traditionally each machine bundles compute, memory, and storage together. Disaggregation breaks them apart: pool all the CPUs, all the memory, and all the storage in the data center as separate shared resources. Any node can reach the giant shared memory pool over the network using a fast protocol like RDMA (Remote Direct Memory Access). The catch: remote pooled memory is slower than local memory.

The research questions: how do you use this huge shared-but-slower memory efficiently, and how do you do it automatically — gaining the benefits without forcing every application to be rewritten? (UCLA's own Harry Xu's group works heavily in this space.)

Systems for Machine Learning

A closely related frontier: building systems to support ML training and inference efficiently, correctly, and with fault tolerance. This is fundamentally a distributed-systems problem — training a huge model spans thousands of nodes, so the moment a node fails you must answer the course's oldest question in a new setting: what do we do when a node dies mid-computation?

Byzantine Faults & Blockchain

The whole course assumed the worst a node could do was crash — fail silently but never lie. There is a strictly harder world.

Key Concept — Byzantine Faults

A Byzantine node is not just crashed — it is malicious: it can actively violate the protocol, send conflicting messages to different peers, and lie. Tolerating Byzantine behavior (Byzantine Fault Tolerance, BFT, and Byzantine consensus) is much harder than tolerating crashes and requires stronger protocols with stronger assumptions about how many nodes can misbehave.

This matters enormously for peer-to-peer systems like blockchains, where there is no trusted coordinator and participants genuinely may be adversarial. The starting point is the foundational Byzantine fault tolerance / Byzantine consensus literature.

Fresh Spins on Classic Problems

Even decades-old problems still yield new results. Two examples the professor found striking:

1

Better vector clocks. The classic vector clock supports random access to its components — but in practice we never randomly access one component; we only compare whole clocks. Exploiting that, recent work structures the vector clock as a tree, achieving much better performance for the operation we actually use.

2

Statistical consensus. Classic consensus is deterministic: "tolerate F faults using 2F+1 nodes." But what does "F faults" even mean in a real fleet? A Berkeley group argues we should think probabilistically — nodes have some crash probability, so design for statistically very high availability instead. In a 5-node, fault-tolerant-up-to-2 system, there is still a small chance all 3 remaining fail; statistical thinking confronts that directly rather than ignoring it.

Why "F Faults" Is an Odd Abstraction

The deterministic "tolerate exactly F faults" framing is convenient for proofs but disconnected from reality, where nodes fail with probabilities, not in fixed counts. Reframing consensus around crash probabilities is a genuinely different way to specify the goal — a reminder that even our most settled abstractions are choices that can be revisited.

Part VI

How to Study & What's Next

Exam Strategy

The lecture closed with concrete advice for the final, and it maps directly onto the three lessons. Take it seriously — it tells you what the exam actually rewards.

The Exam Tests Two Kinds of Understanding

Every lecture matters, but two themes recur in exam questions. (1) Trade-offs, goals, and specifications — why does a system make a given trade-off, and why does it matter? (2) Implementation — how does a mechanism actually work to provide its guarantee? You need both: the conceptual "why" and the mechanical "how."

A Big Chunk of the Exam Is About the Assignments

There will be a significant exam component asking detailed questions about the assignments. The expectation is that you fully understand your own implementation and the assignment specifications — the details of your solution, why it works, and what it guarantees. If you leaned heavily on AI while coding, go back and re-read what you built until you genuinely understand it. The exam will not ask about some "canonical" reference solution you never wrote — it asks about your solution, which means understanding your own code is the whole game.

A Study Plan, Mapped to This Page

Work through every lecture, with extra weight on points the professors flagged in class (often the in-lecture questions). For each system, be able to state its goals and non-goals (Lesson 1), articulate its specification (Lesson 3), and trace how its implementation upholds that spec under failure (Lesson 2). Then make sure you can explain your assignment solutions line by line. Parts II and VI of this page are your conceptual scaffolding; Parts I, III–V are the systems and frontiers that hang on it.

Putting It All Together

This last lecture is the course folding back on itself. The opening review — NCas's stale reads and the diamond pattern — was not a random warm-up; it is Lesson 1 in action. NCas sacrifices real-time ordering (a correctness non-goal) to win read performance (its real goal), and the diamond pattern shows what breaks when you add caching without respecting the system's correctness specification. The whole course, distilled, is the discipline of making those sacrifices on purpose.

From there the three lessons become a single method. You choose goals and non-goals (Lesson 1) so you know which trade-offs you are allowed to make. You write those goals down as a precise specification (Lesson 3), which both decouples client-reasoning from builder-reasoning and lets you rank systems against one another. And you think defensively (Lesson 2) to make sure your implementation actually holds the spec when messages drop, nodes crash, and load spikes. Goal → spec → robust implementation: that is the arc behind every system in the course, from Paxos to Spanner to NCas.

The frontiers then show the same method applied where the answers aren't known yet. Serverless relocates the state problem and leans on idempotence — defensive design in a new environment. Spark trades storage for recomputation — a goal-driven trade-off. Metastable failures are what happen when a defensive mechanism (retries) is itself the trap. Verification is the attempt to prove the implementation matches the spec — and even it is only as good as its assumptions. Weaker consistency, disaggregation, Byzantine consensus, statistical consensus — each one revisits a course concept by changing a goal, an assumption, or a specification. Once you see the field this way, every paper in the tour is the same three lessons, asked at a harder difficulty.

The One Sentence to Keep

There is no system that is best in all cases — so the entire craft of building distributed systems is choosing your goals, writing them down as a specification, and defending that specification against everything that can go wrong.

What's Next

This is the final lecture, so "next" is the final exam — and then the field itself. The professor's parting advice: work on what genuinely interests you, because the field is vast enough that good, careful work always finds an audience. Have a great summer.
