Azure Storage & the Rise of Cloud-Native Apps

An interactive guide to Lecture 15 — how modern apps are stitched together from cloud services, and how Azure Storage layers a file system, a sharding engine, and replication into one production storage platform.

Part I

How Applications Are Built Today

Before we dive into the star of today's lecture — a real production storage system called Azure Storage — we need to understand the world it lives in. The way software is built has shifted, and that shift is exactly why a system like Azure Storage exists and looks the way it does.

In the past, an application was usually a monolith: one big program, running on machines you owned, that did everything itself — accepted requests, stored data, ran the business logic, all in one codebase. Today, applications are increasingly assembled out of cloud services you don't own and didn't write. To understand that style of application, you first need a vocabulary for the kinds of data these services store.

The Three Storage Types

Azure Storage is built to serve three different shapes of data. The whole rest of the lecture is about how one system manages all three at once, so let's pin down what each is.

Key Concept — Blobs, Tables, Queues

A blob ("Binary Large OBject") is an unstructured file — just raw bytes the system doesn't try to interpret. Audio, video, images, arbitrary text dumps. Amazon's S3 is blob storage.

A table is structured data with a schema: rows and columns, like a SQL table. Each row is one data item — think user profiles or business records.

A queue is an ordered channel for message delivery: one component puts messages in, another takes them out.

The interesting one is the queue, because it does more than you might first guess. The obvious use is data movement: a sensor streams temperature readings into a queue, and a downstream component consumes them. That covers streaming pipelines and publish/subscribe ("PubSub"), where one writer fans a message out to many readers.

But queues are also used for control flow, not just data flow. You can wire compute components together with queues to build a workflow: a user makes a request, that request becomes an event, and the event flows from stage to stage through queues. Here the queue carries a "do the next step" signal, not a payload of data. This dual role — moving data and orchestrating events — is what makes queues the glue of cloud applications.

From Monolith to Composition of Services

Armed with blobs, tables, and queues, here is the modern pattern. Instead of one monolith, you build your application as a composition of services, each doing one job, connected by these storage types:

1

Ingest with blobs. A crawler scrapes a website (say, posts from a social network) and dumps the raw text into blobs — no need to structure it yet.

2

Orchestrate with queues. A serverless function picks posts off the blobs and passes them down a chain of services via queues — one expands links, another scores each post for usefulness.

3

Annotate with tables. Each stage consults structured tables of metadata (annotations, scores) to do its job, and writes structured results back to tables.

4

Produce blobs for clients. The final stage writes processed results back into blobs that a separate search service can query on the other end.

The Word for This: Orchestration

These compositions of services are called workflows or orchestrations. They come in recurring shapes — fan-out (one stage spawns many), fan-in (many results merge into one), timers, and so on. The whole industry is moving from "one program does everything" toward "many small services, composed."

Serverless & the Provider's Multiplexing Trick

The other half of this shift is compute that you also don't manage. The traditional model is: you rent a virtual machine (VM) from a cloud provider and run whatever you want on it — but you're responsible for deploying it, scaling it, and keeping it alive.

Key Concept — Serverless

Serverless means you hand the provider your code — typically a function that runs in response to an event — instead of running a VM yourself. The provider spawns VMs to run your function when load arrives, kills them when load drops, and maintains them. You offload all deployment responsibility. Examples: AWS Lambda, Azure Functions / Durable Functions.

Why do providers love this? Because of multiplexing. You pay per invocation — a few cents per second your function actually runs. Behind the scenes the provider can pack many different customers' serverless workloads onto a single physical machine. When one workload goes idle, another runs in its place. The provider squeezes far more useful work out of each machine and pockets the savings. (Keep this idea of packing many tenants onto shared hardware in mind — it returns when we discuss Azure Storage's design goals.)

The Pros and the Cons

This cloud-native style is a genuine trade-off, not a free win. Here is the balance sheet that came out of the lecture discussion.

BenefitsCosts
You depend on others for most functionality, so you do less work.You depend on others — if their software is unreliable, you take the hit, and it's out of your control.
No deployment, management, or operations to run yourself.Vendor lock-in: no standards across providers, so switching platforms is painful.
Providers multiplex many tenants, lowering cost per invocation.The app becomes scattered — hard to analyze, hard to reason about end-to-end correctness and performance.

A Real Cautionary Tale

When the AWS us-east-1 region went down over a simple DNS issue, every company relying on that infrastructure was effectively offline for a day. That's the dark side of "let someone else run it": a single dependency failing can take your whole business with it.

The Big Idea of Part I

Modern distributed applications are no longer monoliths — they are orchestrations of cloud services glued together with blobs, tables, and queues, running on serverless or VM compute. This is powerful and convenient, but it makes the application scattered, locked-in, and hard to reason about end-to-end. Fast, reliable storage systems like Azure Storage are what make this whole style possible.

Part II

Azure Storage: Goals & Architecture

Where We're Headed

Part I gave us the why: cloud apps need a storage service that can hold blobs, tables, and queues for many tenants, fast and reliably. Azure Storage is that service. It has run in production in Microsoft Azure since 2008, and the paper we follow was published at SOSP 2011. The rest of the lecture builds it up layer by layer.

Design Goals — What Are They Trying to Achieve?

As with every system in this course, we start by asking what do they want, then look at how they get it. Azure Storage's stated goals:

1

Strong consistency. They argue this is essential for ease of use — without it, clients can't predict what a read will return. (This is the classic strong-vs-eventual debate; Azure picks the strong side.)

2

Availability and partition tolerance. They claim to want all three of strong consistency, availability, and partition tolerance at once — a claim we'll scrutinize hard at the end.

3

Finding data in a huge store. Azure Storage isn't one user's storage — it's a single service shared across all users. So "where does this account's data live?" is a real problem to solve.

4

Disaster recovery. Like Dynamo, they replicate across data centers so that if an entire data center is lost, the data survives.

5

Scalability, load balancing, and multi-tenancy. Serve thousands of users by packing them all into the same big storage — never dedicate a whole machine to one tenant.

Why Multi-Tenancy Obsesses Them

Multi-tenancy is the same multiplexing idea from serverless: cram many accounts onto shared machines so no resource is wasted. Machines cost money to buy, power, and cool, so leaving them idle is pure loss. The goal is to utilize hardware as close to 100% as economically safe. (Spoiler: Azure aims for ~70% — high, but deliberately not 100%, as we'll see.)

Two Different Kinds of Tasks

Managing a big distributed service involves two very different categories of work, and recognizing the split is key to understanding what Azure Storage does — and what it deliberately ignores.

Application / Distributed-Systems TasksCluster-Management Tasks
How is data stored, sharded, partitioned?Detecting that a node has failed or is healthy
What code does each node run?What to do when a node joins or leaves
Recovering from a fault to preserve consistencyMonitoring, deployment, updating servers
Load-balancing the data itselfWhich rack a node goes on; how nodes find each other

The left column is what this class is about — the distributed-systems layer on top. The right column is handled by a separate cluster manager (Kubernetes, or Azure's "Fabric Controller"). Azure Storage explicitly offloads all cluster-management duties to the Fabric Controller. It only worries about the left column: given a substrate of nodes it can assume are healthy (or knows when they're not), how do we build reliable storage on top?

Stamps and the Location Service

Now the architecture. The top-level unit is the storage stamp.

Key Concept — Storage Stamp

A stamp is a cluster of storage nodes (each with disks and compute). It's the coarsest level of sharding — sharding at the level of accounts. The hard rule: one account lives in exactly one stamp. Many accounts can share a stamp (that's multi-tenancy), but a single account is never split across stamps.

Why force one account into one stamp? To keep data lookup dead simple. Azure Storage is queried by URL — your account name in the URL tells the system exactly which stamp's cluster to search. The cost: your account can't grow bigger than a single stamp. But stamps are so large (petabytes, thousands of accounts) that this only bites truly enormous accounts, who must manually split themselves across several account names.

To map an account to its stamp, there's a location service: it knows which stamps exist and which accounts live where, and a DNS lookup talks to it. When a request arrives, you hit the location service first, learn the stamp, and only then go find the data.

Have We Seen This Before?

The location service is a centralized point that decides when to migrate accounts and knows where everything lives — exactly like the view service or the MapReduce master from earlier in the course. So what do we need from it? It must be fault-tolerant, off the hot path (so its slowness doesn't hurt every request), and — because it can afford to be a bit slower — strongly consistent. It runs a Paxos-based protocol to achieve this. It doesn't need to scale much, since it's consulted rarely.

One more design number: a stamp is kept at only ~70% utilization. Push higher and you lose headroom to absorb failures, migrate load, or recover. Seventy percent sounds conservative until you learn that typical industry utilization is often only 10–20% — so 70% is already aggressive.

The Three Layers Inside a Stamp

Each stamp is itself built from three stacked layers. This layering is the elegant core of the whole system: each layer does exactly one thing.

The three layers inside a stamp. We'll spend Part III on the stream layer and Part IV on the partition layer; the front end we ignore.
  • Stream layer (bottom): a distributed, fault-tolerant, replicated file system. It stores binary files (called "streams") and knows nothing about blobs, tables, or queues. It's the durable substrate.
  • Partition layer (middle): maps the high-level abstractions (blobs, tables, queues) down onto streams, and provides transactions and sharding.
  • Front end (top): a stateless router that finds the right partition and stream servers. We won't discuss it further.

Two Kinds of Replication

Azure Storage replicates data in two ways at once. Understanding why it needs both is one of the lecture's central insights.

Intra-Stamp (inside one stamp)Inter-Stamp (across stamps)
Synchronous, on the critical path of every requestAsynchronous, off the critical path, per-object
Guards against frequent small failures: a disk dies, a switch fails, a machine crashesGuards against rare big disasters: an entire data center / stamp is lost
Happens all the time → must have very low latencyHappens periodically, in batches → must use network bandwidth optimally
Recover without crossing data centersRecover from catastrophe, possibly with manual intervention
Why Both? Different Failure Profiles

Small hardware failures happen constantly, so intra-stamp replication must be cheap and instant — you don't want to phone another data center every time a disk dies. Big disasters are rare but devastating, so inter-stamp replication can afford to be slow and batched, optimizing bandwidth instead of latency. The two are complementary: together they cover the whole spectrum of failures, from a single dead disk to a flooded data center.

Part III

The Stream Layer — A File System Built on Append

Mental Reset

For all of Part III, forget blobs, tables, and queues entirely. The stream layer is a replicated, fault-tolerant file system. The only question it answers is: how do we store binary files reliably and fast? The partition layer (Part IV) will later map the high-level abstractions onto these files.

The Neat Trick: Append, Not Write

A normal file system gives you two operations: read and write (overwrite data anywhere). The stream layer's key design decision — and a trick used in many high-performance systems — is to instead offer only read and append. You can read existing data and you can add new data at the end, but you can never overwrite what's already there.

Why give up arbitrary writes? Because immutability buys enormous simplicity and speed, as we'll see when failures hit. But first, the data model.

Streams, Extents, and Blocks

Key Concept — The Three Levels

A stream is a "file" — but really it's just an ordered sequence of pointers to extents. The stream holds no data itself.

An extent is a chunk of actual data, roughly a gigabyte in size, made of a sequence of blocks. Extents are the unit of replication — each one is stored in 3 replicas.

A block is the smallest unit of read or write, on the order of a few megabytes (e.g. 4 MB). You always read or write a whole block — never a sub-block fragment, because validation/checksums happen at the block level.

A stream is a list of extent pointers; an extent is a list of blocks. New data is appended to the last (open) extent.

Sealing: How Immutability Works

Here is the mechanism that makes append-only work. An extent can be sealed. Once sealed, it is immutable forever — no more appends, the data never changes again. At any moment, only the last extent in a stream is unsealed; everything before it is frozen.

So a file grows like this: you keep appending blocks to the current open extent; every so often you seal it and start a fresh open extent at the end. To add a new extent pointer to a stream, you must first seal the previous one — there's no way to have two open extents. This invariant is the backbone of the whole design.

Why This Makes Pointer Operations Magic

Because extents are immutable and live independently somewhere in the cluster, the stream is just metadata over data. Consider two hypothetical operations on a stream:
• Reverse the order of the extents → just rewrite the pointer list. No data moves. Blazing fast.
• Modify the first block of every extent → since extents are immutable, you must create a brand-new copy of every extent (E1′, E2′, …) and re-replicate all of them. Catastrophically slow.
This is counterintuitive — you'd guess modifying one block per extent is cheaper than touching all of them. The append-only design flips the cost: rearranging pointers is trivial, while mutating data is expensive. The partition layer will exploit this constantly.

How Replication Actually Works

Now the implementation. There are three players:

  • Client — actually the partition layer above, not an outside user.
  • Stream Manager (SM) — a Paxos cluster that owns the stream namespace and all the pointers, and assigns extents to extent nodes.
  • Extent Nodes (EN) — many nodes that know nothing about streams; they just store blocks and extents.

To create a new extent, the client asks the Paxos-backed stream manager, which picks a replica set (three extent nodes), records the mapping, and hands it back. Then the client does the actual data writes directly with those nodes — the stream manager steps aside.

1

Client wants a new extent → asks the Stream Manager (Paxos), which chooses 3 extent nodes as the replica set and stores the mapping.

2

Client sends an append to the primary extent node.

3

Primary writes locally, forwards to the 2nd replica; the 2nd forwards to the 3rd. This is a replication chain, just like primary-backup view chains earlier in the course.

4

Each writes, acknowledgements flow back, and once all three have the data the primary returns success to the client.

The Big Idea: Paxos Off the Critical Path

Why doesn't running Paxos in the stream manager destroy scalability? Because the stream manager is consulted rarely — only when a new extent (≈1 GB) is created, which adds just one pointer. The flood of actual data — millions of megabyte-sized block appends — flows directly between client and extent nodes, never touching Paxos. By decoupling pointer management (rare, Paxos) from data writes (constant, direct), Azure pushes the expensive consensus protocol entirely off the hot path. This decoupling is the single most important idea in the stream layer.

When a Node Fails: Seal and Move On

What happens if, mid-append, a replica fails? The client either times out or gets an error. Now the append-only design pays off spectacularly:

1

Client sends an append; one of the three replicas fails to respond (timeout after, say, 100 ms, or an explicit error).

2

Client tells the Stream Manager to seal the extent at the last successfully-replicated point. Anything written beyond that point is simply discarded — the metadata never records it, so for all purposes it never existed.

3

The SM (highly available via Paxos — survives even multiple node failures) hands the client a fresh extent on a new replica set, avoiding the failed node.

4

Client resumes appending to the new extent. The paper reports this whole reaction takes ~20 ms.

Why Sealing Sidesteps the "Did It Write?" Problem

A subtle danger: maybe the failed replica did apply the append but its response was lost. Doesn't that risk inconsistency? No — because we seal at the agreed point. All replicas (and the client) agree on how far the data is valid; anything past the seal is invisible. Even if a replica secretly wrote more, no one's metadata points to it, so it's as if it doesn't exist. The client just re-appends that data to the new extent. Immutability + sealing turns a messy partial-failure into a clean "freeze here, continue there."

From the client's perspective, a write that hit a failure simply "took 100 ms longer" — an almost seamless experience. Append + read + seal together deliver high availability and great performance: they push Paxos off the critical path and let the system recover from failures in milliseconds.

Part IV

The Partition Layer — Serving Blobs, Tables & Queues

Climbing Up a Layer

Now we can bring blobs, tables, and queues back. The stream layer gave us a replicated file system. The partition layer's job is to map those high-level abstractions onto streams, provide transactions, and — crucially — shard and load-balance. Note the clean split of duties: the partition layer does sharding; the stream layer does replication. The partition layer replicates nothing itself.

A key implementation fact: all data in Azure Storage is stored as big object tables. Every blob for an account lives in one "blob table," every queue in a queue table, and so on. So underneath, everything is a giant table — which raises the question that drives this whole layer: how do you scale serving a single enormous table?

The partition layer has three components:

  • Partition Manager (PM): assigns partitions to partition servers and load-balances across them.
  • Partition Servers (PS): shard servers; each serves a set of partitions. A given partition is served by exactly one server (so it can offer consistency and transactions).
  • Lock Service: a Paxos log service (like Chubby / ZooKeeper). It elects the partition manager and hands out leases that decide which server owns which partition.

Range Partitions

Key Concept — Range Partition

The partition layer shards a table by sorting it and slicing it into contiguous key ranges. Each slice is a range partition — a shard of the table — and is served by one partition server. To scale, you simply hand different ranges to different servers.

Why a Lease, If Each Server Owns Disjoint Data?

You might ask: if partitions are disjoint, why need a lock at all? Because ownership isn't known ahead of time and must move (for load balancing or failover). The lock service holds, per partition, a small record (think a ZooKeeper file named P1) saying "partition server #1 owns this, for this lease period." The lease guarantees exactly one server serves a partition at a time — which is what makes single-server consistency and transactions sound.

The Commit Log + Checkpoint Trick

Now the elegant heart of the partition layer. Instead of storing the table as a literal data structure and overwriting it on every update, Azure stores updates as an append-only commit log plus periodic checkpoints.

Key Concept — Log + Checkpoint

Every write (insert / update / delete) is appended to a commit log as "I changed row R to value V." Periodically the system writes a checkpoint: a materialized snapshot of the whole table. The current table = latest checkpoint + replay the commit log on top of it.

A range partition stores its state in two mirrored forms — on disk (for durability) and in memory (for speed):

On Disk (three streams)In Memory
Metadata stream — pointers to all the streams needed to load this partition—
Commit log stream — all inserts/updates/deletes since the last checkpointMemory table — in-memory copy of the commit log (updates since checkpoint)
Row data stream — the checkpoint of the actual table dataCache of the latest checkpoint
Three Streams Per Partition

Each partition is exactly three streams in the stream layer — metadata, commit log, and row-data (checkpoint). Because a partition is "just three streams," it can be exported to storage and re-imported on a different partition server effortlessly — that's how ownership moves during load balancing.

Serving a Partition

When a partition server takes ownership of a partition, it loads it like this:

1

Read the metadata stream to find which streams to pull.

2

Load the latest checkpoint (row data) into memory.

3

Replay the commit log on top of the checkpoint → now memory holds the fully materialized, up-to-date partition.

4

Updates arrive → append to the commit log (disk) and the memory table, then return success. Reads check the in-memory log first, then the in-memory checkpoint, so they always see the latest data.

5

When the log grows past a limit, write a fresh checkpoint that squashes everything in — then the old log can be safely dropped.

Why This Hybrid Is So Good

The magic is that the commit log maps perfectly onto the stream layer's append operation. Every write is just an append — which, as Part III showed, is blazing fast and often skips Paxos entirely. So updates are nearly free. The checkpoint then prevents the log from growing without bound (fast loading, no need to replay millions of events). Together you get: fast updates (cheap appends), fast recovery (replay the log), fast loading (frequent checkpoints), and an easy way to keep memory and disk in sync. It's the best of both ends of the speed/durability spectrum — and it works precisely because the layer below is append-only.

Load Balancing: Splitting and Merging

Range partitions can grow large, and a single server can be overloaded — by CPU, storage bandwidth, or storage size. The fix is load balancing: split hot partitions (too many requests) into two, and merge cold ones, guided by profiling. Here's a split:

1

The partition server checkpoints the partition, creating a materialized view.

2

It stops serving traffic for that partition.

3

It creates new streams (metadata, commit log, data) for the two resulting partitions.

4

It updates the new partitions' metadata streams and resumes serving both — possibly moving one to a different partition server.

Splitting Is Fast — Because No Data Moves

Creating those new streams is fast, even though it goes through Paxos (~20–100 ms). Why? Because you only create new pointers to existing extents — the data on disk never moves. Re-replicating gigabytes of data would be catastrophic; instead you just re-point. This is the same append-only payoff from Part III, and it rests on the system's deepest design decision:

The Decoupling Insight

Azure Storage decouples how data is replicated (the extent / stream level) from how it is served and sharded (the partition level). Replication lives entirely below; sharding lives entirely above and only manipulates pointers. That separation is what makes load balancing cheap, recovery fast, and the whole system scalable.

Part V

Inter-Stamp Replication & the Big Lessons

Inter-Stamp Replication: The Disaster Safeguard

We promised two kinds of replication. Part III–IV covered intra-stamp (the synchronous, on-the-critical-path replication done by the stream layer). Inter-stamp replication is the other half: disaster recovery across data centers.

All operations are sent asynchronously to another data center, in the background, in batches. The remote data center does not serve reads — it just holds a copy of the data. If a stamp is lost, someone must manually intervene to promote the secondary copy. This isn't a day-to-day mechanism; it's a safeguard so that a natural catastrophe doesn't permanently erase, say, all of a bank's data.

Lesson 1: One Unified Service → Free Multiplexing

Azure Storage serves blobs, tables, and queues through a single infrastructure. Beyond the obvious simplicity (one team, one codebase, one logical way to handle everything) and the ability to batch requests across types, there's a subtler performance win:

Multiplexing for Free

The three data types have different resource appetites: blobs are big and want lots of disk; tables benefit from I/O parallelization across many disks; queues mostly live in memory. Run them on the same infrastructure and their demands balance out, so every machine's CPU, disk, and memory get used near their best capacity. Mixing complementary workloads is exactly the multiplexing/utilization goal from Part I — and here you get it for free.

Lesson 2: Compute–Storage Separation

This ties back to Part I's cloud-native applications. Because Azure Storage exists as a fast, separate storage service, applications can separate compute from storage: run your logic in VMs or serverless, keep all your data in Azure Storage, and scale each independently. Need more disk? Add storage. Need more CPU? Add compute. You no longer over-provision one just to get the other.

What Makes This Even Possible?

Separating compute from storage means every data access now crosses the network to reach Azure Storage — a real performance overhead. This only works because systems like Azure Storage (and Dynamo) are fast and efficient enough to absorb that network round-trip. If storage were slow, no large application could tolerate the latency, and compute–storage separation would collapse. Fast storage is the enabling condition for the entire cloud-native style.

Lesson 3: Did They Actually Beat CAP?

Recall their bold claim: strong consistency and availability and partition tolerance, all at once. The CAP theorem says you can't have all three when a network partition occurs — under a partition you must sacrifice either consistency or availability. So did Azure Storage beat CAP?

No — And Here's the Tell

Azure Storage runs centralized, Paxos-based services in many places: the location service, the stream manager, the partition lock service. Paxos sacrifices availability to keep consistency under partitions. So strictly speaking, the "all three" claim is impossible — it's partly a marketing framing, and you should be critical of it.

The Real Insight Behind the Claim

What they actually achieve is that every centralized, availability-sacrificing service sits off the data / critical path. The Paxos services are consulted rarely (creating an extent, moving a partition, locating an account) — not on every read or write. So most of the time the system is extremely fast and available; you only pay the consensus cost in rare cases. They didn't beat CAP — they hid CAP's cost by keeping consensus off the hot path. This is the recurring theme of the entire lecture.

D is wrong because the data path really is strongly consistent — the cleverness is where the consensus cost is paid, not pretending it doesn't exist.

Putting It All Together

This lecture was really one story told at five altitudes, and they all rhyme.

We opened with cloud-native applications (Part I): apps are now orchestrations of services glued together with blobs, tables, and queues, running on serverless compute that providers multiplex across tenants. That created a demand — a fast, reliable, multi-tenant storage service — which Azure Storage exists to fill.

Part II laid out Azure Storage's architecture: accounts sharded into stamps via a Paxos-backed location service, and each stamp built from three layers with two complementary kinds of replication. Then we descended into those layers. The stream layer (Part III) is an append-only replicated file system whose single deepest trick — streams are pointers, extents are immutable, sealing freezes them — makes pointer operations free and data mutation expensive, lets failures be handled by "seal and move on" in 20 ms, and pushes Paxos off the critical path. The partition layer (Part IV) builds blobs/tables/queues on top using a commit-log-plus-checkpoint hybrid that maps every write onto the stream layer's fast append, and shards via range partitions that split and merge cheaply because only pointers move.

The thread tying every part together is decoupling expensive-but-rare work from cheap-but-constant work. Replication is decoupled from sharding. Pointer management (rare, Paxos) is decoupled from data writes (constant, direct). Consensus is decoupled from the hot path. That's how Azure Storage can run Paxos all over the place yet still be fast — and why, in Part V, its "we beat CAP" claim is really "we hid CAP's cost off the critical path."

Stepping back to the course: we've now seen many ways to shard. Modulo hashing, consistent hashing, Chord, and Dynamo all shard in a decentralized way, where nodes collectively decide placement. Azure Storage is the opposite — sharding happens at two layers (stream and partition), each decided by a centralized manager — yet it stays scalable by pushing those centralized controllers off the data path. Same goal (scale, fault tolerance), opposite philosophy.

What's Next

Azure Storage is the first of our closing tour of real production systems. The lecture sequence continues with Spanner (Google's globally-distributed, strongly-consistent database with synchronized clocks), then Delos and Newcastle, before the final lecture. Watch for the same recurring move you saw today — keeping consensus and centralized coordination off the critical path while delivering strong guarantees — and notice how each system makes a different bargain among consistency, availability, and the cost of coordination.
