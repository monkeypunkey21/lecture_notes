Fault Tolerance & Primary-Backup Replication

An interactive guide to Lecture 5 — how a service survives a crashed machine by keeping a live backup, when to sync them, and how the view service and split-brain problem keep everyone agreeing on who is in charge.

Part I

What We Actually Want From Fault Tolerance

Why Are We Learning This?

A distributed system is a collection of separate computers (we call them nodes or processes) that cooperate over a network to behave like a single service. The whole reason we spread a service across many machines is that any one machine can crash — its power dies, its disk fails, someone trips over the cable. The promise of distribution is that the service keeps running even when individual pieces fall over. Fault tolerance is the machinery that delivers on that promise.

So far in this course we have built up two tools. First, logical clocks let many copies of a service (a replicated state machine, or RSM — several nodes all running the same code so they stay identical) apply updates in the same order. But ordering alone does not save you when a replica dies. Second, we introduced snapshots: a node periodically writes its entire state to stable storage (a checkpoint) so that after a crash it can be restored. This lecture asks the harder question: snapshots can recover a crashed node, but how good is that recovery, and can we do better?

Key Concept — Replicated State Machine (RSM)

An RSM is a set of replicas that each run the same deterministic code on the same sequence of inputs. Because the code is deterministic and the input order is identical, every replica ends up in exactly the same state. That equivalence is the foundation of every fault-tolerance scheme in this lecture: a backup is useful precisely because it is a state machine identical to the primary.

Recovering From a Snapshot, Step by Step

Before we critique snapshots, let's nail down exactly how recovery works for a single node. (For replicated state machines we do not actually need the fancy coordinated, multi-node snapshots from earlier lectures — a plain single-node checkpoint is enough here.) Picture a process that takes a checkpoint, keeps working, and then crashes:

1

Detect the failure. Something in the system must first notice that the node has stopped responding. Nothing can recover a crash that nobody has observed.

2

Restore the latest checkpoint. A new or restarted node reads the most recent checkpoint off stable storage and loads that entire state back into memory.

3

Rebuild ephemeral state. Things that don't live in the checkpoint — open network connections, channels to other nodes — have to be re-established from scratch.

4

Resume. Replay any recorded in-flight messages, then continue processing as normal.

The Problem When Failures Are Frequent

This scheme is correct, but it is time-consuming. Every recovery means re-reading a whole checkpoint and rebuilding all connections. If failures are rare you barely notice; if they are frequent, the cost of constantly restoring snapshots becomes punishing. That cost is exactly what motivates the rest of the lecture.

The Trade-offs Every Fault-Tolerance Scheme Must Make

The professor asked the class to brainstorm what properties we want from a fault-tolerance mechanism. The answers fall into a small, important set. The first one is absolute; the rest are dials we trade against each other.

The Big Idea — Correctness Is Non-Negotiable

Correctness is the one guarantee we never sacrifice. No matter how clever the scheme, it must never return a wrong answer or let the service diverge from what a single correct machine would have done. Everything else — speed, cost, freshness — is a trade-off. System design is the art of choosing where to land on those trade-offs, because there is no design that wins on every axis at once.

Once correctness is locked in, three measurable trade-offs remain:

1. Cost on normal execution

This is the overhead you pay in the good case, when nothing has failed. Recall the goal of coordinated snapshots from earlier: instead of "stopping the world" to take a snapshot, we wanted the system to keep working while the snapshot was taken. The leftover overhead — extra latency, CPU, memory, or network traffic on the critical path — is the cost on normal execution.

2. Recovery freshness

After a failure, how far back in time do we resume from? If you checkpoint once an hour and crash at minute 59, you lose 59 minutes of work. Fresher recovery means losing less.

3. Time to recover

How long does it take to get from "we noticed the crash" back to "we are serving requests again"? Restoring a big checkpoint and rebuilding channels takes time.

Trade-offHow Snapshots Score
Cost on normal executionLow — occasionally saving state off the critical path is cheap; requests keep flowing.
Recovery freshnessStale — you only have state as of the last checkpoint, so you lose everything since.
Time to recoverSlow-ish — you must reload the checkpoint and re-establish all channels.
Where This Is Heading

Snapshots are cheap when healthy but slow and stale when recovering. The mechanism in Part II — primary-backup replication — flips this profile: it pays a higher cost during normal execution in exchange for recovery that is instant and perfectly fresh. Keep these three dials in mind; we will re-score the new scheme against them at the end.

Part II

Primary-Backup Replication

First, a Recap: The Client-Server Model

The setting for this lecture is the everyday client-server pattern, the key abstraction for the control plane of a distributed system. A client sends a request; a server listens for requests, processes them (one at a time, or concurrently), and sends back a response. That's it. Everything below is about making the "server" half survive a crash without the client noticing.

The plain client-server loop. Primary-backup replication hides a second machine behind the "server."

The Primary and the Backup

Here is the central idea of the lecture.

Key Concept — Primary-Backup Replication

We run two replicas: a primary and a backup. To the client they look like a single server — the client only ever talks to the primary. The backup silently keeps a copy of the primary's state. If the primary crashes, the backup is promoted and takes over with the exact same state, so the client barely notices. (You can have several backups, but the assignment uses just one.)

The client talks only to the primary. The primary syncs its state to the backup. The dashed box is the boundary between "inside the service" and "the outside world" — a boundary that becomes crucial in Part III.

Handling the Two Kinds of Failure

With two replicas there are exactly two failures to worry about, and each has a simple answer:

1

The primary fails. Promote a backup to be the new primary. Because the backup already holds an identical copy of the state, it can start serving immediately.

2

The backup fails. Add a fresh machine as a new backup (or, if none is available, run with no backup for now and accept the reduced safety).

That sketch raises the two questions that drive the rest of the lecture, and they are worth stating precisely now because we will spend Parts III–V answering them:

The Two Questions That Define the Protocol

(1) When should the primary sync with the backup? Syncing on every single operation would be correct but slow. Can we sync less often?

(2) What information should the primary send when it syncs? The whole state every time? Just the change? The answers to these two questions are primary-backup replication.

Correct. The backup has been continuously kept in sync, so at the moment of promotion it already is a current copy of the service — no checkpoint to reload, no log to replay. That is exactly why primary-backup gives instant, fresh recovery, in contrast to snapshots which must reload an old checkpoint. Option A describes snapshot recovery; option C would be hopelessly slow and isn't how this works.
Part III

When to Sync, and What to Send

Walking Through a Server Loop

To answer "when should we sync?", the lecture uses a concrete example: the master server from MapReduce (a system that splits a big computation into many small tasks handed out to worker machines). The master sits in a loop, and each pass through the loop looks like this:

// master server loop — assigns tasks to workers
while true:
    msg  = receive()          // 1. a worker asks for work
    task = pick_new_task()    // 2. choose a task to give it
    mark_assigned(task)        // 3. record locally that it's taken
    respond(msg.sender, task) // 4. TELL the worker — leaves the service!

Assume the primary and backup start this loop perfectly in sync. The professor walked the class through "what if the primary crashes right here?" at each line. Trace it carefully — this trace is the heart of the lecture:

1

Crash after receive()? No problem. The primary only pulled a message off its queue and parsed it — a purely local change. The backup, being in sync, has the same message in its own queue. If promoted, it pulls that same message and proceeds identically.

2

Crash after pick_new_task()? Still fine. Choosing a task is deterministic and local; the backup would make the identical choice.

3

Crash after mark_assigned()? Still fine, and this is the subtle one. The primary changed its local state, but the backup never saw it. Yet because the backup holds the same queue, it would pull the same message, pick the same task, and mark it assigned itself. The local change is reproducible, so divergence is harmless. No outside party has seen anything.

4

Crash during respond()? Now we are in danger. The response leaves the service and reaches a worker. The outside world has now observed a result. If the primary crashes here and the backup (which never recorded this assignment as externalized) takes over, the two have diverged in a way that is visible to others and cannot be undone.

Externalization & Observable Consistency

That trace reveals the single rule that governs when to sync. The dividing line is whether a state change has become visible outside the primary-backup group.

The Big Idea — Sync Before You Externalize

We cannot tolerate a primary failure after a state update becomes externally visible. Once anything escapes the service boundary — a response to a client, a request to another service, a write to a shared disk — it can never be taken back. Therefore the primary must sync with the backup just before any externally-visible action, so that both replicas already agree on it. Conversely, as long as a change is purely internal, the primary is free to stay out of sync, because the backup can always reproduce internal work itself.

Key Concept — Observable (Observational) Consistency

We judge the system from the viewpoint of an outside observer. If, from the external world's perspective, nothing has changed, then internal divergence simply doesn't matter — the primary can crash, the backup can take over, and no observer can tell the difference. We only owe consistency at the boundary where the system reveals itself to the outside.

Why does crashing at step 3 truly cause no problem? Because the message the primary pulled came off its own queue, and the in-sync backup has an identical queue. The primary "consumed" the message privately and crashed before telling anyone. When the backup is promoted, it pulls that same first message from its own queue and continues — it never even knows the primary got partway through. Same input, same deterministic code, same result.

Footgun — Messages Can Be Sent Twice

This protocol can send the same response twice. Suppose the primary syncs, sends a response, then crashes; the backup, promoted, also sends that same response. The client now receives it twice. That is acceptable — it is the client's job to recognize a duplicate (e.g., via a request ID) and ignore the second copy. Sending too many times is a solvable nuisance. The unacceptable failure is the opposite: a state change that diverges and can't be reconciled. The protocol deliberately errs toward duplicates.

What to Send: Whole Snapshot or Just the Delta?

Now the second question. When the primary syncs, what does it actually transmit?

The naïve answer — "send a full snapshot of the entire state every time" — is correct but slow. A core assumption of distributed systems is that sending data over the network is expensive compared to a local operation, so we avoid it whenever we can. Shipping the whole state on every sync would crush the cost-on-normal-execution dial.

The better answer is to send a delta: just the operation that changed the state, not the whole state. But that only works under conditions, and there is one case where a full snapshot is genuinely unavoidable:

Send the full state when…Send just the delta (operations) when…
A brand-new backup joins (bootstrapping). It knows nothing, so it must receive everything once.The backup is already up and in sync — then each new operation is enough to keep it current.
Expensive, but a one-time cost per new backup.Cheap and ongoing — but requires same order and determinism.
Two Conditions for Shipping Operations Instead of State

Sending only the operations works only if: (1) they are applied in the same order on both replicas (unless every operation commutes), and (2) request processing is deterministic. If an operation uses randomness, reads the wall clock, or otherwise behaves non-deterministically, replaying the same request would produce a different result and the backup would drift apart. To fix that, the primary must also ship along any extra state the backup needs to reproduce the result exactly — for example, the random number that was drawn.

Putting the Sync Protocol Together

When to sync: before any state change becomes externally visible. How to sync: on a new backup's arrival, send the complete state; once it's running alongside the primary, send only operations, plus any captured non-determinism (random draws, etc.). With that, we have a fault-tolerance scheme that recovers instantly and freshly if the primary fails — at the price of a network round-trip on every externally-visible operation.

Aside: Why Not All Operations Are Equal (Workloads)

The lecture took a useful detour here. If we must sync before externalizing, a natural follow-up is: does the primary really have to inform the backup for every request? The answer depends on what the request does, which leads into a short tour of how API methods differ.

Consider an Instagram-style service: get_posts, get_followers, follow, delete_post, send_message. Even a simple key-value store has at least a GET and a PUT/set. These methods differ along several axes:

  • Read vs write — does it modify state, or just fetch it?
  • I/O-bound vs CPU-bound — does it touch lots of state (I/O-heavy) or mostly compute (CPU-heavy)?
  • Coordination — does it touch a single item, or items scattered across the state (requiring more coordination)?
  • Load characteristics — some methods are called far more often than others, with spikes (Black Friday), daily cycles, and hotspots (a few celebrity accounts get most of the traffic).
There Is No Universal Optimization

System design is about making the right assumptions about your workload. Caching is the classic example: it helps enormously when you repeatedly read a small set of keys, but it is detrimental when writes are frequent (every write invalidates cached entries) or when reads are spread across many keys (low hit rate). An optimization that wins on one workload can lose on another — which is why companies tune the same read/write primitives differently rather than shipping one "optimal" design.

For our assignment, the relevant takeaway is concrete: a key-value store is typically read-heavy (an 80/20 or more skew toward reads is common). That matters for syncing, because reads don't change state. This unlocks the read optimization:

The Read Optimization (Optional)

The primary does not need to inform the backup for reads. Why is that safe? Because if the backup is externally consistent with the primary, a promoted backup would produce the identical response for that read anyway — so the primary can serve it without ever telling the backup. (For the assignment, the professor recommends not implementing this — its safety relies on subtle assumptions — unless you want extra difficulty.)

Part IV

The View Service — Who Is the Primary?

The Problem We Quietly Created

We declared "when the primary fails, promote a backup." But step back and ask a programmer's question: the client was talking to the primary, and it just crashed — who does the client talk to now? The client needs to know, at all times, which replica is the primary.

Why Hard-Coding the Primary Fails

The obvious "solution" — bake the primary's address into the client's code — does not work. The whole point is that the primary gets replaced when it fails. A hard-coded client would keep talking to a dead machine forever, with no way to discover the replacement. We need a reliable, dynamic way to learn who the primary is.

Views and View Numbers

The answer is a new component: the view service.

Key Concept — The View Service and a "View"

The view service maintains the current membership of the primary-backup group. That membership is called a view. Each view contains a view number (incremented every time membership changes), the current primary, and the current backup. Whenever something changes — a crash, a new backup joining — the view service produces a new view with a higher number. Clients and replicas ask the view service "what's the current view?" to learn who is in charge.

View 1: S1 alone. View 2: S2 joins as backup. View 3: S1 died, so the backup S2 was promoted to primary and S3 joined as the new backup. The number only ever goes up.

What triggers a new view? A primary crash, a backup crash, or a request to add or change a backup. To detect crashes, the view service periodically exchanges heartbeat ("are you alive?") messages with the replicas.

Why Only a Backup May Be Promoted

When the primary dies, the new primary must be up to date. The view service guarantees this with one rule: only a node that was already a backup can be promoted. A backup has been kept in sync, so it is provably current. This is exactly why the view service — not some arbitrary outside entity — must be the one to choose backups: it is the only party that knows which nodes are caught up (it learns this from the heartbeats with the primary and backup).

The View-Change Protocol

Transitioning from one view to the next is a small protocol with three steps:

1

The view service announces the new view to all replicas (broadcasts the new membership).

2

The new primary syncs with the new backup, if there is one — bringing the fresh backup up to date (a full-state transfer for a brand-new node, as we saw in Part III).

3

The new primary acknowledges the new view to the view service, confirming the backup is ready. Only then is the view change complete.

A Hole We Deliberately Ignore

What if the new primary dies in the middle — after the announcement but before it can acknowledge? Then we are stuck. In a real system you would handle this (typically with manual override and restart). For the assignment, we simply assume the primary does not crash during the view change, to keep things tractable.

The Key Win: The View Service Is Off the Critical Path

At first this feels like we have just moved the single point of failure. Before, "if the primary fails, we're stuck"; now, "if the view service fails, we're stuck." Have we gained anything? Yes — and the reason is subtle and important.

The trick is that the client does not contact the view service before every operation. Instead, the client contacts the view service once per session, caches the view, and then fires all its requests directly at the primary. The view service only re-enters the picture when the cache is stale.

Key Concept — Caching the View & When to Invalidate It

The client caches the current view across many operations. It invalidates the cache when an operation fails — i.e., it gets no response, or it gets an explicit "I am not the primary" rejection from the replica it contacted. Only then does it go back to the view service to learn the new primary.

The Big Idea — Keep the Fault-Tolerant Part Off the Hot Path

Because the view service handles only the rare "who's the primary?" lookup and not the steady stream of real requests, it is not on the critical path. A component that sees little traffic adds almost no overhead to normal execution and is therefore much easier to make fault-tolerant — you can afford to make it slower and more careful. This principle — push the part that needs strong fault tolerance off the hot path — is exactly what real systems like Apache ZooKeeper exploit to get high reliability without sacrificing performance.

Part V

Split Brain — When Two Nodes Both Think They're Primary

The Failure That Breaks Everything

Here is the scenario that motivates the final, costly piece of the protocol. Recall from earlier in the course that a network partition is a failure where messages between two groups of nodes are dropped or arbitrarily delayed — the nodes are alive, but they can't talk to each other. Watch what a partition does to our scheme:

1

We're in a healthy view: S1 is primary, S2 is backup, and a client has cached this view.

2

A network partition cuts S1 off from the view service. S1 is perfectly healthy, but its heartbeats no longer arrive.

3

The view service can't tell "crashed" from "unreachable" (a deep theme of this course). It assumes S1 is dead and promotes S2 to primary in a new view. S2 agrees and starts acting as primary.

4

Meanwhile the client still holds the old cached view and keeps sending requests to S1 — which is alive and still believes it is the primary, and happily serves them.

The Disaster — Split Brain

Now two nodes both believe they are the primary: S1 (talking to the stale client) and S2 (talking to the view service). Two partitions of the system disagree about who is in charge, each acts independently, and their states diverge arbitrarily. This is the split-brain problem, and it shatters correctness — the one guarantee we swore never to give up.

Split brain: the partition leaves S1 serving a stale client while S2 is promoted by the view service. Two "primaries," two diverging states.

The Fix: The Primary Must Get Backup Acknowledgements

How do we forbid split brain? Several tempting ideas don't fully work — checking reachability through other servers fails if those are partitioned too; telling every client about a view change fails because you can't wait for unreachable clients to agree. The robust answer turns out to be a rule we already half-stated:

The Big Idea — The Primary Must Forward Every Operation to the Backup and Wait for an Ack

Before the primary responds to a client, it must forward the operation to the backup and receive an acknowledgement. This closes the split brain: the new primary was always a backup, so the old primary will inevitably try to contact it. When S1 forwards a request to S2, S2 — which now knows from the view service that it is the primary — replies "you are not the primary anymore." S1 then returns a failure to the client, the client invalidates its cached view and goes to the view service, and discovers the real primary S2. No two nodes can serve as primary at once.

Why does this work so cleanly? Because of the promotion rule from Part IV: only a backup can become the new primary. The old primary always talks to what it thinks is its backup, and that backup is exactly the node that may have been promoted. So the old primary cannot help but encounter the new primary on its next forward — there is no way for some stranger node to become primary behind the old primary's back. The forced conversation is what makes a mistaken old primary discover the truth.

The Price: This Is Slow, and Fundamentally So

This is precisely why primary-backup replication has a high cost on normal execution — every single operation requires a network round-trip to the backup before the client hears anything. Worse, if the new primary S2 is itself unreachable, the old primary S1 blocks and waits, unable to make progress. This isn't sloppy engineering: combining network partitions with crashes makes certain things provably impossible in distributed systems, so we must accept a suboptimal, sometimes-blocking outcome. A later lecture introduces a more symmetric protocol that can keep progressing even when some nodes are down.

Legal and Illegal View Transitions

The split-brain analysis directly constrains which view transitions the assignment may allow. The rule: you may change the primary or the backup, but never lose the chain of an up-to-date node.

Valid transitionsIllegal transitions
Add a backup: (S1, —) → (S1, S2)Change both roles at once: (S1, S2) → (S3, S4)
Replace a failed backup: (S1, S2) → (S1, S3)Transition to a view with no primary
Promote backup on primary failure: (S1, S2) → (S2, S3)Demote a primary into a backup & invent a new primary from nowhere
Promote with no new backup yet: (S1, S2) → (S2, —)Move the view number backward in time

The thread tying all the legal moves together: every new view must keep a node that was already in sync — which is exactly why you can never swap both the primary and the backup in a single step, and why view numbers only ever increase.

Putting It All Together

This lecture is one continuous argument, each part forcing the next into existence. We started by asking what we actually want from fault tolerance and found three dials — cost on normal execution, recovery freshness, and time to recover — all sitting beneath the one rule we never break: correctness. Snapshots score well on cost but badly on freshness, which set us hunting for something fresher.

That something was primary-backup replication: keep a live, in-sync backup so recovery is instant and perfectly fresh. To make it work we answered two questions. When to sync? Only before a change becomes externally visible — the principle of observable consistency, traced through the MapReduce loop line by line. What to send? A full snapshot to bootstrap a new backup, then cheap deltas afterward, as long as execution is ordered and deterministic.

But "promote a backup" silently assumed the client could find the new primary, which forced the view service into existence — a registry of views (primary + backup + an ever-increasing number), kept reliable precisely by sitting off the critical path through client-side caching. And the view service's inability to distinguish "crashed" from "partitioned" gave birth to split brain, whose only robust cure is the rule that the primary must forward every operation to the backup and wait for an ack — which is exactly why this scheme is fast to recover but expensive during normal execution.

The Trade-off, Re-scored

Compared to snapshots, primary-backup flips the profile: it pays a heavy normal-execution cost (a backup round-trip on every externalizing operation) in exchange for recovery that is instant and maximally fresh. There is no free lunch — only a different, deliberate point in the design space. Choosing that point well, given your workload, is the whole craft of system design.

What's Next

Two threads were left dangling on purpose. First, primary-backup is asymmetric and blocking — it has a built-in hierarchy and grinds to a halt when the new primary is unreachable. In two lectures we will meet a more symmetric replication protocol that keeps making progress even when some nodes are down, and which can even be used to make the view service itself reliable.
