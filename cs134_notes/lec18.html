Caching Microservices the Right Way

An interactive guide to Lecture 18 — how NuCache makes cache invalidation automatic, correct, and off the critical path for real microservice deployments.

Part I

Microservices — The World This System Lives In

Every system we study in this class solves a problem that only exists because of how real software is built. So before we can understand NuCache — the system this lecture is about — we have to understand the kind of application it runs inside. That application is almost always built out of microservices. If the term is new to you, don't worry; we'll build it from the ground up.

Why Do Microservices Exist?

Imagine you're building a hotel-booking website. The naive way is a monolith: one giant program that handles searching for hotels, taking reservations, processing payments, sending emails — everything — all in a single codebase, deployed as one unit. That works fine when the company is small. It stops working when you have hundreds of engineers, because everyone is editing the same enormous program, and a bug anywhere can take the whole thing down.

The microservices answer is to split the application into many small, independent services, where each service does one thing. One service handles hotel search. Another handles reservations. Another handles payments. They run on separate machines and talk to each other over the network.

Key Concept — Microservice Architecture

A microservice architecture structures an application as a collection of small, independently deployed services that communicate over the network. Each service owns its own logic, its own data (often backed by its own key-value store or database), and its own deployment — and crucially, each can be written in a different programming language.

That last point — different languages per service — is one of the main benefits. The search team can use Go, the payments team can use Java, and neither has to agree on a single language. But it immediately creates a problem: if two services are written in different languages and store their data in different formats, how do they talk to each other?

How Services Actually Communicate

The answer is a serialization library plus a well-defined API. Serialization means turning an in-memory object (a Go struct, a Java object) into a language-neutral stream of bytes that travels over the network, then turning it back into a native object on the other side. Popular libraries for this are gRPC and Apache Thrift.

On top of serialization, services expose APIs and call each other with RPCs (remote procedure calls) — something we've seen earlier in this course. There are two communication styles worth distinguishing:

Key Concept — Two Ways Services Talk

RPC (request/response): service A calls service B and waits to hear back — like a function call across the network. Queues (fire-and-forget): service A sends an event into a queue and moves on without waiting for a response. Use RPC when you need the answer; use a queue when you only need to notify.

Each service is its own little world: it has its own state (a KV store or database behind it), it's deployed on its own machines, and it can independently be sharded (split across machines for capacity), replicated (copied for fault tolerance), and multithreaded. A separate team owns the deployment and upgrades of each service.

Service Meshes and Sidecars

Beyond their actual business logic, every microservice has to repeatedly do a set of generic chores: discover which other services exist on the network and how to call them, authenticate requests, serialize data, encrypt traffic for privacy, and monitor per-request metrics. Re-implementing all of that inside every service would be wasteful and error-prone.

So the industry factors it out into a service mesh.

Key Concept — Service Mesh & Sidecar

A service mesh is a software layer that implements the generic, application-wide microservice tasks (discovery, auth, encryption, serialization, monitoring). It is typically realized as a sidecar: a small companion process deployed next to each service that intercepts all incoming and outgoing communication and performs these tasks on the service's behalf. Examples: Dapr, Istio.

The mental picture: every service ships with a little buddy process (the sidecar) glued to its side. Every byte the service sends or receives passes through the sidecar first. This matters for NuCache, because the sidecar is exactly the natural place to slot in new interception logic without touching the service's own code.

Why This Matters Later

Keep the sidecar in mind. When NuCache says it intercepts communication "very easily on top of the sidecar," it means it can observe and modify every service call without the service team rewriting anything. That is the whole reason NuCache can be "plug-and-play."

Why Does Industry Love Microservices?

Companies like AWS, Netflix, and Twitter run microservice graphs with hundreds or thousands of services. Why pay all this complexity? The benefits the class surfaced are, above all, organizational:

1

Fine-grained modularity. Each team knows exactly the one service it owns and works on. You don't need to understand the whole company's codebase.

2

Independent iteration. A team can upgrade or redeploy its service whenever it wants, without coordinating a giant company-wide release.

3

Independent scaling and SLAs. You scale only the services that need it, and each team owns its own performance guarantees (SLAs) instead of being blocked by the whole monolith.

4

Fault localization. When something breaks at 3 AM, you only page the team that owns the broken service — not everyone who knows the monolith. Faults stay contained and are easier to fix.

The "It's 3 AM" Story

The professor relayed why a real microservice engineer values them: in a monolith, when the app crashes you must wake the engineers who understand the whole thing. With microservices, whichever service crashed, you page only that service's team. Fault tolerance and fault localization are the recurring answer to "why X?" in this class — and microservices give both.

Part II

The Microservice Graph and What It Costs

We keep drawing microservices as a graph: nodes are services, and an edge from service A to service B means "A can call B." This picture is intuitive, but the lecture spends real time warning that it is dangerously incomplete. Understanding why is the key that unlocks NuCache's hardest design constraints.

The Graph Is Lossy — Calls Are Dynamic

Ask the precise question: what does an edge from Service 1 to Service 2 actually mean? It's tempting to say "Service 1 calls Service 2." But can we be more precise — and is "more precise" even possible?

An Edge Does NOT Mean "Always Calls"

An edge means a service might call another. It does not tell you whether the call happens on every request, only some requests, in what order, or in parallel. If Service 1 sometimes routes to Service 2 and sometimes to Service 3 (say, for A/B testing), the graph drawing is identical — yet the behavior differs per request.

This is what we mean when we say the graph is lossy: there is no static graph that captures all the information, because service calls are dynamic and messy at runtime. A service might try Service 2 first, fall back to Service 3 if the data isn't there, fall back again if a service is down, do load balancing, and so on. The exact set and order of calls depends on the request and on live conditions.

A student asked the natural question: why not just make engineers declare the dependencies up front? The trouble is that even a complete list of services a service could call doesn't tell you which it will call for a specific request, or in what order. So you can't take the static list for granted when optimizing.

The Big Idea — No Ahead-of-Time Graph

In a real microservice deployment, you do not know the call graph ahead of time. Calls are dynamic, data-dependent, and unordered from the outside. Any system that wants to be deployed for real — NuCache included — must work without assuming it knows who calls whom. This single assumption shapes almost every hard decision in the rest of the lecture.

The graph hides one more thing: a single node may actually be a service that is sharded and replicated across many machines for scalability and fault tolerance. So the clean picture is hiding both dynamism and internal multiplicity.

The Price You Pay

Microservices aren't free. The class enumerated the drawbacks, and they sort into two buckets — performance and reasoning:

DrawbackWhy It Hurts
Increased latencyA request now traverses a whole stack — sidecar → network → sidecar — at every hop. Serialization and encryption add cost on top.
Many moving partsMore components means more independent things that can fail or misbehave.
Hard to reason aboutTo know what the whole application guarantees, you must compose the separate specs of every team's service. No single place holds the whole picture.
Hard to debug / programYou can't open one IDE and write the whole app, or use one type system across it. Every service hides behind its API.
This Lecture's Focus: Performance

Of all these drawbacks, this lecture zeroes in on latency. Every hop through serialization, encryption, network, and sidecar adds delay — and a deep graph stacks many hops. NuCache is fundamentally a performance system: it tries to skip hops entirely.

How Could We Make Microservices Faster?

The class brainstormed ways to cut microservice latency. They're worth knowing because they frame the design space NuCache lives in:

1

Fusion / merging. Combine related services into one, doing the work all at once instead of in a chain of separate RPCs.

2

Co-location. Run related services on the same machine or data center so calls stay local. A more intense version runs them in ultra-lightweight containers on one machine — minimal serialization, no encryption, no network.

3

Speculative execution. For services that frequently produce stragglers (bad tail latencies), fire off two copies and take whichever finishes first — a hedge that raises your odds of a fast response.

4

Brute-force the primitives. Minimize serialization cost, minimize encryption cost, offload frequent operations to specialized hardware (e.g., NICs).

5

Caching. If a call's result hasn't changed, skip the call entirely and reuse the saved answer. This is the path NuCache takes.

Part III

Caching and the Coherence Problem

We just saw that caching is one way to cut microservice latency. Now let's make "caching" precise, see exactly what it buys us, and then meet the problem that makes it hard — the problem NuCache exists to solve.

What Caching Buys Us

Key Concept — Caching a Service Call

A service call that has no side effects (it only reads, it doesn't write) can be cached: its result is saved, and the next identical call returns the saved result without re-issuing the call — as long as the downstream data hasn't been modified.

Concretely: imagine a hotel-reservation app. The front-end service calls the hotel service to get hotel details. We put a cache at the front-end that saves the hotel service's responses. The next time the front-end needs the same hotel info, it finds the result in the cache and returns it immediately — completely skipping the hotel backend and everything below it.

Caching Skips Whole Subtrees

A study by Alibaba found that on production traces, 40% of microservice call graphs avoid calling up to hundreds of services thanks to caching. A single cache hit near the top of a deep graph can short-circuit an enormous amount of downstream work. That is why caching is such a powerful lever for latency.

The Coherence Problem

Here is the catch. A cached value is only valid while the underlying data hasn't changed. The instant some write modifies the data at a backend service, every cache holding a result derived from that data is now stale — it would return a wrong answer.

The Hard Part — Keeping Caches Coherent

When a write happens at some backend service, we must tell every cache that depends on that data to invalidate (discard) its copy. In a distributed setting, with a deep dynamic graph, finding all those caches and notifying them is expensive — and getting it wrong means clients silently read stale data.

This is the coherence problem, and it's the same fundamental tension we've seen with caches everywhere in this course: caching speeds up reads, but every write now has to do extra work to keep the caches honest.

What People Did Before NuCache

To appreciate NuCache's contribution, look at the unsatisfying options that existed before it:

Prior approachWhat it gives — and its flaw
Backend-only cachingOnly cache right in front of your database, saving the database round-trip. Safe, but it misses the huge wins of caching high up the graph where you'd skip whole subtrees.
Custom invalidation protocolHand-write a bespoke invalidation scheme per application. Can be correct and fast, but it's enormous manual effort for every single app — not automated, not reusable.
TTL (time-to-live) per serviceThe only existing automation: each cache entry expires after a fixed time. Simple, but stale — within the TTL window you can read outdated data, breaking correctness. And how do you even pick the right TTL?
Why TTL Isn't Good Enough

With TTL, suppose you cache a hotel result with a 10-minute expiry. If the hotel's data changes 1 minute later, every read for the next 9 minutes returns stale data. TTL trades correctness for automation — and it forces you to guess an expiry time, which badly complicates reasoning about the consistency of the end-to-end application.

Where We're Headed

NuCache's pitch: automatic invalidation that is also correct (never serves stale data the original system wouldn't have), and that works across any application without per-app custom protocols. The rest of the lecture is how it pulls that off.

Part IV

NuCache: Goals and a Definition of Correctness

We now have the problem fully framed. NuCache is a system that provides automatic cache invalidation for microservice applications. Before any mechanism, we have to nail down two things: what we want from such a system (its goals), and what it even means for it to be correct. The correctness definition is the intellectual heart of this lecture.

What Should NuCache Want?

The class derived NuCache's goals from first principles by asking "what would you demand from a caching layer you'd actually deploy?" Here's the consolidated list:

1

Correctness. The application with caches must behave the same as the original application without them. No new, surprising behaviors.

2

Performance — on both paths. The hot path (cache hit) must be faster. But equally, the cold path (cache miss) must add almost no overhead — because if misses become slow, no one will adopt the system. You want "good hot and good cold."

3

Low overhead. The cache-management machinery itself must not add much cost (CPU, network, latency) to the system.

4

Support real deployments. Must handle sharding and dynamic call graphs — no ahead-of-time knowledge of who calls whom (recall Part II).

5

Plug-and-play. No modifications to the application code or the database. You add NuCache and get benefits.

6

Incrementally deployable. Add caching to just a small part of the graph and still get benefits — no global coordination, because that would be unacceptable in a real deployment.

Connecting Back

Goals 4 and 5 are direct consequences of Parts I and II. "Plug-and-play with no code changes" is achievable because of the sidecar (Part I). "Support dynamic call graphs with no ahead-of-time knowledge" is mandatory because the graph is lossy (Part II). The goals aren't arbitrary — they fall out of the world microservices live in.

Defining Correctness with Execution Traces

Goal #1 says "behaves the same." But a microservice app isn't a single function with one input and one output — it's a swarm of services handling many concurrent requests. So how do we formalize "the same"?

We use the same tool we've used all course for consistency models: execution traces. We monitor the requests flowing between services and between clients and services, and record them as a trace. Then:

Key Concept — Correctness as Trace Containment

Let E_cache be the set of all execution traces observable in the cache-enabled system, and E_orig the set of traces observable in the original (no-cache) system. NuCache is correct if E_cache ⊆ E_orig: every behavior the cached system can exhibit was already a possible behavior of the original system. Caching may never add a new execution that the original couldn't produce.

This is the trace-based, "subset of executions" style of definition we've used for linearizability and serializability. The intuition the class offered — "running with and without NuCache gives the same output" — is exactly right; traces are just how we express "same output" when there's no single input/output pair.

The Reordering Insight — Why NuCache Can Be Fast

Now watch a problem appear, and then watch a beautiful insight dissolve it.

The problem. Front-end has a cache for the hotel service. Request A: upload a review — it goes in, updates the database at the backend, returns to the client, and finishes. Then, after A finishes, Request B: get reviews — it hits the cache and returns the cached reviews.

The Stale-Read Bug

Request B's cached answer might not include Request A's freshly-uploaded review. In the original system, since B started strictly after A finished, B would always see A's review. So the cached system produced an execution the original couldn't — it violates E_cache ⊆ E_orig. To fix it the obvious way, we'd have to invalidate the cache on the critical path of every write (like the Chubby system does), paying that cost on every read or write. That kills the performance benefit.

Here is the key question the professor poses: does the client actually care? Think about what a client can observe. A client only ever sees the execution from its own perspective. It cannot talk to other clients, so it has no way to know that "A finished before B started" in real time.

The Big Idea — Reorder Across Clients

This is the same idea as sequential consistency. Clients do not require real-time ordering across each other — only the order of their own operations matters to them. So NuCache is allowed to reorder operations across different clients (never within a single client). For each cache-enabled trace, if some cross-client reordering of it lies inside E_orig, the trace is acceptable.

Why is this legitimate? Because in the real system there was never a guarantee that A finished before B anyway. A tiny bit of network delay, or different thread scheduling, could have let A and B overlap. The clients couldn't tell. So we can simply ignore real-time ordering across clients — the original system would have had alternative executions where the order came out the other way.

The Payoff: Off the Critical Path

Given this relaxed correctness condition, NuCache can push all cache management off the critical path. Requests are processed with essentially no added latency; invalidation happens asynchronously, in the background, and still satisfies correctness. This is the whole reason NuCache can be both correct and fast.

Is this just sequential consistency? (A subtle "no")

The professor flagged this as the question he got in nearly every job interview. The honest answer is not exactly. Classical consistency conditions (sequential consistency, linearizability) are defined for systems with two roles: clients and servers. But in a microservice environment, a service is also a client of other services — the roles blur. There is no straightforward lifting of sequential consistency into the microservice setting. NuCache therefore uses a custom correctness condition that borrows the insight of sequential consistency (ignore cross-client real-time order) but is not literally sequential consistency. The professor noted this as an open and interesting research direction: what is the right generalized correctness condition for microservices?

Part V

The Diamond Problem and Dependency Tracking

The reordering insight handles ordering across independent clients. But there's a nastier pattern where reordering doesn't save us — where caching can produce an execution the original system simply could not. NuCache needs a separate mechanism for it. This is the diamond pattern.

The Diamond Pattern

Consider a graph where one entry service S1 can call both S2 and S3, and both S2 and S3 can reach the same downstream service S4. Drawn out, the call paths form a diamond. The danger arises within a single request:

Trace it carefully, all inside a single request:

1

S1 calls S2. S2 calls S4 and writes a new value to key K.

2

Control returns up to S1. Now S1 calls S3 to read something from S4 — the very key K.

3

S3 has a cache of S4's data. Because NuCache deliberately keeps invalidation off the critical path, that cache may still hold the old value of K. S3 returns stale data.

Why Reordering Can't Save Us Here

In a real system, if you write K and then read K, the read must see the new value. There's no legal execution where it returns the old one. So this stale read is an execution that was not observable in the original system, even after any reordering. Unlike the cross-client case of Part IV, we cannot reorder our way out — this is a genuine bug we must prevent.

Same Bug as "Mega-Clients"

This is exactly what would go wrong if clients weren't truly independent — if a single "mega-client" controlled several mini-clients and passed information between them. The diamond is the in-graph version of that: information flows from the write (via S2/S4) back up and into the read (via S3), even though S2 and S3 never directly talk. The fix for the diamond also fixes the mega-client scenario.

The Fix: Dependency Tracking

NuCache's solution is elegantly conservative. Every cache item is tagged with all the downstream services that were visited to produce it. And every in-flight request keeps a running record of all the services it has visited so far.

Key Concept — Dependency Tracking

When NuCache saves a cache item at S3, it records "this item depends on S3 and S4" (the services consulted to build it). When a later request reaches S3 and is about to check that cache, it compares: have I already visited any service this cache item depends on? If the request's visited-set intersects the item's dependency-set, NuCache ignores the cache item entirely and re-issues the call.

Walk it through on the diamond. The request goes S1 → S2 → S4 → back to S1, and now heads to S3. Its visited set is {S1, S2, S4}. The cache item at S3 depends on {S3, S4}. They intersect at S4 — so NuCache refuses to use the cached read. It re-runs the call and gets the fresh value. Bug avoided.

A Causality Guarantee — and a Conservative One

This is a form of causality tracking: the request remembers everything it has touched, and if a cache item could have been affected by something the request already did, it declines to trust the cache. The assumption is deliberately conservative: "if I've gone back to S4 in this request, assume I might have modified any key in S4's database — I might have destroyed all of it." So if a cache item depends on S4 and I've visited S4, I won't even look at it. Better to lose a cache hit than serve a stale read.

Crucially, dependency tracking is what lets NuCache keep invalidation off the critical path (Part IV) while still being correct on the diamond: it doesn't need real-time invalidation messages to catch the diamond, because the request's own visited-set already tells it when not to trust a cache.

Part VI

Implementation and Evaluation

We've built up the why and the what. Now the how: the concrete machinery that realizes correct, off-critical-path invalidation, plus how we measure whether it actually works.

The Components: Cache Manager and Wrappers

NuCache adds three kinds of components to an existing application, and modifies none of the original code or database:

Key Concept — NuCache's Three Pieces

The cache — just an off-the-shelf cache (e.g. Redis); nothing custom. Wrappers — a library that wraps every call a service makes to another service or to its database, intercepting it to do bookkeeping. Cache manager — one thread per microservice shard, with a FIFO input message queue. It processes messages one at a time, which gives a total order — and that order is the key to correctness.

The wrappers feed the cache manager: whenever a wrapper intercepts something interesting, it adds a message to the cache manager's FIFO queue. The cache manager, being a single serial thread per shard, processes those messages in a strict total order. (For sharded services there's one cache manager thread per shard, but for this class you can picture a single thread per service.)

The Three Message Types

Everything the cache manager learns arrives as one of three messages on its queue:

MessageWhen the wrapper sends it
start callWhen a request begins processing at a service. Initializes that request's wrapper state.
end callWhen a request finishes. Carries the request's read-set (every key it read + every downstream call it made) and its return value.
invalidate callAfter a request writes a key, to tell the cache manager that key was modified.

Two subtleties the lecture stressed. First, the wrapper accumulates reads into a read-set during the call and only ships the whole set at end call — not one message per read — for performance. Second, downstream calls are added to the read-set too, not just database reads, "to capture the transitive relationships of read" (this is what feeds the dependency tracking of Part V).

Trace: A Cache Miss, Step by Step

Picture five timelines: the front-end service thread, the front-end cache thread, the front-end cache-manager thread, the hotel service thread, and the hotel cache-manager thread. Solid arrows are inter-service (network) communication; dashed arrows are local. A front-end request needs hotel info:

1

Check the cache. Before calling hotel, the front-end wrapper marks "this call happened in this request" and checks its cache. Miss — the item isn't there.

2

Make the real call. The front-end issues the inter-service call to the hotel service.

3

start call. Before hotel begins executing, its wrapper sends start call to the hotel cache manager.

4

Read, accumulate. Hotel reads key ID from its database. The wrapper adds ID to this request's read-set — it does not message the cache manager yet.

5

end call. Hotel finishes. The wrapper sends end call to the hotel cache manager carrying the read-set ({ID}) and the return value (the hotel info). The hotel info also flows back to be (potentially) saved upstream.

6

Decide whether to save. The cache manager checks: between this read and now, was there any concurrent write to ID? If no write happened, the read is provably still valid, so the cache manager tells the upstream (front-end) cache manager to save the value. The front-end cache is set to the hotel info.

7

Next time: hit. The subsequent identical request finds the value in the front-end cache and skips the hotel call entirely.

Why Saving Is Conditional

A student asked: is the cache populated immediately at end call? No. The cache manager first verifies no concurrent write touched the key. A cached read is only valid "until a write happens to this key" — until someone modifies ID, this is "a fantastically good cache result that can stay there." If a write did overlap, NuCache may decline to save. The read-set + return value are carried precisely so the manager can make this decision after the fact.

Trace: Invalidation, Step by Step

Now suppose, sometime after the front-end has happily cached the hotel info, a write lands at the hotel service:

1

Some request writes a new value to key ID at the hotel backend. The front-end doesn't know yet and may keep hitting its cache — which, by the reordering rule of Part IV, is fine from that client's perspective.

2

When the write completes, the wrapper sends invalidate call to the hotel cache manager: "key ID changed."

3

The hotel cache manager looks up every completed request that had read ID and that it had told an upstream cache to save. It keeps a dictionary: for each key, which cached calls depend on it.

4

It tells each relevant upstream cache manager to evict those cache entries. The front-end deletes the stale hotel info from its cache.

5

The next front-end request now misses, so it re-issues the hotel call and fetches fresh data.

Why Not Just Push the New Value Down on Invalidate?

A student asked: when invalidating, why not also send the new value so the upstream cache can update instead of evict-then-refetch? Because a cache in front of a database shares the database's read/write-by-key API — but a cache in front of a service does not. The cached value might be any arbitrary function of the underlying data (e.g. the service returns factorial(5), not 5). To know what the new value translates to upstream, you'd have to re-run the service's arbitrary code — which isn't free, and may call yet more services. So NuCache evicts rather than recomputes.

Why the Cache Manager — Not the Front-End — Saves

It would seem simpler to let the front-end write to its own cache directly after it gets a response. NuCache deliberately routes the save through the cache manager instead. Here's the execution that shows why:

Saves Must Not Overtake Invalidates

Suppose the front-end saves to its cache itself. There's no guarantee the save lands before a concurrent invalidate — the service might be overloaded, delaying the save. If the save arrives after a delete/invalidate, the stale item gets re-inserted and stays cached forever (until some later invalidation). By funneling saves and invalidates through the cache manager's single FIFO queue, NuCache enforces a sequential order so a save can never overtake an invalidate. The cache manager is the ground truth for what's in the upstream cache.

One more imprecision: reads vs. concurrent writes within a call

Because the read-set is only sent at end call (for performance), the cache manager learns that a request read some keys but not the precise time each read happened. Now imagine two backend threads: thread 1 runs a request that starts, reads key K, then ends; thread 2 writes K (an invalidate) somewhere during thread 1's window. Even if thread 1's read actually happened after the write (so it saw the correct, new value), the cache manager cannot tell — from its view the read could have happened anywhere between start and end, possibly before the write. To stay safe, NuCache must conservatively invalidate the whole call and refuse to cache it. This is a deliberate performance-for-precision trade: in the rare case a write overlaps a request, NuCache may miss a cacheable result. (This relies on the local database being linearizable so that "read after write" genuinely sees the write.)

Evaluation — What to Measure, and What They Found

The lecture treated "what should we even measure?" as a skill worth practicing — it's the first question of any systems-research project. The metrics that matter for a system like NuCache:

MetricWhy you care
Cache hit rateHow often caching actually helps.
Latency — read & write, hit & missTwo bifurcations: reads vs. writes, and the hot path (hit) vs. the cold path (miss). You must verify misses stay cheap.
ThroughputHow many requests the whole application can process per unit time.
Overhead of new componentsCache managers and wrappers cost CPU, storage, and network. The single-threaded cache manager can become a bottleneck; does the system scale with more shards?
Invalidation timeEven though invalidation is off the critical path, is it fast (milliseconds) or does it take minutes on a big graph?
CorrectnessValidate that behavior matches the original system.

The class then predicted how NuCache would help three classic graph shapes — the chain, the fan-out, and the fan-in — and matched it to results:

1

Chain → biggest latency win. A cache near the top of a long chain skips many sequential network hops at once, collapsing a long path into one cache hit.

2

Fan-in → biggest throughput win. When many callers funnel into one backend, caching their results means that potential bottleneck service stops being the bottleneck, freeing capacity for the whole system.

3

Low overhead, confirmed. A latency-distribution plot (showing all percentiles, not just the tail) found that with 0% cache hits, NuCache's extra work — checking the queue and the cache — adds less than 1 ms in the best case and only a few ms even at the worst tail. Adoption-safe.

4

Invalidation is fast. About 2 ms per hop; a chain of 5 takes roughly 10 ms to fully invalidate — comfortably off the critical path.

A Reusable Evaluation Skill

Notice the structure: measure the benefit (hit rate, latency/throughput wins by graph shape), measure the cost (overhead on the cold path, cache-manager bottleneck, invalidation time), and validate correctness. Benefit, cost, correctness — that triad is how you evaluate almost any systems addition, not just NuCache.

Putting It All Together

This lecture is a single, tight argument, and every part is load-bearing for the next. Trace the chain of reasoning:

Parts I–II set the stage. Modern applications are microservices — many small, independently deployed, multi-language services talking over the network through sidecars. They bring huge organizational wins (modularity, independent scaling, fault localization) but pay a steep latency tax, because every call crosses serialization, encryption, network, and sidecar. And the call graph is lossy and dynamic: we can't know ahead of time who calls whom. That single fact — no ahead-of-time graph — constrains everything that follows.

Part III names the lever and the obstacle. Caching can skip entire subtrees of the graph (Alibaba: 40% of graphs avoid hundreds of calls), making it the most powerful latency tool. But caches go stale on writes — the coherence problem — and the prior art was unsatisfying: backend-only caching misses the big wins, custom protocols don't generalize, and TTLs serve stale data.

Part IV is the intellectual core. NuCache defines correctness as trace containment (E_cache ⊆ E_orig), then makes the crucial observation — borrowed from sequential consistency — that clients can't observe real-time order across each other. That licenses reordering across clients, which in turn lets NuCache push all cache management off the critical path. This is why it's simultaneously correct and fast.

Part V patches the one hole reordering leaves. The diamond pattern lets a write and a stale cached read of the same key happen within a single request — an execution no reordering can legalize. Dependency tracking fixes it: tag each cache item with the services it depends on, track each request's visited services, and refuse any cache item whose dependencies intersect what the request has already touched.

Part VI makes it real. Wrappers (riding the sidecar) emit start / end / invalidate messages into a per-shard cache manager whose single FIFO queue imposes the total order that keeps saves from overtaking invalidates. Off-the-shelf caches store the data; the cache manager is the ground truth. And the evaluation closes the loop on the goals: sub-millisecond cold-path overhead, big latency wins on chains, big throughput wins on fan-ins, fast off-path invalidation.

The One Thing to Remember

NuCache makes microservice caching automatic and correct by weakening correctness exactly as much as clients can't tell — reordering across clients to push invalidation off the critical path — and then plugging the one remaining leak (the diamond) with conservative dependency tracking. Pick the weakest correctness condition your clients can't distinguish from the strong one, and you buy enormous performance freedom.

Notice how this lecture re-uses the toolkit of the whole course: execution traces and "subset of behaviors" for defining correctness (as with linearizability and serializability); the sequential-consistency insight that real-time cross-client order is unobservable; FIFO total order as the backbone of a protocol (as in so many of our systems); causality tracking via dependency sets; and sharding/replication as the path to scale. NuCache is a synthesis lecture by design — it's the last main system of the class, and it shows the course's ideas working together on a fresh problem.
