Interoperability — Making Systems Talk to Each Other

An interactive guide to Week 8 — from booking a flight across hundreds of airlines, to the $193 million units bug, to the eternal tug-of-war between interoperability and changeability.

Part I

Why Systems Must Talk to Each Other

Every quality attribute you have studied so far in this course — availability, performance, modifiability, memory consumption — can be measured by pointing at a single running system and evaluating it. This lecture introduces the one quality that breaks that rule. It cannot exist in isolation, because it is fundamentally about a relationship between systems. To see why it matters, we start with a story.

A Story: Bob Books a Flight

Bob wants to go on vacation. He opens a flight portal, types in his destination, and immediately sees hundreds of flights from hundreds of different airlines, side by side, ready to compare. He picks one and books it. Later his flight gets cancelled, so he walks up to a travel agent at the airport. The agent finds an alternative — but on a different airline, because Bob's original airline has nothing available. Bob is automatically rebooked onto that competitor's flight, and off he goes. Bob is happy.

This feels mundane. It is actually a small miracle. For Bob to see every airline's flights in one place, and then be rebooked onto a rival company's plane, every one of those airline systems has to be able to exchange meaningful information with every booking system. They were built by different companies, in different countries, in different decades, in different programming languages. They were never designed together. Yet they cooperate.

The Hidden Machinery — GDS

The system that makes this possible is a Global Distribution System (GDS) — one of the oldest IT systems still running. There are three major ones: Amadeus, Travelport, and Sabre. A GDS creates an interface where every airline system and every booking system can talk to each other. (Fun fact from lecture: the invention of GDS is closer in time to the Wright Brothers' first flight than to today.)

One airline famously does not use GDS: Southwest. That is why you cannot see Southwest's prices on Google Flights — Google Flights is itself a GDS consumer. Why Southwest opts out is a clue we will return to in Part IV.

Defining Interoperability

Key Concept — Interoperability

Interoperability is the degree to which two or more systems or components can usefully exchange meaningful information in a particular context. The key words are "meaningful" and "usefully": it is not enough to ship bits and bytes from one place to another — that is a different, lower-level problem. Interoperability requires that the receiving system can correctly interpret the data stream and act on it.

It Takes (at Least) Two

Interoperability is the only quality in this class that does not exist on a single system. You cannot ask "how interoperable is this one program?" the way you ask "how fast is it?" or "how much memory does it use?" It is always evaluated over a pair or a set of systems. Keep this in mind — it is exactly why this quality has unusual tradeoffs.

Why You Should Care

The professor called this "probably one of the most useful lectures in this class," and the reason is blunt: your system is not an island. In the real world, essentially no useful system is built as a self-contained island, and any company that builds such systems will want to hire you to handle exactly this. Interoperability lets you reuse existing services instead of reinventing them:

  • Email — you do not implement a mail server; you send your data to an existing email service that delivers the messages.
  • Single sign-on — let users log in with their Google, Microsoft, or GitHub account instead of building auth from scratch.
  • Payments — you really do not want to build a payment system yourself.
  • Cloud storage, content delivery, analytics, AI services — all consumed through someone else's interface.
  • Sharing to social media, importing exported data, supporting web + desktop + mobile clients on one backend.
Interop Is Harder Than Reusing a Library

A natural question from lecture: how is this different from just reusing a library? When you call a library you package its source code into your own application — you control the version, and you know exactly when it changes. With interoperability you are calling a different runtime system you do not control. The remote API can change at any moment (think of the OpenAI API changing under you). You get less control and a moving target, which makes interop a genuinely harder problem than library reuse.

A Bonus: Better Usability

Users love being able to export data from another system and import it into yours. Letting data flow in is itself interoperability, and it removes friction. This is especially valuable in medical systems, where portable records are slowly becoming an expectation.

Part II

Shared Interfaces & the REST Standard

So GDS makes hundreds of incompatible systems cooperate. How? Rather than handing you the answer, the lecture asked the class to invent a design principle from scratch. The class converged on the same idea the real engineers did, and it is worth walking through their reasoning because it is the heart of the whole topic.

The Design Principle: A Shared Interface

The class's proposals circled around one phrase: an agreed-upon contract. A specific interface that all the systems agree to implement and agree not to break. GDS defines, say, an interface for passenger data and flight booking, and then every airline and every booking system implements it. The procedure has three concrete steps:

1

List all the data that needs to be exchanged. (For flights: routes, dates, prices, passenger details, and so on.)

2

Define the interface format — the shared data format every participant will use.

3

Implement serialization and deserialization on every end, so each system can turn its internal representation into the shared format and back.

The Big Idea — Abstract, Language-Independent Interfaces

The shared interface must be platform- and language-independent. It must expose no implementation details — it has to be so abstract that any system can implement it no matter how it works inside. One participant might be written in Python, another in C++, and a third still running COBOL that hasn't been touched in decades. The format has to support all of them.

REST and Its Rivals

The modern standard for this kind of "meta-interface" is REST — Representational State Transfer. It is a stateless web protocol, meaning the server does not keep track of the client's state between requests. REST organizes everything around four HTTP verbs that map onto the four CRUD operations:

HTTP VerbCRUD ActionMeaning
POSTCreateCreate a new resource
GETReadRead an existing resource
PUTUpdateUpdate an existing resource
DELETEDeleteDelete a resource
Before REST, the Web Was the Wild West

To your generation REST may feel unremarkable, but before it there was no agreement: everyone invented their own URL patterns and their own meanings for HTTP calls. REST got people to agree on how to structure URLs and what each verb means. That agreement is the interoperability win.

A subtle but classic question: GDS exposes a "Get Flights" endpoint that you call with POST, not GET. Why?

Why POST to "Get Flights"?

Because you are not simply reading a fixed resource — you are creating a search request. The query arrives with a large set of parameters (origin, destination, dates...) that the server executes to build a result. Creating something on the server is a POST, even if the thing you ultimately want back is data.

REST is the most common choice but not the only one:

  • RPC (Remote Procedure Calls) — more powerful; can expose functions beyond plain CRUD, including complex calculations. Good for heavy transactional data, often within an organization between cooperating components.
  • SOAP — a stateful, XML-based protocol with built-in security and error handling. Widely used in enterprise; big APIs like GDS often offer both a REST and a SOAP interface.
  • GraphQL — a more modern standard with server-side schema definitions and structural conformance, good for large, complex, integrated data sources. Rising in popularity.

Schemas: Making the Contract Machine-Checkable

Step 2 said "define the interface format." In practice you write that format down as a schema — a JSON or YAML file describing the structure of a valid document. A schema turns your interface from a verbal promise into something a computer can automatically verify a request against.

// JSON schema fragment for an airport code field
{
  "type": "string",
  "pattern": "^[A-Z]{3}$"
}

That regular expression, dragged straight out of CS 35L, reads: ^ anchors the start of the string, [A-Z] means an uppercase letter A–Z, {3} means exactly three of them, and $ anchors the end. So the field must be exactly three uppercase letters and nothing else — perfect for an IATA airport code like LAX. Schemas can also constrain values (a price with minimum: 0, because a negative price makes no sense) and constrain structure (mark departureAirport and price as required).

Best Practice — Publish and Validate Your Schema

When you offer an API, publish your schema (put it on your site) so other developers can download it and check their requests before sending them. And on your side, actually validate incoming requests against the schema. If you skip the negative-price check and blindly interpret it as a balance change, you may end up handing the customer free money. You should check the price against your database anyway — defense in depth.

Part III

Semantics & the $193 Million Bug

The class's design principle — define a shared, language-independent format — sounds complete. It is not. There is a second half that, when forgotten, destroys spacecraft.

The Mars Climate Orbiter

NASA built a probe to study the climate of Mars. JPL (in Pasadena, around the corner from campus) wrote the flight software; a subcontractor, Lockheed Martin, wrote the ground software. The command interface between them was defined with a schema — they followed the class's design principle exactly. And then the spacecraft was lost.

1

Lockheed Martin's ground software sent thruster commands in pound-force (US customary units), its default.

2

NASA's flight software interpreted those same numbers as Newtons (SI units), its default — NASA is an international research organization.

3

The numbers matched the schema perfectly. But the meaning was off by a factor of about 4.45, so the thrust was wildly wrong.

4

The spacecraft was lost, costing the US taxpayer roughly $193 million.

Syntax Without Semantics

The interface had syntactic interoperability — both sides agreed on the shape of the data (a number in a field). What it lacked was semantic interoperability — agreement on what the number means. The bits were exchanged correctly; their meaning was not shared. That gap cost $193 million.

Fixing It: Document the Meaning

So we extend the design principle. The interface specification must include not just the format, the actions, the parameters, and the output (the syntax), but also the semantics: what the data and actions actually mean. As the professor put it: if a field "represents a plant," nobody can tell whether you mean a factory or the leafy thing in your window — unless you say so.

The Big Idea — Document Semantics, Not Just Syntax

Data exchanged between systems must also be interpreted the same way. Spell out, for every field and every action: units (Newtons or pound-force?), whether price includes tax, date format (DD/MM or MM/DD?), the coordinate / reference frame for robotics, a shared dictionary of terms, the side effects of each action, usage restrictions (who is allowed to call it), and error handling (what errors can occur and what each means). Then write integration tests and include concrete examples of correct and incorrect usage.

Here is what richly-documented semantics looks like for a single field of a flight-booking request. Notice how much hides behind an innocent word like "flight number" or "date":

Worked Example — Semantic Specification of a Booking

flightNumber — must be the number of the operating airline, not the selling/code-share airline. (Code-share partners sell the same seat under different numbers; "DL 974" may also appear as another airline's code.)
dateTime — the departure date and time of the departing flight, in the time zone of the departure airport (not arrival time, not your local booking time zone).
seat — the seat as labeled on the airplane (e.g. 16C).
traveler — the full name of the traveling person, in ASCII.
Side effects — the payment method is charged when the seat is marked sold; the booking can still be cancelled within one hour.
Usage restriction — only authorized booking systems may call this.

Guidelines for Interface Documentation

Focus on how elements interact and their externally visible effects — never the implementation details. Exposing internals is dangerous twice over: people will depend on them so you can never change them, and it leaks information (e.g. which database you use) an attacker can exploit. Keep documentation minimal, use-case oriented, and readable: documentation only gets read if reading it is faster than making the error and noticing the difference.

Why This Matters More Than Ever: MCP and AI Agents

The lecture argues semantic documentation is about to become even more important because of MCP — the Model Context Protocol, described as "the REST API of AI agents." With MCP you give an agent a semantic view of the tool calls it can make: here are the calls available, and here is what each one means.

The Audience Changed — From Humans to AI

Historically you wrote semantic documentation for humans to read. With MCP you write it for an AI to read. That raises the bar: the spec must be even more comprehensive and unambiguous, because the model knows fewer things and cannot fill gaps the way an experienced engineer can. And the readability advice still applies — if your documentation is too long to fit in the model's context window, it still won't work. Software engineering didn't change; only the reader did.

Part IV

The Interoperability / Changeability Tradeoff

We now have a recipe for a good interoperable interface. But interoperability has a dark side, and it is the deepest insight of the lecture — the professor noted that even CMU master's students often struggle with it, so if it feels slippery, you are in good company.

Why GDS Cannot Change

Recall requirements change over time (a lesson from the very first lecture). Airlines now have new requirements GDS never anticipated: bookings without assigned seats (Southwest!), priority-boarding add-ons, merging multiple seats into a "sky couch" for premium customers, and above all capturing customer data so they can upsell. American Airlines reportedly lost money on the seats themselves and made money on the loyalty program — but GDS only lets you sell one individual seat and shares no customer data with the airline. So why can't GDS just add these features?

The Big Idea — Once Deployed, an Interface Is Almost Impossible to Change

The GDS interface is implemented by hundreds of airline and booking systems across hundreds of countries, many built decades ago by people who no longer work there, in code nobody fully understands (search Reddit for GDS to find developers complaining exactly this). Once an interoperable interface is in production, every change must be coordinated across all of those systems simultaneously. That is why some airlines like Southwest never joined, and why others are now trying to leave — and why the agent rebooking your flight is still typing into a 30-year-old, unmaintained terminal application.

Designing for Change: Dynamic Add-Ons

The class then tried to redesign GDS to be more changeable. Several ideas surfaced, building toward the answer:

IdeaWhat It Buys YouThe Catch
Split into several specialized APIs grouped by typeFetch only the info you actually wantStill a fixed set of endpoints to agree on
Add optional fields (e.g. priority boarding)Backwards compatible — use it or ignore itYou still must enumerate one fixed list up front, and it can still change
A generic add-on list (the winner)Airlines can offer anything without changing the interfaceHarder to implement; security and consistency risks

The maximally extensible answer is a dynamically defined add-on. Instead of anticipating every product an airline might sell, you define one flexible shape and let airlines populate it:

// A dynamically defined add-on
{
  "price":       // price excluding tax, charged when selected
  "name":        // name shown to the user, UTF-8
  "description": // short text helping the user decide
  "id":          // unique id, starts with flight number, ASCII
}

Now the airline can sell priority boarding, a VIP pass, or "shake hands with the pilot" — anything that fits price, name, description, id — and the interface never has to change. We no longer anticipate what airlines sell, only that whatever they sell fits this format.

Expand: the assumptions still baked into this design

The flexible add-on is powerful but not free of assumptions. It assumes add-ons are independent: that the total is just the sum of individual prices. It assumes price is a plain number, not, say, a percentage increase that depends on something else. And it still does not capture the airline's real prize — rich customer data. Flexibility always smuggles in some fixed assumptions; the skill is choosing which ones.

Why Hasn't GDS Done This?

Three reasons from lecture. (1) Implementation burden — a dynamic list means every UI in hundreds of systems must parse and render arbitrary add-ons; that is the main reason. (2) Security — a free-form description rendered as HTML invites cross-site-scripting attacks. (3) Less interoperability — add-ons defined differently per airline can mislead users, and GDS could be liable for misinterpretation. (4) Economics — GDS providers earn fees regardless, so they have little incentive to do the hard work while airlines remain pressured to stay listed on Google Flights.

Two Faces of Interoperability

This reveals that interoperability has two distinct quality attributes, pulling in opposite directions:

Runtime InteroperabilityDesign-Time Interoperability
How much data can be exchanged and interpreted correctly at runtime.The tradeoff between effort to implement and variability of the interface.
Measured on the running pair of systems.Measured by how widely the interface can realistically be adopted.
A Tradeoff, Not a Synergy

Many quality pairs synergize: faster performance enables higher availability; high modularity enables tighter access control and thus security. Interoperability and changeability are the opposite — they almost always oppose each other. The more interoperable your interface, the more systems must support it, so the harder it is to change. The more changeable (variable) you make it, the harder it is to implement, so fewer systems adopt it — lowering practical interoperability. Your job as a designer is to find the right point on this curve: simple enough that people will implement it, flexible enough to cover current and future needs.

Expand: why would anyone open up their interface at all?

If publishing an interface costs effort, what's the incentive? From lecture: (1) money — charge for access via API keys (the OpenAI model). (2) data — every request tells you what people want, which is enormously valuable. (3) ecosystem / moat — NVIDIA's GPUs are roughly comparable to AMD's on raw FLOPS and bandwidth, but NVIDIA gave away CUDA for free. That initially raised costs with no direct revenue, but the whole AI stack was built on CUDA, creating a moat: everything now runs faster on NVIDIA, and dependents can eventually be monetized. Openness can be a long game.

When You Can't Share One Interface: Adapters

Sometimes two systems were designed independently and use different formats — say one speaks XML and the other JSON, and now a merger or a customer demand forces them to interoperate. You don't rewrite either system; you insert an adapter that translates between the two interfaces and localizes the change to one place.

An adapter sits between two independently-designed systems, translating XML to JSON (and back), confining the incompatibility to one component.

Can an adapter handle semantic interoperability too, not just syntax? Sometimes. If there is a clean mapping — e.g. between units — translation is easy. But if the two sides have genuinely different side effects, there may be no mapping at all, and the adapter cannot fully bridge them.

Real-World Case — Health Care Records

Patients want to move from one hospital to another and bring their electronic medical record. Options: (1) a shared "GDS for medical records" — better usability but hard to get every hospital to agree on one format; (2) peer-to-peer interfaces; (3) manual transfer (printouts, or the old CDs) — which is what most systems actually do today, at the cost of higher labor and re-entry. Semantic interop here leans on standardized terminologies like SNOMED CT (Systematized Nomenclature of Medicine — Clinical Terms) and semantic data models like RDF and the Web Ontology Language (OWL), plus adapters to standardize units and terms.

Part V

Code Review Practices (Carried Over)

The lecture closed by finishing the previous session's code review. These are concrete techniques for making code easier to read and safer — they connect to interoperability through a shared theme: reduce the cognitive effort needed to understand and trust an interface or a function.

Guard Clauses: Reduce Nesting by Returning Early

A common smell is deeply nested if/else chains. You can't always replace them with a switch (the conditions may span multiple attributes), and a dictionary of closures works but is heavier. The simplest fix is a guard clause: handle the exceptional/early cases first and return immediately, so the rest of the function runs at a shallow nesting level.

# Before: deep nesting, lots of elses
def process(person):
    result = None
    if not person.is_dead:
        if person.has_account:
            if person.balance > 0:
                result = charge(person)
    return result

# After: guard clauses exit early, shallow nesting
def process(person):
    if person.is_dead:        return None
    if not person.has_account: return None
    if person.balance <= 0:   return None
    return charge(person)
Nesting Level Is a Readability Metric

Research shows the smaller the nesting level, the easier code is to understand — and nesting level 15+ is "literally the worst," where nobody can follow what's happening. The same guard-clause move applies to error handling: check and log the error cases first, then write the normal path un-nested at the bottom.

Meaningful Abstractions

Given two versions of a function that compute the same thing, the shorter one that names its sub-expressions wins. Capturing parts of the behavior into meaningful variable names lets a reader understand the logic without reading the code — it costs far less working memory and cognitive effort than tracing a dense expression.

Defensive Programming: Pre/Post-Conditions as Assertions

Describe each function's preconditions and postconditions — that is the visible contract you advertise — and then actually check them with assertions. You advertise the contract and automatically verify it at the same time.

Key Concept — Assertions and the Optimized Build

Most languages provide assertions that can be compiled out of production code: Python's python -O (optimized interpreter flag) strips assert statements; GCC has -DNDEBUG. Why strip them? They add runtime overhead. The strategy: in performance-critical contexts (a satellite, a medical device) keep assertions during development to catch bugs, but at the system boundary keep the user-input checks while optimizing away the internal "are my components wired correctly" assertions. The goal is to find your bugs before you ship, then trust the internals and only validate user input in production.

Don't Crash on Bad User Input

For user input, do not use a bare assert that crashes the app. Show a clear error or throw a handled exception and apply safe/graceful degradation — tell the user their input was incorrect rather than killing the program.

The Boy/Girl Scout Rule

"Always leave the campground cleaner than you found it." Applied to code: when you touch a module, leave it a little better — add a useful assertion, reduce the cognitive load. But only if you truly understand the module first.

Refactor on a Safety Net — and Don't Be Sheldon

The worst developer is the one over-confident in their refactoring who breaks the module. Before judging code "bad," find the reason it was written that way — there may be a real one (the Sheldon-Cooper-reviewing-a-paper trap). And before you refactor at all, you need comprehensive tests: you can only safely "juggle the modules" if you have a safety net of tests underneath.

Putting It All Together

This lecture told one continuous story. Part I established that no system is an island — the moment your software must cooperate with another system you can't fully control, you need interoperability: the useful, meaningful exchange of information across a pair or set of systems. Bob's seamless flight rebooking, powered by GDS, was the existence proof.

Part II reverse-engineered how that cooperation is achieved: define a shared, language- and platform-independent interface; list the data, fix the format, and serialize on both ends. REST (and its rivals SOAP, RPC, GraphQL) standardizes the verbs; schemas with regexes and constraints turn the contract into something a machine can verify and publish.

Part III delivered the painful correction: a perfectly valid syntactic contract is not enough. The Mars Climate Orbiter matched its schema and still cost $193 million, because pound-force met Newtons. Interoperability needs semantic documentation — units, side effects, restrictions, errors, examples — and that need is intensifying as MCP makes AI agents the new readers of our specs.

Part IV exposed the catch: the very thing that makes an interface interoperable — wide adoption — is what makes it nearly impossible to change. GDS is frozen because hundreds of systems depend on it, which is why Southwest opted out and others are fleeing. Designing for change (dynamic add-ons) trades implementation effort and security for flexibility. Interoperability and changeability are genuinely opposed, and good design is choosing the right point between them; adapters and shared ontologies (SNOMED CT, RDF, OWL) help bridge the gaps you can't avoid.

Part V brought it home to the code itself: guard clauses, meaningful abstractions, contract assertions, and the Boy Scout rule all serve the same master as good interface design — minimizing the cognitive effort and risk for whoever has to read and trust your work next, whether that reader is a teammate, a partner company's system, or an AI agent.

The One Thing to Remember

Exchanging bytes is easy; exchanging meaning is hard. Always document the semantics of your interface — not just its syntax — or you risk the $193 million mistake. (The exit-ticket password says it all: semantics.)

What's Next

You'll apply this directly on the project: you'll be asked to document parts of your API — including the semantic view, not just the format — because that is where the real failures hide. Watch for interoperability resurfacing when the course revisits architecture and the upcoming Scrum lecture, and keep the interoperability-vs-changeability tradeoff in mind whenever you weigh quality attributes against each other.
