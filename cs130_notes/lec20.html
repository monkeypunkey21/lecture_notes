Designing Monopoly: An End-to-End Case Study

The Week 10 exam-prep lecture — walking the full software design cycle on a UCLA-themed online Monopoly, from requirements through architecture, UML, design patterns, and symbolic-execution testing.

This lecture was deliberately built as a live system design interview — the exact exercise you will face when applying for software jobs. The professor took one concrete problem, build a multiplayer online Monopoly, and ran it through every aspect of the course: turning stakeholder needs into requirements, choosing an architecture, modeling it in UML, picking design patterns, and planning tests. Along the way the class competed against an AI (Claude Opus 4.8) that had been asked to do the same thing.

Because the final exam is itself a case study of this shape, this page is organized as the design cycle itself. Each part is a stage in that pipeline, and the connections between stages — how a requirement forces an architecture, how an architecture forces a pattern — are the real lesson. We teach every term from scratch, so no prior software-engineering vocabulary is assumed.

The Through-Line of This Lecture

Software design is a chain of decisions driven by requirements. You identify what the system must do and how well it must do it, then you generate multiple design options for each sub-problem and choose between them by how each one scores against those requirements. Every part of this lecture is one link in that chain.

Part I

Requirements — Saying What, Not How

Why Start With Requirements?

Before you write a single line of code — or even decide on a database or a server — you have to know what the system is supposed to do. That sounds obvious, but the discipline of requirements engineering is harder than it looks, because the natural temptation is to immediately start describing how you would build it. Requirements are the first stage of the cycle precisely because every later decision (architecture, patterns, tests) is justified by them. If you skip them, you have nothing to design toward.

Key Concept — Requirement

A requirement is a statement of what the system must do or how well it must do it, written so that it leaves the development team maximum freedom in how to implement it. Requirements describe the problem; design decisions describe the solution.

The professor pushed on this distinction repeatedly. When a student suggested "the system should keep track of the game state," he flagged it as drifting into implementation territory: who keeps track of state is a design decision. Why does this matter? Because, as he put it, the developer "will probably be more experienced... in the right ways to implement than the people specifying the requirements." Over-specifying the requirement ties the implementer's hands for no reason.

Common Mistake — Smuggling In Implementation

"The game keeps track of state" and "each player should have a..." both leak how into a what. The fix is to describe the requirement from the perspective of someone using the system: instead of saying who stores the balance, say "the game shall display the balance of each player." Display is observable behavior; storage is an implementation choice.

Functional Requirements

A functional requirement states a specific behavior the system must exhibit — an input/output relationship a user can observe. For our Monopoly board, the class brainstormed the things the UI must show. Notice that all of them are phrased as observable display behaviors, not storage mechanisms:

Key Concept — Functional Requirement

A functional requirement describes what the system does: a concrete, testable behavior. "The game shall display the board layout" is functional. Contrast this with quality attributes (Part II), which describe how well it does those things.

The "Shall" Form

Requirements are conventionally written as "shall" statements. This is not bureaucratic decoration — the fixed grammar forces precision and makes each requirement individually checkable. The class converged on a single functional requirement covering the board display:

# Functional requirement (shall statement)
The game shall display:
  • the board layout (which fields exist, what event each triggers)
  • the balance of each player
  • the location of each player
  • who owns each field
  • how many houses are on each field
  • whose turn it is
Why Phrase It As "Display"

Every item became a display requirement. By describing the system "from the perspective of somebody actually using the game," you specify the observable behavior and stay silent on internals — exactly the freedom-preserving move from the concept box above. This is the single most repeated piece of advice in the requirements segment.

One more lens the professor applied: he asked specifically for requirements that are architecturally significant — ones that "can really impact our architecture." Not every true statement about the system deserves to be a tracked requirement; the ones worth your attention early are those that will constrain how you build it. This idea returns in Part III, where a single interoperability requirement rules out an entire architectural option.

User Stories vs. Acceptance Criteria

Two other requirement formats came up, and the professor was careful to keep them separate because students kept blending them.

Key Concept — User Story

A user story states a need from the point of view of a specific user: "As a <concrete role>, I want <goal> so that <benefit>." It has three parts: who, what, and — crucially — the "so that" giving the motivation.

Key Concept — Acceptance Criteria

Acceptance criteria describe how you can test that the system conforms to a requirement. They use "the player can see..." (a checkable assertion), not "the player wants to see..." (a desire). Acceptance criteria have no "so that" clause — they are about verifiability, not motivation.

The cleanest demonstration was when a student offered "given that the game has started, the player wants to see the board." The professor diagnosed it as a mix of two formats and split it into the two clean versions:

1

As a user story: "As a player, I want to see the board (including its layout) so that I know what consequences will occur." Note the concrete role, the want, and the "so that."

2

As acceptance criteria: "Given that the game has started, the player can see the board including its layout." Note the precondition ("given that..."), "can see" instead of "wants to see," and no "so that."

Don't Mix The Formats

The recurring error all lecture was stapling a "so that" benefit onto an acceptance criterion, or phrasing a testable check as a desire. Keep them in separate sections of your design doc. A later example showed the professor sharpening the role from a vague "user" to a "game master" — the more concrete the user, the better the story.

Part II

Quality Attributes — How Well, Not Just What

Functional requirements say what the system does. But two systems can do the same things and still be wildly different: one is fast, secure, and accessible; the other is none of those. Quality attributes (also called non-functional requirements) capture the how well. They are usually the architecturally significant requirements — the ones that actually decide your structure — which is why the professor said our quality attributes turned out to be "much more useful" than Claude's for shaping the design.

Key Concept — Quality Attribute

A quality attribute is a measurable property of how well the system performs its functions — performance, interoperability, modifiability, usability, accessibility, security, robustness, and so on. These are the requirements most likely to drive your architecture.

The Anatomy of a Quality-Attribute Scenario

You cannot just write "the system should be fast." A quality attribute is only useful if it is measurable, and the professor gave an explicit recipe for making it so: describe a concrete scenario, then attach a measure.

The QA Recipe — Scenario + Measure

To specify any quality attribute: (1) describe a concrete scenario (what happens, under what stimulus), then (2) attach a precise, independently-measurable metric. In this course we call them the stimulus and the measure; in the formal six-part framework these map to the stimulus and the response measure.

The Measure Must Be Independent

When a student proposed measuring modifiability by "testability," the professor rejected it: the two are correlated, so the metric becomes circular — you'd be defining one quality attribute in terms of another. Pick a metric you can measure on its own, like developer-hours.

Interoperability

Because this is a networked multiplayer game, the most important quality attribute is interoperability — different clients exchanging data and agreeing on what it means. A key insight: interoperability "is not just about the sending. It's also about the interpretation of the data." Two clients can successfully transmit bytes and still disagree about the game state.

A student suggested a 50% threshold as the measure. The professor pushed back hard — would you be happy if only half your fields showed up correctly for your opponent? The bar belongs much higher:

# Interoperability quality-attribute scenario
Scenario: Data exchanged between any two Monopoly clients.
Measure:  The exchanged data shall be interpreted correctly
          with an accuracy of 100%.
# "There's no excuse for Monopoly not to have correctly
#  interpretable data."

Modifiability & Extensibility

We want "a gazillion different themes" — UCLA-themed, Christmas-themed, and so on. The ability to change a system is called modifiability (sometimes "changeability").

"All Software Can Be Changed" — So Be Precise

You can't just say "the game rules shall be modifiable" — by definition, all software can be changed; that's why it's software. A good modifiability requirement names exactly which aspect may change and attaches a concrete metric. Some rules should stay fixed; only specified aspects are changeable.

The class enumerated the changeable aspects — graphics shaders for seasonality, the properties/fields, the number of dice, the chance and community-chest cards, and the currency types — and then chose a measure in developer time:

# Modifiability scenario (developer-time measure)
Changing the set of properties / number of dice / cards
shall take a developer less than 1 week.
Runtime Modifiability Is Stronger

The professor singled out Claude's wording — "new chance cards can be added without recoding the game" — as an especially good modifiability requirement. Why? It is a runtime modifiability attribute: no code changes at all. This stronger form will, in Part V, eliminate every factory pattern as a valid option. A requirement that sounds like a small wording choice quietly decides your design.

Extensibility — A Special Case

Late in the lecture the professor distinguished a sibling term. Extensibility is the special kind of modifiability where you only add something new, without changing or removing anything existing. General modifiability covers all three — adding, removing, or replacing. So:

Extensibility is a subset of modifiability: extension adds without altering what exists.

Usability & Accessibility

Two more architecturally significant attributes surfaced. Usability is specified with the same scenario+measure recipe:

# Usability scenario (learnability / discoverability)
A new user shall be able to learn how to play a complete
game successfully within less than 5 minutes of tutorial.

This is really a discoverability requirement — how easily a newcomer finds out how the system works. (Note: the professor caught himself mislabeling a network-latency property as accessibility; "the system continues to operate under latency" is actually robustness, not accessibility. The categories matter.)

When a Standard Becomes a Legal Constraint

The best discussion of the lecture: is "follow accessibility standard WCAG 2.1 AA" a design decision or a requirement? The professor's answer — it depends on whether you have a choice.

Key Concept — Legal Constraint

A legal constraint is a requirement you do not get to negotiate, because the law imposes it. For commercial / public-serving systems, accessibility standards (WCAG 2.1 AA) are legally mandated — if you don't comply, "you can be fined, and you will be fined." Because you have no freedom to choose otherwise, it is a requirement, not a design decision.

Choice Is What Separates Requirement From Design

This is the deep idea behind the whole requirements/design split. If meeting WCAG were optional — say you chose AA to please your target audience when AA wasn't required — it would be a design decision. The moment the law removes your choice, the same line of text becomes a constraint you design around. "We do not have the choice to affect them, so we consider them requirements." This is exactly why UCLA is updating its systems to comply.

Part III

Architecture — Choosing Among Design Options

With architecturally significant requirements in hand, the next stage is design: deciding the high-level structure that satisfies those requirements. The professor framed the whole activity as generating multiple alternatives per sub-problem and then evaluating each against the requirements — never picking the first idea.

Requirement vs. Design Decision — The Exam-Critical Distinction

The professor explicitly flagged this contrast as something to "remember for the exam," using examples Claude generated:

Requirement (the "what")Design Decision (the "how")
"Players see balance updates as soon as money changes hands." (functional + quality mix)"Use a publish-subscribe event bus to propagate updates."
"New chance cards can be added without recoding the game." (a clean runtime modifiability requirement)"Store card definitions in a database loaded at startup."
Describes observable behavior / quality, leaving freedom.Commits to a specific mechanism or structure.
Not Everything Is A Requirement

The professor rejected Claude's "the game state is consistent for every connected player" as a functional requirement — "it's just not, really." Be skeptical: a true-sounding sentence isn't automatically a well-formed requirement. The good ones are concrete, testable, and architecturally meaningful.

Generating Design Options Per Sub-Problem

The first sub-problem: how do clients exchange data? The class generated two alternatives, then weighed each against the requirements rather than committing immediately.

1

Option 1 — Publish/Subscribe. When a player moves, the event is published and all subscribers are automatically notified. Pub/sub is generally good for modifiability — it's "scalable in terms of events" if you want to add new event types.

2

Test it against our requirements. Pub/sub helps general modifiability — but is it the kind of modifiability we need? Our modifiability need is about cards and properties, not adding event types. The professor's verdict: pub/sub's strength "doesn't really apply to the situation." A pattern being good in the abstract doesn't make it good here.

3

Option 2 — Peer-to-peer (decentralized). No central server; clients talk directly. This separates the data-exchange sub-problem from the networking sub-problem, which is still unsolved at this point.

"Generally Good" ≠ "Good Here"

The recurring move: an option has generic advantages (pub/sub is modular and scalable), but you must check whether those advantages line up with your specific requirements. The whole point of having written precise quality attributes in Part II is so you can answer this question instead of cargo-culting a popular pattern.

Client-Server vs. Peer-to-Peer

The networking sub-problem produced a genuine trade-off the professor called "a really interesting situation":

Client-ServerPeer-to-Peer
Con: you must provide and run your own server → operating cost goes up.Pro: no server to pay for.
Pro: can be more scalable/performant — the server fans out messages.Con: every client must send messages to every other client, putting heavy drain on each client's local network.
Pro: a central authority — you can actually monetize the game.Con: hard to monetize — peers could just send each other a "you bought it" message and the seller disappears.

That last row decided it. The business goal — "we want to code this and become rich" — means you need a server you control so people can't bypass payment. The class chose client-server. Note how a business requirement (monetization) settled an architectural decision that performance alone left ambiguous.

When a Decision Gets Blocked: Ask the Scrum Master

Choosing client-server surfaced a requirement nobody had considered: what is our operating budget? You can't decide whether you can afford your own server without knowing the budget. This is a teaching moment about Agile process.

Key Concept — Scrum Master

In an Agile/Scrum team, when a decision is blocked by something outside the team's authority (here, the budget), you escalate to the Scrum Master, whose job includes removing impediments. The Scrum Master goes and figures out whether the budget exists — and in our story, came back with: yes, there's enough operating cost budgeted for a server.

Design Surfaces Hidden Requirements

This is a real pattern: making a design decision can reveal a requirement you never elicited (the budget). Requirements and design aren't strictly sequential — they iterate. When a design decision exposes an unconsidered requirement, that's normal; you go get the answer and continue.

Part IV

UML Modeling — Drawing the Design

Once you've made high-level architectural decisions, the next interview-standard step is to write a design doc and model the system. In this course you model with UML (Unified Modeling Language). The professor noted the sequence diagram is "the most important one for this" problem, but the class also produced component and class diagrams.

Component Diagrams — And What a Component Is

A student-drawn component diagram showed a client app, game server, game database, event bus, and game engine, wired with different connector types — WebSockets for live updates, REST + JSON for request/response, SQL between engine and database. Using different connectors for different purposes earned praise. But it also exposed a precise definition.

Key Concept — Component

A component is an independently deployable runtime unit, like a process. By this definition the "game server" is not a component — it's a device on which the other components (game engine, database process, event bus) are deployed. The database, running in a separate process, is its own component.

A corrected component diagram. The "game server" is a deployment device, not a component; connectors are labeled by protocol.
An Event Bus Needs Subscribers, Too

The diagram showed publishing to the event bus, but nothing subscribing from it — "just publishing would not be enough." A pub/sub connection only delivers value when something is on the receiving end. Also: a separate process for the database must show how it communicates (which protocol), not just that an arrow exists.

Where Claude went wrong on the component diagram

The professor critiqued Claude's component diagram as not following UML standards. The main faults:

  • It drew the database inside a process rather than as its own separately-running component.
  • It didn't specify how components communicate — over the network? which protocol/wire format?
  • It mislabeled relationships with stereotypes like <<observer>>, <<update>>, <<provide>> — these are not valid component-diagram views ("absolutely terrible," in his words). Model/controller/observer stereotypes belong to other diagram types, not a component diagram.

Sequence Diagrams — Modeling Interactions Over Time

A sequence diagram shows the messages exchanged between participants over time, and it was the right tool to capture a subtle accessibility-driven design: supporting multiple simultaneous UIs on the same game logic.

One Game Logic, Many UIs (Dynamic Registration)

For accessibility, the same game logic must drive a 2D visual UI for sighted players and a text UI for blind players. The professor's solution from a real prior project: each UI dynamically registers itself with the game logic. Once registered, whenever the balance changes, every registered UI receives an "balance changed" update. The sequence diagram captures exactly this register-then-receive-updates protocol — which is the Observer pattern in action.

Part V

Creational Design Patterns — How Objects Get Built

Now a detailed-design sub-problem: the game aggregates many fields, chance cards, and community-chest cards. How are those objects created? The professor first cleared up a vocabulary confusion, then walked the options.

Key Concept — Object vs. Interaction

An object is an instance of a class — a programming-language construct. "The user creates the object" confuses an interaction (the user uses a UI) with a software design decision (which code instantiates the class). The design question is: which code is responsible for creating these objects?

Key Concept — Design Pattern

A design pattern is a common, well-accepted solution to a recurring problem. You don't always need one — but for recurring problems like "how do we construct this object structure," a catalog of patterns (factory method, abstract factory, builder, ...) gives you proven options.

The Object-Creation Problem

The class surfaced several options for creating the cards and fields:

1

The Game creates them directly — the game object is responsible for instantiating its own community-chest cards, chance cards, and fields.

2

Load them from a database — read card/field definitions from storage at startup. (This solves a different sub-problem; see below.)

3

A Factory / Abstract Factory creates them — a dedicated creational pattern encapsulates the instantiation logic.

4

A Builder constructs them from a spec — "if you take a spec and then you build it, it's a builder."

Factory vs. Builder — Evaluated Against the Requirement

Here the lecture pays off the modifiability requirement from Part II. Recall the strong, runtime form: a non-technical game master must add new cards/properties without changing any code. The professor walked the class to the key insight:

The Insight — Factories Require Code Changes

If a non-technical user must add new cards without touching code, then none of the factory patterns apply — adding a new product to a factory means editing the factory's code. The factory's pro is flexibility; its con is that "code changes are necessary." A builder, by contrast, takes a spec as input and constructs the object from it — no code changes needed to support a new card the game master designs.

OptionProCon (vs. our requirement)
Abstract Factory / Factory MethodFlexible object creationAdding a new card requires code changes — fails the "non-technical user, no recode" requirement
BuilderTakes a spec as input → no code changes to add a new card(Satisfies the runtime-modifiability requirement)
Where Even Claude Failed

The professor's punchline: Claude picked a factory for card creation — even though Claude had itself written the requirement that adding cards "should not require changes to the code." A factory contradicts that requirement. The lesson: always trace your pattern choice back to the actual requirement; don't reach for the famous pattern reflexively.

Orthogonal Sub-Problems

A crucial clarification: "load from a database" and "use a builder" are not competing options — they answer different questions.

Decompose Into Orthogonal Sub-Problems

Storage (disk? database?) and construction (factory? builder?) are orthogonal sub-problems — they're complementary, not alternatives. "Who is responsible for creating the object" is yet a third axis (a responsibility-assignment question). The disciplined process: split the design into independent sub-problems, list the options within each, evaluate those options against your requirements, then decide each axis separately.

This Mirrors The Whole Lecture

This is the same move as Part III, where data-exchange and networking were separate sub-problems. Recognizing that two "options" actually answer different questions is half the skill of system design — it prevents false either/or choices and lets each axis be optimized on its own.

Part VI

Testing — Coverage and Symbolic Execution

The final stage of the cycle is planning how to test the system. The class chose to focus on the functional requirement of game logic. Two questions organize good testing: what makes a test good? and how thoroughly are we exercising the code?

Two Properties of Good Tests

Key Concept — Two Properties of a Good Test

(1) Specification-dependent, not implementation-dependent: a good test checks the specification (what the system should do), not the particular implementation — so refactoring doesn't break it. (2) High coverage: the test suite should exercise as much of the code's behavior as possible.

Coverage Criteria

Coverage metrics quantify how much of a program your tests exercise. The professor was emphatic about terminology and listed a hierarchy:

"Line Coverage" Does Not Exist In This Course

Say statement coverage, not "line coverage." They are different: one source line can hold several statements (or none), so there's no 1-to-1 mapping. On an exam, "line coverage" is the wrong term.

The coverage hierarchy. Achieving a stronger criterion implies the weaker ones beneath it.
CriterionWhat it requires
Statement coverageEvery statement is executed at least once.
Branch coverageFor every if, both the true and false branches are taken.
Path coverageEvery feasible combination of branches (every path through the program) is exercised — minus paths that are infeasible.

Symbolic Execution — Finding Inputs for Path Coverage

How do you determine whether a path is feasible, and what inputs would drive the program down it? You draw a control-flow graph and run symbolic execution.

Key Concept — Symbolic Execution

Symbolic execution walks the program with symbolic (not concrete) values, tracking two things at each statement: the path condition — the set of conditions that must hold for control to reach this statement — and the effect the statement has on program state (e.g. an assignment updates a variable's symbolic value). Its purpose: find concrete test inputs that drive execution down each path, so you can achieve 100% path coverage.

The professor traced this worked example live:

if (x > 0) {        // branch F1
    y++;
}
if (y > 5) {        // branch F2
    x++;
}
1

Entry. The first if is always reached, so its path condition is simply true. Evaluating the condition has no effect on state.

2

Inside the first if (F1 true). Path condition: x > 0. Effect of y++: y = y + 1. The program state now records the new symbolic value of y.

3

Reaching the second if. This statement is reached from two incoming paths: F1-true (x > 0 ∧ y = y+1) and F1-false (x ≤ 0, the negation of the true branch's condition, with no effect since y++ never ran). Its path condition is the OR of these two.

4

Inside the second if (F2 true), on top of F1-true. Path condition: x > 0 ∧ (y + 1) > 5 — note y+1, because y was already incremented on this path. Effect of x++: x' = x + 1.

5

Inside the second if (F2 true), on top of F1-false. Here y was never incremented, so the path condition is x ≤ 0 ∧ y > 5, with effect x' = x + 1. Each distinct path yields a distinct path condition — and solving each one gives a concrete test input.

Why The "y + 1" Matters

The subtlety the professor drilled: on the path where the first branch ran, y has already changed before the second condition is evaluated. So the second condition is y + 1 > 5, not y > 5. Symbolic execution carries the accumulated effects forward — that's what lets it compute exactly which inputs reach each statement.

Test Adequacy — How Much Coverage Is Enough?

Path Coverage ≠ Exhaustive Testing

Reaching 100% path coverage is usually feasible: paths are finite unless you have unbounded recursion or reflection — plain if/else/loops keep the path set finite (infeasible paths are simply discarded). But exhaustive testing — covering every single execution case — is impossible, because the space of inputs/states is effectively unbounded. Don't conflate the two.

Choose Adequacy By Requirements

How much coverage is "enough" is a judgment call set by your test adequacy criteria, which depend on your requirements. A spacecraft system warrants 100% path coverage; a digital Monopoly does not — you only need the level of confidence appropriate to the use case. Once again, requirements drive the decision — the same theme as every other part.

In Part III those requirements selected an architecture: we generated pub/sub vs. peer-to-peer, client-server vs. peer-to-peer, and let the requirements (and a budget unblocked by the Scrum Master) decide — landing on client-server for monetization.

In Part IV we modeled the design in UML — components as independently deployable units, and a sequence diagram capturing the multi-UI registration that the accessibility requirement demanded.

5

In Part V the runtime-modifiability requirement reached all the way down into detailed design: it eliminated factories and selected a builder, while we kept storage and construction as orthogonal sub-problems.

6

In Part VI we tested, choosing an adequacy criterion (again, by requirements) and using symbolic execution to compute the path conditions that yield inputs for path coverage.

The Single Skill The Exam Tests

Notice the same diagnostic recurs everywhere: distinguish the "what" from the "how." Requirements vs. design decisions, specification vs. implementation in tests, observable behavior vs. internal storage, a choice vs. a legal constraint. The professor even beat the AI on exactly this — Claude wrote "no recoding" as a requirement, then picked a factory that contradicts it. Holding "what" and "how" apart, and tracing every design choice back to a requirement, is the heart of the course.
