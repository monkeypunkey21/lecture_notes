Modeling with UML: Class & Sequence Diagrams

An interactive guide to Week 2 — how to capture the structure and the behavior of a software system in pictures, and how to read those pictures back into code.

Part I

Why Model? Interface vs. Implementation

Why Are We Learning This?

Source code is precise, but it is also enormous and overwhelming. A real system can be hundreds of thousands of lines long. If a new teammate asks "how does this system fit together?", handing them the source code is almost useless — they would spend hours reading before they understood even one collaboration between two classes.

Modeling is the discipline of drawing a system at a higher level of abstraction than the code. A good diagram throws away the noise and keeps only what matters, so a reader can grasp in a minute what would take half an hour to dig out of the source. This lecture covers the two most useful diagrams from UML (the Unified Modeling Language): the class diagram, which shows the static structure of a system, and the sequence diagram, which shows how objects interact over time.

The Big Idea

A diagram is an abstraction of the code, not a transcript of it. Its whole value comes from what it leaves out. Your goal is never to "model everything" — it is to model the important aspects so a human can reason about the system.

Interface vs. Implementation

Before any notation, the lecture draws one foundational distinction. Every class has two faces:

Key Concept — Interface vs. Implementation

The interface is the means by which a class is accessed by other classes — its function names, argument types, and return types. It is the API: what you want other developers to know. The implementation is how the class actually fulfils the contract those signatures promise. It is what you want to hide, so other developers cannot build on details that might change.

This split is the reason class diagrams are useful. A class diagram deliberately shows you only the interface of each class. It tells developers what they should care about without exposing what they should not. (Later in the course this becomes the information hiding principle.)

Why This Matters Later

Because the interface is meant to stay fixed while the implementation changes underneath, a class diagram is supposed to capture concepts that persist, not implementation details that churn. Keep this in mind when we get to relationships — some of them describe code, others describe intent.

Subclassing vs. Subtyping

A second distinction that trips people up: subclassing and subtyping look similar but mean different things.

SubclassingSubtyping
Inherits behavior (actual implementation) from another class.Inherits only the interface — the shape — with no behavior.
Java: class A extends BJava: class A implements B
C++: class A : public BC++: a class with only abstract (pure virtual) methods.

Java is praised here precisely because it separates the two with dedicated keywords: extends for inheriting behavior, implements for conforming to an interface. In a language like Python, with no static typing, there is no separate interface keyword; in C++ you simulate an interface by writing a class whose methods are all abstract. Java is the language used for most of this course, so its vocabulary maps directly onto the UML notation we are about to learn.

Design Guidance

For flexible design you should prefer subtyping over subclassing — favour implementing interfaces over inheriting behaviour. Inheriting behaviour couples you tightly to a parent's implementation; conforming to an interface leaves you free to change how you fulfil it.

Check Your Understanding
In Java, class FuelTank implements Refuelable establishes which relationship?
A
Subclassing — FuelTank inherits Refuelable's behavior.
B
Subtyping — FuelTank conforms to an interface without inheriting behavior.
C
Composition — FuelTank is made up of Refuelable parts.
D
Nothing — implements has no meaning in UML.
Correct: B. The implements keyword conforms a class to an interface — it inherits the shape (method signatures) but no behavior. That is subtyping. extends (A) would be subclassing, which inherits actual behavior. And UML absolutely represents implements — it is drawn as interface realization (a dashed line with a hollow triangle).
Part II

Anatomy of a Class Box

Now the notation. In a class diagram, a class is drawn as a rectangle. Any rectangle in a class diagram is, by default, a class. But the rectangle is divided into compartments, and the dividers carry meaning.

The Three Compartments

The three compartments: name (top), attributes (middle), methods (bottom). The order is fixed.
1

Top — the name. Always present. By convention you name classes with nouns (write Transaction, not Transact) because a class represents a thing, not an action.

2

Middle — attributes. The member variables of the class. Optional.

3

Bottom — methods. The functions defined on the class. Optional. A method is written as name(params): returnType; you may omit void but it can be clearer to keep it.

Don't Swap the Compartments

The attribute compartment always sits above the method compartment. The lower compartments are optional, but you may never reorder them. Attributes on top, methods on the bottom — every time.

Visibility Markers

Each attribute and method carries a visibility marker — a single symbol that says who is allowed to access it. These come straight from Java's access modifiers.

SymbolVisibilityWho can access
+publicAny class.
-privateOnly the same class.
#protectedThe same class and its subclasses.
~packageAnything inside the same Java package (Java's default visibility).
The Default-Visibility Gotcha

In Java, if you write void m() with no modifier, it is package-private, which maps to the tilde ~ in UML — not public. Don't be confused by missing keywords in Java source: absence of a keyword is itself a visibility level.

Multiplicity, Constraints & Generics

A few more annotations live on the box:

  • Multiplicity — a number in the top-right corner says how many instances of the class can exist (e.g. 1 for a singleton).
  • Constraints — written in curly braces, e.g. {width > 0}. These can express preconditions (assumed true before a method runs), postconditions (guaranteed after), and invariants (always true).
  • Parameter direction — out before a parameter marks it as a return-by-reference value, as in classic C-style out-parameters.
  • Templates / generics — a dashed rectangle in the corner names a type parameter (e.g. ElementType) that gets bound to a concrete type, modelling FruitList extends ArrayList<Fruit>.
Why Pre/Postconditions Are Worth the Effort

The professor flagged this as increasingly important in the age of AI agents: when you describe an operation in, say, an MCP server's documentation, an agent cannot know your preconditions unless you state them. Carefully documenting what an operation assumes and guarantees is becoming a core design skill, not a formality.

Check Your Understanding
A Java method is declared as boolean validate(byte[] data) with no access modifier. How is its visibility shown in UML?
A

B
- private — no modifier means hidden.
C

D

Correct: C. In Java, omitting the access modifier gives package visibility — accessible only within the same package — which UML draws with the tilde ~. It is a common mistake to assume "no keyword" means public (A); it does not.
Part III

The Six Relationships

Boxes are only half the story. The real expressive power of class diagrams is in the lines between boxes. There are six relationship types, and they form a layered hierarchy of increasing strength. Some correspond directly to programming-language constructs; others add semantic intent the code alone can't express.

Key Concept — The Strength Hierarchy

The relationships nest: composition is a stronger kind of aggregation, which is a stronger kind of association. Each inherits the properties of the one below it. "Pure association" or "pure aggregation" means just that level and nothing stronger.

1. Dependency — "A uses B"

A dependency is the weakest relationship: class A uses class B, but only temporarily. B might appear as a method argument, a local variable, or a thrown exception — it is on the stack for the duration of a call, not stored as a field. A cannot compile or run without B, but A does not hold onto B.

Notation: a dashed line (with an open arrowhead) from A to B.

Javaclass Train {
  void addStop(ButtonPressedEvent e) { // only used here
    ...
  }
}

Here Train depends on ButtonPressedEvent: it appears only as an argument, so the two interact temporarily — yet Train won't compile without the class being present.

2. Association — "A has a long-term reference to B"

An association is stronger: it is a structural, long-term relationship. Usually A stores a reference to B as an instance member, so B exists across A's whole lifecycle. A station that always knows its button has an association with that button.

The Subtle Case: Unnavigable Association

The precise definition of association is "a long-term structural relationship" — it does not require either side to hold a reference. An account and its clear-text password are in a long-term relationship even though, for security, neither stores a pointer to the other. So a reference is a strong hint of association, but its absence does not rule association out.

3. Aggregation — "A is the whole, B is a part" (loosely)

An aggregation adds a whole–part hierarchy on top of association. One object is the whole, the other is a part — but the part can still exist independently. A classroom has chairs: the chairs are part of the classroom, yet a chair still exists if you remove it from the room. Other examples: a department and its professors, a roster and its students.

Notation: a solid line with a hollow diamond on the whole end.

4. Composition — "B cannot exist without A"

Composition is the strongest whole–part relationship: the part's lifetime is bound to the whole. When the whole is destroyed, the parts are destroyed with it. A house is made of rooms — destroy the house and the rooms go too. A person has a head; the head doesn't meaningfully exist apart from the person.

Notation: a solid line with a filled diamond on the whole end.

Aggregation (hollow diamond)Composition (filled diamond)
Whole–part, but parts survive on their own.Whole–part, and parts die with the whole.
Classroom & chairs; department & professors.House & rooms; folder & files; motherboard & its private inner class.
No destructor coupling.In C++, destroying the whole calls the parts' destructors.
The Filesystem Example

Files in a folder are a clean composition: you can move a file between folders, but if you delete the folder, the files inside are gone. Movability does not break composition — what matters is the lifetime coupling on destruction.

5. Generalization & 6. Realization

The last two are the inheritance relationships — and they pair exactly with the subclassing/subtyping split from Part I.

Left: generalization — solid line, hollow triangle (class extends class). Right: realization — dashed line, hollow triangle (class implements interface). The arrow always points to the parent.

Generalization is inheritance of behaviour — Java's extends. The arrow (a hollow triangle) points from the subclass to the superclass, drawn with a solid line. Realization is implementing an interface — Java's implements. Same hollow triangle, but a dashed line, and it inherits only the structural interface, no behaviour.

Solid vs. Dashed: A Strict Rule

Between a class and an interface you must use realization (dashed). Between two classes you must use generalization (solid). One exception: an interface inheriting from another interface uses a solid line, since that is true inheritance. Picking the wrong line style is a common exam mistake.

Check Your Understanding
A Motherboard class defines a private class IdBus { ... } as an inner class no other class can see. What relationship connects them?
A
Dependency — the two only interact temporarily.
B
Association — Motherboard holds a long-term reference.
C
Aggregation — a whole–part relationship where the part survives alone.
D
Composition — IdBus cannot exist outside Motherboard.
Correct: D. A private inner class is a guaranteed beacon for composition: no other class even knows IdBus exists, so it literally cannot exist outside Motherboard. That lifetime coupling is the defining property of composition. Aggregation (C) would be wrong because the part here cannot survive independently.
Part IV

Navigability & Reading Relationships off Code

Navigability: Which Way Can You Travel?

An association line can carry an arrow to show navigability — which object holds a reference to which. If a Shape stores a reference to a Color, the arrow points from Shape to Color: you can navigate from a shape to its colour.

MarkingMeaning
Arrow toward BThe source stores a reference to B; you can navigate A → B.
Cross (×) on an endAn explicit claim: this side does not store a reference.
No arrow, no crossNo claim made — the diagram is silent about whether a reference exists.

Examples the lecture used: an employee and boss reference each other (double-navigable). A vote and politician — your vote stores the politician's name, but whether the politician keeps a list of voters is unknown, so we leave it unmarked. An account and clear-text password should be doubly non-navigable (a cross on each end) — they are in a long-term relationship, but for security neither stores a reference to the other.

Beacons: Reading Diagrams off Source Code

A recurring exam-style exercise: given a snippet of Java, draw the diagram. The professor's advice is to read the syntax first, then layer in obvious semantics. Certain code shapes act as beacons that strongly suggest a particular relationship.

1

A stored field / reference → an association (navigable at least one way). The reference lives across the object's lifetime.

2

An argument, local, or thrown exception → a dependency. Something on the stack is inherently temporary. A method that throws InvalidChecksumException creates a dependency, not an association — the exception is never stored.

3

A private inner class → guaranteed composition. Nobody else can reference it, so its lifetime is bound to the enclosing class.

4

A field holding a collection of some type (e.g. List<Employee>) → often aggregation, because the whole point of the container is to group the parts into a semantic whole. But this one needs your judgement, not just the syntax.

Syntax vs. Semantics — The Hard Part

Aggregation is "more about semantics than code." Two divisions each holding a List<Employee> look identical to a list-of-anything, but you label it aggregation because you understand a division is meant to group employees. Note also: the standard library (List, etc.) is usually not modelled — the diagram is an abstraction, so we say "a division has a list of employees" without exposing that it is implemented with ArrayList.

A Stack-vs-Heap Heuristic

If an object lives only on the stack (a local for one call), lean toward dependency — it's inherently temporary. If it lives on the heap and is held long-term, it's more likely a real association. This heuristic distinguishes dependency from association, not association from aggregation (both of those typically sit on the heap).

Check Your Understanding
A ChecksumValidator has a method validate() that throws InvalidChecksumException. What is the relationship to the exception class?
A
Association — the validator is permanently linked to the exception.
B
Dependency — the exception is only thrown, never stored.
C
Composition — the exception cannot exist without the validator.
D
Generalization — the validator extends the exception.
Correct: B. The exception is thrown and caught but never stored as a field, so the interaction is temporary — a dependency. The validator nonetheless won't compile without the exception class being referenced. It is not an association (A) because nothing keeps a long-term reference.
Part V

Sequence Diagrams — Behavior Over Time

Why Are We Learning This?

Class diagrams show static structure — relationships that hold for all instances, things you can often read straight off the code. Sequence diagrams show something the source code hides: the order in which objects send messages to each other over time. To reconstruct that flow from source you might read thousands of lines for half an hour; a sequence diagram conveys it in about a minute. That's why the professor would ask an AI assistant to draw a sequence diagram before a class diagram.

The Big Idea

Class diagrams model relationships between classes. Sequence diagrams model interactions between objects (instances). This is the distinction students confuse most. The same class can appear as two separate instances in one sequence diagram — two distributors, two passengers — because we are tracing concrete interactions, not the general structure.

Lifelines & Activation Bars

Each participating object gets a column. At the top is a box naming it; a dashed lifeline drops down representing the passage of time (top = past, bottom = future). Because these are objects, we write a colon: :Station is an unnamed instance of Station; start:Station is a named one. The colon is what signals "instance, not class."

A passenger calls pushButton() on a station (solid arrow = synchronous). The orange activation bar shows the station is busy; the dashed arrow is the return.

The fat bar on a lifeline is the execution specification (or activation bar): the period the object is "doing something" — a method of it is on the stack, or it is waiting for a response. Real people are usually drawn permanently active (they're always alive and busy). Activation bars can nest when an object calls a method on itself (a self-message).

Messages: Synchronous vs. Asynchronous

An arrow from one lifeline to another is a message — one object calling a method on another. The arrowhead style matters:

ArrowKindMeaning
Solid line, filled arrowheadSynchronousCaller waits for the response (a normal method call).
Solid line, open arrowheadAsynchronousCaller does not wait — fire-and-forget, e.g. a message to a microservice with a callback.
Dashed line, open arrowheadReply / returnThe response to a prior synchronous call.

The dashed return arrow can be labelled with the returned value(s). Because UML isn't tied to one language's semantics, a return can even carry multiple values — total, description := enterItem() — which Python could express in one line but Java cannot.

Combined Fragments: loop, alt, opt & more

A sequence diagram shows example executions, not all of them — the goal is to highlight the important flows (a normal path, a buggy path). But combined fragments let one diagram capture several behaviours at once. They are drawn as rectangles with a cut corner spanning the lifelines they affect, tagged with a keyword.

KeywordBehaviour
loopRepeat the enclosed messages for each item / while a condition holds.
altAlternatives — like if/else if/else, multiple guarded sections.
optOptional — an if with no else; runs only if its guard is true.
parParallel — sections that execute concurrently.
criticalA critical region, for concurrency / threading concerns.

The top three — loop, alt, opt — are by far the most common in practice. Worked example from lecture: an order loops over each item; inside, an alt dispatches items worth more than $10,000 to a careful (reliable, pricier) distributor and cheaper items to a regular distributor; afterward an opt sends a confirmation only "if needed."

Expand: the loop + alt example as pseudocode 
pseudocodefor item in order.items:        // loop fragment
    if item.value > 10000:       // alt fragment
        carefulDistributor.dispatch(item)
    else:
        regularDistributor.dispatch(item)

if needsConfirmation:           // opt fragment
    order.sendConfirmation()

Found, Lost & Create Messages

Three special messages handle the edges of a flow:

1

Found message — drawn from a filled dot into a lifeline. It starts a flow when we don't want to (or can't) model who sent the first message. Useful for "something calls dispatch() on the order" without modelling the client.

2

Lost message — the mirror image: a message sent to a recipient we choose not to model.

3

Create message — a dashed arrow labelled create that ends at the top of a new lifeline, marking the moment an object comes into existence. Destruction is shown by a large × on the lifeline.

Don't Force a 1-to-1 Mapping With Code

Sequence diagrams and code don't always line up exactly. UML has no native for-each loop, so a loop over a list is drawn with an explicit index a[i] even when the real Java/Python code iterates with no visible index. The diagram captures the behaviour, not the literal syntax.

Check Your Understanding
In a sequence diagram you see a solid line with a filled arrowhead from :Sale to :Payment. What does this tell you?
A
An asynchronous message — Sale does not wait for a reply.
B
A synchronous message — Sale calls a method on Payment and waits.
C
A return value being sent back from Payment to Sale.
D
A generalization — Sale inherits from Payment.
Correct: B. A solid line with a filled arrowhead is a synchronous message — a normal blocking method call where the caller waits. An open arrowhead (A) would be asynchronous; a dashed line (C) would be the return. Generalization (D) belongs to class diagrams, not sequence diagrams.

Putting It All Together

Everything in this lecture flows from one idea introduced at the very start: a model is an abstraction that keeps what matters and discards the rest. The interface-vs-implementation distinction from Part I is the engine that drives all of it — class diagrams exist to expose interfaces while hiding implementations.

From there the two diagram types divide the labour. The class diagram answers "what is the system made of?" — its boxes (Part II) carry the interface of each class, and its lines (Part III) carry the six relationships, layered from the temporary dependency up through association, aggregation, and composition, plus the two inheritance flavours, generalization and realization, that mirror the subclassing/subtyping split. Part IV taught you to read these straight off code by spotting beacons: a stored field means association, an inner class means composition, a thrown exception means dependency — while reminding you that aggregation often needs human judgement about intent.

The sequence diagram (Part V) answers a question the class diagram can't: "what happens, and in what order?" It trades static structure for a time-ordered trace of messages between objects — the same class can appear twice — using activation bars, synchronous vs. asynchronous arrows, and loop/alt/opt fragments to compress a thousand lines of control flow into a glance.

When to Reach for Which

Reach for a class diagram to reverse-engineer or plan the shape of a codebase — modern AI assistants reliably generate Mermaid or PlantUML class diagrams from source. Reach for a sequence diagram first when you need to understand a collaboration, because that behaviour is the part hardest to recover from reading code.

UML Is Semi-Formal — Use It Wisely

UML is a semi-formal, visual, widely-understood language: precise enough to communicate intent, but not a formal logic you can mechanically verify (languages like Alloy do that). It is a hodgepodge — just because a notation exists doesn't mean you should use it — and it scales badly: a diagram of "everything" becomes unreadable. Model the important aspect, not the whole system.
