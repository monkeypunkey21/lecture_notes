Creational Design Patterns

An interactive guide to Lecture 7 — how Factory Method, Abstract Factory, and Builder let us create objects without baking the decision into code that shouldn't know about it.

Part I

The Problem of Creating Objects

Why Are We Learning This?

In a software engineering class, the goal is not to write code that merely works today. The professor was emphatic about this: we want code that is easy to change, easy to maintain, and flexible in the future. A program that does the right thing but collapses the moment a requirement shifts is a failure by the standards of this course.

It turns out that one of the most common places where code becomes rigid and painful to change is in the act of creating objects. Creating the right kind of object, in the right place, based on some runtime decision, sounds trivial — but done naively it scatters knowledge of "which classes exist" across your entire codebase. Every pattern in this lecture exists to contain that knowledge in one well-chosen place.

Key Concept — Creational Design Pattern

A creational design pattern is a reusable solution to the recurring problem of how to create objects. Unlike the behavioral patterns from last week (observer, strategy, mediator), creational patterns focus specifically on the moment of instantiation — deciding which concrete class to build, and hiding that decision from the code that uses the result.

Recap: The Patterns Behind Us

Before introducing anything new, the professor recapped last week's behavioral patterns. We define each from scratch here so that nothing is assumed.

PatternWhat problem it solves
ObserverA one-to-many relationship: when one object (the subject) updates, many observers need to be notified. The subject holds a list of observers; observers register with the subject and get pushed a notification on change. The intent is to decouple the listeners from the subject.
StrategyBehavioral variation: several interchangeable algorithms. Instead of many subclasses, you delegate the varying behavior to a strategy interface with multiple implementations. "Favor composition over inheritance."
MediatorReplaces a tangled web of object-to-object dependencies with a central hub. Collaborations between objects move into a mediator object, leaving the other components reusable and independent.
A Pattern Is a Triplet

The professor stressed an idea that runs through the whole lecture: a design pattern is not just a solution. It is a triplet of problem + context + solution, and crucially it carries an intent. You only call something an "observer" if your goal is to decouple listeners from a subject — even if an identical class structure shows up for some unrelated reason, it isn't an observer unless that intent is present.

Cognitive Load: The Hidden Currency

Why does scattering creation logic everywhere actually hurt? Because of cognitive load. When code that changes for one reason is mixed into the same method as code that changes for a different reason, anyone wanting to make a small change must first understand the whole method. And you should never change a method you don't fully understand.

The professor's rule: structure code so that a person needs to understand the minimum amount of logic to make a given change, removing every other distraction. He added a modern twist — what confuses a human also confuses a large language model. Code that is hard for a person to load into their head is hard for a model to load into its context window. Reducing human cognitive load reduces a model's token-and-attention load too.

The Big Idea

Separate code that changes together from code that changes for different reasons. Almost every design pattern in this lecture is a specific technique for doing exactly that — isolating one "design decision" into one place so the rest of the system never has to know about it.

Part II

The Simple Factory

We build up gradually. Before the named patterns, the professor walked through two failed attempts and one modest fix. Watch how each step removes one specific problem.

Two Motivating Examples

First, ducks. Last week's running example had a MallardDuck, a DecoyDuck, a RobotDuck, and more. Suppose throughout our program we keep writing code like "if the use case needs a mallard, make a mallard; if it needs a robot duck, make a robot duck." A student nailed the flaw: it's not only long, you have to "check them all procedurally," and biologists keep discovering new ducks. Every new duck means edits at every creation site.

Second, pizza — the example that carries the rest of the lecture. An orderPizza method picks a pizza type from a string, then runs the same preparation steps on it:

// The naive version: creation and preparation tangled together
Pizza orderPizza(String type) {
    Pizza pizza;

    // --- this part VARIES when we add pizza types ---
    if (type.equals("cheese"))    pizza = new CheesePizza();
    else if (type.equals("greek")) pizza = new GreekPizza();
    else if (type.equals("pepperoni")) pizza = new PepperoniPizza();

    // --- this part STAYS THE SAME ---
    pizza.prepare();
    pizza.bake();
    pizza.cut();
    pizza.box();
    return pizza;
}
The Footgun — Mixed Rates of Change

The if/else block changes whenever the menu changes. The prepare/bake/cut/box steps do not. Yet they live in the same method, so anyone adding a new pizza must read and understand the preparation logic too — code that has nothing to do with their change.

The tempting wrong fix

A student suggested moving the constructors into the Pizza class itself. The professor called it "an interesting design" — then asked the class to find the flaw. The problem: the superclass would now know about its own subclasses. That creates a cyclic dependency (Pizza depends on CheesePizza, which extends Pizza), so neither can be compiled or tested in isolation. We also want superclasses to stay as general and reusable as possible, usable in contexts where those specific subclasses don't even exist.

Extracting a Factory Class

The first real insight: don't put the construction logic in the Pizza class — put it in a separate class. This is the simple factory (technically an idiom rather than a "real" pattern, the professor noted).

class SimplePizzaFactory {
    Pizza createPizza(String type) {
        if (type.equals("cheese"))     return new CheesePizza();
        else if (type.equals("pepperoni")) return new PepperoniPizza();
        else if (type.equals("clam"))     return new ClamPizza();
        else if (type.equals("veggie"))   return new VeggiePizza();
        return null;
    }
}
1

The Pizza class no longer references any subclass, so it can be compiled and tested on its own. No cyclic dependency.

2

The decision "which string maps to which pizza" is now hidden inside SimplePizzaFactory. That is the one design decision this class owns.

3

Clients call createPizza(...) wherever they need a pizza. They never see the if/else. Adding a new pizza touches exactly one method.

Same Behavior, Different Shape

A student asked whether anything actually improved — and the professor said this was a great point. The refactored code runs the exact same if/else at runtime and does literally the same thing. Software design is not about changing behavior; it is about relocating code so that the place needing a change is easy to find and understand in isolation. We traded "everything visible in one method" for "behavior hidden behind a layer of indirection."

This trade has a real cost, and the professor was honest about it: to understand the whole system you now have to look in more places. We made global comprehension slightly harder in exchange for making local changes much easier. That is why these patterns are not always worth it.

Part III

The Factory Method Pattern

When the Process Itself Varies

The simple factory handles "which pizza." But now the company expands to multiple locations — New York and Chicago — and each prepares pizza differently. A Chicago pizza might need a longer bake time for its thicker crust; the cut step might differ by style. Now two things vary: the type of pizza and the preparation process.

The Wrong Move — Subclass Explosion

A naive fix is to make a subclass per combination: ChicagoCheesePizza, NYCheesePizza, ChicagoPepperoniPizza... This multiplies the pizza types by the number of locations and quickly becomes unmanageable. We want to vary location without duplicating every pizza type.

The solution a student arrived at: make the pizza store itself abstract, with an abstract creation method that each location overrides.

abstract class PizzaStore {
    // The factory method — abstract, decided by the subclass
    abstract Pizza createPizza(String type);

    // Shared, unchanging workflow lives here in the base class
    Pizza orderPizza(String type) {
        Pizza pizza = createPizza(type);  // deferred to subclass
        pizza.prepare();
        pizza.bake();
        pizza.cut();
        pizza.box();
        return pizza;
    }
}

class NYPizzaStore extends PizzaStore {
    Pizza createPizza(String type) { /* NY-style pizzas */ }
}
class ChicagoPizzaStore extends PizzaStore {
    Pizza createPizza(String type) { /* Chicago-style pizzas */ }
}

The Pattern's Anatomy

Key Concept — Factory Method

The Factory Method pattern defines an interface for creating an object, but lets subclasses decide which class to instantiate. An abstract creator declares an abstract factory method; each concrete creator overrides it to produce a concrete product. The pattern "defers instantiation to subclasses."

Two parallel hierarchies. The creator side (usually inheritance) decides which product (usually interface realization) to instantiate. Only the concrete creator knows the concrete product.

Factory Method vs Strategy — The Key Distinction

The professor posed his favorite kind of question: structurally, Factory Method and Strategy look almost identical. Both take an abstract class, delegate to an abstract method, and let subclasses implement varying behavior. So what's the difference?

It's the Intent, Not the Structure

Behaviorally they are nearly the same — both are "behavioral variation via subtyping." The difference is that Factory Method's varying behavior is specifically the creation of objects. We use the name "factory method" precisely so that when a competent designer sees it, they know not just the solution but the intent: "I'm varying how objects get created." Saying "I used Strategy to create objects" is technically true but miscommunicates intent — patterns are shared vocabulary for design ideas.

This is the same logic as Observer: you only call it an Observer if the goal is a one-to-many, decoupled relationship. The name carries the why, not just the what.

Part IV

The Abstract Factory Pattern

When You Create a Whole Family

New problem: some franchise locations have "gone rogue" and started using inferior ingredients. To enforce quality, the corporation gets into the ingredients business — now we don't just build pizzas, we also build their ingredients, with regional variations. New York uses one kind of dough, sauce, cheese, and veggies; Chicago uses another.

These ingredients are not unrelated products that each deserve their own factory. They are a family — created and used together in the same context. Whenever you make dough you also want the matching sauce and cheese. That semantic relatedness is what motivates a single factory with many creation methods.

interface PizzaIngredientFactory {
    Dough   createDough();
    Sauce   createSauce();
    Cheese  createCheese();
    Veggies[] createVeggies();   // can mix several — real logic, not blind forwarding
}

class NYIngredientFactory implements PizzaIngredientFactory {
    Dough  createDough()  { return new ThinCrustDough(); }
    Sauce  createSauce()  { return new MarinaraSauce(); }
    Cheese createCheese() { return new ReggianoCheese(); }
    // ... red peppers (shared NY+Chicago), thick crust (NY only), etc.
}
Key Concept — Abstract Factory

An Abstract Factory provides an interface for creating a family of related products without specifying their concrete classes. The professor described it precisely: it is a compound pattern that bundles multiple factory methods into one class. Whereas a Factory Method makes one kind of product, an Abstract Factory makes a coordinated family — and the single decision "which factory" (NY? Chicago?) selects the whole consistent set at once.

The key benefit of co-locating the family: to understand "what makes a New York pizza a New York pizza," you look in exactly one place — the NYIngredientFactory. And to add a brand-new location like Los Angeles, you just write LAIngredientFactory and pick its ingredients. As the professor put it, "nobody even knows there are multiple locations" — no other code knows LA exists. That is the localization of change the pattern buys you.

The Principle Underneath: Dependency Inversion

Why does this work so cleanly? Because the pizza store depends only on the abstract ingredient interfaces, never on concrete ingredient classes. It has no idea what concrete products or concrete factories exist.

Key Concept — Dependency Inversion Principle

Robert C. Martin's Dependency Inversion Principle: concrete classes should never depend on concrete classes — they should always be coded against abstract interfaces. The professor called this one of the most important principles in object-oriented design, because depending on interfaces rather than implementations minimizes the blast radius of any change.

Open–Closed, and Where the Pattern Breaks Down

The professor switched to a second Abstract Factory example — GUI window kits (a WindowKit with Motif and OpenLook variants, each producing windows and scrollbars). He used it to probe which changes the pattern makes easy and which it makes hard.

Key Concept — Open–Closed Principle

Bertrand Meyer's Open–Closed Principle: software should be open for extension but closed for modification. You should be able to add new behavior by writing new classes (subclassing), without ever opening up and editing an existing class.

Adding a whole new family (a Mac window kit) is the easy case: you just add new concrete classes — MacWindowKit, MacWindow, MacScrollBar — and change nothing existing. That is the Open–Closed Principle in action.

The Hard Change — A New Product Type

But adding a new type of product (say a Button) is painful: you must add a createButton() method to the abstract factory interface and to every concrete factory implementing it. One small request now forces edits across all existing classes. The professor's lesson: each pattern is designed for one direction of growth and resists another. Use a pattern only when the changes you expect are the ones it makes easy.

He also flagged two structural limits the class voted on. Abstract Factory assumes all products share a uniform interface — if one concrete product needs different constructor arguments (a shell connection vs. a server connection), you've "broken the interface promise" and the pattern no longer fits. And it assumes products are created independently; if one product is itself a composition of several others, that's a job for a different pattern (the Builder, next).

Part V

The Builder Pattern

When a Product Is Made of Many Parts

The last open question from Part IV — "what if a product is a combination of a window, a scrollbar, and a button?" — names exactly the problem the Builder solves. Abstract Factory assumes products are created independently. Builder is for objects assembled from multiple parts in a sequence.

Key Concept — Builder

The Builder pattern separates the construction process of a complex, multi-part object from the way its individual parts are built. A director defines the order in which parts are assembled; a builder (with one factory-method-like operation per part) knows how to build each concrete part. The client talks to the director through a deliberately simple interface.

Director and Builder

The client uses a director; the director calls buildPart() on the builder in a fixed order. The director encapsulates the sequence; the builder encapsulates how each part is made.
1

The client asks the director to construct(). It interacts through a simple interface and stays ignorant of part details.

2

The director calls buildPart() for each part, in a predetermined order. The order lives in the director, isolated from how parts are built.

3

The builder has one factory-method-style operation per part, each producing a concrete piece of the final product.

Two Decisions, Cleanly Separated

Builder's payoff is decoupling two independent decisions: the order/structure of assembly (owned by the director) and the how-to-build-each-part logic (owned by the builder). If you later need to reorder steps or drop a part, that change is localized in the director. The professor noted builders can also be richer than a fixed sequence — a builder might consume a JSON-style specification and assemble parts according to it.

Part VI

Choosing a Pattern — Consequences & Judgment

Same Problem, Different Sub-Problems

All three patterns answer "how do I create objects?" but each is tuned to a different sub-problem and carries different consequences (think of consequences as the quality attributes you gain or lose by choosing it).

PatternSub-problem it fitsConsequence
Factory MethodOne product type, simple object structure; an independent creation decision.Most simplicity, least extensibility/flexibility.
Abstract FactoryA family of related products created/used together; the decision is co-located (NY vs Chicago).Most extensibility and variation points, but most complexity and classes.
BuilderOne product assembled from many parts in a sequence.Communicates the product's parts; encapsulates assembly order and state.
Their Shared Consequences

The professor said all three patterns (in ~99% of cases) share the same pair of consequences: they decouple the use of an object from its creation (the user code only sees abstract interfaces; the creation code never sees how the object is used), and they make it easier to add new products. The cost is also shared: you always get more complexity than if you'd used no pattern at all.

Because pattern names carry intent, an onboarded engineer who sees "Abstract Factory" instantly knows there's a family of related products; seeing "Builder" tells them there's a multi-part sequential build. They needn't read the code to grasp the design. This is exactly how the professor framed structuring an AI agent skill: a main file that says when to load each pattern's knowledge, with the detailed structure/behavior looked up on demand — the same way a human should memorize when to apply something and then look up the details.

The Real Skill: Avoiding Both Extremes

The Central Trade-off

Every pattern is a choice between simplicity (fewer classes and locations to understand the whole system) and changeability (abstractions that let you extend without modifying, at the price of more locations to comprehend but fewer to touch for a given change). Your job as an engineer is to judge, for this situation, whether flexibility, performance, or comprehensibility matters most.

Under-engineeringOver-engineering
Too little thought about design. Inflexible, hard-to-change solutions where change becomes expensive. Common from lack of time, bad management, or lack of skill.The "second system effect" — students fresh from a class like this add patterns everywhere. Premature abstractions: interfaces with a single implementation forever, adding files and hiding details with no payoff.
Typical of those who haven't studied design.Typical right after studying design.
The Cost of Over-Engineering

Unnecessary flexibility isn't free. Every extra class must be maintained, documented, and tested. Chasing the "perfect" design can waste a week you could have spent shipping — especially fatal for a startup that needs revenue. Don't treat a pattern as a hammer that makes everything look like a nail.

The professor's procedure for deciding: do a change impact analysis. Anticipate which design decisions are likely to change, how likely, and how large the blast radius would be — then pick the pattern designed to absorb that kind of change. If you never expect new products, the "easier to add products" benefit is worthless to you, so the pattern may be pure overhead. He noted change impact analysis is the kind of thing that shows up on exams.

Abstract Pattern, Concrete Instance

A design pattern is an abstract solution to an abstract problem in an abstract context. Applying it means recognizing that your concrete problem and context are instances of the abstract ones, then tailoring a concrete solution. The professor observed that AI tools are good at the tailoring step but weak at the recognition step — they often won't reach for the right pattern (e.g. State for a Monopoly game) unless explicitly told to think in patterns. That recognition is where human judgment still earns its keep.

Putting It All Together

This lecture told one continuous story: a pizza company growing in complexity, and the creational patterns that let its code grow gracefully alongside it. Trace the arc and you can derive every pattern from the pressure that produced it.

1

Naive if/else in orderPizza mixed creation with preparation — code that changes for different reasons living together, maximizing cognitive load.

2

Simple Factory pulled the creation decision into one class, killing the cyclic dependency and isolating "which pizza string maps to which class." Same runtime behavior — better structure.

3

Factory Method handled a second axis of variation (location) by deferring the createPizza step to subclasses of the store — behavioral variation whose intent is creation.

4

Abstract Factory bundled multiple factory methods to produce a coordinated family of ingredients, with one decision (NY vs Chicago) selecting the whole consistent set.

5

Builder covered the remaining case: a single product made of many parts in a sequence, with a director owning the assembly order.

Underneath them all ran the principles: separate what changes together from what changes apart; depend on abstract interfaces, not concrete classes (Dependency Inversion); stay open for extension but closed for modification (Open–Closed). And tying it back to last week, the lecture's deepest theme is that a pattern is a problem + context + solution with an intent — the name is shared vocabulary that tells a reader the why, which is why "Factory Method" and "Strategy" matter as distinct labels even when their code looks the same.

Most of all: a pattern is a tool with consequences, and the engineer's real skill is the judgment to pick the right one — or none at all — between the twin failures of under- and over-engineering.
