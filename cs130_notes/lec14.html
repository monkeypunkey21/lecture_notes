Part I

Pair Programming — Two Heads, One Keyboard

Most of this lecture is about a single uncomfortable truth: software is built by people, not by machines, and people have memory limits, blind spots, and very different skills. Every practice we study today exists to work with those human limits rather than against them. We start with the smallest unit of teamwork — two developers sitting at one machine.

Why Are We Learning This? The "It Must Be Half Speed" Trap

Here is the intuition almost everyone has the first time they hear about pair programming: if you put two developers on one keyboard, you've paid for two people but you only get one person's worth of typing. So the team's output should be half of what those two people produce working separately. That intuition is wrong, and understanding why it is wrong is the whole point of the practice.

Key Concept — Pair Programming

Pair programming is an Extreme Programming (XP) practice in which two developers work together at one machine, so that no line of code is ever seen by only one pair of eyes. Empirically, two junior developers pairing produce roughly the same total output as two working alone — sometimes more — even though only one is typing.

The Two Roles: Driver and Navigator

The work is split into two named roles, and keeping them distinct is what makes the practice effective.

DriverNavigator
Hands on the keyboard. Actually types the code.Describes the high-level approach: "now we go into this module and implement this interface."
Focused on the line being written right now.Watches the bigger picture and scans for bugs.

To get the most out of pairing, you swap roles regularly so each person practices both. Why does this combination beat two solo developers? The professor gave several concrete reasons.

1

Bugs are easier to spot in code you didn't type. When you write code yourself, your brain "knows" what it meant to say and glosses over the actual mistake. The navigator, reading fresh, catches the fault the driver's eyes slide past — and catches it immediately, before it sinks into the codebase.

2

Junior developers have more bugs to begin with. Because faults are detected the moment they appear instead of hours later in debugging, the pair spends far less time hunting down problems — which is exactly where juniors lose the most time.

3

Design decisions get discussed in the moment. Two people with different experience debate an approach before committing to it. That avoids the much larger cost of making a wrong design choice and having to unwind it later.

4

Knowledge transfers and team spirit improves. Both people now understand that code. Rotating who pairs with whom spreads understanding across the team and shrinks the "bus factor" (the number of people whose disappearance would sink the project).

Connection — This Reappears with AI

The professor frames AI-assisted coding as pair programming with the AI as the driver and you as the navigator. But there's a catch: it is hard to be a good navigator if you have never been a driver. You can only guide and review code well if you have enough hands-on coding experience to recognize when something is wrong. Hold onto this — it is the bridge to Part VI on code comprehension.

Part II

Continuous Integration — Merge Early, Merge Often

Why Are We Learning This? Avoiding the "Big Bang"

Imagine each teammate works alone on their own branch for the whole quarter, then everyone tries to merge into main in the final week. The result is what the professor calls a big bang: a tangle of merge conflicts where features that were never tested together suddenly collide, and you end up rewriting large amounts of code under deadline pressure. Continuous integration exists to make this disaster impossible.

Key Concept — Continuous Integration (CI)

Continuous integration is the software-development practice of regularly merging feature branches into the main branch — ideally as soon as each small piece of work is done. The principle: a bug written yesterday is easier to find than a bug written a month ago, so the shorter your feature branches live, the less likely a painful big-bang merge becomes.

The Common Misconception: CI Is Not Just a Tool

Many students think "continuous integration" means GitHub Actions, Travis, or Jenkins — the automated pipelines that run your tests. Those tools support CI, but they are not what it is.

Common Mistake — Confusing CI With the Pipeline

CI is fundamentally a practice: frequently merging code so features get integrated and tested together as early as possible. The automated pipeline is a helper, not the definition. You can run GitHub Actions all day and still violate CI if every branch lives for a month before merging.

The Supporting Practices

Several things make frequent merging painless:

1

Short-lived feature branches. A branch older than one week that hasn't merged into main is a warning sign. The professor's rule of thumb: if you follow good user-story design, you won't have these.

2

Version and configuration management. Track everything so merges are reproducible.

3

Keep build + test time under ten minutes. If running your tests takes half an hour, merging becomes a chore people avoid. Fast feedback keeps the merge habit alive.

Connection — INVEST Makes CI Automatic

This builds directly on user-story design. If your stories follow INVEST (Independent, Negotiable, Valuable, Estimable, Small, Testable), each one is tiny and merges into main the moment it's done. Small, testable, independent stories are short-lived branches — so good story-writing gives you continuous integration for free.

The Google Scale Problem

Google's codebase is so large that running all tests can take ~5 hours, far over the ten-minute ideal. With thousands of commits landing in that window, when a test fails you can't tell which of a thousand merges broke it. Google built clever heuristics to bisect the culprit — a reminder that the ten-minute rule is an ideal small teams can actually hit, while giants have to engineer around it.

Part III

Lean Software Development — Eliminating Waste

Why Are We Learning This? From a Toyota Factory to Your Repo

Lean software development isn't a software idea originally — it comes from Toyota's car-manufacturing process, sometimes called Kaizen. Where most carmakers optimized to keep their expensive machines fully loaded, Toyota optimized to reduce waste: parts arrive exactly when needed ("just in time") instead of piling up in heaps. The software-engineering community copied these ideas into a development methodology.

Key Concept — Waste

Waste is any time or effort spent on something that does not deliver value to the customer. Lean's central goal is to eliminate the seven wastes of software development. Crucially, waste isn't just your own lost time: every line of code written must be read by someone later, so unnecessary code wastes everyone's attention too.

The Seven Wastes

Walk through these as a checklist you can apply to your own project right now.

WasteWhat it is & why it hurts
1. Extra featuresCode nobody needs. You waste your time writing it and everyone's time reading and being confused by it.
2. Partially done workLong-lived, unmerged feature branches. Time invested with no value delivered, and it confuses anyone waiting on it. (Note how this is the same problem CI attacks.)
3. RelearningForcing people to re-acquire knowledge. Assign tasks to whoever is uniquely qualified so they don't have to relearn.
4. HandoffsEach person who must touch a task adds relearning and task-switching. Shorten the chain — ideally a direct line from customer to developer.
5. Task switchingSwitching tasks costs ~15–20 minutes each time to reload everything into working memory. Interruptions are mental-effort waste, and mental effort is your most valuable resource.
6. DelaysWaiting for approvals, management sign-off, or a slow test environment. No developer or user story should sit waiting.
7. DefectsThe worst waste. The longer a defect lives, the more damage, lost money, and context-switching it causes — it compounds every other waste.
"Stop the Line" and Why Defects Are Worst

At Toyota, detecting a defect triggered a stop-the-line response: halt production and let the whole organization fix it before making any further progress, because once one defect exists you don't know what else might be wrong. Software teams echo this — when a bug is found, fix it fast before anything worse happens. This is why defects sit at the top of the waste hierarchy.

The Same Principles Apply to AI Agents

The professor noted that "assign work to whoever is uniquely qualified" reappears in multi-agent AI systems: give a task to the agent that has the knowledge it needs. Understanding these human-process principles academically helps you recognize them in AI-based development too.

Cycle Time and the Kanban Board

How does Lean actually reduce waste? By minimizing cycle time.

Key Concept — Cycle Time

Cycle time is the time between the end user notifying you of a problem (or requesting a feature) and that feature being deployed. Lean's whole aim is to shrink it — so Lean avoids long planning meetings. There are no two-hour planning-poker sessions; you just take the highest-value item next.

Lean uses a Kanban board to track work as it flows through columns. The figure shows the four columns and the order in which a Lean team prioritizes moving items.

A Kanban board. The team prioritizes pulling from the right: move Review→Done first (almost finished), then In-Progress→Review, and only then pull a new high-priority item from To-Do.

The flow rule: once an item leaves To-Do, get it to Done as fast as possible. So you first finish what's nearly done. A developer who pushes their feature to Review next reviews someone else's work; only if there's nothing to review do they start a new story — always the highest-priority one they are qualified to do. This produces the most customer value globally.

Lean vs XP — Opposite Philosophies on Planning

This is the surprising contrast. In XP, you tell the customer when a feature will arrive. In Lean, you don't promise dates at all — you ask the customer to be patient and simply always deliver the highest-value item next. Lean trades predictability for raw throughput of value.

Expand: The five values of Lean

Lean is less a fixed set of steps and more a set of values:

1. Identify value — understand what the customer actually needs.
2. Map the value stream — find and eliminate waste.
3. Establish smooth flow — minimize interruptions between columns.
4. Establish pull — pull the highest-priority items to Done first.
5. Seek perfection / build quality in — use TDD, code reviews, high standards, because shipping low quality creates more waste, defects, and delays. (Toyota was historically famous for exactly this quality reputation.)

And empower teams: assign each task to the person uniquely qualified, because developers have different skills and cognitive limits — they are not interchangeable machines.

Part IV

Agile Design & Managing Risk

Why Are We Learning This? The Great Misreading of Agile

The Agile Manifesto says "working software is the primary measure of progress" and "working software over comprehensive documentation." A huge number of people read that and concluded: design is waste; skip the UML, the design docs, the tests, and just ship features. That is a misreading, and clearing it up is the heart of this part.

Common Mistake — "Agile Means No Design"

The Manifesto also says: "continuous attention to technical excellence and good design enhances agility." Agile never says testing or design docs are useless. It says they must be used deliberately — only when they enhance your ability to deliver high-quality software faster. Drawing 20 UML diagrams per feature just to "feel useful" is not agile; using design to avoid bugs and align the team is.

The Big Idea — Deliberate, Not Ceremonial

The best architectures, requirements, and designs emerge from self-organizing teams. Rather than a manager mandating "always write a design doc," you trust expert teams to choose the tools, languages, and processes best suited to their software. Agile works brilliantly with people who deeply understand their craft — and badly when imposed top-down. Quality attributes (performance, security, availability) matter exactly to the degree they deliver observable value to the customer, and no more.

What goes wrong when teams genuinely skip design? You get a big ball of mud — software so tangled it's painfully hard to change. Without planning for change, you never build the abstractions and design patterns that make change cheap, and you can't bolt them on later. You also get a small bus factor, because everyone just heads-down implements features and stops having design conversations.

Risk-Driven Design: The Middle Path

So how much upfront design should you do? Not "all of it" (big upfront design), not "none of it" (tiny upfront design). The answer the professor endorses comes from George Fairbanks' book Just Enough Software Architecture.

Key Concept — Risk-Driven Design

Risk-driven design: identify the most critical aspects of your software — the things that, if you get them wrong, the product fails — and focus your design effort there. The amount of risk determines the amount of upfront design. A casual mobile game carries little risk; a medical device, space shuttle, or financial trading app carries a lot and demands much more upfront design.

Risks are usually decisions that are hard and expensive to change later: the programming language, target platforms (iOS/Android), component architecture and connectors, interfaces, and quality attributes. The professor worked through several domains live — here is the pattern of "what's the highest risk?":

DomainHighest-risk things to get right early
Online shopData consistency, security & privacy (credit cards!), and changeability — competitors ship features fast.
GamesTarget platforms (console vs mobile vs Apple Silicon — some games never planned for M1/M2/M3), latency & frame rate, and usability (the highest one, often overlooked).
Medical devicesCorrectness (bugs can literally kill), reliability & robustness (work in hot/cold, intermittent power), testability.

Risk Storming

To find these risks as a team, use risk storming (from Simon Brown — riskstorming.com): draw an architecture diagram, then everyone silently writes risks on sticky notes and attaches them where relevant, color-coded red/yellow/green for high/medium/low priority. Then you reveal and discuss. Silence first prevents anchoring; the team's different experts (security, database) surface different risks. Do it early — and it's genuinely fun.

Why AI Struggles Here

Risk-driven design needs real-world understanding, and that's where current AI is weak. The professor's example: ask an AI "the car wash is two minutes away, should I drive or walk?" and it says "walk, it's close" — missing that the entire point is to bring your car. AI doesn't grasp what makes software succeed in the real world. Identifying what truly matters is your advantage.

Connection — Responding to Change Requires Planning for It

"Responding to change over following a plan" doesn't mean abandoning planning — you can only respond to change if you have planned for change. The design patterns, information hiding, low coupling, and high cohesion you learned earlier are exactly the tools that let architecture absorb change. As Robert C. Martin put it: "A good architect maximizes the number of decisions not made" — and those flexible architectures never emerge by accident; you must invest effort to build them.

Technical Debt

If you only ever pull items off the feature backlog — always shipping the next user story — your architecture slowly rots. The professor's image: a customer asks "why does adding one more window take so long?" because the system has become a teetering mess.

Key Concept — Technical Debt

Technical debt is the result of short-term-oriented decisions that make future changes more costly or impractical — code smells, missing documentation, skipped refactors, architectural shortcuts taken to ship a feature on time. Like financial debt, it accrues "interest": the longer it sits, the more it slows you down.

The recommended remedy is a technical-debt backlog alongside your feature backlog. When you're forced to cut a corner to hit a release, you log a technical-debt item: "I committed this debt; I'll pay it back." Two people own the two backlogs: someone who deeply understands user needs owns the feature backlog; a software architect skilled in design and quality owns the technical-debt backlog.

Strategy A — A little each sprint (preferred)Strategy B — A dedicated debt sprint
Every iteration mixes a few debt items with feature work.Pause feature work for one whole sprint to pay down debt.
Customer always sees a steady stream of features.Customer wonders "did you do anything this sprint?"
Default recommendation.Only when debt has grown faster than you can absorb; best timed right after a release.

Part V

Code Review — Expectation vs Reality

Why Are We Learning This? Code Review Doesn't Do What You Think

Historically, code review meant Fagan inspections: a group gathered and read source code aloud line by line in a long meeting until everyone understood every line. Extremely effective at finding bugs — and extremely slow and wasteful. Modern practice instead assigns 1–2 reviewers to look over a new feature. But here is the twist that the research reveals.

Microsoft Research (Christian Bird and colleagues) asked developers what code review is for. The top expected answer was finding bugs. Then they measured what review comments actually do.

Developers expect review to find bugs, but only ~15% of real comments are about defects; the largest share (~30%) is code improvement, readability, and maintainability.
The Big Idea — Review Is About Readability, Not Bugs

Only about 15% of real code-review comments concern defects. The biggest categories are code improvement, understanding, and long-term maintainability: naming, missing comments, style, organization, alternative solutions. And it gets sharper — when the authors rated the comments they received, only about 33% were judged useful at all. Code review's real value is knowledge sharing and readability, not bug-catching. Embracing that is what makes review effective.

Why does this matter? Because the goal of source code is not only to be understood by a machine but by humans, who must change it. As Robert C. Martin says: "Any fool can write code that a machine can understand. Good programmers write code that humans can understand." At Google, code review explicitly is not for finding bugs (they have tests and verification for that) — it's to ensure developers write code other developers can understand.

What Actually Makes Reviews Work

Research-backed guidelines the professor highlighted (a later 2015 Microsoft study found the review tool is open ~5–6 hours/day but developers spend only ~30 minutes/day actually reviewing):

1

Use 1–2 reviewers. Google found one is the sweet spot; Microsoft found two. Either way, not 4–5 — large groups (like Fagan inspections) become deeply inefficient.

2

Cap a review at ~one hour. Review quality degrades significantly after about an hour of continuous reviewing.

3

Keep pull requests to ~200–400 lines. This is the key counterintuitive finding (next callout).

Footgun — The Giant Pull Request

You'd think a bigger PR gets more review time. The opposite is true. Past ~400 lines (and certainly past ~1000), reviewers stop reading and just skim — "looks good to me" — often leaving one nitpick comment as evidence they "reviewed" it. A giant PR therefore gets less real scrutiny. Submit the smallest coherent feature that provides value; stack small PRs instead of bundling unrelated changes.

Connection — Small PRs Are CI and Lean Again

Small, focused pull requests review faster (~20–30 min), so they don't sit waiting — that's Lean's "eliminate delays" and CI's "short-lived branches" showing up again. Notice how INVEST → small stories → short branches → small PRs → fast reviews is one connected chain running through the whole lecture.

Ramp-Up and Google's Readability Test

New reviewers learn fast but generally need 6–12 months to become productive contributors. At Google you must pass a code readability test for a language before you're allowed to review code in it, and each piece of code has a designated owner overseeing its reviews — institutional ways to keep review quality high.

Part VI

Code Comprehension — The Skill of Reading Code

Why Are We Learning This? The Skill That's About to Matter Most

The professor's prediction: over the next five years developers will spend an increasing share of time reading code and a shrinking share writing it, because AI agents write much of the code. (He notes friends at Meta say they've barely written code in months.) Since you'll be reviewing AI output that can run to tens of thousands of lines, fast, effective code comprehension is your highest-value near-term skill — and it's barely taught. There are two ways to read code.

Bottom-Up Reading

Key Concept — Bottom-Up Comprehension

Bottom-up reading starts from the concrete and builds abstractions by reading line by line, top to bottom: understand line 1, then line 2, then line 3… You assemble meaning from the details upward. It's the approach novices naturally favor. (When you lack mental "schemas" for the domain, you fall back to chunking — grouping a few lines into a meaningful unit, then combining chunks into higher-level meaning.)

The cost: you must read every line and hold a lot in working memory, even lines irrelevant to your actual question. It's slow.

Top-Down Reading and Beacons

Key Concept — Top-Down Comprehension

Top-down reading starts from a hypothesis, not from line 1: "I think this code does X." You then hunt for evidence — called beacons, key lines or patterns that confirm or refute the hypothesis. You start abstract (a UML-like, big-picture view) and drill down only to the details that answer your question. Research finds it's vastly superior in ~99% of cases where people can do it.

Here is the hypothesis-driven loop a top-down reader runs — and you can practice it on any code today:

1

Start with a question, not an answer. Ask "why am I reading this code?" — e.g. "is this function buggy?" or "where is the database connection?" Form a hypothesis: "the system must have a DB connection module."

2

Decide what evidence would answer it. What beacon would confirm the hypothesis? For a DB module: SQL endpoints, dbConnect() methods, identifier names like PaymentCalculator.

3

Scan for those beacons instead of reading every line — e.g. search the codebase for "payment" or "overtime." Top-down readers don't read everything; they navigate to the relevant parts.

4

Confirm, then form the next hypothesis — repeat until you know enough to make the change, review the code, or implement the feature.

What Counts as a Beacon?

Beacons are signposts that reveal what code does and why. The class brainstormed these — note how many are things you've already studied:

BeaconWhy it helps
Meaningful identifier namesA class named OvertimeCalculator announces its purpose. Misleading names are worse than none — so name carefully, for the reader's sake.
Design patterns"Singleton" or "Factory" instantly tells you the structure and collaborations. Studying patterns builds the schemas top-down reading relies on.
TestsAn executable, unambiguous spec that's always up to date because it runs regularly — the best ground truth of what the code should do. You can even write a test to probe your hypothesis.
Comments & documentationREADMEs, design docs convey high-level intent — but may be outdated, so trust with care.
Assertions (pre/post-conditions)A precondition + postcondition assertion tells you exactly what a function expects and provides, and is checked at runtime so it stays true. Put them in production code, not just tests.
Code structureNested for-loops hint at the algorithm; structural patterns suggest intent (lower-level signal).
Diagrams (incl. AI-generated)UML/architecture views give the high-level map. AI is now good at drafting roughly-UML diagrams that save hours of orientation.
How to Become a Top-Down Reader

You move from bottom-up to top-down by building domain knowledge. You can only form a good hypothesis if you understand what the software is about and know the domain's vocabulary and design patterns — which is exactly why undergraduates default to bottom-up. So before reading code, first understand the requirements: the user stories, functional and quality requirements, architecture docs, and a system-level sequence diagram. That foundation lets you generate good hypotheses.

Opportunistic Processing — Toggle Between the Two

Experts toggle: when reviewing AI-written code you can't read all 10,000+ lines, so use top-down to find the most critical, highest-risk code, then switch to careful bottom-up reading for just that piece. (Note the callback: "highest-risk code" is risk-driven design from Part IV applied to reading.) This is how practitioners actually review AI code today.

Clean Code — Writing for the Reader

Code review's job is partly to decide what code is clean. There's no single definition; the professor collected several, all pointing the same direction — clean code is code a human can understand and confidently change.

SourceDefinition of clean code
Dave ThomasCan be read and enhanced by a developer other than its original author. (You know you understand it when you feel comfortable changing it.)
Michael FeathersAlways looks like it was written by someone who cares — about the person who will read it.
Ward CunninghamEach routine turns out to be pretty much what you expect — i.e. your top-down hypotheses come true.
Grady BoochSimple and direct; reads like well-written prose; never obscures intent.
The professorConvinces the reader the code is likely free of bugs and gives contributors confidence that changes won't break anything unexpected.

Two phrasings worth keeping: there are two ways to write code — one that obviously has no bugs, and one that has no obvious bugs; the first is much harder and the goal. And violated assumptions should be caught early — which is exactly what assertions do.

The Big Idea — Code Is Read Far More Than It's Written

Research found code is read 10–14 times more often than it's written, and developers spend about 48% of their time just understanding code. So every minute spent making code easier to read pays back many times over. Beware the broken-window effect: once one piece of unclean code lands, developers write more unclean code around it ("if that passed review, why should I bother?"). This is why you avoid even small technical debt — it multiplies.

Putting It All Together

Every topic today rests on one foundation: software is built by humans with real cognitive limits, and good process works with those limits. That single idea ties the lecture into a graph rather than a list.

Pair programming exploits a quirk of human attention — we find bugs in others' code faster than our own — to make two people roughly as productive as two solo developers, while spreading knowledge and shrinking the bus factor. Continuous integration exploits another quirk — a fresh bug is easier to find than an old one — so we merge small, short-lived branches constantly to avoid the big-bang merge.

Lean generalizes this into a war on waste, with defects as the worst waste and cycle time as the metric, all visualized on a Kanban board where work flows rightward to Done. Agile design corrects the dangerous misreading that "agile means no design": instead, risk-driven design spends design effort precisely where the cost of getting it wrong is highest, and a technical-debt backlog keeps short-term shortcuts from compounding into a big ball of mud.

Then the lecture turns from writing to reading. Code review, despite everyone expecting it to find bugs, mostly transfers knowledge and improves readability — and only works when PRs stay small (200–400 lines) and reviewers stay few (1–2) and fresh (under an hour). Finally, code comprehension names the skill underneath all of it: top-down, hypothesis-and-beacon reading beats line-by-line bottom-up reading, and clean code is simply code written so other humans can form true hypotheses about it.

Watch how one chain threads through the entire lecture: INVEST stories → small short-lived branches → continuous integration → small focused pull requests → fast effective reviews. The same value — finish small things fast, never let work pile up unseen — drives pairing, CI, Lean, reviews, and clean code alike.

The Through-Line

Design patterns, information hiding, low coupling, and high cohesion — everything from earlier in the course — are the tools that let you plan for change and write code that broadcasts its intent through beacons. "A good architect maximizes the number of decisions not made," and good code maximizes the number of questions a reader never has to ask.
