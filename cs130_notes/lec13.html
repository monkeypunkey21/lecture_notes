From automatically generating test inputs to the human process of shipping software — why teams abandoned the waterfall, what the Agile Manifesto really said, and the day-to-day practices that make great engineers.

Part I

Closing Out Symbolic Execution

The lecture opened by finishing a topic from earlier in the course before pivoting to its main theme. We will do the same: a quick, self-contained recap of symbolic execution, because the professor explicitly flagged that "you'll probably have some question like this on the final exam." This is the one piece of formal verification the course wants on a software-engineering exam.

Why Are We Learning This?

Imagine you want to test a program thoroughly — specifically, you want 100% path coverage, meaning every possible route through the program's branches gets exercised by at least one test. Writing those test inputs by hand is exhausting: for each path you'd have to reason backwards through the conditions and find numbers that force execution down exactly that route. Nobody wants to do that. The whole point of symbolic execution is to let a machine do it automatically.

Key Concept — Symbolic Execution

Instead of running a program on concrete numbers, we run it on symbols (like x and y). As we walk each path through the program, we record two things: the path condition (the conjunction of all branch conditions that must hold to take this path) and the effects (how variables get reassigned along the way). The full result for a path is the path condition combined with those effects.

Path Conditions, Effects, and Feasibility

A program with branches has multiple paths. For each path we write down its path condition conjoined with the effects of every statement executed before it. We then combine the paths — joined with logical OR, because a real run takes either one path or another. Two questions fall out of this:

1

Is the path feasible? A path is infeasible if its path condition is a logical contradiction — no input could ever satisfy it. The professor's example: a path requiring x > 3 together with a quadratic equation that has no solutions for x > 3. Those two facts cannot hold simultaneously, so that path is infeasible. The opposite branch did have solutions, so it is feasible.

2

What concrete input exercises it? For every feasible path, the combined formula is handed to an SMT solver — an automated tool that finds concrete values satisfying a set of logical constraints. The solver hands back actual numbers (e.g. "x = 2 exercises statement one; x = 1 exercises statement two"). You could solve these by hand with "human intelligence," but the automatic route is the SMT solver.

Key Concept — SMT Solver

SMT stands for Satisfiability Modulo Theories. An SMT solver takes a system of logical constraints (equalities, inequalities, boolean structure) and either produces a concrete assignment that satisfies them all, or proves that none exists. In symbolic execution it is the engine that turns a path condition into a usable test input.

The Trap: Updating a Variable Used Later in a Condition

Here is the subtle point the professor spent the most time on. Consider a program where a variable z is incremented on one branch and decremented on another, and then a later condition tests z:

if (x < y)        // first decision
    z = z + 1;
else
    z = z - 1;
// the two branches join here
if (z < ...)      // second decision uses z
    ...

When you write the path condition for the second decision, you must not use the original z. On the path that took x < y, the second condition becomes (z + 1) < ...; on the path that took x ≥ y, it becomes (z - 1) < .... The path condition incorporates the effects of previous statements, so the symbol you constrain is the value of z after the first if, not the initial one.

Common Mistake — Stale Variables

If a condition uses a variable that was reassigned earlier on the path, forgetting to substitute the updated value gives you the wrong path condition. Rule of thumb: whenever a condition follows an assignment to a variable it mentions, replace that variable with its post-assignment expression before solving.

Counting Paths Without Drawing Them

The class worked an example with two sequential decision points, where both branches of the first decision flowed into the second decision. That gives 2 × 2 = 4 paths. When two decisions are in sequence on the control-flow graph, you multiply their branch counts to get the number of end-to-end paths.

Check Your Understanding

Correct: B. The path condition incorporates the effects of earlier statements, so the z in the later condition is the value after the decrement, giving (z - 1) > 5. Part II

The Waterfall Model — and Why It Breaks

Now the lecture turns to its real subject: the process of software development. The professor framed this with a career argument. As writing code increasingly gets done by AI, the skills that set graduates apart are the non-coding ones — choosing the right development process, making high-level decisions. "Working software is the only software making money," and getting there is about process, not just typing.

Why This Matters

This isn't abstract theory. The professor was explicit that these process skills "will set you apart from graduates who have not studied these topics," and that they show up past the interview stage when you actually have to ship.

The Five Stages of Waterfall

Historically, software was built in a strict, plan-driven sequence. Each stage finishes completely before the next begins — like water falling down a series of ledges, never flowing back up.

The waterfall model. Each stage fully completes before the next starts — feedback only arrives at the very bottom.

What Goes Wrong

The class brainstormed the failures of this model. They cluster around one root cause: feedback arrives far too late.

1

Requirements change during development. The environment shifts (suddenly you have many customers, so now you need scalability), or — crucially — the user "didn't really know what they wanted until they saw it." If requirements change while you are in the testing stage, enormous effort has gone into creating zero value.

2

Bugs are found late, when they are most expensive. If you only test after implementing the whole system, the developer who wrote the buggy code may have left the company — or simply forgotten why they wrote it that way. The earlier a bug is found, the cheaper it is to fix.

3

You learn hard truths only at deployment. Only when you deploy do you discover the system is too slow, uses too much memory, or doesn't work on the target platform.

4

Customer feedback comes only at the end. You can't make money until you ship — but maybe a "half-baked solution" would have been enough, and you'd have learned that months earlier.

"If You're Not Embarrassed..."

The professor quoted a piece of startup wisdom: "If you're not embarrassed by the first version of your product, you've released too late." The whole problem with waterfall is that it makes an early, embarrassing-but-informative release structurally impossible.

Nuance — Waterfall Isn't Always Slower

A tempting conclusion is "waterfall is just slow." The professor pushed back: some alternative processes are actually slower than waterfall in certain situations. The real failing is rigidity and late feedback, not raw speed. Whether an alternative is faster genuinely "depends on the situation."

And waterfall remains the right choice for some safety-critical or hardware-bound systems, so D is wrong too.
Part III

The Agile Manifesto

Building On Part II

Agile is the historical reaction to waterfall's late-feedback problem. Everything in this part is an attempt to pull feedback as early as possible.

A Ski Resort in Utah

Around the year 2000, software engineers across the industry were fed up with the waterfall process that essentially every big company practiced. A group of software-engineering "gurus" — all highly experienced people — gathered at a mountain ski resort in Utah to fix it. There was a twist: they discovered they disagreed on almost every proposed solution. So instead of agreeing on a method, they wrote on a whiteboard only the things they all agreed on — the least common denominator. The result, never really meant to be grand, became the Agile Manifesto.

The Four Values

The manifesto names four value pairs. The phrasing is precise and the structure matters: it says it values the things on the left more than the things on the right — while still affirming that the things on the right have value.

We value this MORE......over this (still valuable!)
Individuals and interactionsProcesses and tools
Working softwareComprehensive documentation
Customer collaborationContract negotiation
Responding to changeFollowing a plan
The Great Misreading

Many people read these four lines literally and absolutely: "the only right way is no processes, no documentation, no contracts, no plan." That is not what the authors meant. The manifesto explicitly says "there is value in the items on the right." A process that emphasizes the right side over the left simply isn't agile — but documentation still helps onboarding, a plan still lets you anticipate change, and a contract still gives you a legal basis. The nuance is the whole point.

The Principle Everyone Ignored

Behind the four values sit twelve principles that the professor argued are what Agile is actually about. A few stood out:

Key Principles

Satisfy the customer through early and continuous delivery. Welcome changing requirements, even late, and harness change for the customer's advantage. Deliver working software frequently — every couple of weeks to a couple of months. Business people and developers work together daily. Working software is the primary measure of progress.

Two principles were singled out as the ones people forgot:

  • "Continuous attention to technical excellence and good design enhances agility." After the manifesto, many companies sprinted in the opposite direction — "we should have no plan, no design at all, the gurus told us to just respond to change." That led straight to the big ball of mud the course covered in the information-hiding lecture.
  • "Simplicity — the art of maximizing the amount of work not done — is essential." This does not mean "do no work." It means find the work that creates no value and eliminate it.
Self-Organizing Teams

"The best architectures, requirements, and designs emerge from self-organizing teams." There is no single company-wide process every team must follow — teams pick what fits their needs. Real example: Google leans more on self-organizing teams, while Apple follows more of a company-wide process, partly because Apple is closer to hardware and must co-design across product teams. Proximity to hardware reduces how agile you can be.

Sprints and Iterations

How does "respond to change" become a concrete practice? You split work into small chunks so that at the end of each chunk you have a working, deployable increment.

Key Concept — Iteration / Sprint

An iteration (or sprint) is a timeboxed period, usually 1–4 weeks, with a specific set of work — the sprint backlog — to complete. At the end you review it, check it against requirements, show it to a real customer or a product owner (someone who understands what the customer truly needs), and iterate on that feedback. This is the core engine of most agile processes — except lean/Kanban.

Where Agile Doesn't Fit

Agile assumes you can deploy increments and that your team is experienced. Both can fail. You can't ship a spacecraft system every two weeks — launches happen maybe once a year. Safety-critical and medical-device systems are hard to build without a plan. And the manifesto's authors had decades of experience and deep implicit knowledge; a junior engineer who skips planning can't learn. Some agile ideas are genuinely harder for juniors.

Part IV

User Stories & the INVEST Criteria

Agile work is organized around user stories — small, customer-facing descriptions of a feature. But not every story is well-formed. INVEST is a six-letter checklist (covered earlier and on the midterm) for what makes a user story good. Here the professor connected each letter directly to why it helps agile planning.

LetterMeansWhy it helps agile planning
IndependentDoesn't depend on other storiesLets you schedule and estimate stories individually; a dependent story must wait for the one it needs
NegotiableDetails can be discussedIf a story were non-negotiable, why even have a self-organizing team talking about it? Negotiability is what makes the conversation worth having
ValuableDelivers value to the customerThe planning game exists to maximize customer value — estimating a story with no value is wasted effort
EstimableEffort can be estimatedObvious: if you can't estimate it, you can't fit it into a sprint
SmallFits inside one sprintSmaller chunks give more scheduling flexibility — exactly like OS scheduling, small jobs pack a sprint more optimally
TestableYou can verify it's doneIf you can't test it, you can't know the value was delivered — the burden shifts onto the customer
Independence Does Double Duty

Independence helps in two ways the class teased out: a dependent story has to be scheduled after its dependency, and dependent stories can't be reliably estimated or scheduled on their own. Both scheduling and estimation break when stories are tangled together.

Exam Note

The professor said "negotiable" and "testable" are somewhat less central to the planning game specifically, but still valuable across the agile process as a whole. INVEST was designed specifically to support these planning activities — that connection is the point.

Part V

Extreme Programming — Small Releases & the Planning Game

Extreme Programming (XP) was one of the earliest and purest implementations of agile principles. It was proposed by Kent Beck — also the creator of JUnit and the originator of test-driven development. The name comes from taking the development practices Beck believed were good and dialing them to their most extreme form. Its goals: maximize the business value added by every development activity, embrace change, and minimize risk. XP shines in uncertain environments where you can't anticipate the future.

Key Concept — "Minimize Risk" in XP

When Beck says "minimize risk," he means the risk of uncertainty: not knowing whether things will work, what you actually need, or which approach is right. You attack it by collecting as much feedback as possible and minimizing planning/documentation to just what's needed to deliver value. It does not mean all risk — XP won't protect you from being sued or from your software having bugs.

Practice 1: Small Releases

Release as often as possible, each release an increment that delivers value to the customer — always shipping the highest-business-value features first. Don't wait months or years for a finished product; release working software in short cycles. The professor's own undergrad team picked this practice, and its real benefit is the discipline it forces: it makes you constantly ask "what is the minimum work I can do to add new value?"

For Your Capstone Project

Even without a real customer, you can practice small releases — and earn bonus points for getting feedback from a non-developer. Think: what is the smallest release you could ship next week, and who outside your team could react to it? It also keeps you honest about how delayed you are, so you don't cram everything into the final week.

Practice 2: The Planning Game

The planning game aligns business needs with technical capabilities. A customer wants a feature but has no idea whether it's a trivial reuse of an existing library or a six-month research project. Only the developers can estimate that — so the team estimates effort and forwards it to the customer. XP implements this through planning poker:

1

The team gets cards labeled with Fibonacci numbers (not arbitrary numbers). They first calibrate: agree on a reference story that is, say, a "5."

2

Everyone silently estimates the effort of a story and places a card face down. Silence prevents anchoring on the first person to speak.

3

All reveal at once. The people with the highest and lowest estimates debate — "I know a library that does exactly this, so it's a 2" versus "you missed a requirement, there's a hard problem here, so it's a 13."

4

Estimate again with the shared insights, repeating until the team reaches consensus.

These Are Not Hours

The planning game does not produce a number of work-hours. The Fibonacci numbers are imaginary units (sometimes called "t-shirt sizes"). Different developers work at different speeds — "we call the most productive ones the 10x developers for a reason" — so estimating in shared abstract units reaches consensus far more easily than "I'd take an hour" versus "I'd take a week." The two real goals: reach a usable estimate, and share experience so everyone knows the edge cases and the shortcuts teammates have spotted.

Velocity

Key Concept — Velocity

A team's velocity is the sum of story points it can fit into one sprint. It answers "how many user stories can we realistically finish in the next increment?" — which is what you then promise the customer. Because the units are imaginary, one team's velocity of 200 versus another's 100 says nothing about who is faster; they just defined a "5" differently. Velocity is never a cross-team performance metric.

Check Your Understanding
Team A has a velocity of 200 and Team B has a velocity of 100. What can you conclude?
A

B

C

D

Correct: B. Story points are imaginary, team-relative units, so velocities are not comparable across teams — the professor stressed velocity is not indicative of performance. Options A and C both assume the points map to a universal scale or to hours, which is precisely the misunderstanding the planning game is designed to avoid.
Part VI

Test-Driven Development

The professor's favorite XP practice — "in my capstone I was known as test-driven Toby." Test-Driven Development (TDD) keeps you focused on the next feature, avoids scope creep, follows a rhythm, and produces high test coverage. And "you can't move fast if you have low test coverage."

The Counter-Intuitive Rhythm: Red, Green, Refactor

1

Red. Write a test first — and watch it fail. The test describes the feature before the feature exists.

2

Green. Write the smallest possible change that makes the test pass. No more production code than is sufficient to pass.

3

Refactor. Now clean it up into a tidy implementation, with the test as a safety net.

The TDD cycle: write a failing test, make it pass minimally, then refactor — and repeat.
The Big Idea — TDD Is a Design Tool, Not a Testing Tool

Despite "test" being the first word, TDD's main benefit is not finding bugs. It is a design approach. By writing the test first, you decide how a function will be used before you decide how it's built. The test is living documentation written before the implementation, and it forces you to design the interface from the caller's point of view.

Why TDD Improves Design — Through Information Hiding

The deeper reason connects to the course's information-hiding principle. Writing the test first means you define a module's contract — its interface — before writing any implementation. You start from "what is the easiest way to use this module?" Because you haven't made implementation decisions yet, the interface naturally hides design decisions. You begin from the interface, so the interface comes out clean.

No Debugging, Higher Productivity

TDD feels like extra work, but it boosts productivity — especially for junior developers. You never write code without a purpose: every line exists to pass a test. And the feedback loop is ~30 seconds. "A bug doesn't live longer than 30 seconds" — when a test fails, you wrote the offending code seconds ago, so it's still in working memory. There is essentially no debugging.

When TDD Produces Terrible Design

The professor posed the advanced question: when does following the TDD rules strictly lead to awful design? Two answers emerged.

1. The large-scale architecture problem

TDD goes test → implement → test → implement, module by module. Inside a module the design is great. But across modules, TDD never makes you think about which large-scale design decisions are likely to change — the thing that should define your module hierarchy. You just started with a test.

The Recommendation

Use TDD after you've designed the high-level architecture and its interfaces. First do the design work — decide how your top-level structure hides important design decisions — then implement each module individually with TDD. Don't TDD your entire application from scratch.

2. The "tower to the moon" problem

TDD starts from a very simple solution and adds complexity incrementally. That works only when the real solution is an incremental improvement on simpler ones.

The Big Idea — You Can't Reach the Moon by Building Taller Towers

If your problem needs a spaceship, you'll never get there by building taller and taller towers and feeling like you're getting closer. TDD only works for problems whose solution is an incremental improvement on simpler solutions. It fails for: well-known algorithms (just implement the one you learned in CS 180); ACID-compliant or distributed databases (you can't build up A → C → I → D); and systems with non-binary outcomes like image recognition or non-functional properties like response time and availability, where a better solution needn't pass every old test.

Robert C. Martin's Three Rules of TDD

(1) Don't write production code until you have a failing unit test. (2) Don't write more of a unit test than is sufficient to fail. (3) Don't write more production code than is sufficient to pass the test. The reason to see the test fail first: if you've never watched it fail, you can't trust it isn't a false-positive tautology that always passes.

Correct: B. You can't build an ACID/distributed database by passing A, then C, then I, then D — the correct design must be conceived whole, like a spaceship rather than an ever-taller tower. A and C are exactly the incremental, well-structured problems TDD excels at, and the professor noted TDD actually pairs extremely well with AI (you write the test, the AI makes it pass), so D is wrong.
Part VII

Collective Code Ownership & Coding Standards

Collective Code Ownership and the Bus Factor

The goal of collective code ownership is to raise the team's bus factor.

Key Concept — Bus Factor

The bus factor is the number of team members who could suddenly become unavailable — "hit by a bus," get sick, handle a family emergency, or leave — without the project stalling. A low bus factor (e.g. one person owns the critical feature) is a single point of failure. You raise it by making sure everyone feels comfortable changing every module.

Under XP, "every person may and should feel comfortable making changes" to any module — no code is owned by a single person. The benefits: fewer bugs (more eyes on the code), no single point of failure, faster feature delivery, fewer bottlenecks. The research (Laurie Williams, ~2010) supports that it prevents knowledge silos and keeps progress going when an owner leaves.

The Failure Mode — and Google's Counterpoint

The pure version has a flaw: "if everybody is responsible, nobody feels responsible," which yields mixed styles and spaghetti code. Google does something subtly different: every module has one designated owner who must review every pull request. This isn't the opposite of collective ownership — anyone can still make changes — it just ensures one person feels responsible for keeping changes consistent with past design decisions. The research even found higher team cohesion in non-XP teams, though the professor suspects a modern replication might differ.

Coding Standards

The cure for the spaghetti-code risk is another XP practice: coding standards.

Key Concept — Coding Standards

The goal of coding standards is that when you look at a piece of code, you cannot tell who wrote it. If you can identify the author, people are using different styles, which hurts readability. When all code looks the same, patterns are easier to recognize and the code is easier to understand. Standards cover indentation, capitalization, variable naming, and commenting.

The Standard Itself Doesn't Matter — Consistency Does

Open-source projects host endless debates about whether a space goes before or after a bracket. The professor's point: it does not matter what the standard is, only that everybody sticks to it. The moment that matters most is when you merge into main.

Automate It — Pre-Commit Hooks & AI

Make standards mandatory with git pre-commit hooks: before a commit is allowed, run a command that checks tests pass and the code conforms. This stops you from accumulating non-conforming code you "promise to fix later" (you never do). And if your team uses AI, put the standards in a markdown file (e.g. a CLAUDE.md) so the agent conforms — which also stops it from "randomly" restyling your code and producing a huge diff.

The Big Idea — Use AI Effectively

The professor closed with Jensen Huang's commencement line: "You're not going to be replaced by AI — you're going to be replaced by somebody who uses AI more effectively than you." Coding standards in your AI config are one concrete way to use AI well and set yourself apart.

Check Your Understanding

Correct: B. If you can identify the author, styles differ and readability suffers; uniform code lets everyone recognize patterns faster. There is no "one correct" style (A is the very debate the professor called pointless — only consistency matters), and the point is precisely to make authorship invisible, the opposite of C.

Putting It All Together

This lecture spanned from a verification technique to a philosophy of how software gets made — but a single thread runs through it: get feedback as early and as cheaply as possible.

Symbolic execution (Part I) automates the search for test inputs so you can reach 100% path coverage without burning human effort — feedback about your code's behavior, cheaply. That same instinct, applied to the whole development process, is what dooms the waterfall model (Part II): by deferring testing, customers, and deployment to the very end, waterfall makes every feedback signal arrive when it is most expensive to act on.

The Agile Manifesto (Part III) is the industry's correction — pull feedback forward through short, working iterations — but only if you read it as a ranking of priorities, not a license to abandon design. INVEST (Part IV) makes the unit of agile work, the user story, schedulable and estimable, which feeds directly into Extreme Programming's (Part V) small releases and planning game, where the team converts uncertainty into shared estimates and velocity.

Finally, the day-to-day practices tie back to the course's deeper principles. TDD (Part VI) is really the information-hiding principle in motion — design the contract before the implementation — but it only works for incrementally-improvable problems and only inside a pre-designed architecture. Collective code ownership and coding standards (Part VII) protect the team itself: a high bus factor and invisible-authorship code keep the project alive and legible when people inevitably come and go. The recurring meta-lesson: as AI writes more code, your edge is knowing which process and practice to apply, and why.
