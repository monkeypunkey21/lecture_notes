Testing — Finding the Bugs Before They Find You

An interactive guide to Lecture 10 — why we test, the levels and styles of tests, equivalence classes, the difference between strong and brittle tests, and how coverage measures what we've actually checked.

Part I

Why Test At All?

Before we learn how to test, we need to feel why it matters. The cheapest way to convince an engineer that testing is non-negotiable is a horror story — a real one, with a real price tag.

The Knight Capital Disaster

Knight Capital was a high-frequency trading firm — the kind that buys a stock and sells it milliseconds later for a sliver of profit. By 2012 they were the largest trading house on Wall Street, handling roughly 17% of all volume on the major exchanges, about $121 billion of trades a day. Then, on August 1st, 2012, they deployed a new software version to support a new exchange feature.

1

8:00 a.m. — Everyone in the company received mysterious warning emails. Nobody acted on them. (Why would you panic over a vague warning at 8 a.m.?)

2

9:31 a.m. — One minute after the market opened, Knight's program began flooding Wall Street with buy orders at $148 million a minute. It would not stop.

3

They tried to roll back to the previous version. This made it worse: the program now traded at eight times the volume. They had eight servers; seven had the new code, but the eighth had never received a critical patch — so rolling back put all eight on the broken path.

4

28 minutes to find the off switch (trading systems don't have a big red button). In those 28 minutes they lost $440 million — more than they had in cash reserves. The company was effectively bankrupt.

The root cause: an old feature flag that, nine years earlier, had activated "Power Peg" — dead code that deliberately simulated a dumb trader (buying high, selling low). That code had been dormant for nine years. The new feature reused the same flag. Flipping it on reawakened the nine-year-old dead code. It earned a place in the Guinness Book of Records as the most expensive software failure in history.

The Lesson the Class Drew

Students brainstormed three engineering principles that would have caught this: (1) test in a separate sandbox environment before deploying to production, (2) use automated static analysis to flag dangerous patterns, and (3) delete dead code you no longer need. Today's lecture is about the first one: testing before deployment.

What Testing Actually Is

It's tempting to treat "testing" as a vague good habit. The lecture gave it a precise definition you should be able to recite.

Key Concept — Testing

Testing is the execution of a program with selected inputs in a controlled environment, with assertions on its behavior and output.

Unpack that definition word by word, because every clause does work:

  • Execution of a program. Testing is dynamic analysis — you can only test code you can actually run. (Contrast with static analysis, which inspects code without running it.)
  • Selected inputs. You can't test exhaustively — most programs have an effectively infinite input space. You must deliberately choose a small, telling set of inputs.
  • Controlled environment. You need observability: you can only assert on something you can actually see. A stock-trading platform is hard to test precisely because it's hard to run safely in a controlled way.
  • Assertions on behavior. The whole point. You state the expected behavior, then check it — usually on outputs, but sometimes on other observable properties.
Testing Can Only Prove Bugs Exist

A famous slogan (Dijkstra): testing can prove the presence of bugs, but never their absence. Because testing is dynamic, it only ever exercises the finite inputs you picked. This connects to theory of computation: deciding whether an arbitrary non-trivial property holds for all inputs is undecidable (Rice's theorem) — at best semi-decidable. So no test suite, however large, certifies "bug-free."

The Three Goals of Testing

Why write tests? The obvious goal is finding bugs — but the lecture pulled out three reasons, and the third is the deepest.

1

Reveal bugs. The headline purpose. Find faults before your users — or Wall Street — do.

2

Document what the code should do. When you join a new project, the tests are the best place to learn the intended behavior. And because tests are executed, they can't silently rot out of date the way comments do. They are an executable specification.

3

Cognitive offloading → confidence to change. The professor's favorite. Tests give you the confidence to make changes without manually re-verifying everything by hand. That frees your limited cognitive load — and your teammates' — to focus on the actual design problem. Tests are how a team safely keeps moving.

The Big Idea

You are not paid to write code, and you are not even paid to write tests. You are paid to think about what might happen in the future. Tests are cognitive offloading: they let you and your team stop holding the whole system in your heads, so your attention goes to design instead of fear.

Verification vs Validation

Quality assurance splits into two questions that sound similar but are not. The course calls this "V&V."

VerificationValidation
"Did we build the system right?""Did we build the right system?"
Does the implementation conform to the specification?Does the specification actually solve the real-world problem?
This is what testing (today's topic) does.This is what manual testing with real users does — not today's focus.

Today we live entirely in the verification world: we assume a specification and check whether the code matches it.

Testing Is Not Debugging

People blur these two, so the professor drew a sharp line:

Key Concept — Testing vs Debugging

Testing is the act of detecting and revealing errors — it surfaces a symptom. Debugging begins only after a fault has been revealed: it's the process of tracing that error to its root cause and fixing it. No symptom yet? Then you're not debugging.

Check Your Understanding
A developer adds a new feature, runs the suite, and one test turns red. Which describes what happens next, in order?
A
Debugging revealed the fault; testing then traces it to its source.
B
Testing revealed the fault; debugging then traces it to its source and fixes it.
C

D

Correct: B. Testing is detection — the red test is the revealed symptom. Debugging is what follows: tracing that symptom to its root cause and fixing it. A reverses the two roles. C confuses V&V (those are QA categories, not a detect-then-fix sequence). D is the classic fallacy — testing can show a bug is present but never that all bugs are gone.
Part II

Levels of Tests

Not all tests look at the same amount of code. The level of a test is essentially a question of size: how much of the system does this one test exercise at once?

Unit, Integration, System

Key Concept — The Three Levels

Unit tests exercise one small piece — a single class or method in isolation. Integration (component) tests check that several units work together correctly. System tests are end-to-end: a user supplies input and you assert on output the user can actually see.

Think of a chest of drawers. Each drawer slides on its own (units pass), but if the frame is misaligned you still can't open one of them (integration fails). The professor's running example: every unit test is green, yet the combination of components doesn't do its job. You need tests at every level.

The terminology isn't perfectly crisp. A single class or method is always a unit test. A larger chunk might be called a "component test" — really just a big unit test — and the categories can overlap. The professor prefers the clean trio: unit → integration → system.

The test pyramid: many small, fast unit tests at the base; fewer, larger system tests at the top. Size grows upward.

If Integration Tests Are "Better," Why Keep Unit Tests?

A natural question: end-to-end tests prove the whole thing works, so why bother with unit tests at all? The class surfaced four good reasons:

1

Bug localization. A failing system test tells you "something is broken" but not where. A broken unit test points to a tiny, specific unit — the bug's location is almost handed to you.

2

Speed. Change one unit, run only that unit's tests. The professor's e-book tutorial suite takes ~30 minutes end-to-end; re-running just one unit's tests is near-instant.

3

You can test before integrating. While you're still developing a component, the rest of the system may not exist yet. You literally can't run an integration test until things are integrated — but you can unit-test the piece in your hands today.

4

Coverage of the input space. The deepest reason. A system test only exercises the paths the current composition forwards down to a sub-component — a narrow slice of that sub-component's possible inputs. But software grows and changes; future code will drive different inputs into that sub-component. Unit tests let you cover a much wider input range for the piece directly, so it stays correct as the system evolves.

From Acceptance Criteria to Tests: Gherkin, Cucumber & CI

Earlier in the course you learned to write user stories with good acceptance criteria. The payoff arrives here: well-written acceptance criteria translate almost mechanically into tests.

Gherkin is the language of Cucumber, an end-to-end testing framework. It takes structured natural language and, using deterministic rules (not an LLM), turns it into executable tests. The structure mirrors acceptance criteria exactly:

# Gherkin — Given / When / Then
Given the user has no prior passwords    # precondition / setup
When  they submit "Sunfl0wer!" as new   # the action under test
Then  the password should be accepted   # the assertion / oracle
  • Given — the preconditions: everything that must be true before you exercise the feature.
  • When — the action under test: the sequence the system actually performs.
  • Then — the assertions: what the outcome must be.
Why This Matters For the Exam

When you write acceptance criteria, make each Given describe a precondition, the When the action, and the Then the assertion. Do this carefully and the translation to tests becomes "easy — barely an inconvenience." The exam's user-story question rewards exactly this discipline.

One caveat the professor stressed: Cucumber pays off only when you have a non-technical stakeholder on the team — a product owner who can read the plain-language criteria, confirm "yes, that's what I want," and even write more. Without such a stakeholder (as in this class's projects), Cucumber's overhead isn't worth it.

Once you have tests, run them in a continuous integration (CI) system — GitHub Actions, Travis, Jenkins — on every merge to the main branch, so main is always continuously verified. This is the automated version of "test in a sandbox before deploying" that would have saved Knight Capital.

Check Your Understanding

Correct: B. A system test only drives the inputs that today's composition happens to forward downstream — a narrow slice of any sub-component's possible inputs. Since software grows, future paths will feed it different inputs, so you unit-test the component directly across its full input range. Unit tests are also faster and better at localizing bugs (D is backwards — they're faster, not slower). C is invented.
Part III

Black Box vs White Box

The next classification cuts a different way. It's not about size — it's about how you design the test: do you look inside the code or not?

A Philosophy of Design, Not a Property of the Test

Key Concept — Black Box vs White Box

A black-box test is designed without knowing the implementation. You derive it purely from the specification / interface / contract — the method signature, parameters, and documented behavior. A white-box test is designed while looking inside the code, choosing inputs to cover its paths, branches, and statements.

It's About How You Came Up With the Test

Crucially, black vs white box is a property of the test's design process, not of the finished test. Once written, two tests can look identical; you often cannot tell from the code alone which philosophy produced it. The difference is the information you had in hand: black box = "here is the interface only"; white box = "here is the interface and the source."

This connects directly to the information hiding principle from earlier in the course. Black-box tests respect the interface — exactly the public contract that information hiding exposes — and stay ignorant of the hidden internals.

Why Have a Mix? The Trade-offs

Strengths of black-box tests

  • No implementation bias. If you know how the code is written, you tend to write tests that confirm your implementation rather than challenge it. Working from the spec alone keeps you honest.
  • Less brittle. Because they make no assumptions about internals, black-box tests usually don't have to change when the implementation changes.

Strengths of white-box tests

  • They target risky internal structure. Maybe there's one critical if condition, or one rarely-taken branch, that hides the bug. If you don't look at the code, you don't know that branch exists — so you can't deliberately write a test that exercises it. White-box design ensures every branch gets covered.
Use Both — They Catch Different Things

Black-box tests catch "the spec isn't met" without bias and survive refactors. White-box tests catch "this specific branch is wrong" by guaranteeing structural coverage. A good suite designs from both hats. We'll wear the black-box hat first in Part IV, then switch to the white-box hat in Part VI.

A is wrong because white-box doesn't imply touching internals (that's a separate sin called brittleness); C wrongly conflates this axis with the size/level axis from Part II.
Part IV

Equivalence Classes & Boundary Values

Now we put on the black-box hat. The core question of input selection: a program has infinitely many possible inputs, so which few do we pick? The answer is equivalence partitioning plus boundary value analysis.

Key Concept — Equivalence Classes

An equivalence class is a group of inputs the program should treat the same way. If two inputs fall in the same class, testing one is as good as testing the other. So you pick one representative per class instead of testing every value. The valid inputs typically form one class; the invalid inputs split into several — one per way of being invalid.

The Worked Example: A Password Validator

The system is PasswordValidator.validPassword(username, password). A password is valid only if all four criteria hold:

// PasswordValidator — the contract (black box)
boolean validPassword(String username, String password)

// 1. length is 8–20 characters (inclusive)
// 2. not a password used previously (needs a PasswordDatabase)
// 3. has ≥1 uppercase, ≥1 lowercase, ≥1 digit, ≥1 special char
// 4. must not contain the username
The Mark of a Great Engineer

Before writing a single test, the professor asked: "What else do you need to know about this system?" A good engineer starts coding; a great engineer asks questions first. Here the missing piece was the existence of a separate PasswordDatabase class (with functions to check past passwords and to store new ones) — and, as we'll see, an under-specified detail about case sensitivity.

Criterion 1: Length — Why Four Tests, Not Three

Focus on "8–20 characters." There are three equivalence classes: too short, valid length, and too long. So why not just three tests? Because bugs hide at boundaries: the place developers most often slip is < vs <=. To pin down both boundaries exactly, you test the values right on either side of each:

Boundary value analysis for the 8–20 length rule. Three equivalence classes, two boundaries → four tests, one straddling each boundary.

With only three tests you can nail just one boundary; with four — testing 7, 8, 20, and 21 — you confirm both the lower and upper boundary land in exactly the right place. The other three criteria are held valid in these length tests so they don't interfere.

Criterion 3: Character Classes — Four Tests, Not Five

The password needs an uppercase, a lowercase, a digit, and a special character. The instinct is five tests (one all-valid, plus one missing each). But the all-valid case is already covered by a valid-length test from criterion 1. So you only need to add four negative tests: one missing uppercase, one missing lowercase, one missing a digit, one missing a special char.

Build the Suite Incrementally

Notice the pattern: each new criterion adds only the tests not already covered. This is how you optimize for the smallest suite that still gives confidence. A more exhaustive engineer might test all combinations (e.g., 16 or 48 tests) — and that's a legitimate choice for a truly critical feature. But for a password validator (a bug here won't bankrupt you, unlike a flaw in password storage), the lean per-criterion suite is the better trade-off. Testing is part science, part art, calibrated to how critical the feature is.

Criterion 4: The Under-Specified Case

"Must not contain the username" needs one test (e.g., username testuser embedded in the password). But it exposes a hidden ambiguity: if username is testuser and the password contains TestUser, is "contains" case-sensitive or not? The spec doesn't say.

The Big Idea — Testing Is a Dialogue

Systematically writing tests forces you into exactly these under-specified corners. When you hit one, the right move isn't to guess — it's to ask the stakeholder who owns the requirements. Testing is not a one-way street; it's a conversation with whoever defines the spec. Good tests surface the questions the requirements forgot to answer.

Check Your Understanding
For the "8–20 characters" rule, why does black-box testing prefer four tests over three?
A

B
Three tests can pin down only one boundary; four (7, 8, 20, 21) pin down both boundaries, where off-by-one bugs hide.
C

D
You always need exactly N+1 tests for a range of length N.
Correct: B. There are three equivalence classes (short / valid / long) but two boundaries. Each boundary needs a value just inside and just outside it to catch the common < vs <= slip — hence 7&8 and 20&21. A miscounts (there are three classes, not four). C invokes path coverage, which is a white-box notion irrelevant to this black-box reasoning.
Part V

Strong, Weak & Brittle Tests

Picking good inputs is half the battle. The other half is the assertion: what does the test actually check? This is where tests succeed or fail at their job.

Oracles and Strong Assertions

Key Concept — The Oracle

The oracle is the part of a test that decides, for a given input, what the correct output is. A strong oracle has as many (correct) assertions as possible: it should fail for every incorrect program. The more relevant aspects of the specification you assert, the more bugs you catch.

Two opposing forces define a good test, and balancing them is the heart of the craft:

A test should......but a test should NOT...
fail for every incorrect program (strong, not permissive — a weak test lets broken behavior pass)fail for any correct program (not too strong / brittle — it must pass for all valid implementations)
A Test That Lies

The professor's Gandalf meme: "You shall not pass!" — pointed at a lying test, one that passes when it should fail (it fails to catch the bug it was meant to catch). A common cause is a test that fails for the wrong reason. To guard against this, write the test before the implementation: see it go red, implement the feature, watch it go green. That's TDD (Test-Driven Development), and seeing the red first proves the test can actually fail.

Testing State: A Sequence of Steps

The first three criteria were single calls with single assertions. But "must not be a password used previously" means the system has state — and testing stateful behavior gets trickier, because you need a sequence of steps.

// Testing the "no reuse" rule — a stateful sequence
// 1. sanity-check: the chosen password is valid to begin with
assert validator.validPassword("testuser", pw) == true;

// 2. change the user's password to pw (now pw is "used")
database.changePassword("testuser", pw);

// 3. pw must now be rejected as previously-used
assert validator.validPassword("testuser", pw) == false;

Why is the step-1 assertion there? To make the test fail for the right reason. If pw were accidentally invalid (a typo, or a changed definition of validity), step 3 would still see "rejected" — but for the wrong reason. Asserting validity up front guarantees that a later rejection is genuinely caused by reuse, not by a bad starting password.

Making the assertion stronger

You can strengthen this test without writing a whole new one — cheaper than a separate test, and it catches more:

  • Repeat with a second new password — maybe reuse-rejection only works for the most recent password, not earlier ones. Cycle through multiple changes.
  • Also assert on the hash — if there's a hashPassword function, assert the stored value equals the hash. Semantically linked assertions belong together.
  • Use assertion failure messages — e.g. "hashPassword incorrect" vs "validPassword incorrect" — so when one of several assertions fails you instantly know which behavior broke.

The Brittle Test — Testing Internals

Now the cautionary version. Suppose the PasswordDatabase stores passwords as a dictionary from username to a list of passwords, and a test reaches in to check that the new password appears in that dictionary:

// ⚠ A brittle test — reaches into internals
database.changePassword("testuser", pw);
assert database.passwords.get("testuser").contains(pw);  // 👎
Why This Is the Worst Test

Two problems. First, it asserts on an implementation detail — the exact data structure used for storage — not the contract. Change the dictionary to anything else and the test breaks even though behavior is correct. Second, security: storing the clear-text password is a serious flaw; the password should be hashed. Asserting on the stored hash is slightly better, but if you hard-code a specific hash function you're still tied to an implementation detail that can change.

Key Concept — Weak vs Brittle

A weak test lets broken behavior pass (too permissive). A brittle test rejects correct implementations (too strict — it breaks on a refactor that didn't change behavior). The goal: accept all correct implementations, reject all the important incorrect ones.

Information Hiding Applies to Test Code Too

The way passwords are stored is hidden behind the contract — no client, including a test, should depend on it. A brittle test is a violation of information hiding. This is why the professor calls this "the most interesting insight in testing," and why information hiding had to come first: brittle tests only make sense once you understand it.

The Big Idea — Bad Tests Slow You Down

Real student feedback from a prior course: every tiny UI change broke their end-to-end tests, so the tests "gave us anxiety, not confidence." Good tests speed you up; bad (brittle) tests slow you down. The fix: test behavior, not implementation — and for parts of the system still in flux (a UI that may change next week), delay writing brittle tests until that part stabilizes. You're paid to anticipate the future: design tests that won't break when correct code legitimately evolves.

Check Your Understanding
A test asserts that a private dictionary field contains the new password after changePassword. Behavior is correct, but the test fails after the team swaps the storage structure. What kind of test is this?
A

B

C

Correct: B. The behavior is right yet the test fails — that's the signature of brittleness: it rejects a correct implementation because it depends on a hidden storage detail, violating information hiding. It's the opposite of weak (A), which would pass broken behavior. D is the common confusion the lecture warned against: white-box means designing inputs from the code's structure, not asserting on internals.
Part VI

Coverage — Measuring What You've Tested

Now switch to the white-box hat. We've been choosing inputs from the spec; now we look inside the code and ask: have we actually exercised all of it? The yardsticks are test adequacy criteria.

Key Concept — Test Adequacy Criteria

Adequacy criteria answer "have we tested enough?" by measuring coverage. They guard against two failures: writing too few tests (missing behavior) and writing redundant tests (wasting execution time). There are three, in increasing strength: statement, branch, and path coverage.

Why Efficiency Matters

If a suite takes longer than ~10 minutes, developers stop running it on every change — they'll just push and hope. So you want the minimum tests that reach your chosen coverage level. More tests beyond that add quality-assurance overhead without proportional benefit.

Statement, Branch, and Path Coverage

Consider this function with three independent if statements (no early returns):

void f(int x) {
  if (cond1) { x++; }
  if (cond2) { x--; }
  if (cond3) { x = x; }
}
Key Concept — The Three Coverages

Statement coverage: has every statement been executed at least once? Branch coverage: for every condition, has it evaluated to both true and false? Path coverage: has every possible route through the program been executed?

How many tests does each need for this function?

1

Statement coverage → 1 test. A single input where all three conditions are true runs every statement (x++, x--, x=x) at least once. Done.

2

Branch coverage → 2 tests. Each condition needs both a true and a false evaluation. One test with all conditions true plus one with all conditions false covers every branch. (This clean "just 2" works because there are no return statements interrupting the flow — it is not the general rule.)

3

Path coverage → 8 tests. Three independent two-way conditions give 2×2×2 = 8 distinct routes through the function. You'd need all eight true/false combinations to reach 100%.

The Counter-Intuitive One Is Branch Coverage

Students kept guessing "8" for everything. The trap: statement coverage feels like it needs many tests but here needs just one; branch coverage feels like it needs eight but here needs two. The reason branch ≠ statement here is the missing else branches — when there are no else blocks, the "false" outcome of a condition runs no extra statement, so statement coverage can be hit without ever evaluating a condition to false. (When conditions do have else branches, statement and branch coverage tend to coincide.)

A cumulative worked example

Now run three concrete tests against this function and track coverage as it accumulates. Each test is the (cond1, cond2, cond3) tuple:

After running...StatementBranchPath
Test 1: (T, T, T)100%50%1/8
+ Test 2: (F, F, F)100%100%2/8
+ Test 3: (F, T, T)100%100%3/8

Read the story in the table. Test 1 (all true) runs every statement → 100% statement immediately, but only exercises the "true" side of each condition → 50% branch, and one route → 1/8 path. Test 2 (all false) fills in every "false" branch → 100% branch, and adds a second route → 2/8 path. Test 3 adds only a third distinct route → 3/8 path, but contributes nothing new to statement or branch coverage.

The Lesson of Test 3

If your goal had been 100% statement coverage, test 1 alone sufficed. If 100% branch coverage, tests 1–2 sufficed. Test 3 added neither — only a marginal path bump that still doesn't reach 100% path coverage. The systematic move is to pick a target criterion and write the minimum tests to hit it, rather than adding tests that don't advance your chosen goal. Order matters too: running only test 2 first would not give 100% statement coverage, because its all-false inputs skip x++, x--, and x=x.

Control Flow Graphs: Doing This Systematically

How do you reason about coverage rigorously instead of by eyeballing? Draw a control flow graph (CFG).

Key Concept — Control Flow Graph

A CFG is a graph from the program's entry to its exit. At every branching point it shows how control can split, and it labels each path with the statements executed along it. It makes coverage mechanical: branch coverage = cover every outgoing edge of every condition; path coverage = cover every full route from entry to exit (all combinations of those edges).

CFG for f(x): each diamond is a condition with a true edge (executes the statement) and a false edge (skips it). Branch coverage = every edge taken; path coverage = every entry-to-exit route (2×2×2 = 8).
Picking a Realistic Target

In real codebases, full path coverage can mean tests that "run all weekend" — combinatorially infeasible. Branch coverage is the practical minimum bar: it guarantees every condition has been tried both ways. Even full statement coverage is rare in industry. So you pick the strongest criterion you can afford for the criticality of the code, then write the minimum tests to reach it.

Check Your Understanding
For f(x) with three independent ifs and no else or return, how many tests give 100% statement, branch, and path coverage respectively?

B

Correct: B. One all-true test runs every statement (statement = 1). Adding an all-false test makes every condition evaluate both ways (branch = 2) — this clean "2" works only because there are no early returns. Path coverage needs every route: 2×2×2 = 8. A is the seductive wrong answer the class kept blurting; D fails because all-true never evaluates any condition to false, leaving branch and path coverage incomplete.

Putting It All Together

This lecture moved from motivation to mechanism to measurement, and the pieces interlock. The Knight Capital disaster (Part I) is the whole course in one story: untested dead code, reused behind a flag, cost $440M in 28 minutes. Testing — execution with selected inputs and assertions — is verification's answer, and its deepest value isn't catching that one bug but the confidence to change safely.

That confidence only materializes if you test at the right levels (Part II): unit tests for fast, localized, future-proof coverage; integration and system tests for the assembled whole. You then design tests through two complementary lenses (Part III): the black-box hat derives tests from the contract — and in Part IV that meant carving the input space into equivalence classes and probing their boundaries, where off-by-one bugs live. The password validator showed both the technique and the meta-skill: a great engineer asks the spec questions first, and treats testing as a dialogue when the requirements turn out ambiguous.

Part V was the moral center: a good test has a strong oracle (fails for every incorrect program) yet is not brittle (passes for every correct one). The line between them is information hiding — assert on the contract, never on hidden internals — which is why bad tests give anxiety and good tests give velocity. Finally the white-box hat (Part VI) gave us a ruler: statement < branch < path coverage, made systematic with a control flow graph, always balanced against what's affordable.

The Through-Line

Black-box thinking (equivalence classes, boundaries) chooses which inputs to test. White-box thinking (coverage, CFGs) checks which code you've reached. Strong-vs-brittle thinking decides what to assert. A professional suite uses all three — and respects information hiding so that tests survive the future you're actually paid to anticipate.
