Counting Paths: Coverage, Mutation & Symbolic Execution

An interactive guide to Week 6 — how we measure whether tests are good, how to count the paths a program can take, and how to find the inputs (and the dead code) automatically.

Part I

Coverage Criteria — Measuring a Test Suite

Why Are We Learning This?

Suppose you wrote a function and then wrote some tests for it. A natural, uneasy question follows: are these tests any good? "Good" is vague, so engineers replace it with something measurable. A coverage criterion is a yardstick that asks: of all the things in the program that could be exercised, how many did your tests actually exercise? If your tests never run a line of code, that line could be completely broken and your tests would never notice. Coverage turns the fuzzy goal "test thoroughly" into a percentage you can put a number on.

This lecture builds the entire machinery for answering "how many tests do I need, and which inputs should they use?" Coverage is the first rung. By the end we will count paths exactly, then automate the whole thing with symbolic execution.

Three Criteria: Statement, Branch, Path

There are three coverage criteria in this course, and they form a hierarchy of strictness. Each counts a different kind of "thing" the tests must hit.

Key Concept — The Three Coverage Criteria

Statement coverage asks that every line (statement) of the program runs at least once. Branch coverage asks that every conditional takes both its true and its false outcome at least once. Path coverage asks that every complete route through the program from entry to exit is executed at least once.

Why does the professor call path coverage the most restrictive? Because a single path threads together one choice at every branch, and there are usually many combinations of choices. Hitting all of them implies you also hit every individual branch, which in turn implies you hit every statement. We say path coverage subsumes the other two: achieving it automatically achieves them.

Path coverage is the strictest: it contains (subsumes) branch coverage, which contains statement coverage.

Worked Example: Hitting 100% Branch Coverage

Consider a function with three independent conditions stacked in it — one checking x > 1, one checking y > 2, one checking z > 3. "Independent" means the outcome of one says nothing about the others. We want a set of inputs that makes every branch go both ways.

if (x > 1) { ... }
if (y > 2) { ... }
if (z > 3) { ... }
1

Test case A: x=0, y=0, z=0. Every condition is false, so this single test covers all three false branches at once — that is 50% branch coverage (three of the six branch outcomes).

2

Test case B: x=2, y=3, z=4. Every condition is true, so this covers all three true branches at once — the remaining 50%. Together the two tests reach 100% branch coverage.

3

Why this trick works: because the conditions are independent, "all false" and "all true" are both feasible inputs. The professor warned this "all-low then all-high" strategy is a good first attempt but does not always work — if conditions depend on each other, "all true" might be impossible.

Notice two classifications the professor drew out of this example. Because we picked these inputs by reading the code, this is white-box testing (testing informed by the implementation, as opposed to black-box testing, which only knows the spec). And because we test one function in isolation, this is unit testing — not component, integration, or system testing.

Control-Flow Graphs and the for-Loop Problem

To reason about paths, we first draw a control-flow graph (CFG): a diagram where rectangles are straight-line statements, diamonds are conditions, and arrows show how control moves. A loop becomes a cycle in the graph.

A subtlety the professor stressed: a for loop has no dedicated CFG shape. But a for loop is just a while loop with extra pieces bolted on through syntactic sugar (convenient surface syntax that expands into something more basic). So we desugar it into three parts: the initialization (i = 0), the condition (i < 2), and the increment (i++) which happens at the bottom of the loop body before jumping back to the condition.

// for (i = 0; i < 2; i++) { if (...) value += 2; }
i = 0;                  // init  (drawn first, once)
while (i < 2) {          // condition diamond
    if (...) value += 2;  // loop body
    i++;                // increment, then back to condition
}
return value;           // false branch of condition exits here
CFG for a desugared for loop. The back edge from i++ to the condition is the loop. The false branch leaves the loop and returns.

For a program like this with a small fixed loop, the professor asked: what is the maximum number of complete paths? For the "complex fun" example used in class, the answer was 4 — and that is exactly the minimum number of tests you would need for 100% path coverage.

Connection — Why This Sets Up Part II

Once you can draw a CFG and unroll loops, counting paths becomes mechanical. Part II turns that mechanical procedure into formulas: 2^k for branches, and finite sums for nested loops.

Part II

Counting Paths — From 2^k to Nested Sums

The Core Formula: 2^k and Loop Unrolling

Here is the foundational rule. Suppose a loop-free program executes k branches, each of which is a genuine true/false choice, and none of the resulting paths is infeasible. Then the number of complete paths is:

Number of paths through a loop-free program with k independent true/false branches.

The intuition: each branch doubles the number of possible routes, because for every route so far you can append "true" or "false." Two branches give 2×2 = 4; three give 8; and so on.

What about loops? Loops are the hard part because the number of branch executions depends on how many times the loop runs. The technique is loop unrolling: in your head, replace the loop with an equivalent loop-free program where the body is written out once per iteration. If the loop runs n times and the body contains k independent branches each iteration, the unrolled program has n × k branches, so the path count becomes 2^(n·k).

The Mental Move — Unroll the Loop

Whenever you need to reason about branch executions inside a loop, "unroll" it: imagine the equivalent program with no loop, where the body is copied out once per iteration. Then the plain 2^k rule applies to the copies. The professor returned to this move again and again.

Nested Loops and Finite Sums

Things get richer with nesting, because an inner loop's bound can depend on the outer loop's counter. Consider an outer loop running n times, and on iteration i an inner loop runs up to 3^i times. The total number of inner executions is a sum:

3¹ + 3² + 3³ + ... + 3ⁿ

The professor showed how to collapse this without memorizing the formula. Call the sum S. Multiply by 3 (shifting every term up by one), then subtract the original — almost everything cancels, leaving only the endpoints. Solving gives a closed-form formula (one with no "..." you can type straight into a calculator):

Closed form for 3¹ + 3² + ... + 3ⁿ. Equivalently written 3^(n+1)/2 − 3/2.

The same subtract-and-cancel trick recovers the famous sum 1 + 2 + ... + n = n(n+1)/2 when an inner loop simply counts up to i rather than to 3^i. The lecture re-derived it by writing the sum forwards and backwards, adding the two copies (every column sums to n+1, and there are n columns), then dividing by two.

Expand: re-deriving 1 + 2 + ... + n by hand

Write the sum twice, once forwards and once backwards, then add column by column:

  S =   1   +   2   + ... + (n-1) +   n
  S =   n   + (n-1) + ... +   2   +   1
─────────────────────────────────────────
 2S = (n+1) + (n+1) + ... + (n+1) + (n+1)

There are n columns, each equal to n+1, so 2S = n(n+1) and therefore S = n(n+1)/2. No memorization needed — just the trick.

Conditions That Don't Actually Exist

Here is the most surprising idea of the part. Not every if in the source adds paths. Look at a loop whose condition is (2^k mod 2) == 0. Since 2^k is always even, 2^k mod 2 is always 0, so the condition is always true. There is no input that makes it false. When you unroll this loop, that if simply isn't there — it contributes one route, not two, and adds nothing to the path count.

Common Mistake — Counting Phantom Branches

An if in the source code does not always mean a real branch. If a condition can only ever be true (or only ever false), it adds no paths — it is effectively dead code. Counting it as a real true/false branch over-estimates the path count badly (e.g. 2^n instead of 2^(something smaller)).

An even subtler version: an inner loop compares A[i] < k. Unrolled, the condition appears 2^n − 1 times — but it only ever compares against the same k using values A[0], A[1], ..., A[n-1]. There are only n distinct conditions, indexed by the outer loop. So even though the condition is written 2^n − 1 times, the number of distinct paths is only 2^n, governed by how many distinct values of A[i] exist. The lesson: count distinct feasible decisions, not textual occurrences.

Part III

Oracle Strength — Why Coverage Isn't Enough

The Hole in Coverage

Imagine a company with one rule: every test suite must reach 100% branch (or path, or statement) coverage. The professor asked the class to find the flaw. Several surfaced, but one is fatal and easy to miss.

Coverage only constrains which paths the test executes. It says nothing about what the test checks. You could write a test that runs every line and every branch and then asserts… nothing at all. Such a test only catches crashes — unhandled exceptions or hangs. It never notices that the program computed the wrong answer, because it never looks at the answer.

Key Concept — Oracle Strength

The oracle of a test is the part that decides pass or fail — its assertions. Oracle strength measures how strong those assertions are: how many real bugs the test could actually detect. Coverage and oracle strength are independent axes of test quality. You can have high coverage with a weak oracle (no assertions) or strong assertions with poor coverage.

Other weaknesses the class raised: chasing 100% path coverage can be infeasible or pure overhead; brittle, unmaintainable tests; and tests that pass on correct-input/correct-output but tell you nothing about where a failure happened. Concurrency makes it worse — a "singleton" with only two branches can still hide enormous behavioral variety. The takeaway: no single metric captures test adequacy.

Mutation Testing — Measuring the Oracle

So how do you measure oracle strength concretely? The class derived the answer through a series of hints. The purpose of tests is to find bugs — so to measure how well tests find bugs, deliberately inject bugs and see if the tests catch them.

Key Concept — Mutation Testing

Mutation testing automatically creates many slightly broken copies of the program — mutants — by mechanically flipping operators (e.g. changing <= to >, + to -). Each mutation is guaranteed to change behavior, so a correct test suite should fail on it. A mutant whose injected bug is caught by some failing test is said to be killed. The mutation score is the fraction of mutants killed.

1

Start from the real implementation. Mechanically mutate one operator at a time — flip <= to >, swap + for -. Each change is known to be a bug because it reverses the intended semantics.

2

Run the full test suite against each mutant. If at least one test fails, the bug was detected — the mutant is killed.

3

If a mutant survives (no test fails despite a guaranteed bug), your assertions are too weak. A high mutation score means strong assertions; you have caught the bugs you knew how to inject.

Limitation — Single-Point Mutations

Standard mutation operators change exactly one place at a time, on purpose: that way the change is guaranteed to be a bug and two changes can't accidentally cancel out. The cost is that some real bugs only appear when two places change together — these are missed. (Bugs requiring multiple simultaneous mutations relate to higher-order mutants.) Mutation testing can never prove you thought of every bug, only that you caught the injected ones.

Two Axes, One Goal

Coverage answers "did the test reach the code?" Mutation answers "would the test notice if the code were wrong?" A research metric called the mutation gap combines both to tell you, at any moment, whether to invest more in coverage or in stronger assertions.

Correct — B. Coverage guarantees the paths execute, but without assertions the test never inspects the result, so it only notices failures that stop the program — unhandled exceptions or hangs. It will happily pass a program that computes the wrong answer. This is exactly why oracle strength (measured by mutation testing) is a separate, necessary axis. A is the seductive wrong answer.
Part IV

Test Fixtures & Black-Box Thinking

Test Fixtures — Independent, Repeatable Tests

Many tests need a fresh object to work with — a new database connection, a new account, a new Rational number. If tests share one object, the result of one test can taint the next, and the suite passes or fails depending on the order tests run in. That is a serious smell.

Common Mistake — Flaky Tests

A flaky test is one whose pass/fail depends on the order it runs in, or on shared mutable state. Tests should each verify program behavior independently; if reordering them changes the outcome, they are not good enough.

Key Concept — Test Fixture

A test fixture is a fresh object built from scratch for every individual test. A setup procedure creates it before each test; a teardown procedure cleans up afterward (e.g. disconnects the database). Every test gets a brand-new, isolated instance, which keeps tests independent and removes duplicated setup code.

Fixtures exist in every framework — JUnit (Java), pytest (Python), Playwright. The syntax differs but the idea is identical: factor the create/destroy boilerplate into setup/teardown so each test starts clean. This pays off most for large suites that all revolve around the same kind of object.

Black-Box Testing — The Bar Comic

Recall the distinction: white-box testing picks inputs by reading the code (Part I); black-box testing picks inputs from the specification and from imagining what a real user might do, without looking inside. Black-box testing leans on equivalence classes — groups of inputs the system should treat the same, so you test one representative of each.

The professor illustrated this with a famous comic. A QA engineer walks into a bar and tests every variation he can dream up:

The QA Engineer Walks Into a Bar

He walks in. Runs in. Crawls in. Dances in. Flies in. Orders 1 beer. 2 beers. 0 beers. 99,999,999 beers. −1 beers. A lizard in a teacup. Every equivalence class — normal, boundary, zero, huge, negative, nonsense — is handled correctly. Testing complete. Then a real customer walks in and asks, "Where's the bathroom?" — and the whole bar bursts into flames.

The punchline is the lesson. The tester covered every input he could imagine, but a real user did something nobody anticipated. You can only black-box test the cases you think of — which is why understanding what users actually need is essential. This is the bridge to requirements engineering and user stories: if you don't understand the user, you can't write the right black-box tests, and no amount of coverage saves you.

Connection — Equivalence Classes Cut the Work

You can't test infinite inputs, so group them into equivalence classes (e.g. "negative", "zero", "huge") and test one from each, plus the boundaries between them. This is the black-box counterpart to the white-box path-counting of Parts I–II.

Part V

Symbolic Execution — Finding Inputs and Dead Code

Infeasible Paths and Dead Code

We've been counting paths as if all of them can actually happen. Some can't. An infeasible path is a route through the CFG that no input can ever trigger — the code on it is dead code. You can't test dead code, and its presence usually signals a bug.

Some dead code is obvious (the always-true condition from Part II). But consider this:

x = 0;
if (x > 1) {   // can this ever be true?
    ...        // INFEASIBLE — x was just set to 0
}

Looking at the condition x > 1 alone, you cannot tell whether the branch is reachable. You have to look at the previous statement: x was just assigned 0, so x > 1 is impossible. The branch is infeasible. The question is how to detect this automatically, for large programs, without slow and expensive human reasoning. That technique is symbolic execution.

Path Conditions and Effects

Symbolic execution runs the program on symbolic values — fresh variables standing for "any input" — instead of concrete numbers. As it walks each path it accumulates two things.

Key Concept — Path Condition & Effect

The path condition of a statement is the conjunction of all the branch conditions that must be true to reach that statement. The effect of a statement is the logical fact we know holds afterward — for an assignment x = e, the effect is x' = e (a fresh x' equals the new value). Symbolic execution propagates both branches at every condition, carrying these formulas along each path.

An infeasible path is then detected purely logically: gather the path condition together with the effects of earlier statements, and check whether they are contradictory. In the example above, the effect x' = 0 contradicts the path condition x' > 1. A constraint solver flags the contradiction — no human inspection required.

A Worked Example

Take a tiny program and compute its symbolic result:

if (x > 1)
    x = x - 2;   // statement 1
else
    x = 2 * x;   // statement 2
1

Statement 1's path condition is simply x > 1 (the if was taken). Its effect is x' = x - 2. Combined: x > 1 ∧ x' = x - 2.

2

Statement 2's path condition is the negation, ¬(x > 1). Its effect is x' = 2x. Combined: ¬(x > 1) ∧ x' = 2x.

3

Combine the branches. Exactly one runs, and the two path conditions are mutually exclusive, so we join them with OR. The whole program's symbolic result is: (x > 1 ∧ x' = x-2) ∨ (¬(x > 1) ∧ x' = 2x).

The lecture also did a harder example: a function returns early when x < 4, then later branches on whether a parabola x² exceeds a line 3x + 1. Because of the early return, every statement past it carries the path condition x ≥ 4. Carrying that forward, one branch demands a condition that x ≥ 4 makes mathematically impossible — a logical contradiction — so that branch is infeasible / dead code. A human might "see" it; symbolic execution proves it systematically for programs far too large to eyeball.

The Big Idea — Automating the Whole Lecture

Symbolic execution ties Parts I–II together and automates them. From the path conditions it can (1) generate the concrete inputs needed to cover every path — answering "which inputs reach 100% coverage?" without guessing — and (2) detect infeasible paths by finding contradictions. It's the engine behind the static analyzers and compiler warnings ("this condition is always false," "possible null dereference") you already see.

In the Real World

When your static analyzer warns about an impossible condition or a null-pointer dereference, symbolic execution is running underneath, accumulating path conditions and checking them with a solver. As a developer you can also use it to auto-select the inputs that cover all paths.

Part VI

Testing Real Systems — When Bugs Cost Everything

Bounded Loops and the Mars Helicopter

NASA's Jet Propulsion Laboratory (JPL), in nearby Pasadena, publishes coding rules every developer must follow. Rule #2: give every loop a fixed upper bound — no loop may run up to an unknown n.

Why? Everything in Parts I–II explains it. If a loop runs n times and you don't know how large n can get, the number of paths is 2^(something·n) with no ceiling — you'd need an unbounded number of tests to cover it. Bounding the loop bounds the path count, which bounds the number of tests, making the program testable at all.

Connection — This Is Why 2^k Matters

The abstract formula from Part II has a concrete consequence: an unbounded loop means unbounded paths means infinite tests. Bounding loops is a direct, practical application of path counting to safety-critical engineering.

JPL built the Ingenuity Mars helicopter — flight on a planet with ~1% of Earth's atmosphere and ~1/3 its gravity. They couldn't fly to Mars and back to fix bugs (a round-trip signal takes 20+ minutes; the craft must be fully autonomous), so they had to test exhaustively on Earth. To simulate the environment they built a pressure chamber (evacuating the air and tuning the gas mix) and hung the helicopter from a spring that offloaded about two-thirds of its weight to mimic Martian gravity.

The Bug That Slipped Through — A Race Condition

Despite all that testing, a concurrency bug survived: a watchdog component waited on signals from other components, but a race condition meant the signals could arrive in a different order than Earth testing ever produced. That ordering blocked humanity's first powered flight on another planet. JPL read the logs, found it, and uploaded a software patch to Mars. Ingenuity then flew — far beyond its design life — and now serves as a weather station. Concurrency / event-ordering bugs are exactly where the hardest defects hide.

Therac-25 — When the User Outran the Tests

The Therac-25 was a radiation-therapy machine that reused code from an older model in new hardware. Operators (nurses) entered the dose through a UI. The fatal scenario was a real-world echo of the bar comic's "grandma asking for the bathroom": operators became so fast at typing that they entered values quicker than anyone had during testing.

That speed left no time for the machine's physical components to move into position — a race between the software state and the hardware. The result was a misconfiguration that delivered roughly a thousand times the intended radiation dose, killing at least six people. The bug wasn't an exotic input; it was ordinary use performed faster than the testers imagined — the same lesson as the comic, now lethal.

The Big Idea — Test the Cases That Matter, Especially Ordering

In real, complex systems you cannot cover every path. The discipline is to identify the paths that matter and to scrutinize concurrency and event ordering, because that is where the worst bugs live. Either guarantee events are always ordered correctly, or make the system gracefully degrade when they aren't.

Putting It All Together

This lecture answered one practical question from six angles: how do we know our tests are good, and how many do we need?

We began with coverage criteria (Part I) — statement, branch, and path — and the subsumption hierarchy that makes path coverage the strict gold standard. To count paths we drew control-flow graphs, desugared for loops, and arrived at the engine of Part II: 2^k branches, loop unrolling for loops, and finite sums for nested loops. The crucial refinement was learning that a textual if is not always a real branch — always-true conditions and repeated comparisons against the same value collapse the path count, so we count distinct feasible decisions.

But Part III exposed coverage's blind spot: it measures where tests go, never what they check. Oracle strength, measured by mutation testing (inject bugs, count the kills), is the orthogonal axis. Part IV grounded testing in practice — fixtures for independent, non-flaky tests, and black-box / equivalence-class thinking, with the bar comic's warning that you can only test what you imagine. Part V then automated Parts I–II: symbolic execution uses path conditions and effects to generate covering inputs and to prove infeasible paths via logical contradiction. Finally Part VI showed the stakes — bounded loops (so path counts stay finite), and the Mars helicopter and Therac-25, where the surviving bugs were concurrency and event-ordering failures triggered by real-world use no one anticipated.

The through-line: no single metric is enough. Combine coverage (reach) with oracle strength (detection), automate input selection with symbolic execution, bound your loops so the math stays finite — and never stop asking what a real human will actually do with your system.
