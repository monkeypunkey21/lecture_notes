Refactoring & Code Smells

How to transform working code into clean code — juggling your classes on a thin thread, but always above a safety net of comprehensive tests.

Part I

What Is Refactoring?

Why Are We Learning This Now?

Over the past several lectures you learned a catalog of design patterns — reusable, time-tested solutions like Strategy, State, and Mediator. They produce code that is clean and extensible. But there is a gap nobody talks about: the code you actually write rarely starts out looking like those textbook examples. It starts out as a tangle of loops, conditions, and classes that grew organically as requirements piled up.

This lecture is the bridge across that gap. Refactoring is the discipline of getting from the messy design you have to the clean, pattern-based design you want — safely, in small steps, without breaking anything. We place it right after design patterns on purpose: a design pattern encapsulates an abstraction that is often hard to see while you are writing individual classes. You only realize "oh, I should have used a Strategy here" once you have written enough code to feel the pain. Refactoring is how you act on that realization after the fact.

The Big Idea

You don't have to get the design right the first time. You write a current version for today's requirements, and when new requirements appear, you refactor toward a cleaner design (often a design pattern) that makes the anticipated change easy. The earlier you refactor, the more it pays off.

Refactoring Is a Semantics-Preserving Program Transformation

Let's define the term precisely, because the word is widely misused. To do that we first need one more term: semantics.

Key Concept — Semantics

The semantics of a program is the complete set of behaviors it can have — its observable input/output behavior. It is what the program means, as opposed to how it is written.

Key Concept — Refactoring

A refactoring is a change to code that preserves its behavior while changing its structure. It makes the internal structure and internal behavior easier to understand and cheaper to modify without changing the observable behavior.

That second half — "without changing the observable behavior" — is the part everyone gets wrong. You have surely seen git commit messages that just say "refactoring" because the author made a huge change and was too tired to describe it. Don't do that. If a change alters what the program outputs, it is not a refactoring.

Common Mistake — Mislabeling Changes

Mixing behavior changes into a "refactoring" destroys its whole value. The discipline only works if you separate structure-only changes from changes that intentionally alter behavior. Keep them in different commits.

Why does this separation matter so much in practice? Because of tests. If a change is a true refactoring, then every test that passed before must still pass after — the observable behavior is identical. So when a test breaks, you immediately know you accidentally changed behavior. If you instead bundle a real behavioral change in with twelve structural tweaks, a failing test tells you nothing: maybe the failure is expected, maybe it's a bug you just introduced, and you can't tell which.

The Refactoring Metaphor

Think of refactoring as juggling your classes and objects on a very thin thread — but doing it above a safety net of comprehensive tests. The tests are what let you take the risk. Without them, you are juggling over concrete.

The Second Law of Software Evolution

Why does code need refactoring at all? Because code degenerates over time if no one actively maintains its clarity. This is sometimes called the second law of software evolution, by analogy with the second law of thermodynamics: just as physical entropy increases unless energy is spent to reduce it, code entropy increases unless effort is spent to reduce it.

There are real reasons this happens. Different people work on different parts of a system at different times. None of them sees all the code at once, so none of them spots the abstraction that would tie it together. Each adds a little more, and the structure slowly decays. Refactoring is the process that reduces entropy in our code — it adds back the clarity that erosion takes away.

Why This Matters — Code Is Read More Than Written

Studies show software engineers spend roughly 58% of their time on program comprehension — reading and trying to understand code, not writing it. Since code is read far more often than it is written, every bit of clarity you add through refactoring pays off many times over.

Part II

Code Smells — A Vocabulary for Bad Design

What Is a Code Smell?

Before you can fix bad design, you need a way to recognize it. That is the contribution of Martin Fowler's book Refactoring: Improving the Design of Existing Code. Earlier design books (like the Gang of Four) showed only beautiful, finished code — readers admired it but couldn't see themselves in it. Fowler's book did something radical: it put ugly code on the page, side by side with its cleaned-up version. The "before" page is where readers finally recognized their own work, and the "after" page showed them the destination.

Key Concept — Code Smell

A code smell is a surface symptom in code that suggests a deeper design problem — a reason to consider refactoring. (Not literally code that "hasn't showered" — it's code that hasn't been refactored for a while.) A smell is a hint, not a guaranteed bug.

This lecture is deliberately about the vocabulary — the names of the smells and the refactorings that address them — far more than the mechanical edits themselves. Why? Because the mechanical edits are trivial and your IDE (or an AI coding agent) can perform them automatically. But to direct that tool, you must be able to say "this is a Divergent Change, apply Extract Class." These names are documented everywhere, so the AI agents you'll use on the job already know them. The vocabulary is the leverage.

Connection — Builds on Design Patterns

Each smell points toward a refactoring, and many refactorings point toward a design pattern you already studied. Smells are the "why," refactorings are the "how," and patterns are often the "where it lands." Watch for that chain throughout this part.

Bloaters — Code That Got Too Big

The first family of smells is about things that have grown larger than the human mind can comfortably hold.

Data Clumps → Introduce Parameter Object

A data clump is a bunch of primitive values that keep hanging around together. If every invoice method takes a startDate and an endDate — amount invoiced, amount received, amount overdue — those two parameters are a clump. Bundle them into a single DateRange object. The code gets easier to read (you now know it's a date range), the object is reusable, and you have a natural home for behavior. For instance, time-zone conversion across a start and end in different zones is awkward as two loose dates but clean as one DateRange that owns the logic.

Primitive Obsession → Value Object

Closely related: primitive obsession is modeling a rich concept with a bare primitive. Suppose an Order has a customer attribute typed as string. A customer is more than a string! Worse, nobody can guess what the string holds — is it an ID? A first name? "First Last" or "Last, First"? Replacing it with a Customer value object removes the guesswork and gives you a place to add meaningful getters and methods later.

The Object-Oriented Growth Pattern

Notice the recurring move: you start by grouping data together, then realize you want to perform operations on that data, so you pull both into a class. Refactoring formalizes this natural drift from loose primitives toward proper objects.

Two more bloater-adjacent refactorings worth naming:

  • Replace Type Code with Class. If a blood group is stored as an integer (O, A, B, AB = 0,1,2,3), replace it with a BloodGroup class holding four static instances — a kind of "quadruple singleton." In UML, an underlined attribute means static: it lives on the class, not on each instance.

Change Preventers — Smells That Make Change Hard

The next family is about how a smell makes future change painful. The two key opposites are Divergent Change and Shotgun Surgery.

Key Concept — Divergent Change

One class is commonly changed in many different ways for many different reasons. It violates separation of concerns — a database concern, a bookkeeping concern, and a UI concern all live in one class — so it has the longest, most-blamed history in the system. Fix: Extract Class, pulling each concern into its own class that can evolve independently.

Key Concept — Shotgun Surgery

The opposite: one change forces you to touch many different classes — a ripple effect spread across modules and methods. These are the worst smells because you will almost certainly miss one of the five places and ship a bug. (Famously documented as a major source of bugs even in safety-critical robotic systems.)

Robert C. Martin's guiding principle captures both: "Put together what changes for the same reasons, and spread apart what changes for different reasons." Divergent Change crams unrelated reasons into one class; Shotgun Surgery scatters one reason across many. Both fight that principle.

Divergent Change vs. Shotgun Surgery — mirror images. One concentrates unrelated reasons; the other scatters a single reason.

A related special case is Parallel Inheritance Hierarchies: a flavor of shotgun surgery where adding a subclass to one hierarchy forces you to add a matching subclass to another. You can often Move Method / Move Field to merge them — though sometimes the concerns are inherently cross-cutting and cannot be cleanly separated at all (more on that in Part V).

Couplers & Dispensables

Two final families. Couplers are smells where classes know too much about each other:

Key Concept — Feature Envy

A method seems more interested in another class's data/methods than its own — it "envies" that class and keeps calling into it. This creates strong coupling. Fix: Move Method to where the data lives, or Extract a new class.

Key Concept — Inappropriate Intimacy

Two classes spend too much time in each other's private data — too much coupling, design decisions not hidden from each other. Fixes depend on the situation: sometimes Move Method, sometimes merge them, sometimes Extract a third class, sometimes introduce a delegating method (e.g. a getManager() on Person so a client never needs to know Department exists).

Dispensables are things that exist without earning their keep:

  • Lazy Class. A class that doesn't do enough to justify the cognitive space it occupies — close to dead code. Fix: Inline Class (the opposite of Extract Class): fold its few members into another class. Or Collapse Hierarchy — if a Salesman barely differs from an Employee, just merge them.
  • Speculative Generality. Abstractions built for changes that will never come. We'll give this its own treatment below.
  • Dead Code. Code that is never executed — pure cognitive overhead. (On Thursday: a story where dead code that "came back to life" cost a firm $440M in 28 minutes.)
Footgun — Speculative Generality (The "Second System Effect")

After learning information hiding and "plan for change," students often overshoot: they build wildly flexible designs for changes that won't happen for 100 years. Fred Brooks (author of The Mythical Man-Month) named this the Second System Effect — your second system, the "good" one, ends up over-generalized to the point that nobody can follow it. Only build the flexibility you will actually need.

Part III

Replace Conditional with Polymorphism

Why Conditionals Become a Problem

This is one of the most important refactorings in object-oriented programming, so it gets its own part. First, the motivation — and it starts with how you were trained.

In your first algorithms class you learned to think in loops and conditions: "this happens, then that happens, and if this condition holds we go here, else we go there." This is a procedural way of thinking, and it's a fine starting point. But it has a hidden cost. When you write an if with its else branch right next to it, you have coupled the two behaviors — they sit in the same module, the same place. If they change for different reasons, they shouldn't be neighbors.

It gets worse when the same condition appears in many places. Every if-branch across the code is really the same concern, replicated; every else-branch likewise. So conditionals tend to breed code duplication: the behavior that depends on the condition is copy-pasted everywhere the condition is checked. Adding a new case means hunting down every switch statement and editing each one — classic Shotgun Surgery.

Connection — Which Pattern Solves This?

The design pattern for behavior variation is Strategy. A close cousin, the State pattern, handles the case where conditions everywhere ask "what state am I in?" — replacing those checks with an object that delegates its state-dependent behavior polymorphically.

Polymorphism Beats Conditionals on Extensibility

Why prefer polymorphism over conditions in real systems? Extensibility. To add a behavior, you just create a new class implementing the interface — no hunting through existing switch statements to add a case. The existing code doesn't change at all.

Worked Example — The Bird Speed Switch

Here is the "before": a single Bird class whose getSpeed switches on a type code.

# BEFORE — one class, switch on a type code
class Bird:
    def get_speed(self):
        if self.type == EUROPEAN:
            return self.base_speed
        elif self.type == AFRICAN:
            return self.base_speed - self.load_factor * self.num_coconuts
        elif self.type == NORWEGIAN_BLUE:
            return 0 if self.nailed else self.voltage
        # ...add a new bird? edit THIS switch — and every other one

The "after" replaces the switch with an inheritance hierarchy. The base class declares get_speed as abstract; each bird subclass supplies its own version.

# AFTER — abstract base + one subclass per bird
from abc import ABC, abstractmethod

class Bird(ABC):
    @abstractmethod
    def get_speed(self): ...

class European(Bird):
    def get_speed(self): return self.base_speed

class African(Bird):
    def get_speed(self):
        return self.base_speed - self.load_factor * self.num_coconuts

class NorwegianBlue(Bird):
    def get_speed(self):
        return 0 if self.nailed else self.voltage
1

Create the abstract base. Bird inherits from ABC and declares get_speed with @abstractmethod — there is no implementation, just the promise that every subclass provides one.

2

Make one subclass per former case. European, African, and NorwegianBlue each implement get_speed with the logic that used to live in one arm of the switch.

3

Delete the switch. Calling bird.get_speed() now dispatches automatically to the right subclass via polymorphism. No condition is evaluated by hand.

4

Reap extensibility. Adding a Penguin is one new class — and you never have to revisit any switch statement. If the same type-code switch appeared in several methods, all of that duplication collapses too.

The switch becomes an inheritance hierarchy. Each subclass overrides get_speed(); the runtime picks the right one.

A Crucial Caveat — Refactorings Go Both Ways

Here is something that distinguishes refactorings from patterns. A design pattern is usually a good idea in one direction — you generally want to move toward it. Refactorings, by contrast, go both ways. Extract Class and Inline Class are opposites; so are Replace Conditional with Polymorphism and its reverse.

Which direction is right depends entirely on your anticipated future changes. Only refactor Bird into a hierarchy if you genuinely expect more bird types to be added. If no new birds are coming, building the hierarchy is wasted effort and added complexity. Likewise, Collapse Hierarchy (merging Salesman into Employee) is right only if you're confident they'll keep evolving together. If you expect them to diverge — to play very different roles later — keep them separate.

The Decision Rule

For every "both-ways" refactoring, ask: do these things evolve for the same reason or different reasons? Same reason → merge them (less cognitive load). Different reasons → keep them apart (independent evolution). The realistic set of anticipated changes is your compass.

Part IV

Refactoring Safely

Levels of Refactoring

Not all refactorings are equal in scope. Fowler's catalog spans a spectrum from tiny local edits to sweeping architectural moves:

1

Data-level — renaming a variable, changing a field's type.

2

Statement-level — reordering or simplifying statements.

3

Routine-level — Extract Method, Inline Method.

4

Class implementation — changing internals without touching the public interface.

5

Class interface — changing the public API (this one will break unit tests; that breakage is expected).

6

System-level — the big architectural moves. The low levels are fully automatable by your IDE; the system level is often still planned and driven by people.

The practical takeaway: the lower-level refactorings are mechanical and tool-driven, but you should still think about the end-state you're aiming for. Even when an AI agent performs the steps, you decide the destination.

The Safety Net of Tests

Everything in this lecture rests on one foundation: a comprehensive test suite. Recall the metaphor — you're juggling above a net. Here's the safe-refactoring checklist:

  • Keep refactorings small — so small that you fully understand what each one does and can be sure it changes nothing.
  • Make frequent checkpoints and commit each refactoring separately.
  • Use compiler warnings as a free second pair of eyes.
  • Re-test after every step. A true refactoring keeps all previously-passing tests green.
  • Know which level you're at. A class-interface refactoring is expected to break unit tests; update them at the same time. Every other level should leave the tests untouched.
  • Match effort to risk. Refactoring the backbone of the world's payment system, or a security agent deployed everywhere, demands far more caution than a hobby script. (See: CrowdStrike shipping on a Friday.)
Study Strategy — The Parking Lot

As you read code, keep a "parking lot": a running list of refactorings you want to make. You can hand that list straight to an AI coding agent, which knows the named refactorings and can execute them — provided you have the comprehensive test suite to verify the result.

Hoare's Two Ways to Write Programs

The late C. A. R. (Tony) Hoare gave the principle that should guide both your code and your refactorings:

Tony Hoare

"There are two ways to write code: write code so simple there are obviously no bugs, or write code so complex that there are no obvious bugs." Aim for the first — code so clear that a reader instantly sees it can't be wrong.

The same standard applies to each refactoring step: make it so small and obvious that you know it introduces no behavioral change. That is exactly why we insist refactoring be semantics-preserving and unmixed with behavioral edits — once the two are tangled, you lose the ability to know any step is safe.

The nuance the lecture stresses: this is special to the interface level; lower-level refactorings (data, statement, routine, class-implementation) should leave all tests passing, which is how you know they preserved behavior. Part V

Information Hiding, Revisited

The lecture closed by reinforcing earlier principles through true/false questions — exactly the style you'll see on the exam. These distinctions are subtle, so let's make them sharp.

Information Hiding vs. Separation of Concerns

Key Concept — Information Hiding

Design your modules so each one hides a design decision that is likely to change from the others. It is primarily a module-decomposition approach — about how you structure modules — not merely about shrinking interfaces.

Key Concept — Separation of Concerns

Separate the sections of your code that implement different concerns. It makes no claim that those concerns must be hidden or that they are likely to change — only that they be separated.

How do they relate? Information hiding is stronger (it demands hiding, and targets decisions likely to change), while separation of concerns is more general (it applies to all concerns). They are tightly correlated but neither strictly implies the other. Example: security is a concern you'd want addressed everywhere, but it's cross-cutting and hard to isolate into one module — and once you decide "the system must be secure," that decision isn't likely to change. So it's a concern worth separating that isn't really a hidden, change-prone design decision.

Exam Footgun — The "Tyranny of the Dominant Decomposition"

Pure separation of concerns is almost always impossible: real systems have cross-cutting concerns (like security or logging). A 1972 idea, later named the "tyranny of the dominant decomposition," says you must pick one way to decompose your system, and that choice forces some concerns to remain tangled — unless you use exotic tools like aspect-oriented programming, which almost nobody uses in practice.

The Getter/Setter Myth

Does information hiding recommend using getters and setters in Java? The intuitive answer is "yes," but the precise answer is not in general.

Making a field private and exposing a public getter and setter hides almost nothing: you've still advertised that the attribute exists and revealed its data type, and you've handed everyone the ability to read and write it. You followed a different principle ("don't make fields public"), but not information hiding. Getters/setters help hide information only in specific cases — e.g. a getter with no setter (reducing the interface), or accessors typed in terms of a more general type that conceals the concrete representation.

Read Exam Quantifiers Literally

The professor phrases questions precisely. "Information hiding always uses getters/setters" is false; "sometimes / in certain cases" can be true. Watch the quantifier — always vs. usually vs. can — and take it literally.

One more from the question set: "By following information hiding, you can reduce inter-module dependencies." True. Because information hiding is fundamentally about structuring modules to hide change-prone decisions, it reduces the assumptions modules make about each other — and therefore their coupling. The Mediator pattern is the showcase example: it hides all the collaborations between objects, collapsing a web of dependencies into one.

Spotting the Strategy Pattern

The lecture spent real time on distinguishing the Strategy pattern from look-alikes — a perfect exam target. The single most reliable test:

The Strategy Litmus Test — Delegation

Strategy requires delegation (usually composition/aggregation): a client object holds a reference to a strategy and calls through it, making the algorithm interchangeable at runtime. No delegation → not Strategy. If you'd draw only inheritance arrows and no association, it isn't Strategy.

Strategy PatternTemplate Method (the look-alike)
Client delegates to a strategy object it holdsSubclasses inherit and override a hook method
Interface carries no behaviorAbstract base defines the algorithm outline (a "Swiss cheese" with holes)
Algorithms swapped at runtime via compositionStructure fixed by the superclass; subclass fills the holes
Strategy body has real complexitySubclass may fill in just one line

So if you see an abstract class that defines a triple-nested sort loop and leaves a single needsSwap() for subclasses to fill — that's Template Method, not Strategy. Tells: (1) no delegation, only inheritance; (2) the base "interface" actually ships a lot of code (you inherit more than you define — the opposite of Strategy); (3) the subclass contribution is trivially small.

Deep dive: what "design" actually means (Plato's chairs)

When a question says "the design keeps control over the algorithm structure," what counts as design? The professor reached for Plato's theory of ideas: every chair you see is an instance of the abstract idea of a chair, which you recognize because you've seen chairs before.

Likewise, design is the recognizable structure and behavior that exists above the specific programming language — patterns and decisions you've seen before. Variable names, specific numbers, and exact syntax are details, not design. But "a triple-stacked loop with a condition inside" (an algorithm outline) or "an abstract class delegated to by a hook" (Template Method) are ideas you recognize — those are design. Design lives at a higher level of abstraction than the language you happen to be using.

Putting It All Together

This lecture tied the whole course together by answering a question the design-patterns lectures left open: how do you actually get from the code you write to the code you admire? The answer is a three-part vocabulary working in concert.

First, code smells give you the language to diagnose bad design — Divergent Change, Shotgun Surgery, Feature Envy, Data Clumps, Primitive Obsession, Speculative Generality, and the rest. Second, each smell points to a named refactoring that treats it — Extract Class, Introduce Parameter Object, Replace Conditional with Polymorphism, Inline Class, Collapse Hierarchy. Third, those refactorings frequently land on the design patterns you studied earlier: Replace Conditional with Polymorphism reaches for Strategy and State; reducing dependencies reaches for Mediator. Smell → refactoring → pattern is the spine of the whole course.

Underneath it all sits one non-negotiable rule that makes the practice safe: a refactoring is semantics-preserving. It changes structure, never observable behavior. That single constraint is what lets your test suite act as a safety net — every previously-passing test must stay green, so any failure flags an accidental behavior change immediately. And one judgment call governs every decision: refactorings go both ways, so you always ask whether the things in question evolve for the same reason (merge) or different reasons (separate), guided by the realistic set of anticipated changes.

The recurring theme, from data clumps to information hiding to Hoare's "obviously no bugs," is that human cognitive capacity — not CPU speed or memory — is the limiting factor in modern software. Every technique here exists to reduce the mental load required to understand and change code. That's the whole point of the juggling-above-a-net metaphor: refactoring lets you keep improving a living system without ever dropping the ball.
