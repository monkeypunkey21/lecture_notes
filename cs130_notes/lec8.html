Four Design Patterns: Singleton, Adapter, Facade & Command

An interactive guide to Week 4's lecture — when you want exactly one of something, how to connect mismatched interfaces, how to hide complexity, and how to package an action as an object.

This lecture is about design patterns: reusable, named solutions to recurring problems in object-oriented software design. A design pattern is not code you copy-paste. It is a description of abstract roles — a small cast of collaborating classes — that you then map onto the concrete classes of your own system. The exam tests whether you can name those roles and explain the problem each pattern solves.

Key Concept — What a Design Pattern Is

A design pattern captures hard-won design experience as a reusable template of abstract roles (e.g. "the adapter," "the receiver") plus the problem they solve and the trade-offs they carry. When you apply a pattern, you map each abstract role onto a real class in your system. Learning a pattern means memorising its roles and the problem it addresses — not memorising one specific implementation.

A theme runs through the whole lecture: every one of these patterns superimposes new roles onto already-existing domain classes. Your Earth class already models a planet; the Singleton pattern just adds a getInstance method and a private static field. The patterns are extra responsibilities layered onto code that already has its own purpose — they are not standalone classes invented from nothing.

Part I

The Singleton — Exactly One Instance

Why Would You Want Only One Instance?

Some objects only make sense if there is exactly one of them in the entire program. A thread pool manages a fixed set of worker threads — having two competing pools defeats the purpose. A cache is supposed to be the one shared place where results live. Logging objects, dialog boxes, and configuration registries are similar: a second copy is at best wasteful and at worst a correctness bug.

The professor's favourite example came from his father, who was writing a planetary simulation and accidentally created two Earth objects. Code that set an attribute on one Earth never saw it reflected on the other, and the simulation silently went wrong. There should only ever be a single Earth. That is exactly when the Singleton earns its place: when a second instance can quietly corrupt everything.

Key Concept — The Singleton Pattern

The Singleton pattern ensures a class has only one instance and provides a global point of access to it. Per the original Design Patterns book, it solves two things: (1) guarantee a single instance, and (2) give global access to it. The professor's strong advice: only reach for the Singleton because of part (1). Part (2) is a trap.

Why not just use a global variable?

The obvious alternative is a global variable holding the one object. The professor pushed the class on why that is a bad idea, and several real problems came out:

  • No access control. A global is reachable from everywhere. If an attacker compromises one component, they reach the shared object too — and any code anywhere can modify it.
  • Too many dependencies. When everything depends on the global, any change to it ripples through the entire codebase. You cannot hide or constrain who knows about the instance.
  • No control over when it is created. A global is typically initialised at program start. You may want to delay creation until it is first needed.

A pure static class (a class with only static methods) is no better. It loudly advertises "there is exactly one of these" to the whole program, so callers build assumptions on that fact. If you later decide you need two instances, you must rewrite every Class.method() call into object.method(). The Singleton hides the one-instance decision behind a method, keeping that decision changeable later.

Building a Singleton in Java, Step by Step

To create an object in Java you write new MyClass() — and you can call it as many times as you like, getting a fresh object each time. To enforce a single instance we have to take that power away from callers.

1

Make the constructor private. Now no code outside the class can call new MyClass(). We have blocked uncontrolled creation.

2

Hold the one instance in a private static field. The class itself keeps a reference to the single object it allows.

3

Expose a public static getInstance(). Since the constructor is private, this method is the only way to obtain the object. It checks whether the instance exists yet; if not, it creates it, then returns it.

public class Singleton {
  private static Singleton instance;   // the one and only

  private Singleton() { }                 // nobody can call new from outside

  public static Singleton getInstance() {
    if (instance == null) {        // lazy: create only on first call
      instance = new Singleton();
    }
    return instance;
  }
}

This is the classical lazy Singleton. "Lazy" because the object is created on the first getInstance() call rather than up front. Callers write Singleton.getInstance() to obtain the shared object.

The Singleton's Bad Reputation

The Singleton is the only pattern whose original authors later wrote a paper saying they regretted including it. By blessing it, they encouraged programmers to chase the "global access" half of the pattern — sprinkling getInstance() calls everywhere. The result is a global variable in disguise, with all the coupling problems a real global has. Use the Singleton only when single-instance is the genuine design goal.

Study Strategy — Use getInstance() Once

The professor's personal rule: call getInstance() in essentially one place, where you obtain the object, then pass that object around like any normal object to the components that need it. This hides the fact that it is a singleton at all. If getInstance() appears in many places across your code, you are doing the Singleton wrong and have re-created a global variable.

The Hidden Bug: A Race Condition

Now switch on "max thinking mode," as the professor put it. The lazy Singleton above looks airtight, but it can create more than one instance when multiple threads call getInstance() at the same time. A thread is an independent stream of execution; a program can run several at once, and they can interleave their statements in unpredictable ways.

Trace two threads both calling getInstance() on a fresh program where instance is still null:

1

Thread A reaches if (instance == null). It is the first call, so the field is null — the condition is true, and A is about to enter the then branch.

2

Before A actually assigns the field, Thread B reaches the same if (instance == null). The field is still null — so B's condition is also true, and B will enter the then branch too.

3

Both threads now run new Singleton(). A stores its object in instance; B's object goes to whoever called B and gets passed around. Two instances now exist — the exact failure the Singleton was supposed to prevent.

Because B reads instance before A writes it, both checks see null and both construct an object.

Four Ways to Fix It — A Tour of Trade-offs

The fix evolves through four implementations, each trading one quality attribute for another. This progression is the heart of the Singleton section.

1. Synchronized method (thread-safe but slow)

Mark getInstance() with Java's synchronized keyword. This enforces a lock: only one thread can be inside the method at a time. The race disappears — the first thread creates the instance, later threads see the now-non-null field and just return it.

public static synchronized Singleton getInstance() { /* ... */ }
The Cost — Locking on Every Call

Correctness is restored, but now every call to getInstance() must acquire the lock and wait — even long after the object exists and we are only reading it. If many parts of the code call getInstance() frequently, this lock contention drains performance. We only ever needed to synchronise the first creation.

2. Eager instantiation (fast but always allocates)

Create the object when the class is loaded, before any getInstance() call. Code that runs at class-load time has no race condition, so no locking is needed.

private static Singleton instance = new Singleton();  // eager

The trade-off flips: we gain speed but the object is always created, even if it is never used. That wastes memory in cases where the singleton might not be needed at all.

3. Double-checked locking with volatile (the clean version)

The professor called this the cleanest Java Singleton. The idea: synchronise only on the first creation. Check whether the instance is null; only if it is do you enter a synchronized block, and inside that block you check again (double-checked) in case another thread created it while you were waiting.

private static volatile Singleton instance;

public static Singleton getInstance() {
  if (instance == null) {              // 1st check: skip lock once created
    synchronized (Singleton.class) {
      if (instance == null) {          // 2nd check: another thread may have won
        instance = new Singleton();
      }
    }
  }
  return instance;
}

The volatile keyword is the linchpin. It is a field modifier that makes reads and writes of instance behave correctly across threads, so one thread always sees the value another thread wrote. After the object is created, all subsequent reads skip the lock entirely — synchronisation happens only on the first write. The cost here is simply more, trickier code to maintain; functionally it solves the thread-safety problem.

Aside: why not just mark the field final?

final in Java (and C++) allows exactly one assignment to a variable, which sounds perfect for a singleton. But final is a statically checked property: the compiler must be 100% certain there is exactly one assignment. For a member field assigned inside getInstance() based on a runtime condition, the compiler cannot prove this, so it will not let you make the lazily-initialised field final. volatile is the right tool because it is about cross-thread visibility, not single-assignment.

One Pattern, Many Implementations — "Context Tailoring"

Even a pattern as simple as the Singleton has several implementations, each with different trade-offs in performance, memory, and code complexity. There is no one-size-fits-all version. You pick the implementation that satisfies your system's quality-attribute requirements — the professor calls this context tailoring. A pattern is a starting point, not a finished answer.

Scope of the Guarantee

Making the Singleton thread-safe only guarantees safe creation of the single object — the most important safety property. It says nothing about concurrent calls to the object's other methods. If multiple threads mutate the shared instance's attributes, you must make those operations thread-safe separately, case by case.

Part II

The Adapter — Making Mismatched Interfaces Fit

Where We Are

The Singleton controlled how many of an object exist. The next three patterns are about connecting objects whose interfaces don't line up — starting with the Adapter, which translates one interface into another.

Two Plugs, One Wall

An interface is the set of method names (and their meanings) that one component exposes for others to call. When two components have different interfaces, they cannot talk directly. The everyday analogy: you fly to the UK with a laptop charger whose plug doesn't fit the British wall outlet. You don't rewire your laptop and you don't rewire the building — you slip an AC power adapter between them that translates one plug shape into the other.

Software hits this constantly. You have your own system with an interface that makes sense for you, and you must connect to an external vendor (or a legacy subsystem) whose interface is different. Rewriting your whole system is wasteful, and the vendor can't rewrite theirs because their other customers depend on it. The two interfaces are separate by design and you can't unify them. So you insert an adapter that translates between them.

Key Concept — The Adapter Pattern

The Adapter converts the interface of one class into another interface that the client expects. It lets classes work together that otherwise couldn't because their interfaces are incompatible. The adapter is the only component that knows the mapping between the two interfaces.

The Roles and Structure

The classic (object) adapter uses delegation. The roles are:

  • Target — the ideal interface the client wants to use.
  • Adaptee — the existing class with the "wrong" interface.
  • Adapter — a class that implements the Target interface but internally owns an Adaptee and forwards each call to it, translating along the way.
  • Client — uses the adapter purely as a Target, with no idea an adapter exists (information hiding).

The professor's example: you have a Duck interface that can quack() and fly(), with a MallardDuck implementing it. Separately you have a Turkey interface that can gobble() and fly() — similar idea, different interface. To use a turkey where a duck is expected, build a TurkeyAdapter that implements Duck but holds a Turkey:

public class TurkeyAdapter implements Duck {
  private Turkey turkey;                  // owns the adaptee

  public TurkeyAdapter(Turkey t) { turkey = t; }

  public void quack() {
    turkey.gobble();                     // syntax: quack → gobble
  }
  public void fly() {
    for (int i = 0; i < 5; i++)   // semantics: a turkey flies a shorter hop
      turkey.fly();
  }
}
The client calls the adapter as a Duck; the adapter quietly forwards to a Turkey.

Two Kinds of Adaptation: Syntax vs. Semantics

The duck/turkey example reveals that adaptation happens at two levels:

Adapting SyntaxAdapting Semantics
Different names for the same operation.Same name, but different meaning.
quack() → call gobble(). A direct one-to-one rename.fly() exists on both, but a turkey's hop is shorter — so the adapter calls turkey.fly() five times to match a duck's range.
Same Name ≠ Same Meaning

When two interfaces share a method name, do not assume they mean the same thing. fly on a Turkey is not fly on a Duck. A good adapter must handle both kinds of mismatch — translate the names (syntax) and translate the behaviour (semantics) so the result genuinely satisfies what the client expects.

A real adapter can also do non-1-to-1 translation: one target method might fan out to several adaptee calls, or combine multiple adaptee calls into one. The duck/turkey case happens to be one-to-one per method, but more complex mappings are common.

Object Adapter vs. Class Adapter

What we built above is the object adapter: it uses delegation (the adapter owns an adaptee). There is a second flavour, the class adapter, available in languages like C++ that support multiple inheritance: the adapter inherits from both the Target and the Adaptee.

Class Adapter Is the Riskier Choice

Multiple inheritance brings confusion and the dreaded diamond problem — when a class inherits from two classes that share a common superclass, that superclass effectively appears twice, and a method call becomes ambiguous about which copy it refers to. The class adapter usually avoids the diamond (its two parents typically have very different, unrelated interfaces, which is the whole reason you're adapting), but it's still a use of multiple inheritance, so prefer the object adapter when you can.

Part III

The Facade — One Simple Door to a Complex Subsystem

Hiding a Mess Behind a Simple Interface

Imagine a home theater built from many classes: a projector, a DVD player, an amplifier, a popcorn popper, lights, a screen. To do the high-level thing a user actually wants — watch a movie — you must orchestrate a long, ordered sequence: turn on the popper, start popping, dim the lights, lower the screen, turn on the projector, set its input to DVD, switch it to widescreen, power up the amplifier, set its input, set surround sound… and when the movie ends, run it all again in reverse.

No outside caller should have to know that sequence, or its exact order, or how it changes when you upgrade the equipment. And listening to streamed audio would be a second, equally tangled sequence over the same parts. The complexity needs to be hidden.

The fix is to introduce a new class — a Facade — that becomes the single point of entry. It exposes high-level operations (watchMovie(), endMovie(), listenToCD(), listenToRadio()) and internally knows exactly how to translate each one into the right sequence of low-level calls. From the outside you only ever talk to the facade.

The client talks only to the facade; the facade fans each high-level call out to the subsystem. The client never learns the subsystem classes exist.
Key Concept — The Facade Pattern

The Facade provides a unified, higher-level interface to a set of smaller, more concrete interfaces in a subsystem, making that subsystem easier to use. It is a direct application of the information-hiding principle: it both simplifies the interface and decouples the client from the subsystem, hiding the existence of the sub-components entirely.

Only Group Things That Belong Together

A facade should unify a related family of subsystems. If you wrap a pile of unrelated interfaces behind one facade, you've grouped together things that change for different reasons — which creates a maintainability problem rather than solving one.

Facade vs. Adapter — A Frequently Asked Question

Both wrap multiple classes and both sit "between" a client and other code, which is why students confuse them. The distinction is about intent and context:

AdapterFacade
Intent: convert one interface into another.Intent: simplify a complex interface.
Two interfaces of similar complexity, both genuinely valuable and both still needed.One useful high-level interface plus a much more complex low-level one whose complexity you just want to hide.
You introduce it to integrate with a system you don't control (a vendor, a legacy subsystem).You design it yourself, at design time, for your own system.
It's About the Problem, Not the Label

Design patterns exist to help you find a solution. If you simplify three classes into one without ever calling it "a facade," you've still solved the problem — whether you name it a facade is largely academic. Naming it does communicate intent ("use this, not the inner classes"), which matters for big subsystems. And beware over-abstraction: "every problem in CS can be solved by adding a layer of indirection — except the problem of too many layers of indirection." A facade that hides too many configuration options can make the system less useful; deciding what to expose is a requirements-engineering question, settled before you design the interface.

Correct — this is the Adapter's home turf. The intent differs — convert (adapter) vs. simplify (facade) — so the choice is not cosmetic (D).
Part IV

The Command — Packaging an Action as an Object

Where We Are

The Adapter and Facade are static patterns — they reduce dependencies in your code's structure. The Command is the lecture's first behavioral pattern: its job is runtime flexibility, letting you decide at runtime which action runs.

Decoupling the Action from the Actor

Picture a universal remote control for a smart home. It has programmable buttons, and the home is full of wildly different devices: a stereo, a TV, ceiling lights, outdoor lights, even a sprinkler. The trouble is that they all have different interfaces: the sprinkler has openValve()/closeValve(), the TV has on()/off(). They don't map to each other.

We want the remote to activate and deactivate any device, and we want to add new devices later without recompiling or modifying the remote. So the remote must not depend on the concrete device classes at all.

Don't Reach for a Big Switch Statement

The tempting hack — a giant switch over "which device is programmed here" — is exactly what to avoid. Switch statements are "the enemy of changeability": they violate the single-choice principle, because adding a new device forces you to edit every switch that enumerates the cases. Avoid switch statements whenever you can.

Key Concept — The Command Pattern

The Command pattern wraps a request as an object. It defines a common interface — typically just execute() — so that wildly different actions on wildly different receiver objects can all be invoked uniformly. This decouples the object that triggers an action from the object that performs it.

The Roles and the Code

  • Command — an interface with execute() (and often undo()). Many concrete command classes implement it.
  • Receiver — the device that actually does the work (the Light, the Stereo). Each command holds a receiver and translates execute() into the right call on it.
  • Invoker — holds a command in a "slot" and blindly calls execute(), knowing nothing about receivers or concrete classes.
  • Client — creates the receivers, creates the concrete commands (wiring each to its receiver), and loads them into the invoker.
// The Command interface — every action looks the same to the invoker
public interface Command { void execute(); }

// A concrete command: holds its receiver (the light)
public class LightOnCommand implements Command {
  private Light light;
  public LightOnCommand(Light light) { this.light = light; }
  public void execute() { light.on(); }   // translate execute → light.on()
}

// The invoker: a remote slot that just calls execute()
public class SimpleRemoteControl {
  private Command slot;
  public void setCommand(Command c) { slot = c; }
  public void buttonPressed() { slot.execute(); }
}

And the client wires it together — creating the receiver, the command, and loading the invoker:

SimpleRemoteControl remote = new SimpleRemoteControl();
Light light = new Light();
remote.setCommand(new LightOnCommand(light));  // configure the slot
remote.buttonPressed();                          // later: just fire it
Command Is "Strategy, Turned Up"

This looks a lot like the Strategy pattern: you plug in an object that decides what happens, then call execute() when you want. Think of Command as a richer Strategy — it carries more variation and a receiver object it ultimately translates the call to. Its essence is separating the creation of an action from its execution: you define the command in one place in the code and fire it from another, completely different place.

Command vs. Mediator

Both involve many objects with different interfaces, so their class diagrams look similar. The key difference is flexibility over time:

MediatorCommand
A central hub that knows the order and combinations of interactions between many "colleague" objects.No central object that knows all the combinations; each command is self-contained.
Essentially static: the interaction logic is fixed at coding time. To change behaviour you edit the mediator and recompile.Dynamic: you can pick which command a button maps to at runtime, giving genuine runtime configurability.
Don't Confuse Command with Facade Either

A student asked whether Command resembles the Facade. It does not. The Facade is a static pattern whose purpose is reducing dependencies in your code's structure. The Command is a behavioral pattern whose purpose is runtime flexibility — letting the user customise what happens when a button (or a framework's before/after hook) fires. In dynamically-typed languages, a command is often just a closure (an anonymous callback) rather than a concrete class.

Undo and Command Histories

Because each action is now a self-contained object, Command makes undo natural. Add an undo() method to the interface so every command knows how to reverse itself. But supporting a history of undos needs one more thing: a data structure to remember the order in which commands ran.

1

Every time you execute() a command, push it onto a history stack.

2

When the user presses Undo, pop the top command off the stack and call undo() on it.

3

Because a stack is last-in-first-out, undos reverse the actions in exactly the opposite order they were performed — the behaviour users expect.

A more elaborate cousin, the Memento pattern, is purpose-built for undo, but the Command stack handles the common case. Commands are also frequently stored in lists: a client loads a sequence of commands (perhaps from a configuration file) and the system runs them in order — a flexible way to script behaviour.

Worked Question — How Does the Remote Tell Two Lights Apart?

If both the kitchen light and the living-room light are instances of the same Light class, how does a button know which one to control? Answer: not by class, but by which object instance you stored in the command. Somewhere in the code you create the kitchen Light and the living-room Light separately; when you build the command, you must store the right instance. In a real app you'd keep a directory of lights with display names and let the user pick the specific instance.

Putting It All Together

Four patterns, four distinct problems — but they share a single deeper principle: information hiding. Each one decides what the rest of the system is allowed to know, and hides the rest behind a controlled interface.

  • Singleton hides the decision "there is exactly one of these" behind getInstance(), so that decision stays changeable and access stays controlled.
  • Adapter hides the fact that an incompatible class is being used at all — the client thinks it's talking to the Target interface, never the Adaptee.
  • Facade hides an entire complex subsystem behind a few high-level operations — the client never learns the sub-components exist.
  • Command hides which action will run and on which receiver, so the invoker can fire actions it knows nothing about, configurable at runtime.

A second unifying thread is the static vs. behavioral split. Singleton, Adapter, and Facade are structural/static patterns: they shape dependencies in your code's structure. Command is the lecture's first behavioral pattern: it shapes what happens at runtime. That contrast — reduce dependencies vs. add runtime flexibility — is exactly why Command should never be confused with Facade even though their examples can look alike.

The most important meta-lesson is context tailoring, demonstrated most vividly by the Singleton's four implementations. A pattern is never a finished answer you paste in. It is a named starting point with known roles and known trade-offs; you choose the variant — synchronized, eager, double-checked-volatile — that meets your system's specific quality-attribute requirements. And remember the recurring caution: patterns superimpose roles onto your existing domain classes (your Earth, your Light), and even good patterns like the Singleton are dangerous when overused.

The Big Idea

A design pattern is a reusable solution expressed as abstract roles plus the problem it solves plus its trade-offs. Mastery means naming the roles, recognising the problem in the wild, and tailoring the implementation to your context — not memorising a single canned version.
