Bayesian Network Inference & Modeling

An interactive guide to Lecture 15 — the questions you can ask a Bayesian network, how to answer them without exploding, and the surprising bridge from probability back to logic.

Part I

The Network and the Question It Hides

Why Are We Learning This?

In the last lecture we learned how to build a Bayesian network: a set of variables, arrows showing which variables directly influence which, and a small table of probabilities attached to each variable. A Bayesian network is a compact recipe. But a recipe is only useful if you can cook from it. The whole reason we build one is so we can later ask it questions: "Given that both tests came back positive, how likely is it the patient actually has the disease?" The act of answering such a question is called inference, and it is the engine that turns a static model into a reasoning system.

This lecture is about two things that always go together. First, inference — the kinds of questions you can pose and how a machine answers them. Second, modeling — how you sit down with a real story and turn it into a network in the first place. We start with inference, because once you see what questions a network can answer, you understand why you would ever build one.

Key Concept — Bayesian Network

A Bayesian network is a directed acyclic graph whose nodes are random variables. Each node carries a CPT (conditional probability table) giving the probability of that variable's values given each combination of its parents' values. An edge from A to B means "A is a direct cause / influence of B." Together, the structure plus the CPTs define a full probability distribution over every variable — without you ever writing that giant distribution down.

The Running Example: A Condition and Two Tests

The professor anchors the whole inference discussion in one small network with five binary variables. Keeping it concrete lets us actually compute numbers by hand. Here is the cast:

  • C — Condition: does the person have the medical condition (a disease)? More common in males (5%) than females (1%).
  • S — Sex: the population is 55% male, 45% female.
  • T1 — Test 1: a test for the condition; its result is caused by whether C is present.
  • T2 — Test 2: a second test, whose accuracy also depends on S (it is more accurate on females than males).
  • A — Auxiliary: a special "constructed" variable we will explain shortly; it lets us say "both tests agreed" as a single piece of evidence.
The five-variable network. Condition C causes both tests; S also influences T2. The purple node A is an auxiliary variable that records "both tests agreed."
Reading a CPT Number

The professor pauses on the number 20% in Test 1's table — "is this a false positive or a false negative?" It is the probability the test reads negative given the condition is present: a missed sick person, i.e. a false negative. The mirror entry — positive given no condition — is the false positive. Test 2's table is split into male and female halves precisely because its error rates differ by sex; that is what the edge S → T2 encodes.

The Ghost in the Machine: the Joint Distribution

Before asking any question, hold one image in your head. Five binary variables means 2⁵ = 32 possible "worlds" — full assignments like (S=male, C=yes, T1=pos, T2=pos, A=yes). The joint probability distribution is a table with one row per world and a probability in each row, all summing to 1. From last lecture you know how to compute any row: multiply together the relevant CPT entries. This 32-row table is the complete truth about the network.

The Trap We Must Avoid

Every query could be answered by building this joint table and reading off rows. With 5 variables that is fine — 32 rows. But the table is exponential in the number of variables: 100 variables would need 2¹⁰⁰ rows, more than atoms in the universe. The entire field of inference algorithms exists to answer questions without ever materializing this table. Keep this tension in mind for the rest of the lecture.

Part II

Four Kinds of Question

A Bayesian network can be interrogated in several distinct ways, and the homework will drill all of them. They form a natural ladder, each a little more demanding than the last: prior marginals → posterior marginals → MPE → MAP. We will climb it one rung at a time.

Rung 1 — Prior Marginals

A marginal is the distribution over one variable (or a small set), ignoring the others. Prior means "before we have observed any evidence." So a prior marginal asks: for variable C, what is P(C = yes) and P(C = no), averaged over everyone in the population?

Conceptually you compute it from the joint table by summing over all rows where C = yes. For this network that comes out to P(C = yes) ≈ 3.2%, so P(C = no) ≈ 96.8%. Do the same for Test 1 and you find P(T1 = positive) ≈ 21.92%: pull a random person off the street, run Test 1, and about 22% of the time it reads positive — even though only 3.2% are truly sick, because the test makes mistakes.

Key Concept — Marginal vs Prior

Marginal = a distribution over a small subset of variables, with the rest "summed out." Prior = computed with no evidence asserted. Tools like SamIam (the professor's own software you will use on the homework) display these as monitors — little boxes next to each node showing its current marginal probabilities.

A Detour: the Auxiliary Variable A

Why does the network carry that fifth variable A? Suppose you want to assert a compound fact: "both tests came out the same — either both positive or both negative — but I don't know which." You cannot express that by fixing T1 and T2 individually, because you don't know their values; you only know they match.

The trick is to add an auxiliary node A whose CPT is deterministic: A = yes with probability 1 exactly when T1 = T2, and 0 otherwise. Its table is all 0s and 1s — it encodes a logical constraint rather than any real uncertainty. Now "the tests agreed" becomes the single, clean piece of evidence A = yes.

Constraints as Deterministic CPTs

This is a recurring move: when evidence is a relationship between variables rather than a value of one variable, introduce an extra node whose CPT is 0/1. The probabilistic machinery then handles the logical constraint automatically. It quietly foreshadows Part IV, where the whole network gets recast in logical, Boolean terms.

Rung 2 — Posterior Marginals

Now we add evidence. A posterior marginal is the same kind of question as a prior marginal — give me the distribution over one variable — but computed after observing some facts E (the professor also calls evidence β). Suppose we observe T1 = positive and T2 = positive.

Conceptually, evidence carves the joint table down to only the rows consistent with it, then renormalizes so those rows sum to 1 again. We are now answering questions about that reduced table — the posterior — instead of the full prior table.

1

Prior: before any test, P(C = yes) ≈ 3.2%. The disease is rare.

2

Observe T1 = positive and T2 = positive. Keep only joint-table rows where both hold.

3

Posterior: P(C = yes | both positive) jumps to ≈ 45.3%. Two positive tests dramatically raise our belief — but note it is not near-certainty, because the disease was so rare to begin with.

4

Side effect: the same evidence shifts other variables too. P(S = male | both positive) rises from the prior 55% to ≈ 83%, because the condition (which the positive tests now suggest) is much more common in males.

This Answers the Day-One Mystery

In the first probability lecture, a disease test came back positive yet P(disease) stayed surprisingly low, and students asked why. Posterior marginals are exactly the answer: a rare prior plus an imperfect test means even a positive result leaves real doubt. Here, two positive tests still only reach ~45%, not ~99%.

Rung 3 — MPE (Most Probable Explanation)

Marginals report a distribution per variable. Sometimes you instead want the single most likely complete scenario. An MPE query asks: given the evidence, what is the one full assignment of all the remaining variables that has the highest joint probability?

With evidence A = yes fixed, the other four variables (S, C, T1, T2) range over 2⁴ = 16 combinations. MPE searches those 16 and returns the winner. Here it is C = no, S = female, T1 = negative, T2 = negative, with probability ≈ 47%. The answer is not a number alone — it is a specific instantiation plus its probability, with a guarantee that no other full assignment is more probable.

Rung 4 — MAP (Maximum a Posteriori)

MAP is the generalization of MPE. Instead of finding the best assignment to all variables, you, the user, name a subset you care about, and MAP finds the most likely joint state of just those — averaging (marginalizing) over everything else. With evidence A = yes and asking only about {S, C}, MAP returns S = male, C = no with probability ≈ 49.3%.

The Big Idea — MPE is the Easy Special Case

If you have a MAP solver, you get MPE for free: just tell MAP that your "subset of interest" is all the variables. So MPE ⊆ MAP. Yet we give MPE its own name because, from a complexity standpoint, maximizing over all variables is genuinely easier than maximizing over a chosen subset while summing out the rest. MAP is the harder problem. Naming the easy case lets us reach for the cheaper algorithm whenever it applies.

Part III

How Inference Stays Cheap: Tree Width

Why Are We Learning This?

We keep insisting we must not build the exponential joint table. So how do real algorithms dodge it, and when do they succeed? The answer is one of the most important practical facts in the whole field: the cost of inference is governed not by the number of variables but by a structural property of the graph called tree width. Get that small, and a million-variable network is still tractable.

The professor notes there are two broad families of exact inference algorithm — too many to cover, more suited to a graduate course — but you should know they exist by name:

FamilyIdea
Variable eliminationSum out variables one at a time, reusing intermediate results.
ConditioningFix ("condition on") certain variables to break the graph into easy pieces, then recombine.

The Master Formula

Both families share the same cost profile. Let:

  • m = number of variables,
  • d = number of values per variable (2 for binary),
  • w = the tree width of the network — a number measuring "how connected / loopy" the graph is.
# Cost of answering a marginal (prior or posterior):
cost  =  O( m  ·  d^w )
         └─ linear   └─ exponential in tree width only
The Big Idea — Pay for Connectedness, Not Size

Inference is linear in the number of variables but exponential only in the tree width. If w is small, you win, no matter how many variables there are. If w is 50 or 100, exact inference is hopeless. The number of variables is almost a red herring; the network's shape is what determines whether you can reason at all.

Tree width comes from graph theory. We won't define it formally, but two named network shapes pin down its value and give you intuition.

Three Structures You Must Recognize

Three structures. A tree gives every node one parent (w = 1). A polytree allows multiple parents but no undirected loops (w = max parents, k). A general DAG can have loops and arbitrarily large tree width.

Trees — w = 1

A tree Bayesian network gives every node exactly one parent. Its tree width is 1, so the formula collapses to O(m · d) — pure linear time. You can have a million nodes and still answer queries fast, never touching the impossible joint table. This is the headline payoff promised earlier.

Polytrees — w = k

A polytree still has no undirected loops, but a node may have several parents. Its tree width equals k, the maximum number of parents any node has. So if no node has more than 5 parents, w = 5 and inference is still very efficient. Polytrees are also called singly connected networks.

General DAGs — multiply connected

Anything more tangled — with undirected loops — is a general DAG, also called a multiply connected network. Here tree width can be large, and exact inference may become infeasible. (More sophisticated methods sometimes still cope, including the logic-based one in Part IV.)

Correct. Cost is O(m · d^w), and a tree has w = 1, so the cost is just O(m · d) — linear in the million variables. The joint table (option A) would indeed be astronomically large, but the whole point of inference algorithms is that we never build it. The number of variables alone never makes inference exponential; only large tree width does.
Part IV

The Bridge Back to Logic: Weighted Model Counting

Why Are We Learning This?

Earlier in CS161 we spent weeks on logic — Boolean formulas, satisfiability, model counting. It might have seemed like a separate world from probability. This part delivers the punchline that ties the course together: you can answer probabilistic queries using nothing but symbolic, logical manipulation. The technique is called a reduction — recasting one problem as another you already know how to solve — and it is one of the most powerful ideas in computer science.

The Ladder: SAT → #SAT → WMC

Start with a Boolean expression over Boolean variables, e.g. (A ∨ B) ∧ ¬C. With 3 variables there are 2³ = 8 possible worlds (truth assignments). Three increasingly informative questions:

1

SAT: "Is there at least one world that satisfies the formula?" A yes/no answer. For (A ∨ B) ∧ ¬C, yes.

2

#SAT (sharp-SAT): "How many worlds satisfy it?" Now we count the satisfying rows rather than just detecting one.

3

WMC (weighted model counting): Each satisfying world carries a weight; instead of counting 1 per world, we sum the weights of all satisfying worlds. Plain #SAT is just WMC with every weight equal to 1.

Key Concept — Weighted Model Counting (WMC)

Given a Boolean formula and a weight for each satisfying assignment, WMC = the sum of the weights of all satisfying assignments (models). The reduction we are about to build encodes a Bayesian network as a formula whose models correspond to the network's worlds, and whose weights are the very probabilities we want. Answering a probability query then becomes: do a weighted model count.

The Encoding: From Network to Formula δ

From the Bayesian network we extract a Boolean formula the professor calls δ. Its variables come in two kinds:

  • The original network variables (here A, B, C — the network's nodes, written as Booleans).
  • One extra parameter variable Pᵢ per number (parameter) in the CPTs. A parameter like θ(A) or θ(C | A) gets its own indicator Pᵢ.

Now look at any model (satisfying assignment) of δ. Some of the Pᵢ appear positively (true) and the rest negatively (false). The professor's worked model has A true, B false, C true, exactly three of the Pᵢ true and the other seven false. The reading is: a positive Pᵢ means "this CPT parameter is present in this world," and the three present parameters turn out to be exactly the CPT entries you'd multiply to get that world's probability.

A model of δ: the three positive Pᵢ pick out exactly the three CPT parameters whose product is that world's probability. Multiplying their weights reproduces P(world) exactly.

Weights Live on Literals, Not Worlds

There are exponentially many worlds, so we cannot list a weight for each. The elegant fix: specify weights per literal (per variable-sign), then define a world's weight as the product of its literals' weights. That is only a linear number of weights. The recipe is strikingly simple:

LiteralWeight
Original network variable (A, ¬A, B, ¬B, C, ¬C)1
Positive parameter literal Pᵢthe CPT number θ it stands for
Negative parameter literal ¬Pᵢ1

Because the network literals all weigh 1, and the absent-parameter literals all weigh 1, a model's weight reduces to the product of just the present parameters — precisely θ(A) · θ(¬B|A) · θ(C|A) for the worked model. That is exactly the probability that world should have. Sum the weights of all eight models and you recover every row of the probability table. Probabilistic inference is weighted model counting.

The Big Idea — Probability as Pure Symbol-Pushing

This reduction means you can compile the Boolean formula δ into a special Boolean circuit (in a normal form with the properties smoothness, decomposability, determinism) and answer probabilistic queries by manipulating that circuit. Three further perks: (1) the encoding shown is the simple "vanilla" version — smarter ones exist; (2) it is not bound by tree width, so it can sometimes win even when standard algorithms drown; (3) it generalizes beyond Bayesian networks — even to inference in probabilistic programming languages.

"Aren't We Adding Exponentially Many Variables?"

A student worried the Pᵢ blow up the variable count. They do not. We add one parameter variable per CPT entry, so the total number of variables tracks the size of the network (the total size of the CPTs), which is itself only n · d^(k+1) — linear in the network, exponential merely in a node's parent count k. Compare that to the joint table's d^n. The encoding stays linear in the size of the network.

Worlds absolutely do have well-defined probabilities (option C is false) — that product is the probability.
Part V

Modeling: Turning a Story into a Network

Why Are We Learning This?

So far we assumed the network already existed. But in practice you have to build it from a messy English description of some domain. The professor gives a disarmingly simple but genuinely useful recipe — three steps that, once you internalize them, keep your modeling organized no matter how complex the story.

Key Concept — The Three Steps of Modeling

Step 1: Define the variables and their possible values. Step 2: Add the edges — for each variable, ask "what are its direct causes?" and draw arrows from those. Step 3: Fill in the CPTs — the actual probability numbers. Always in that order: variables, then structure, then numbers.

Worked Example: Flu, Cold, and Tonsillitis

The story: Flu causes fever, body aches, chilling, and a sore throat. Cold causes chilling and a sore throat. Tonsillitis causes a sore throat and can be associated with fever. We want a network for medical diagnosis.

1

Variables. Three diseases — Flu, Cold, Tonsillitis — and four symptoms — Fever, Body aches, Sore throat, Chilling. Each is binary (present / not present). Seven variables in all.

2

Edges — one node at a time. Don't scatter arrows randomly. Visit each symptom and ask "what diseases directly cause this?" Chilling ← Cold, Flu. Body aches ← Flu only. Sore throat ← all three. Fever ← Flu, Tonsillitis. All arrows point from diseases (top) down to symptoms (bottom).

3

CPTs. Here the story gives no numbers, so we'd fill these later (Part VI shows how). Structure first; numbers can come from data.

A Tactic Worth Remembering

"Go variable by variable and ask: what are this variable's direct causes?" This single discipline prevents the classic beginner error of throwing in arrows by gut feeling. Direct causes become parents; everything else is left to the conditional independencies the structure already implies.

The Bipartite Pattern: Causes → Effects

The resulting graph has a recognizable shape: two layers, with edges flowing only from the top layer to the bottom. This is a bipartite structure — diseases on top (causes), symptoms below (effects).

A bipartite diagnosis network: diseases (causes) on top, symptoms (effects) below, edges flowing only downward. This cause→effect, two-layer pattern recurs across many domains.
The Same Shape, Everywhere

This isn't just for medicine. Google built a bipartite network ("Rephil," millions of nodes) to detect what a web page is about: the top layer is topics (sports, elections...), the bottom layer is words on the page. Give it evidence about which words appear, and query which topic is most probable. Same structure, totally different domain — that is the power of recognizing patterns.

Query Variables vs Evidence Variables

Two more terms you will meet in the tools. Evidence variables are the ones you observe and assert; query variables are the ones you ask about. Crucially, which is which is task-specific, not baked into the network:

TaskEvidence (observed)Query (asked about)
DiagnosisSymptoms ("I have fever + aches")Diseases ("do I have the flu?")
PredictionDisease ("I have tonsillitis")Symptoms ("will I get a sore throat?")

Part VI

Filling the Numbers: Learning & Sensitivity

Where Do CPT Numbers Come From?

Step 3 of modeling — the actual probabilities — has three possible sources. The professor lays them out:

SourceHow
Problem statementThe numbers are simply given to you in the story (you'll see this in the cow example).
Subjective beliefsAn expert estimates them by hand — the historical method when the field began.
Learning from dataEstimate the numbers automatically from a dataset of past cases.

Data: Complete vs Incomplete

A dataset is a table: columns are the network's variables, rows are cases (e.g. patients). Patient 1 might have Cold = yes, Flu = no, Tonsillitis = ? (unknown), Chilling = yes, and so on. That question mark is the whole story.

Key Concept — Complete vs Incomplete Data

A row is complete if every variable's value is known; a dataset is complete if every row is. The moment any cell is a "?", the dataset is incomplete. This distinction decides how hard learning is.

Why the "?" Hurts

With complete data, learning the CPTs is easy — one pass over the data, a closed-form count-and-divide. With incomplete data there is no such shortcut; you need an iterative algorithm. The most famous is EM (Expectation-Maximization), which you'll run on the homework (using a tool, not by implementing it). EM essentially fills in the missing values and re-estimates, round after round.

What Are We Optimizing? Maximum Likelihood

Suppose two groups learn two different CPT sets — CPTs-A and CPTs-B — for the same structure, giving networks A and B. Which is "better"? Answering this defines the objective that learning algorithms chase.

1

Take network A. For each row (patient) in the data, ask A: "what probability do you assign to this exact case?" Call them P₁, P₂, …, Pₙ.

2

Multiply them: score(A) = P₁ · P₂ · ⋯ · Pₙ. This is how well network A "explains" the data — its likelihood.

3

Do the same for network B to get score(B). Choose the network with the larger score — the one that assigns the data the highest probability.

The Big Idea — Maximum Likelihood

The maximum likelihood principle: pick the parameters that maximize the probability of the observed data. As a student neatly put it, you maximize P(data | network). With complete data this maximum has a clean closed form; with incomplete data, EM iteratively climbs toward it. This single principle underlies an enormous fraction of statistical machine learning.

Sensitivity Analysis

The final new concept comes from a second story — confirming pregnancy in artificially inseminated cows, with a tree-structured network of five variables (Pregnancy → Progesterone level → three tests: Scanning, Blood, Urine). Crucially, that story hands you every number, including an 87% procedure success rate, so you can fill all CPTs straight from the text.

Once the network is built, sensitivity analysis asks: how much does a query's answer change when I tweak a parameter? If nudging one CPT entry barely moves your conclusion, that number can be sloppy; if a tiny change flips your diagnosis, you'd better measure it carefully. It tells you which parameters actually matter — invaluable when your numbers are uncertain estimates.

Putting It All Together

This lecture completed the arc of Bayesian networks by joining the two halves of the story: how to ask questions of a network and how to build one in the first place.

We began (Part I) by re-grounding ourselves: a network secretly defines a giant joint distribution we must never materialize. That tension motivated the four queries (Part II) — prior and posterior marginals, then MPE and its harder generalization MAP — which are simply different ways of summarizing or maximizing that hidden distribution. The two-positive-tests example finally explained the day-one mystery: rare priors plus imperfect tests leave real residual doubt.

Part III answered the practical worry hanging over everything: can we actually compute these answers? Yes — as long as tree width is small. Cost is linear in the variable count but exponential only in tree width, which is why trees (w = 1) and polytrees (w = k) are tractable even at massive scale, while tangled DAGs may not be.

Part IV delivered the lecture's intellectual climax: a reduction of probabilistic inference to weighted model counting — pure logic. The very SAT and model-counting machinery from earlier in the course turns out to answer probability queries, with CPT parameters reappearing as literal weights. This is the course's logic and probability threads tied into a single knot, and it even sidesteps the tree-width barrier.

Parts V and VI turned to construction: the three-step modeling discipline (variables → edges → CPTs), the recurring bipartite cause→effect pattern, the query-vs-evidence framing, and finally where CPT numbers come from — problem statements, expert beliefs, or learning from data via the maximum likelihood principle (closed-form for complete data, EM for incomplete). Sensitivity analysis then told us which of those numbers actually deserve our care.

The Through-Line

A Bayesian network is a compression of an exponential distribution into a linear-sized object. Everything today exploited that compression: inference reads answers off it without expanding it, tree width measures when that's possible, the WMC reduction re-expresses it in logic, and learning chooses its numbers to best fit data. Build small, reason big.
