Part I

The Neuron — The Building Block

Why Are We Learning Neural Networks?

For most of this course we have studied model-based AI: we wrote down a representation of the world — logical sentences, Bayesian networks, search spaces — and then reasoned over it. The professor calls this the "represent and reason" paradigm. Neural networks belong to a different family entirely: model-free approaches, also called function fitting or curve fitting. Instead of hand-building knowledge, we collect data and let an algorithm tune a flexible function until it matches that data.

Why does this matter? Over the last several years, neural networks have driven nearly all of the visible progress in AI. Two areas dominate: image analysis (does this image contain a cat? where are the faces? what is the self-driving car seeing?), powered by convolutional neural networks (CNNs), and sequence-to-sequence translation for speech and text, powered by recurrent neural networks (RNNs). This lecture covers the foundation underneath all of them.

Key Concept — Model-Free vs. Model-Based

A model-based system encodes knowledge a human supplied (rules, probabilities, structure) and reasons over it. A model-free system, like a neural network, starts with a generic, highly flexible function and learns its parameters from labeled data. Neural networks are trained in a supervised fashion: every training example comes with the correct answer attached.

Scope of This Lecture

We focus on one slice: feed-forward neural networks trained with supervision. "Feed-forward" means signals only ever flow forward, from inputs toward outputs — a neuron can never take its own output back as an input. Architectures like CNNs and RNNs (and the broader debate about whether model-free is "enough") come in the next lecture.

The Anatomy of a Single Neuron

A neuron is a tiny computation unit. Think of it as a machine that takes in several numbers, mixes them together in a specific way, and emits a single number. Concretely, a neuron has four ingredients:

  • Inputs — a bunch of numbers a_j arriving on "wires." Each input is either a raw input to the whole network, or the output of some earlier neuron.
  • Weights — each wire carries a number w_ij, "the weight on the wire going from neuron j into neuron i." Weights are not set by hand; they are learned from data.
  • Bias — one extra number b attached to the neuron itself, added in regardless of the inputs.
  • Activation function — a function applied at the very end that transforms the combined number into the neuron's output a_i.

The inputs and outputs of a neuron are always numbers, never symbols. The neuron's whole job is to turn a list of incoming numbers into one outgoing number.

A neuron i: weighted inputs flow in, the bias b is added, the activation g is applied, and one output aᵢ flows out — and may fan out to many later neurons.

Step One: The Weighted Sum

Before the activation function ever runs, the neuron first collapses all of its inputs into a single number. Call this intermediate value in_i. It is a weighted sum: multiply each incoming activation by the weight on its wire, add them all up, and then add the bias.

# The combined input to neuron i, before activation
in_i = ( Σ_j w_ij * a_j ) + b
     # sum over every input j, then add the bias b

So the two-step recipe inside every neuron is:

1

Combine. Form in_i by multiplying each input a_j by its weight w_ij, summing the products, and adding the bias b. This is one plain number.

2

Activate. Pass that number through the activation function: a_i = g(in_i). The result is the neuron's output — again one plain number, which can be fed into many downstream neurons.

A Concrete Example

Suppose a neuron (call its output a₄) has three inputs a₁, a₂, a₃, a bias b, and weights 2, −1, 0 on those three wires. Then its combined input is:

in = 2·a₁ + (-1)·a₂ + 0·a₃ + b
   = 2·a₁ - a₂ + b      # the 0-weighted input drops out entirely

a₄ = g(in)                  # apply the activation function

Notice what the weights express: a weight of 2 says "this input matters a lot, and positively." A weight of −1 says "this input pushes the answer the other way." A weight of 0 says "ignore this input completely" — which is why a₃ vanishes from the formula. Learning a neuron is choosing these numbers.

The Output Fans Out

A neuron has exactly one output value, but that single value can be sent to many other neurons at once. One number, many destinations. This is what lets us wire neurons into a network where each layer feeds the next.

Part II

Activation Functions — The Nonlinear Twist

Why Do We Need an Activation Function At All?

So far a neuron just computes a weighted sum plus a bias — that is a linear operation. If every neuron were purely linear, then stacking neurons into layers would gain us nothing: a chain of linear functions is itself just one big linear function. The activation function g is the nonlinear ingredient that lets stacked neurons represent rich, curved, complicated relationships rather than only straight lines. It transforms the single number in_i into a new single number.

The lecture introduces four classic activation functions. Two are sharp on/off switches; two are smoother and matter more in practice.

The Step and Sign Functions

The step function is the oldest. It compares the combined input against a threshold T: output 1 if the input reaches the threshold, otherwise 0.

# Step (threshold) activation
u(x) = 1   if x >= T
       0   otherwise

The sign function is similar but emits −1 instead of 0 for the "off" case, and uses a fixed threshold of 0 rather than a tunable T:

# Sign activation
g(x) = 1    if x >= 0
      -1    otherwise
Step (jumps 0 → 1 at threshold T) and sign (jumps −1 → +1 at 0). Both are hard switches with a discontinuity.

The Sigmoid and the ReLU

Two more activations matter more in practice. The sigmoid is a smooth S-shaped curve that rises gradually from near 0 to near 1, passing through exactly ½ when its input is 0. Because its output sits between 0 and 1, the sigmoid fits naturally when a neuron's output should be read as a probability. There are variants that let you control how steep the S-curve is.

The ReLU — "rectified linear unit" — is the newest of the four (introduced around 2000) and is the dominant choice today. It is astonishingly simple: pass positive inputs straight through, and clamp anything negative to zero.

# ReLU: rectified linear unit
g(x) = max(0, x)

# negative signals are suppressed to 0;
# positive signals pass through unchanged
Sigmoid (smooth S through ½) and ReLU (flat at 0 for negatives, then slope 1). ReLU dominates modern networks.
Why Smoothness Matters Later

One reason the sigmoid and ReLU beat the step and sign functions is friendliness to gradient descent. As we'll see in Part VI, training works by computing partial derivatives of the network. A function with a smooth, well-behaved slope is far easier to differentiate and optimize than a hard step with a flat-everywhere, jump-once shape. Keep this in mind — it foreshadows the whole training story.

Part III

Network Structure — Stacking Neurons into Layers

From One Neuron to a Layered DAG

A single neuron is limited. The power comes from wiring many of them together. In a feed-forward network we arrange neurons into layers: the raw inputs form layer 0 (the input layer), one or more hidden layers sit in the middle, and the final neuron(s) form the output layer. Signals flow strictly forward — layer 1 reads only from layer 0, layer 2 reads only from layer 1, and so on. Because nothing ever loops back, the whole structure is a directed acyclic graph (a DAG).

A feed-forward network as a DAG. Each layer's neurons read only from the previous layer; the rightmost neuron is the output.

Fully Connected Layers and Network Depth

A neuron need not read from every neuron in the previous layer — it might tap only a few wires. But when a neuron does connect to every neuron in the layer before it, we call that a fully connected layer. Fully connected layers are everywhere; you'll meet them again at the tail end of a CNN next lecture.

The number of layers is the network's depth. This single word explains a major plot twist in the field's history.

Where "Deep Learning" Comes From

Early networks had only a couple of layers, because training networks with many layers ("multi-layer" networks) was extremely hard — so hard that this difficulty is part of why neural-network research nearly died out for years. The breakthrough that revived the field was learning how to train deep (many-layer) networks. That is literally where the term deep learning comes from: not some mysterious depth of thought, but the depth — the layer count — of the network.

Networks Are Universal Function Approximators

A whole network, end to end, just computes a function mapping its inputs to its output. Whenever you have a representation that claims to represent functions, the natural question is: is it expressive enough? If I hand you an arbitrary function over the inputs, can you build a network that represents it?

For neural networks the answer is yes, in a precise sense: they are universal function approximators. Give one an arbitrary target function and any error tolerance you like, and there exists a network that approximates the target to within that tolerance. This is the technical statement behind the loose claim that neural networks are "expressive enough to capture anything." Not every machine-learning representation has this property — it is a genuine strength of neural networks.

The Big Idea

A neural network is just a function — a flexible, nonlinear one — built by composing many simple neurons. Because it is a universal function approximator, it can represent essentially any input-output mapping you want, provided you can find the right weights. Everything else in this lecture is about how to find those weights.

Part IV

Logic Gates & Linear Separability

Building an AND Gate from One Neuron

To build intuition, let's make a single neuron behave like a familiar Boolean function. Start with the AND gate: it takes two inputs i₁, i₂ (each 0 or 1) and outputs 1 only when both are 1. Can we pick weights w₁, w₂ and a threshold T (using a step activation, no bias) so the neuron reproduces this truth table?

Yes. Take w₁ = 1, w₂ = 1, and T = 1.5. Let's trace every input combination:

1

i₁=1, i₂=1: weighted sum is 1·1 + 1·1 = 2. Is 2 ≥ 1.5? Yes → output 1. ✓

2

i₁=1, i₂=0: sum is 1 + 0 = 1. Is 1 ≥ 1.5? No → output 0. ✓

3

i₁=0, i₂=1: sum is 0 + 1 = 1. Is 1 ≥ 1.5? No → output 0. ✓ (symmetric to the previous case)

4

i₁=0, i₂=0: sum is 0. Is 0 ≥ 1.5? No → output 0. ✓

All four rows match the AND truth table. The threshold 1.5 sits between "one input on" (sum = 1) and "both inputs on" (sum = 2), so only the all-on case clears the bar.

The NOT Gate

Negation needs only one input and one output, and the trick is a negative weight. Take w = −1 and T = −0.5. We want input 1 to map to output 0, and input 0 to map to output 1.

# input = 1:  1 · (-1) = -1.   Is -1 >= -0.5?  No  → output 0
# input = 0:  0 · (-1) =  0.   Is  0 >= -0.5?  Yes → output 1

The negative weight flips the input's sign, and the threshold is positioned so that "0" clears it but "−1" does not — exactly inverting the input.

The XOR Problem — Where a Single Neuron Fails

Now try XOR ("exclusive or"): output 1 when the two inputs differ (1,0 or 0,1) and 0 when they match (1,1 or 0,0). Can we find weights and a threshold for a single neuron? When the professor asked the class, the honest answer came back: no, we cannot. And that is provably correct. A single neuron of this kind simply cannot represent XOR.

The Limit of a Single Neuron

A single threshold neuron can only represent linearly separable functions. XOR is the canonical example of a function that is not linearly separable, so no choice of weights and threshold will ever make one neuron compute it. This famous limitation was historically a serious blow to early neural-network optimism.

What "Linearly Separable" Means

Plot the inputs as points in space. With two inputs, the four combinations (0,0), (0,1), (1,0), (1,1) are the four corners of a square. Color a corner filled if its output should be 1 and hollow if 0. A neuron with a threshold draws a straight line (a hyperplane in higher dimensions) and outputs 1 on one side, 0 on the other. So a neuron can compute a function if and only if a single straight line can separate the filled corners from the hollow ones.

Filled = output 1, hollow = output 0. AND's one filled corner can be sliced off by a line. XOR's two filled corners sit on opposite diagonals — no straight line separates them.

This generalizes beyond two dimensions. For three inputs the points become the eight corners of a cube, and the separator becomes a flat plane. The professor's minority function (output 1 when fewer than half the inputs are 1) turns out to be separable by a plane in the cube, so a single 3-input neuron can compute it. XOR, by contrast, stays stubbornly non-separable.

The Fix: Stack More Neurons

The resolution to the XOR problem is exactly why we build networks rather than lone neurons. By cascading just two neurons (a small hidden layer), you can represent XOR. This is the concrete payoff of depth from Part III — layering neurons lets the network carve up the input space with multiple lines, escaping the single-line limit.

Why can a single threshold neuron compute AND but not XOR?

Part V

A Network Is a Function of Its Weights

Unfolding the Output Into the Inputs

Before we can train a network, we need to see clearly what function it computes. Consider a tiny network: two inputs (neurons 1 and 2), a hidden layer with neurons 3 and 4, and a single output neuron 5. The weight from neuron j to neuron k is w_jk, the activation is g everywhere, and biases are all zero. The output is a₅.

The output neuron's value is just its own weighted-sum-then-activate rule applied to its inputs a₃ and a₄:

a₅ = g( w₃₅·a₃ + w₄₅·a₄ )

But a₃ and a₄ are themselves outputs of the hidden neurons, each computed from the raw inputs a₁, a₂. We unfold them — substitute their definitions in:

a₃ = g( w₁₃·a₁ + w₂₃·a₂ )
a₄ = g( w₁₄·a₁ + w₂₄·a₂ )

# Substitute both into a₅:
a₅ = g( w₃₅·g(w₁₃·a₁ + w₂₃·a₂)
       + w₄₅·g(w₁₄·a₁ + w₂₄·a₂) )
The 2-2-1 network. Six weights connect the layers; unfolding writes a₅ purely in terms of the inputs and those six weights.

The Output Depends on Two Things: Inputs and Weights

Look at what the unfolded formula contains: the inputs a₁, a₂ and the six weights w₁₃, w₂₃, w₁₄, w₂₄, w₃₅, w₄₅ — nothing else. So we can write the output abstractly as:

a₅ = f( a₁, a₂ ; w₁₃, w₂₃, w₁₄, w₂₄, w₃₅, w₄₅ )

Two kinds of quantities determine the output: the inputs (which the world hands you) and the weights (which you choose). This split is the key to learning. For a given training example, the inputs are fixed — so the output becomes a function of the weights alone. The professor calls this a "sub-function": fix a₁, a₂ to row 1's values and you get one function f₁ of the weights; fix them to row 2's values and you get a different function f₂. We will minimize over exactly these.

The Big Idea

You control a network's behavior entirely through its weights. The inputs you can't change — they come from the data. But by choosing the weights, you decide what output the network produces for any given input. Training is nothing more than searching for the weights that make the network's outputs match the labeled data. Real networks have millions or billions of weights, making this a massive function-fitting problem.

This is precisely why training optimizes over weights: it is the one set of knobs we control. Part VI

Training — Loss Functions & Gradient Descent

The Loss Function: What We're Optimizing

We have labeled data: rows of inputs paired with the correct outputs. For each example i we have the input I, the label l (what the data says the answer should be), and a third quantity — what the network actually produces when we feed it I. Training means tuning the weights until the network's outputs line up with the labels. To make "line up" precise, we define a loss function: a single number measuring how badly the network's outputs disagree with the labels. Lower loss = better fit.

Key Concept — Loss Function

The loss function is the optimization criterion: the quantity we try to minimize during training. Two common choices are mean squared error (average of squared differences between prediction and label) and cross-entropy. The whole training process is the search for weights that drive this number as low as possible.

For mean squared error, iterate over all N examples, take the difference between the network's output and the label, square it, and average:

# Mean squared error over N examples
Loss(w) = (1/N) · Σ_i ( network(Iᵢ) - labelᵢ )²
        #         └ prediction ┘   └ truth ┘

Here is the crucial link to Part V: each network(Iᵢ) term, once its input is fixed, is a function of the weights. So the entire loss is itself a function of the weights — Loss(w). We control the loss by choosing w, and the learning algorithm's job is to find the weights that minimize it.

Gradient Descent: Following the Slope Downhill

How do we minimize a function of many variables? With gradient descent — the same idea you met in calculus, just scaled up. Imagine a one-variable function y = f(x) and you want to find a peak. Start at some point x₀, compute the derivative (the slope) there, and read its sign: a positive slope means the function rises if you move right, so you take a small step in that direction. Repeat until you stop improving.

1

Initialize. Pick starting weights (often random).

2

Compute the gradient. With many variables, the derivative becomes a vector of partial derivatives — one per weight. This vector is the gradient; it points in the direction the loss changes fastest. (This is the gradient that gives the algorithm its name.)

3

Take a step. Adjust every weight a little in the direction that reduces the loss. The size of that adjustment is the step size (learning rate), often written as a small ε.

4

Repeat — recompute the gradient at the new weights and step again, until a stopping condition is met.

Footgun — Choosing the Step Size

Step size is a delicate tradeoff. Too small and training crawls — it takes forever to converge. Too large and you can overshoot the optimum entirely, jumping clean over it without noticing. Tools expose this as a knob; variants like the popular Adam optimizer differ mainly in how cleverly they choose and adapt these steps.

You Won't Implement This By Hand

In practice, libraries like TensorFlow and PyTorch already implement gradient descent and compute gradients automatically. You specify the loss function; they handle the differentiation and the weight updates. The reason we still study the concept is that you must understand it to make sound training decisions — it isn't magic, it's the same optimization you've already seen in calculus.

Loss vs. Performance Metric — Two Different Numbers

Here is a subtlety that trips people up. The loss is what we optimize during training, but it is usually not the thing we ultimately care about. Suppose you're classifying images as "cat / no cat." You train by minimizing a loss like mean squared error or cross-entropy — but when you ask "how good is this network?", you don't report the loss. You report the performance metric: the real-world objective, such as classification accuracy ("it got 98% of the held-out images right").

Loss FunctionPerformance Metric
The thing you optimize during trainingThe thing you actually care about
Must be smooth/differentiable for gradient descentCan be any objective (e.g. accuracy), even non-differentiable
e.g. mean squared error, cross-entropye.g. % of images classified correctly
Drives the weight updatesTells you whether the result is any good

Why two numbers? Because accuracy ("right or wrong") is a hard, jagged objective that gradient descent can't smoothly slide down. We optimize a smooth proxy (the loss) but evaluate success with the metric we truly want. The next section shows that this distinction is the foundation of how we decide when training is done.

Part VII

The Training Workflow — Epochs, Batches & Splits

Even when a tool handles gradient descent for you, you still have to control the process: how the data is fed in, how it's split, and when to stop. These are the practical knobs the lecture closes on.

Epochs and Batches

An epoch is one full pass through the data: one application of gradient descent that touches every example once (you need the whole dataset both to compute the loss and to compute the gradient). Training runs many epochs — epoch 1, epoch 2, … up to some final epoch K — and you'll see the loss and the metric printed after each one.

Within an epoch, modern practice does not run gradient descent on the entire dataset at once. Instead it splits the training data into batches — small random subsets, with a typical batch size like 32, 64, or 128 rows — and performs a gradient-descent update on each batch in turn. The batches are re-randomized every epoch, so each epoch shuffles the data into fresh groupings.

The data is split into random batches; a gradient-descent update runs on each. Finishing all batches completes one epoch.
Why Batches? Two Orthogonal Reasons

First, efficiency: batches can be processed in parallel on hardware like GPUs. Second, and more profound, randomization: drawing random batches turns plain gradient descent into stochastic gradient descent (SGD), a variant that is meaningful beyond mere speed — it was one of the major advances behind modern progress. The randomness in the batch selection genuinely improves learning, not just throughput.

Train, Validate, Test — Three Roles for Data

You never train on all your data. You split it. The first split is training vs. testing (commonly 80% / 20%): train on the 80%, and after you're completely done, check the metric on the held-out 20% the algorithm never saw. But there's a third, subtler role. Carve a validation set out of the training data (e.g. split the 80% again). You train on what remains and, after each epoch, report the metric on the validation set to monitor progress.

SetUsed during training?Role
TrainingYes — gradient descent runs on itOptimizes the loss / fits the weights
ValidationYes — checked after each epochMonitors the metric to decide when to stop
TestingNo — never touched in trainingFinal, honest evaluation of the finished model
The Test Set Is Sacred — Never Peek

Validation and testing both compute a metric, so they feel similar — but their roles differ sharply. The validation set participates in training: you use it to decide when to stop. The test set must stay completely hidden during the entire training process. It is your "last line of defense," used only once you deliver the final product. Looking at the test set while training is effectively cheating — it leaks information and inflates your reported performance.

When Do You Stop? Avoiding Overfitting

How do you know when to halt? Two naive ideas both fail. "Stop after a fixed number of epochs" is arbitrary. "Keep going as long as the loss keeps dropping" is worse — it leads straight to overfitting: the network memorizes the training data so precisely that it stops generalizing to new examples.

The standard fix uses the validation set. After each epoch, watch the validation metric (e.g. accuracy). Normally accuracy climbs as you train — 80%, 90%, 95%, 96% — but at some point it plateaus, or even starts decreasing even while the training loss keeps falling. That divergence is the signature of overfitting, and it's your cue to stop.

Key Concept — Overfitting

Overfitting happens when a model fits the training data so closely that it captures noise and quirks specific to that data, hurting its performance on unseen examples. The validation set catches it: when validation accuracy stops improving (or degrades) while training loss still drops, the model is overfitting, and training should stop.

Patience & the Composite Stopping Rule

A common refinement is patience: if the validation metric hasn't improved for, say, 15 epochs, give up — patience has "run out." In practice the stopping criterion is often an aggregate: stop on lack of improvement, but also never exceed some maximum epoch count so you aren't waiting forever. High-level tools like Keras expose all of this — batch size, validation set, patience — as parameters you simply pass in.

Putting It All Together

This lecture built the entire feed-forward neural network from a single idea and then showed how to make it learn. Trace the arc:

We started with the neuron (Part I): a unit that takes a weighted sum of its inputs plus a bias. On its own that is purely linear, so we added an activation function (Part II) — step, sign, sigmoid, or the now-dominant ReLU — to inject the nonlinearity that makes deep stacking worthwhile. Wiring neurons into layers (Part III) gave us a DAG that is a universal function approximator, expressive enough to represent essentially any mapping. The XOR problem (Part IV) made the stakes concrete: one neuron can only carve a single straight line through its inputs, so it handles linearly separable functions like AND but fails on XOR — and the cure is exactly the layering from Part III.

Then we shifted from what a network is to how it learns. Unfolding a small network (Part V) revealed the central insight: a network's output is a function of its weights, and the weights are the one thing we control. That reframed learning as optimization. We packaged "how wrong are we?" into a loss function and minimized it with gradient descent (Part VI), carefully distinguishing the smooth loss we optimize from the real-world performance metric we actually want. Finally, the training workflow (Part VII) gave us the operational vocabulary — epochs, random batches and stochastic gradient descent, the train/validate/test split, and validation-based stopping to dodge overfitting.

The Through-Line

A neural network is a flexible nonlinear function whose behavior is governed entirely by its weights. "Learning" means searching weight-space — via gradient descent on a loss function — for the weights that best fit labeled data. Every concept in this lecture, from ReLU to patience-based stopping, exists to make that search work and to keep it honest.

Notice the recurring frame the professor used: syntax and semantics. The structure of neurons and layers is the "syntax" (what a network looks like); the function it computes and how we fit it to data is the "semantics" (what it means). That is the same lens this course applied to logic earlier in the term — first the form, then the meaning — now turned on a model-free, data-driven representation instead of a hand-built logical one.
