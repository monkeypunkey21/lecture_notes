Bayesian Networks: Capturing Belief Through Causal Structure

How a graph of causes plus a handful of numbers can pin down an exponentially large state of belief — compactly, intuitively, and exactly.

Part I

Why Bayesian Networks?

From Logic to Probability — and Now to a Modeling Tool

For a long stretch of its history, AI tried to reason using classical logic: facts are either true or false, and you chain them together with rules of inference. That works beautifully for things you are certain about. But people constantly reason in ways logic cannot capture. "It looks like rain, so I'll grab an umbrella" is not a logical deduction — clouds do not entail rain. This is plausible reasoning, and logic simply has no machinery for it.

The previous lecture introduced the fix the field eventually agreed on: probability theory as the formal basis for beliefs. Instead of "true" or "false," every statement gets a number between 0 and 1, and we have rules (conditioning, Bayes' rule, marginalization) for updating those numbers as evidence arrives. That gave us the foundation. Today's lecture builds the tool that lets us actually specify and work with beliefs over many variables at once: the Bayesian network.

Key Concept — Bayesian Network

A Bayesian network is a high-level, intuitive way to specify a probabilistic state of belief. It is a modeling tool: you describe what causes what and supply a few local probabilities, and the network induces a full probability distribution over all the variables.

The Two Components

Every Bayesian network has exactly two parts, and it is worth fixing them in your mind now because the whole lecture is really just unpacking each one.

1

A directed acyclic graph (DAG). Nodes are variables; directed edges point from one variable to another. We will read this graph as a picture of causality — an edge from A to B means "A directly influences B." Crucially, the graph alone (with no numbers attached yet) already tells us a great deal about which variables are independent.

2

Numeric annotations. For each variable we attach a small table of probabilities describing how that variable behaves given its direct causes. These are the numbers — and later in the course, we will see they can even be learned from data.

Put the structure and the numbers together and they jointly induce a probability distribution: the same kind of object from last lecture, where every possible world (every combination of variable values) gets a probability, and all those probabilities sum to 1.

The Punchline: Compactness

Here is why this matters so much. A probability distribution written out as a flat table is exponential in the number of variables. With 5 binary variables you have 2⁵ = 32 rows; with 30 variables you have over a billion. You could never elicit that table by asking someone to fill in every row.

The Big Idea

A Bayesian network is a compact representation of a distribution. The full distribution is exponential in the number of variables, but the network is generally not — it stores only small local tables. Yet structure + small tables pin down the giant table completely and uniquely.

Where This Is Heading

This sets up the next several lectures. Once a belief state is specified by a network, we ask how to answer queries about it (inference) — this year by compiling networks into circuits reminiscent of neural networks. From there, the course flows into machine learning, where the network's numbers are learned from data rather than handed to us.

Part II

The Notation You Cannot Skip

Before any of the real machinery, the professor deliberately slowed down to fix notation — because this area reuses a small set of symbols in very precise ways, and mixing them up makes everything downstream confusing. Read this part carefully; it is the alphabet for the rest of the lecture.

Variables vs. Values: Big X and Little x

The first convention is a capitalization rule. A variable is written with a capital letter, like X. A specific value that the variable takes is written with the matching lowercase letter, like x.

Key Concept — Variable vs. Value

If X is a variable whose possible values are red, blue, green, then a lowercase x stands for one of those — say red. Writing P(x) is shorthand for "the probability that the variable X takes the particular value x," i.e. P(X = x). This is a single number.

A Distribution Is a Whole Table

Now contrast P(x) (a number) with P(X) (capital). Writing the probability of the variable — not a specific value — denotes the entire distribution over that variable: a small table giving one probability for each possible value.

P(X) is the whole table — one probability per value. P(x) picks out a single cell.

Sets of Variables and Instantiations

Often we care about several variables at once. A set of variables is written in bold, like X (bold). When you take a bold set and fix a value for every variable in it, the result is called an instantiation, written with a bold lowercase x.

Concretely, suppose X has values {red, blue, green} and we add a second variable Y with values {large, small, medium}. Let bold A = {X, Y}. Then a bold instantiation a is a full assignment, for example "X = red and Y = small." Another instantiation would be "X = red and Y = medium." The professor (and we) use state and instantiation interchangeably — they mean the same thing.

Don't Confuse These

Plain capital = one variable. Bold capital = a set of variables. Plain lowercase = a value of one variable. Bold lowercase = an instantiation (a value for every variable in the set). The bolding carries real meaning; losing it is the most common way students misread these formulas.

Binary shorthand: t and t'

For a binary variable T (values true/false) we previously wrote t for "T is true" and ¬t for "T is false." A convenient alternative is to write t for true and t' (t-prime) for false. So t' means exactly the same as ¬t. It can be slightly ambiguous, but context makes it clear, and it shows up constantly in the tables ahead.

Variable Independence

This is the notation that earns its keep. Last lecture we met conditional independence between sentences: P(α | β, γ) = P(α | γ) means "given γ, learning β tells you nothing new about α." Here α, β, γ were individual sentences.

Now we lift this idea to whole sets of variables. Given three variable sets X, Y, Z (all bold), we write

I(X, Z, Y)   # X and Y are independent given Z

The middle set, Z, is the thing that is given (known). The claim is: once you fix the state of Z, telling me the state of Y does not change my belief about any state of X. Formally, for every instantiation, P(x | y, z) = P(x | z).

One Statement, Many Statements

Variable independence is a compact declaration of sentential independence. Suppose X = {A, B}, Y = {C}, Z = {D, E}, all binary. Then X has 4 states, Y has 2, and Z has 4 — so I(X, Z, Y) is actually 4 × 2 × 4 = 32 separate sentence-level independence statements (e.g. "A∧B is independent of C given D∧E", "A∧¬B is independent of C given D∧E", and so on). One line of variable notation packs all 32.

This compactness is exactly why Bayesian networks reach for variable independence. The structure of a network will assert many such independences at once — which is what makes the graph so informative before we ever write down a number.

Part III

The Graph — Capturing Independence Visually

The first component of a Bayesian network is its graph. The goal of this part is to see how drawing a graph of causes automatically commits you to a set of independence beliefs — so the picture does real work, not decoration.

The Running Example: A Burglar Alarm

We extend last lecture's classic story. You own a burglar alarm. It can be triggered by a burglary (its intended job) but it is also sensitive to earthquakes. If the alarm goes off, a neighbor may call you. And separately, an earthquake might produce a radio report. That gives five variables:

E = earthquake, B = burglary, A = alarm, C = call from neighbor, R = radio report. Arrows point from cause to effect.
Key Concept — Directed Acyclic Graph (DAG)

A graph has nodes and edges. Directed: every edge has an arrow pointing from one node to another. Acyclic: you can never follow the arrows and return to where you started — no loops allowed. This DAG is the first component of a Bayesian network.

Edges as Causality

We read each edge as a direct causal influence. An earthquake can cause the alarm to trigger, so E → A. A burglary can too, so B → A. The alarm triggering can cause a call, so A → C. And an earthquake can cause a radio report, so E → R. Viewed this way, the graph is simply a picture of what causes what.

Here is the surprising payoff. If you tell me only your causal picture, I can already read off some of your independence beliefs — with no numbers at all. Two examples from the alarm graph:

1

R and C are independent given A. Normally a radio report (R) would change your belief about getting a call (C): a report suggests an earthquake, which suggests the alarm fired, which suggests a call. But if you already know the alarm's state (say your phone confirms it did not trigger), then the chain is cut — a radio report no longer moves your belief about a call. So I(R, A, C).

2

E and B are independent given the empty set. Earthquakes and burglaries have no common cause and neither causes the other, so knowing nothing else, learning about one tells you nothing about the other: I(E, {}, B).

The Big Idea

Your perceptions of causality, drawn as a DAG, imply a set of independence beliefs automatically. The graph is the cheap, intuitive first half of eliciting your enormous belief table — it constrains the table heavily before a single number is written.

Three Family Terms: Parents, Descendants, Non-descendants

To state the general rule precisely, we need three pieces of graph vocabulary, all relative to a node V.

Parents

The parents of V are the nodes with an edge pointing directly into V — its direct causes. In our graph: parents of A are {E, B}; parents of R are {E}; parents of C are {A}.

Descendants

The descendants of V are every node you can reach by following edges forward from V — everything downstream. Descendants of E: follow E → A → C, giving {A, C} (and via the rule we are about to verify, also R is a child of E but R is reached from E too — careful: descendants of E are {A, C, R}). Descendants of B: {A, C}. Descendants of C: nothing — C is at the bottom.

Key Concept — Non-descendants (the special version)

Literally, the non-descendants of V are everything that is not a descendant. But for our purposes we use a refined version: take the non-descendants and then exclude V itself and its parents. This trimmed set is what the next rule will reference.

Let's compute it for two nodes:

  • Node A. Descendants = {C}. Everything else = {A, B, E, R}. Remove A (itself) and its parents {E, B}, leaving non-descendants = {R}.
  • Node E. Descendants = {A, C, R}. Removing E and its parents (E has no parents) leaves non-descendants = {B}.

Part IV

Markovian Assumptions — Reading the Graph as Independence

We now state the single rule that turns a DAG into a precise list of independence statements. These statements are the Markovian assumptions of the network: all the independences implied directly by the graph's structure.

The Reading Rule

Key Concept — Markovian Assumption

For every variable V: once you know the states of V's parents, information about V's non-descendants does not change your belief about V. In symbols, I(V, parents(V), non-descendants(V)).

Because the rule applies to every node, a network with five variables yields five Markovian assumptions — one per variable. The intuition is causal: your direct causes (parents) screen you off from everything that is neither downstream of you nor one of your causes.

Worked Examples

Let's instantiate the rule on the alarm graph, node by node.

1

V = C (call). Parents(C) = {A}. Non-descendants(C) = {B, E, R}. So I(C, A, {B,E,R}): once you know whether the alarm fired, your belief about getting a call is independent of the burglary, the earthquake, and the radio report.

2

V = R (radio report). Parents(R) = {E}. R has no descendants, so its non-descendants are {B, A, C}. So I(R, E, {B,A,C}): once you know whether there was an earthquake, the report's likelihood is independent of burglary, alarm, and call.

3

V = A (alarm). Parents(A) = {E, B}. Non-descendants(A) = {R}. So I(A, {E,B}, R): knowing the earthquake and burglary, the alarm is independent of the radio report.

4

V = E and V = B (the root causes). E and B have no parents, so each is independent of its non-descendants given nothing. This is exactly the earlier observation that earthquake and burglary are independent I(E, {}, B).

Structure Without Numbers

Notice we stated all five Markovian assumptions using only the graph — no probability tables anywhere. That means: whatever numbers you eventually attach, these independences are guaranteed to hold. Different numbers give a different belief table, but the same structural independences persist. The graph constrains the table; the numbers fill in the rest.

Why This Matters Later

This guarantee is what makes the upcoming "compile to circuits" idea powerful. Because behavior is constrained by your causal model regardless of the numbers, the resulting circuit — trained like a neural network later in the course — carries provable guarantees instead of being a black box.

Part V

CPTs and Inducing the Full Distribution

The graph gave us independences. Now we add the second component — the numbers — and show how structure plus numbers reconstruct the entire belief table. We switch to a second example network with five variables: winter (A), sprinkler (B), rain (C), wet grass (D), and slippery road (E).

Conditional Probability Tables (CPTs)

Each variable gets one CPT — a Conditional Probability Table — giving the probability of that variable's states for every combination of its parents' states. With five variables we have five CPTs. The convention: in a CPT, the variable listed last is the one whose probability we are specifying; the earlier columns are its parents.

A CPT for D with parents B and C. The highlighted row encodes P(d' | b, c') = 0.1.

Parameter Notation: θ

A single number inside a CPT is called a parameter. We write it as θ (theta) with the node's value first and its parents' values after a bar. So the highlighted row above is

θd' | b, c'  =  P(d' | b, c')  =  0.1

Other examples from this network: θ(e' | c') = 1 and θ(a') = 0.4 (variable A has no parents, so its CPT is just a prior). This θ-notation becomes essential during learning, where parameters are the very quantities estimated from data.

Counting Independent Parameters

A binary node with no parents has two parameters (P(a), P(a')) but only one is free — the other is forced because they sum to 1. A binary node with two binary parents has 4 × 2 = 8 entries but only 4 independent ones. This freedom-counting is where the compactness savings actually live.

Filling the Distribution Table, One Row at a Time

Now the central mechanism. The five CPTs induce a full distribution over all 2⁵ = 32 worlds. To find the probability of any one world (one full row of the giant table), use this rule:

The Big Idea — The Product Rule

For a chosen world, go to each of the five CPTs and pull out the one parameter whose row is compatible with that world (matches the values of the node and its parents). Multiply the five numbers. That product is the probability of the world.

Let's trace the world where A = true, B = true, C = false, D = true, E = false. We pull one number from each CPT that matches this assignment:

1

CPT for A (no parents). A is true → pull 0.6.

2

CPT for B (parent A). B is true with A true → pull 0.2.

3

CPT for C (parent A). C is false with A true → pull 0.2. (Be careful here — match both C's value and its parent A's value.)

4

CPT for D (parents B, C). D true, B true, C false → pull 0.9.

5

CPT for E (parent C). E false, C false → pull the matching entry. Multiply all five: 0.6 × 0.2 × 0.2 × 0.9 × … → the world's probability comes out to 0.0216.

One more, the all-false world (everything false): pull 0.4 (from A), 0.25 (B), 0.9 (C), 1 (D), 1 (E), giving 0.09. Do this for all 32 rows and you have reconstructed the entire distribution — from five small tables.

Why does multiplying CPT entries work? (The chain rule)

This product rule is not arbitrary — it is the probability chain rule specialized to the network's independences. The chain rule says any joint factors as P(A) · P(B|A) · P(C|A,B) · P(D|A,B,C) · P(E|A,B,C,D). The Markovian assumptions then let us drop every conditioning variable that is a non-descendant given the parents, collapsing each factor to P(node | parents) — exactly the CPT entries we pull. The result is the product over nodes of θ(node | parents).

P(world) = ∏over nodes V  θ( v | parents(V) )
A Well-Behaved Belief State

The table you reconstruct is not just any table. Every independence implied by the structure is automatically respected by it. So the distribution's behavior is governed and constrained by your perceptions of causality — you could have predicted some of its independence behaviors before ever seeing a number.

Part VI

D-Separation — Testing Independence on the Graph

Why the Markovian List Is Not Enough

Suppose someone hands you a complicated DAG and asks: "Are X and Y independent given Z?" One approach is to list all the Markovian assumptions and look. But the Markovian list is only the directly implied independences. You can often derive more independences by combining them with the laws of probability. So if your independence is not on the list, it might still hold — the list is necessary-to-trust but not complete-to-rule-out.

A Subtle Trap

If an independence appears in the Markovian assumptions, it definitely holds. But if it does not appear, you cannot conclude it fails — it may still be implied. Reasoning "it's not on the list, so it's false" is a genuine error.

Key Concept — D-Separation

D-separation is a graphical test applied to a DAG that detects all the independences — the Markovian assumptions and everything derivable from them — without doing any algebra. Run the test on X, Y, Z and it tells you directly whether X and Y are independent given Z.

Paths and Valves

The intuition is about information flow. To test whether X and Y are independent given Z, look at every path between X and Y. Picture each path as a pipe and each node along it as a valve that is either open or closed (given what we know in Z).

1

A path is blocked if at least one valve on it is closed — one closed valve stops the whole pipe.

2

If every path between X and Y is blocked, then X and Y are d-separated given Z, hence independent.

3

If any path is unblocked (all its valves open), information can flow and we do not have independence.

So the entire test reduces to one question: is a given valve open or closed? And that depends on which of three shapes the valve has. Along a path, a node connects to its two path-neighbors in exactly one of three ways:

The three valve types at a node W (blue). Sequential: a cause flows through W to an effect. Divergent: W is a common cause. Convergent: W is a common effect.
The Same Node, Different Valves

A valve type depends on the path, not the node alone. In the alarm graph, node A is a sequential valve on the path E → A → C, but a convergent valve on the path E → A ← B. Once you fix a specific path, the neighbors are unambiguous, so the valve type is clear.

The Three Valve Rules

Everything is decided "given that we know Z." Here are the rules — two are identical and intuitive; the third is the famous twist.

Sequential valve (cause → W → effect)

Example: E → A → C. Does knowing there was an earthquake change your belief about a call? Yes — through the alarm. But if you already know the alarm's state, the earthquake no longer matters for the call. So:

Sequential valve is CLOSED  ⇔  W ∈ Z   # the middle node is known

Divergent valve (W is a common cause)

Example: R ← E → A. A radio report and the alarm are correlated through their common cause, the earthquake. But if you already know whether there was an earthquake, that correlation vanishes. Same rule as sequential:

Divergent valve is CLOSED  ⇔  W ∈ Z   # the common cause is known

Convergent valve (W is a common effect) — the twist

Example: E → A ← B. Two causes, one shared effect. If you know nothing about the alarm, do earthquake and burglary interact? No — two independent causes of a common effect are independent until you observe the effect. But the moment you know the alarm fired, they start interacting: if you then learn there was no burglary, the earthquake becomes more likely (something had to set it off). This is "explaining away." So the rule flips:

Convergent valve is CLOSED  ⇔  W ∉ Z   # the common effect is UNknown
# (and, in the full rule, no descendant of W is known either)
Why Convergent Is Backwards

For sequential and divergent valves, knowing the node closes it (cuts the flow). For a convergent valve, knowing the common effect opens it — observing a shared symptom makes its competing causes inform each other. This single asymmetry is the heart of d-separation, and it is exactly the "explaining away" reasoning logic could never capture.

Valve typeShape at WClosed (blocks flow) when…
Sequential→ W →W is in Z (known)
Divergent← W →W is in Z (known)
Convergent→ W ←W is not in Z (W and its descendants unknown)

With these three rules, the test is mechanical: for each path between X and Y, label every valve, decide open/closed using Z, and check whether the path is blocked. If all paths are blocked, X and Y are d-separated — and therefore independent given Z.

Putting It All Together

This lecture built a single, coherent pipeline for capturing an enormous belief state without ever writing it out. It starts from the realization (Part I) that probability succeeds where logic fails at plausible reasoning, but a raw distribution is exponential and impossible to elicit directly. The Bayesian network is the fix, and it has exactly two halves.

Parts II–IV are the structural half. After fixing the notation (variables vs. values, sets vs. instantiations, and the compact I(X, Z, Y) form of independence), we drew a DAG of causes. That graph, through the family terms parents / descendants / non-descendants, hands us the Markovian assumptions for free — a precise set of independences implied by the structure alone, before any number exists.

Part V is the numeric half. Each node gets a CPT of parameters θ(v | parents). The graph's independences turn the probability chain rule into a simple product: the probability of any world is just the product of one matching entry per CPT. Structure plus numbers therefore pin down the full 2ⁿ table completely and uniquely — and that table automatically respects every structural independence.

Part VI is the reasoning half. Because the Markovian list misses derived independences, d-separation gives a complete graphical test: trace paths, classify each valve as sequential, divergent, or convergent, and decide open/closed given what you know. Two valve rules are intuitive; the convergent "explaining away" rule is the deep twist that makes probabilistic reasoning richer than logic ever was.

The One-Sentence Takeaway

A picture of causes (the DAG) plus a few local tables (the CPTs) compactly and exactly specifies a full probabilistic belief state — and you can read that state's independences straight off the graph using d-separation.

What's Next

We have specified a belief state; next we need to query it — given evidence, what is the probability of some event? That bridges directly into the course's final theme, machine learning, where the CPT parameters θ are no longer handed to us but learned from data — turning hand-built causal models into trainable, guarantee-carrying systems.
