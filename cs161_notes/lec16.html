Learning Bayesian Networks From Data

An interactive guide to Lecture 16 — estimating parameters with maximum likelihood, filling in missing data with EM, learning network structure, and bridging to supervised learning through arithmetic circuits and cross-entropy loss.

Part I

Where Do the Numbers Come From?

In every lecture so far we were handed a Bayesian network: a graph of variables plus all of its probability tables, ready to answer queries. But that raises an awkward question. A Bayesian network has two pieces — a structure (which variable points to which) and parameters (the actual probability numbers). Where do those come from in the real world? Nobody walks up to a doctor and reads off "the probability of body aches given the flu is 0.83." This lecture is about the other direction: instead of using a network, we build one from observed data.

Key Concept — A Bayesian Network Has Two Parts

The structure is the directed graph: which variables are parents of which. The parameters are the conditional probability tables (CPTs) that fill in the numbers. A network is only fully specified once you have both. Learning means recovering one or both of them from data.

Two Jobs, and Two Ways to Slice Them

Learning a Bayesian network breaks into two distinct jobs. The first is learning the parameters: you already know the structure (perhaps it came from a story or an expert), and you just need to fill in the numbers. The second, harder job is learning the structure itself — figuring out the arrows — which in practice means you must also learn the parameters of whatever structure you pick. This is a deep subject; we will treat the common cases in detail and only sketch the rest.

There is a second, orthogonal way to slice the same territory. We can ask what we are learning the network for. If we want a general-purpose model that can answer any query you might throw at it later, that is model-oriented learning. If instead we know in advance the one specific question we will always ask — "given these inputs, predict that output" — that is query-oriented learning, which turns out to be exactly the supervised learning that dominates modern AI. We build up to that in Parts V and VI.

Why This Matters Later

The query-oriented view is the bridge to neural networks, which we tackle next week. A neural network is, underneath, a circuit whose parameters are tuned to make predictions match labels — exactly the supervised setup we reach at the end of this lecture. Keep that destination in mind.

What a Data Set Actually Looks Like

Throughout the lecture, "data" means a table. Each column is a variable of the Bayesian network. Each row is one example — think of it as one patient, on whom we recorded some observations. So a row is a snapshot of the world: it tells us the value each variable took for that one case. The whole table together is the data set, which we write as D.

A data set D. Row 2 is a complete example (every value known); row 1 is incomplete — the value of Ache is missing (the red ?).

Complete vs Incomplete Data

The single most important distinction in this lecture is whether the data is complete. An example is complete when we know the value of every variable for that case. If even one value is missing — say a patient's chest X-ray was never taken — the example is incomplete. A data set is called complete only if all of its examples are complete; one missing cell anywhere makes the whole data set incomplete.

Why does this matter so much? Because it controls how hard learning is:

Complete DataIncomplete Data
Maximum-likelihood parameters are uniqueMultiple maximum-likelihood solutions can exist
Found by a single pass over the data (closed form)Requires an iterative algorithm (EM)
Fast and well-behavedSlower; the more missing data, the slower it converges
The Big Idea

Whether your data is complete or not is the fork in the road for all of parameter learning. Complete data gives you a clean, one-shot answer. Missing data forces you into an iterative, hill-climbing procedure with no guarantee of a unique answer. Everything in Parts II and III flows from this one distinction.

Part II

Maximum Likelihood on Complete Data

Suppose we know the structure and our data is complete. We want to choose the parameters — the numbers in every CPT. But there are infinitely many possible choices. How do we decide which is best? We need a score: a single number that tells us how good a candidate set of parameters is, so we can pick the highest-scoring one.

Scoring Parameters: The Likelihood

Here is the idea. Pick a set of parameters. Those parameters define a complete Bayesian network, and that network defines a probability distribution over all the variables. Now ask that distribution a pointed question: "How probable do you think the data we actually saw is?" A good set of parameters should assign high probability to the data we observed; a bad set should assign it low probability. That probability of the data is the likelihood of the parameters, and we choose the parameters that maximize it.

Key Concept — Likelihood

The likelihood of a set of parameters is the probability the resulting distribution assigns to the observed data set. Because examples are assumed independent, it is the product of the probabilities of the individual examples:

L(θ) = Pr(e1) × Pr(e2) × ··· × Pr(en)

The maximum-likelihood (ML) parameters are the ones that make this product as large as possible.

To compare two candidate parameter sets, θ1 and θ2, you compute the likelihood of each — the product of how probable each distribution thinks the patients are — and keep whichever scores higher. The remarkable fact for complete data is that you do not have to search at all: there is a direct formula.

The Empirical Distribution

The key tool is the empirical distribution, written PrD. The idea is almost too simple: just read the probabilities straight off the data by counting. The probability the empirical distribution assigns to any full instantiation of the variables is simply the frequency with which that combination appears in the data.

Concretely, suppose our network has three variables — H (health-aware), S (smokes), E (exercises) — and the data set has 16 examples. Many rows repeat: two different people might both be health-aware, non-smoking exercisers. The empirical distribution collapses those repetitions. If a particular combination appears 9 times out of 16, its empirical probability is 9/16. Combinations that never appear get probability 0.

The empirical distribution summarizes 16 raw rows into the distinct instantiations, each weighted by how often it occurred. One pass over the data builds it.

Reading Off the Estimates

Once we have the empirical distribution, every parameter is just a ratio of two of its numbers. Say we want θ = Pr(S' | H), the probability someone doesn't smoke given they're health-aware. By the definition of conditional probability (Bayes conditioning):

Pr(S' | H) = Pr(S', H) / Pr(H)

and we get both of those probabilities from the empirical distribution. Let's trace it with the lecture's numbers.

1

Numerator — Pr(S', H). How many of the 16 examples have someone who is health-aware and does not smoke? Adding up the matching rows (9/16 from one combination plus 1/16 from another) gives 10/16.

2

Denominator — Pr(H). How many examples are health-aware at all? Summing the rows where H is true gives 12/16.

3

Divide. θ = (10/16) ÷ (12/16) = 10/12 = 5/6. The 16s cancel — it is really just "10 cases out of the 12 relevant ones."

4

Repeat. Do the same ratio for every parameter in every CPT. One pass to build the empirical distribution, then a division per parameter, and you are done.

Why the Division?

A student asked exactly this in lecture. The parameter is a conditional probability, and a conditional is by definition a joint over a marginal: Pr(S'|H) = Pr(S', H)/Pr(H). We are not inventing the formula — it is just Bayes conditioning, which we studied earlier in the course, applied to numbers we counted from the data.

For complete data these estimates are not merely convenient — they are provably the maximum-likelihood parameters, and they are unique. So the closed-form "count and divide" recipe is genuinely the best answer the likelihood score can give.

Part III

The EM Algorithm for Incomplete Data

Why the Easy Method Breaks Down

The "count and divide" recipe rests entirely on the empirical distribution. But the moment a row has a missing value — say (H=true, S=false, E=?) — we cannot count it. Which bucket does it belong to? (T, F, T) or (T, F, F)? We genuinely do not know, so we cannot build the empirical distribution, and the whole closed-form approach collapses. We need something cleverer.

The Trap

You might be tempted to just throw away incomplete rows, or to guess the missing value. Throwing rows away wastes data; guessing a single value bakes in an arbitrary commitment. The EM algorithm does something better: it keeps the row but spreads it probabilistically across all its possible completions, weighted by how likely each completion is under the current model.

The Completion Loop

The algorithm is called EM, for Expectation–Maximization — and it is the same algorithm you used in your homework. It is iterative. The trick is a chicken-and-egg resolution: if we knew the parameters we could fill in the missing values, and if the values were filled in we could compute the parameters. EM bootstraps by guessing one, then alternating.

1

Initialize. Take the known structure and invent a starting set of CPTs — typically random parameters. Call these CPT1. They define a Bayesian network and therefore a distribution Pr1.

2

Expectation — complete the data. For each incomplete row, list its possible completions. The row (T, F, ?) has two: (T, F, T) and (T, F, F). Use the current distribution Pr1 to compute the probability of each completion — e.g. Pr(E=true | H=true, S=false) = x, and Pr(E=false | ...) = y. These x, y are weights, not hard choices.

3

Build a weighted empirical distribution. A complete row counts with weight 1. An incomplete row is replaced by its completions, each carrying its weight x or y instead of a full count of 1. Now build the empirical distribution using these fractional weights instead of integer counts.

4

Maximization — re-estimate. With a (soft) complete data set in hand, apply the Part II "count and divide" recipe to get new parameters CPT2, a new network, and a new distribution Pr2.

5

Repeat. Use Pr2 to recompute the completion weights, rebuild the weighted empirical distribution, re-estimate to get CPT3, and so on — until the parameters stop changing.

Soft Counts: The Heart of EM

The whole trick is the move from hard counts to soft counts. In complete data each row contributes a count of 1 to one bucket. In EM, an ambiguous row contributes a fraction (its weight) to each possible bucket. As the parameters improve, those weights sharpen, and the estimates improve with them. Every iteration internally runs probabilistic inference on the current network to compute those weights — which is why each iteration is computationally costly.

Convergence and Local Optima

EM comes with a guarantee: across iterations, the likelihood of the parameters never gets worse. It either improves or stays put, until it converges. That makes EM a kind of local search — it hill-climbs through parameter space, exactly like the local-search algorithms from the search unit of the course.

And that connection comes with the same caveat. Local search can get stuck in a local optimum, and so can EM: where it converges depends on where it started. This is the concrete face of the fact that incomplete data can have multiple maximum-likelihood solutions. The standard remedy is also borrowed from local search — random restarts: run EM several times from different random initial parameters and keep the best result. One more rule of thumb: the more data is missing, the slower EM converges.

Connection — This Builds On Search

If "hill-climbing, local optima, random restarts" sounds familiar, that is by design. EM is local search wearing a probability costume. The intuitions you built studying search algorithms transfer directly: a guarantee of non-worsening progress, sensitivity to the starting point, and restarts to escape bad basins.

Part IV

Learning the Structure

So far we assumed the arrows were given. Now the harder problem: we are handed a data set and asked to learn both the structure and the parameters. Conceptually this splits in two — pick a structure, then estimate its parameters with the tools from Parts II–III. The real challenge is the picking. There are two questions: what score tells us one structure is better than another, and how do we search the space of structures.

Three candidate structures over the same variables. Structure learning is choosing among them (and many more) by score, then estimating each one's parameters.

Why Maximum Likelihood Is the Wrong Score Here

It is tempting to reuse our likelihood score: pick the structure whose best parameters make the data most probable. But this fails badly. A denser structure — more arrows, more parameters — can always fit the data at least as well as a sparser one, because it has more freedom. So pure maximum likelihood will always prefer the most complicated structure it is allowed, fitting the data perfectly including its noise. That is overfitting: memorizing the training data instead of capturing the real pattern.

Common Mistake — Likelihood Alone Overfits

Maximum likelihood rewards fit and nothing else. Since complexity always buys more fit, likelihood alone hands you the most tangled possible network — great on the training data, useless on new data. For parameter learning likelihood was perfect; for structure learning it is dangerous on its own.

The Penalty Term

The fix is to add a penalty for complexity, so the score becomes "fit minus complexity." The structure search then has to earn each extra arrow: a new parameter is only worth it if it improves the fit by more than the penalty it costs. This balances the two forces and prevents overfitting.

The penalty has two ingredients. First, model complexity — typically the number of independent parameters in the structure; you literally count them. Second, the size of the data set, which scales the penalty. The reasoning is subtle but important: with a huge amount of data, fitting it closely is actually fine — there is so much signal that overfitting is not a worry. You only get into trouble with too little data. So the penalty should shrink as the data set grows.

The Big Idea — Fit Minus Complexity

Structure learning is a tug-of-war: likelihood pulls toward complex networks that fit every wrinkle, the penalty pulls back toward simple ones. The penalty grows with the number of parameters and shrinks with the amount of data. This "fit minus complexity" trade-off is one of the most universal ideas in all of machine learning — you will meet it again under names like regularization and the bias-variance trade-off.

Searching the Space of Structures

With a score in hand, how do we find the best-scoring structure? The space of possible graphs is enormous, so we cannot check them all. Instead we reuse machinery from earlier in the course — local search again. Start from some initial structure, which already has a score. Then apply small transformations — add an arrow, delete an arrow, reverse one — and check whether the transformed structure scores better. Move to it if it does, and keep climbing.

Connection — The Search Unit Pays Off Twice

Local search shows up twice in this lecture: once as EM for parameters under missing data, and again here for structures. Both start somewhere, take improving steps, and risk local optima. The score is different (likelihood vs. penalized likelihood) but the search skeleton is identical.

Why does structure learning add a penalty term instead of just maximizing likelihood like parameter learning does?

Part V

Query-Oriented Learning & Arithmetic Circuits

Model-Oriented vs Query-Oriented

Everything so far was model-oriented: we learned a network that could answer any query — you give evidence on whatever variables you like and ask about whatever you like. But often we know the question in advance. Imagine we will always observe inputs E1, E2, E3 and always want to predict a target Q. That is query-oriented learning: we can specialize the learning to do that one job well.

This reframing reveals a familiar structure. We have inputs and a target output, and our data lists inputs alongside the correct answer for each. Often a human supplies that answer — looking at an image and writing "cat" or "no cat." That human-supplied answer is a label, the data is labeled data, and learning to reproduce labels from inputs is supervised learning — the engine behind most real-world AI today.

Model-OrientedQuery-Oriented (Supervised)
Learn a general model for any queryLearn for one fixed input → output query
Score: maximum likelihood / penalized likelihoodScore: a loss function (e.g. cross-entropy)
Data: full instantiations of all variablesData: inputs paired with human labels
Connection — The Bridge to Neural Networks

This whole story is the on-ramp to next week. A neural network is used almost exactly this way: a circuit with tunable parameters, fed labeled data, optimized so its outputs match the labels. Get the Bayesian-network version solid now and neural networks will feel like a variation on a theme rather than something brand new.

Compiling a Bayesian Network Into a Circuit

To do supervised learning cleanly, we borrow a tool from Bayesian-network inference: the arithmetic circuit (AC). Given a network plus a chosen query — say, network over A, B, C, with evidence always on A and C, predicting B — we can compile the pair into a circuit that computes the answer.

Key Concept — Arithmetic Circuit

An arithmetic circuit is a directed graph of just two kinds of internal nodes — addition nodes and multiplication nodes — with two kinds of inputs at the bottom. Evaluating it bottom-up (multiply and add) produces the query answer at the top, in time linear in the size of the circuit.

The two kinds of inputs are the crucial part:

  • Parameters (θ). These come straight from the Bayesian network's CPTs — in the lecture's example there are exactly 10 of them, matching the network's parameters.
  • Indicators (λ). These are how we feed in evidence. There is one indicator per value of each evidence variable: two for binary A (λA, λ¬A) and two for binary C.
Schematic arithmetic circuit: θ parameters and λ indicators feed multiplications, which feed additions, producing the query distribution Pr(B) at the top.

Where does the circuit come from? It is compiled by the very inference machinery we saw earlier — write the network as a Boolean formula with weights, compile that to a circuit (the NNF → AC reduction), and you can answer queries by weighted model counting. We will not redo that derivation; the point is the link between probabilistic reasoning and logical reasoning. And the cost is the inference cost we already know: linear in the number of variables, exponential only in the network's treewidth.

Evaluating With Indicators

Indicators are how evidence enters the circuit, and the rules are simple. For a variable, set the indicator of its observed value to 1 and all others to 0. If the value is unknown, set all of its indicators to 1.

You observe (binary A)λAλ¬A
A = true10
A = false01
A unknown11

So for evidence "A = false, C unknown," you set λA=0, λ¬A=1, λC=1, λ¬C=1. Then evaluate bottom-up. The intuition for the "unknown → all ones" rule: a 1 lets every term through, so leaving all indicators on means "sum over all values of this variable," i.e. marginalize it out — exactly what you want when you have no evidence about it.

Setting both to 1 lets every term involving C pass through the circuit, which is exactly how the circuit sums over (marginalizes out) C. Setting both to 0 (option A) would zero out those terms entirely — the opposite of "no information," it would assert C is impossible.
Part VI

Loss, Gradient Descent & Background Knowledge

We have a circuit that turns parameters and evidence into a prediction. For supervised learning we flip the problem: we don't know the parameters and want to learn them so that, on our labeled data, the circuit's predictions match the labels. We no longer maximize likelihood — we minimize a loss function that measures how far predictions are from labels.

Cross-Entropy Loss

For each example the circuit produces a predicted distribution over the target. Feed in A=true, C=true and the circuit might output Pr(B)=0.7, Pr(B')=0.3. The data, meanwhile, gives a label: it just says B is false. We treat that label as a distribution too — a one-hot distribution that puts probability 1 on the labeled value and 0 everywhere else, here Pr(B)=0, Pr(B')=1. The loss measures the gap between the predicted distribution and this one-hot label distribution.

Key Concept — Cross-Entropy

Cross-entropy takes two distributions over a variable X — predictions and label — and returns one number measuring how far apart they are. The formula sums over the values x of X:

H(pred, label) = − Σx pred(x) · log label(x)

Smaller cross-entropy means closer distributions. We minimize it, averaged over all examples, so the circuit's predictions hug the labels across the whole data set.

One-Hot Labels

The label distribution is "certain" — it knows the answer. For binary B it is just (0, 1) or (1, 0). If B had three values like {small, medium, large}, the one-hot label for "medium" would be (0, 1, 0). Recasting a hard label as a degenerate distribution is what lets us compare it to the model's soft prediction using a single, smooth measure.

Gradient Descent

How do we actually find parameters that minimize the average cross-entropy? With gradient descent. Start with a guess for the parameters. Compute the derivative of the loss with respect to each parameter — that gradient points in the direction the loss increases fastest. To decrease the loss, step in the opposite direction. Repeat until the loss stops dropping. It is exactly the calculus idea of "follow the slope downhill," generalized from one variable to many.

1

Guess parameters. Initialize the θ values (often randomly).

2

Measure loss. Run the circuit on the data and compute the average cross-entropy between predictions and labels.

3

Compute the gradient. Take the derivative of the loss with respect to every parameter to learn which way is "uphill."

4

Step downhill. Nudge each parameter in the direction that lowers the loss, then loop back to step 2 until convergence.

You Don't Hand-Code This Anymore

Years ago you would program these derivatives yourself, with varying success. Today tools like TensorFlow (Google) and PyTorch (Meta) do it for you: hand them the circuit, the data, and the loss function, and they compute gradients and run optimized variants of gradient descent automatically. They are built for neural networks but handle circuits like ours just as well — which is exactly why this generalizes to next week's material.

Background Knowledge — The Holy Grail

Here is where the Bayesian-network framing shines over a raw neural network. Sometimes you already know some parameters — you have background knowledge. Suppose a logical constraint holds: B is true if and only if A is false. That pins down B's CPT entirely — its entries are forced to 0s and 1s (true given A-false, false given A-true, and so on).

Because the circuit's θ inputs are literally the network's parameters, you can drop those known values straight in and remove them from the set you have to learn. You learn fewer parameters. This is effortless here precisely because the circuit was compiled from a meaningful model — every θ corresponds to a real probability you might happen to know.

The Big Idea — Knowledge Beats Data

Injecting background knowledge has two payoffs. First, less data needed: every parameter you already know is one you don't have to learn, so you hit a target accuracy with fewer expensive human-labeled examples. Second, robustness: the more knowledge baked in, the harder the model is to fool or break. This easy knowledge-injection is a genuine advantage of model-based circuits over hand-crafted neural networks, where wiring in constraints is far messier.

Why It's Harder in Neural Networks

A neural network is also a circuit — but a hand-crafted one whose internal weights don't map to interpretable quantities like "Pr(B | A)." So you usually can't just look up a weight and set it to a known value. Integrating background knowledge there is an open challenge, whereas with a compiled Bayesian network it is as simple as copying a number across.

Putting It All Together

This lecture answered the question every previous one ducked: where do Bayesian networks come from? The answer is a layered story, and each layer set up the next.

We began by splitting learning two ways — parameters vs. structure, and model-oriented vs. query-oriented — and identifying the one distinction that governs everything: complete vs. incomplete data. With complete data, maximum likelihood has a beautiful closed form: build the empirical distribution in one pass, then read each parameter off as a ratio (count and divide). When values go missing, that recipe collapses, and the EM algorithm rescues it by replacing hard counts with soft, probability-weighted completions — a hill-climber that inherits both the guarantees and the local-optima pitfalls of local search.

Learning the structure exposed a deeper principle: maximum likelihood alone overfits, because complexity always buys more fit, so we balance fit against a complexity penalty that fades as data grows — and we search structures with local search yet again. Finally, the query-oriented view reframed learning as supervised learning from labeled data. Compiling the network into an arithmetic circuit exposed its parameters and evidence indicators as explicit inputs; swapping the likelihood objective for a cross-entropy loss minimized by gradient descent turned the circuit into a trainable predictor; and because every parameter stays interpretable, we can inject background knowledge to learn from less data and resist being fooled.

Notice how often two ideas recurred. Local search drove both EM and structure learning. And inference — the work of earlier lectures — is the engine inside everything here: EM runs inference each iteration to compute completion weights, and the arithmetic circuit is compiled inference repurposed for learning. Learning and inference are not separate worlds; learning is built on top of inference.

What's Next

Next week: neural networks. The arithmetic circuit, the cross-entropy loss, gradient descent, and PyTorch/TensorFlow were not a detour — they are precisely how neural networks are trained for supervised learning. A neural network is another circuit with tunable parameters fit to labeled data; the difference is that its weights are not interpretable probabilities, which is why injecting background knowledge becomes hard and why robustness becomes a live concern. You now have the conceptual scaffolding to see neural networks as a variation on the supervised-learning theme you just built — not a leap into the unknown.
